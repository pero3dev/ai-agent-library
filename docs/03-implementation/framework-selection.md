# フレームワーク選定ガイド

## この記事の目的

Agent 開発のフレームワークを「名前」ではなく「選定軸」で評価できるようになります。個別フレームワークの機能は変化が速いため、本記事は評価の枠組み(いつ選ぶか・何を見るか)を提供し、具体的な製品比較は行いません。

## 対象読者

- Agent 開発の技術スタックを決める立場のエンジニア・テックリード
- 「フレームワークを使うべきか、生の API でいくか」を迷っているチーム

## 前提知識

- [Agent ループ](../01-concepts/agent-loop.md) — フレームワークが抽象化する対象の最小構成
- [Workflow 型 vs Agent 型の使い分け](../02-architecture/workflow-vs-agent.md) — 必要な抽象度を決める設計判断

## 本文

### 概要: 選定の前に「生 API で小さく作る」

フレームワーク選定は、要件が見える前に行うと失敗します。まず生の LLM API と自前の最小ループ([Agent ループ](../01-concepts/agent-loop.md) の数十行)でプロトタイプを作ってください。得られるものは 2 つあります — (1) 抽象の下で何が起きるかの理解(フレームワークのデバッグに必須)、(2) 自分のタスクに本当に必要な機能の一覧(= 選定基準)。

### 詳細: フレームワークの 3 つの抽象度

| 層 | 提供するもの | 向く状況 |
| --- | --- | --- |
| 薄い SDK・エージェントランタイム | API クライアント + ループ・ツール実行の基本 | 制御を握りたい。要件が独自 |
| オーケストレーション框組 | グラフ・ステート管理・分岐の宣言的定義 | 複雑な Workflow 構成([オーケストレーションパターン](../02-architecture/orchestration-patterns.md))を宣言的に管理したい |
| フルスタック / ローコード | UI・デプロイ・監視込みの統合環境 | 標準的な構成で速く出したい。カスタマイズ要求が弱い |

上に行くほど自由度が高く実装量が多い、下に行くほど速いが枠から出にくい、という通常のトレードオフです。

### 詳細: 選定軸チェックリスト

個別製品は変化が速いため、**軸**を固定して比較します。

1. **可観測性** — ループの中身(各ステップの入出力・トークン・レイテンシ)をどこまで見られるか。デバッグ可能性はここで決まります
2. **状態管理と再開** — チェックポイント・中断からの再開([メモリと状態管理](../01-concepts/memory-and-state.md))を自前で作らずに済むか
3. **Human-in-the-Loop 支援** — 承認待ちの永続化・再開([Human-in-the-Loop 設計](../02-architecture/human-in-the-loop.md))の一次サポートがあるか
4. **ツール接続** — MCP 対応([ツール接続標準](mcp-and-tool-protocols.md))、ツール定義の書きやすさ
5. **マルチエージェント** — 必要な場合のみ。並列・委譲・統合のプリミティブ
6. **ロックインの形** — ベンダー(特定 LLM 専用か)とフレームワーク(抽象が独自すぎて移行不能にならないか)の両方
7. **成熟度と保守** — 更新頻度、破壊的変更の履歴、コミュニティの厚さ。この領域のフレームワークは若く、メジャーバージョンで API が大きく変わることが珍しくありません
8. **評価・テストとの統合** — 評価ハーネス([Agent 評価の基礎](../04-evaluation/agent-evaluation-basics.md))を組み込みやすいか

### 設計判断: 「フレームワークなし」も正式な選択肢

自前ループ + 薄い SDK は、次の条件では合理的な最終形です。

- Workflow 構成が単純(直列 + せいぜいルーティング)
- 制御・監査・権限チェックの要件が強い(抽象の下に隠れてほしくない)
- チームが仕組みを理解していることを重視する

逆に、複雑なグラフ・再開可能な長時間タスク・HITL を複数箇所で使うなら、フレームワークの状態管理を自前で再発明するコストの方が高くつきます。

### 詳細: 移行可能性を設計しておく

どれを選んでも、**ビジネスロジックとフレームワークの境界を保つ**ことが保険になります。ツールの実装・プロンプト・評価データセットをフレームワーク非依存のモジュールに置き、フレームワークには「配線」だけをさせる構成なら、乗り換えのコストは配線の書き直しで済みます。

## 実務での注意点

### アンチパターン

- **デモの豪華さで選ぶ** → デモは最良ケースの見せ方であり、可観測性・再開・保守性というつらい部分は映らない → 8 軸チェックリストで自分の要件と照合する
- **抽象の下を理解しないまま本番へ** → 不具合時にフレームワークの内部を初めて読む羽目になり、障害対応が長期化する → 先に生 API で最小ループを書き、抽象が何を隠しているかを知っておく
- **移行コストをゼロか無限大かで考える** → 「一度選んだら終わり」と構えて選定が進まない、または安易に選んで密結合する → ロジックと配線を分離し、移行を「高いが可能」に保つ

### チェックリスト

- [ ] 生 API での最小プロトタイプを作り、必要機能の一覧を得た
- [ ] 8 つの選定軸で候補を比較した(機能表の再確認日も記録した)
- [ ] 可観測性(各ステップの入出力の追跡)を実際に試した
- [ ] ビジネスロジック(ツール・プロンプト・評価)がフレームワーク非依存に分離されている
- [ ] バージョン固定と更新の運用(破壊的変更への追従)を決めた
