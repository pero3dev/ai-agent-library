# ローカル・オンデバイス LLM の実務

## この記事の目的

端末・エッジでの LLM 実行を、ユースケース選定から品質検証まで設計できるようになります。ローカル実行が向く場面を見極め、実行環境の類型・モデルと量子化形式の選定・クラウド上位モデルとの品質ギャップの検証・ハイブリッド構成(ローカル優先 + クラウド昇格)・配布と更新を、実務判断として持ち帰れる状態を目指します。小型モデルを「どう選ぶか」の戦略は [小型言語モデル(SLM)の活用戦略](slm-strategy.md) が、サーバー側のセルフホストは [セルフホスト推論の実務](../05-operations/self-hosted-inference.md) が正本で、本記事は**端末・ローカルで動かす実行環境**に絞ります。

## 対象読者

- オフライン・プライバシー・低レイテンシの要件で、端末・エッジでの LLM 実行を検討する実装担当
- クラウド API と併用しつつ、一部をローカルに寄せたいプロダクト設計者

## 本文

### 概要: 分担と「ローカルで動かす」意味

本記事は LLM を**端末・ローカル環境で動かす**実務を扱います。小型モデルをどう戦略的に配るかは [小型言語モデル(SLM)の活用戦略](slm-strategy.md)、サーバー GPU でのセルフホストは [セルフホスト推論の実務](../05-operations/self-hosted-inference.md) が正本です。ここでは「オフライン・端末内で動かすか」の判断と、その実行環境の作り方に集中します。

### 向くユースケース

ローカル実行は万能ではなく、**特定の要件で光ります**。

- **オフライン**: ネットワークがない・不安定な環境で動かしたい。外部 API に依存できない現場
- **プライバシー・データ主権**: データを端末外に出したくない。**閉域網**([企業システム環境の制約と対応](../08-coding-agents/se-enterprise-constraints.md))や個人情報を扱う場面
- **低レイテンシ**: ネットワーク往復をなくして応答を速くしたい。その場での即時処理
- **ゼロ限界費用**: 端末で動けば 1 回ごとの API 課金がゼロ。大量・高頻度でコストを気にせず回したい
- 逆に、**高い品質・広い知識・複雑な推論**が要る用途は、後述の品質ギャップからローカルには向きにくいです

### 実行環境の類型

ローカル実行の「どこで動かすか」は複数あります。

| 環境 | 内容 | 特徴 |
| --- | --- | --- |
| 端末 CPU/GPU/NPU | スマホ・PC 上で実行 | 配布が広い。ハード性能に強く依存 |
| エッジサーバー | 現場に置く小型サーバー | 端末より高性能。閉域網に置ける |
| ブラウザ | ブラウザ内で実行 | インストール不要。制約が強い |

- **端末実行**: 個人 PC・スマホの CPU/GPU/NPU で動かします。配布は容易ですが、端末のメモリ・性能がモデルサイズの上限を決めます
- **エッジサーバー**: 現場に置いた小型サーバーで動かし、複数端末から使います。閉域網内に置けるのが強みです
- **ブラウザ**: ブラウザ内実行はインストール不要な一方、使えるモデルサイズ・性能の制約が強めです

### モデルと形式の選定

ローカルで動かすには、**小型モデル**を**量子化した形式**で使うのが基本です(具体のツール/形式は変化が速いため 2026-07 時点の代表例)。

- **小型モデルを選ぶ**: 端末のメモリに載る小型モデルが対象です。顔ぶれ・性能帯は [主要 LLM の全体像](llm-landscape.md) と各公式に委ねます(変化が速い)
- **量子化フォーマット**: ローカル実行では量子化([GPU・AI ハードウェアの基礎](../05-operations/gpu-and-hardware-basics.md))が前提です。代表的な形式に、llama.cpp 系エコシステムで事実上の標準となっている **GGUF**、Apple silicon 向けの **MLX** があります
- **実行ツールの類型**: ワンライン導入のツール(Ollama)、ライブラリ/サーバー(llama.cpp とその `llama-server`)、GUI アプリ(LM Studio・GPT4All)、Apple silicon 特化(mlx-lm)など。多くが**OpenAI 互換 API** を提供し、既存の OpenAI クライアントから叩けます(mlx-lm のサーバーのように「本番非推奨」と明記するものもある)

### 品質ギャップの検証

ローカル LLM の最大の論点は、**クラウド上位モデルとの品質差**です。ここを測らずに導入すると失敗します。

- **差を自社タスクで測る**: 「ローカルの小型・量子化モデルが、自社タスクでクラウド上位モデルとどれだけ差があるか」を評価で確かめます([Agent 評価の基礎](../04-evaluation/agent-evaluation-basics.md))。一般的な評判ではなく、自分のタスクでの実測が要ります
- **量子化の劣化も込みで**: 量子化で品質が落ちる分も含めて測ります([セルフホスト推論の実務](../05-operations/self-hosted-inference.md)の量子化の運用判断)
- **「崖」を見つける**: 小型モデルは得意領域では上位に迫り、苦手領域で急に落ちます([小型言語モデル(SLM)の活用戦略](slm-strategy.md)の崖)。自社タスクがどちら側かを見極めます

### ハイブリッド構成: ローカル優先 + クラウド昇格

多くの実務では、ローカル単独でなく**ローカルとクラウドの併用**が現実解です。

- **ローカル優先・クラウド昇格**: まずローカルで処理し、品質が足りない/確信が低いケースだけクラウド上位モデルへ**昇格**します([小型言語モデル(SLM)の活用戦略](slm-strategy.md)のルーティング、[信頼度と較正(calibration)](../04-evaluation/confidence-and-calibration.md)の確信度)
- **オフライン時のフォールバック**: ネットワークがある時はクラウド、ない時はローカル、という切り替えも設計できます
- **昇格の条件を決める**: 「いつクラウドに上げるか」(確信度・検証失敗・難易度)を決め、ローカルで取れる価値(コスト・レイテンシ・オフライン)とのバランスを取ります

### 配布と更新

ローカル LLM は、**モデルファイルの配布と更新**という固有の運用があります。

- **モデルファイルの配布**: 量子化モデルは数 GB になることがあり、端末への配布・保存が課題です。初回ダウンロードのサイズ・時間、保存容量を織り込みます
- **互換性**: 量子化形式・ランタイムのバージョンとモデルの互換性を管理します。ランタイム更新で古いモデルが動かなくなることがあります
- **更新の運用**: モデルを更新するとき、全端末への配布・切り替え・旧版の扱いを設計します。クラウド API のように「サーバー側で一斉更新」とはいかず、端末ごとの更新になります

## 実務での注意点

### アンチパターン

- **品質ギャップを測らずローカル化する** → クラウド上位モデル基準の品質を期待して裏切られる → 自社タスクでクラウドとの差を実測する
- **端末性能を無視してモデルを選ぶ** → メモリに載らない・遅すぎて使えない → 端末のメモリ・性能に合う小型・量子化モデルを選ぶ
- **量子化の劣化を検証しない** → 品質が静かに落ちる → 量子化レベル込みで自社タスクの品質を測る
- **ローカル単独に固執する** → 苦手タスクで品質が破綻する → ローカル優先 + クラウド昇格のハイブリッドにする
- **モデル配布・更新を設計しない** → 数 GB の配布・端末ごとの更新で運用が破綻する → 配布サイズ・互換性・更新手順を先に設計する

### チェックリスト

- [ ] ローカル化の動機(オフライン・プライバシー・低レイテンシ・ゼロ限界費用)を明確にした
- [ ] 実行環境(端末 CPU/GPU/NPU・エッジ・ブラウザ)を要件で選んだ
- [ ] 端末に載る小型モデルと量子化形式(GGUF・MLX 等)を選んだ
- [ ] クラウド上位モデルとの品質差を、量子化込みで自社タスクで実測した
- [ ] ローカル優先 + クラウド昇格のハイブリッドと、昇格条件を設計した
- [ ] モデルファイルの配布サイズ・互換性・更新手順を設計した
