# インシデント対応

## この記事の目的

Agent 固有のインシデント — 暴走・コスト急騰・品質劣化・危険な操作 — に対して、**検知 → 封じ込め(キルスイッチ)→ 影響特定 → 復旧 → 再発防止** の一連の対応を、インシデントが起きる前に設計できるようになります。特に「デプロイなしで止められる・自律度を下げられる」仕組みの事前準備に重点を置きます。

## 対象読者

- Agent の本番運用でオンコール・障害対応に責任を持つエンジニア・SRE
- 「もし暴走したらどう止めるか」に即答できない状態を解消したいチーム

## 前提知識

- [可観測性とトレーシング](observability-and-tracing.md) — 検知と影響特定の基盤
- [エラー処理・リトライ・フォールバック設計](../02-architecture/error-handling-and-retries.md) — 実行時の自動対処(本記事はそれを超えた事態)
- [Human-in-the-Loop 設計](../02-architecture/human-in-the-loop.md) — 縮退先となる承認フロー

## 本文

### 概要: Agent 固有のインシデントの型

| 型 | 例 | 特徴 |
| --- | --- | --- |
| 暴走 | 無限に近いループ、同一 API への大量呼び出し | 技術メトリクスに出る。速度が速い |
| コスト急騰 | 暴走・利用急増・長大コンテキストの常態化 | 気づくのが遅れると請求額に直結 |
| 品質劣化 | モデル・入力分布・依存データの変化による誤答の増加 | **エラーが出ない**(静かな失敗)。検知が最難 |
| 危険な操作・情報漏えい | 誤った書き込み・削除、機微情報を含む応答 | 件数が少なくても影響が重大。セキュリティ起因は [Agent の脅威モデル概観](../06-security/threat-model-overview.md) で扱う |
| 外部依存の障害 | LLM API・ツール先サービスの障害や劣化 | 自システムに原因がない。フォールバックの有無で影響が決まる |

通常のシステム障害と違うのは、**「正常に動いているように見えるが間違っている」型が多い**こと、そして影響が「誤った副作用の蓄積」として残ることです。対応の全体像は次の流れです。

```mermaid
flowchart LR
    D[検知] --> C[封じ込め<br>キルスイッチ / 縮退]
    C --> I[影響特定<br>トレースから対象抽出]
    I --> R[復旧<br>副作用の補償]
    R --> P[再発防止<br>回帰ケース化 + ガード追加]
```

### 詳細: 検知 — 静かな失敗まで拾う

[可観測性とトレーシング](observability-and-tracing.md) で設計したメトリクスにアラートを張ります。

| シグナル | 疑うもの |
| --- | --- |
| エラー率・タイムアウト率の上昇 | 外部依存の障害、ツールの不具合 |
| ステップ数分布の右シフト | 迷走・暴走の始まり(平均に出る前に裾に出る) |
| コストレート(単位時間あたりトークン)の急増 | 暴走、利用の急変 |
| 品質シグナルの低下(フィードバック・エスカレーション率・サンプリング判定) | 静かな品質劣化。遅行指標であることを前提に見る |
| 特定ツールの呼び出し頻度・失敗率の急変 | ツール先サービスの変化、プロンプト起因の挙動変化 |

品質劣化はどのメトリクスにも出にくいため、ユーザーからの報告を「個別対応して終わり」にせず、インシデント検知の入力として扱う経路(報告の集約とトリアージ)を作っておきます。

### 詳細: 封じ込め — キルスイッチは粒度を分けて事前に作る

インシデント対応の速度は、**止める手段をどれだけ事前に作ってあるか**で決まります。すべてデプロイなしで(フラグ・設定で)発動できることが条件です。

| 粒度 | 仕組み | 使いどころ |
| --- | --- | --- |
| 自動遮断 | 予算・レートの上限による自動停止([コスト管理](cost-management.md) の 3 層上限) | 人間が気づく前の初動 |
| ツール単位の無効化 | 特定ツール(特に書き込み系)だけを外す・失敗を返す | 危険な操作の停止。読み取り系は生かして機能を部分維持 |
| 縮退運転 | 自律度を下げる — 全書き込みを承認必須に([Human-in-the-Loop 設計](../02-architecture/human-in-the-loop.md))、読み取り専用モードに、または Agent を止めて定型応答・人間対応に切り替え | 全停止を避けつつ被害拡大を止める |
| テナント・機能単位の停止 | 影響範囲・悪用元だけを止める | 特定ユーザー・特定機能に限定された事象 |
| 全停止 | Agent 機能全体を止め、代替手段(定型応答・問い合わせ窓口)へ | 原因不明で被害が進行中のとき |

「書き込み系ツールだけ止める」「承認必須モードに落とす」という中間段階があるかどうかで、インシデント時の選択肢がまるで違います。全停止しか手段がないと、判断が遅れがちです(機能を止める影響が大きいため)。

### 詳細: 影響特定と復旧 — 副作用の補償

封じ込めたら、影響範囲をトレースから特定します。「問題の構成(プロンプト・モデルのバージョン)で処理されたセッション」「問題のツールを呼んだセッション」をトレース属性で抽出できることが前提です([バージョニング・デプロイ・モデル更新追従](versioning-and-model-updates.md) のバージョン記録)。

Agent 特有の重い作業が**副作用の補償**です。誤って行われた書き込み・送信・更新を洗い出し、巻き戻すか訂正します。

- 書き込み系ツールの操作ログ(いつ・何を・どのレコードに)が残っていることが、全件人手調査になるかどうかの分かれ目です
- 巻き戻せない操作(外部への送信・通知など)は、事後の訂正連絡で対処するしかありません。**だからこそ、取り消せない操作は平時から承認ゲートの対象にします**([Human-in-the-Loop 設計](../02-architecture/human-in-the-loop.md))

### 設計判断: 事後 — 再発防止を資産にする

- 問題を起こした入力・状況を**回帰ケースとして追加**します([回帰テストと CI 組み込み](../04-evaluation/regression-testing.md))。「同じ失敗は二度と静かに起きない」を積み上げます
- 検知が遅れたなら監視・アラートを、封じ込めが遅れたならキルスイッチの粒度を、復旧が重かったなら操作ログ・補償手段を直します(対応の各段階がそのまま改善対象の一覧になります)
- ポストモーテム(振り返り)を書き、ランブック(対応手順書)を更新します。キルスイッチは**平時に発動演習**をしておかないと、本番で「フラグはあるが誰も試したことがない」になります

## 実務での注意点

### アンチパターン

- **止める手段がデプロイしかない** → 封じ込めまでの時間が伸び、その間も被害(コスト・誤操作)が進行する → フラグでの全停止・縮退・ツール単位無効化・自動遮断を事前に実装し、演習しておく
- **エラー率だけを監視する** → エラーなく完走する誤動作(静かな失敗)を検知できず、ユーザー影響が長期化する → 品質シグナルとステップ数・コストの分布を監視に加える
- **書き込み系ツールの操作ログがない** → 影響特定が「期間内の全セッションを人手調査」になり、復旧が事実上不可能になる → 操作ログ(対象・内容・時刻)を残し、補償手段(巻き戻し・訂正)を設計しておく
- **復旧して終わりにする(ポストモーテムなし)** → 同じ型のインシデントが再発し、対応も毎回ゼロから → 回帰ケース化・ガード追加・ランブック更新をセットで完了条件にする

### チェックリスト

- [ ] 暴走・コスト急騰・品質劣化のそれぞれにアラートがある(分布・品質シグナルを含む)
- [ ] デプロイなしで発動できる停止手段が粒度別(全停止・縮退・ツール単位・テナント単位)にある
- [ ] 予算・レート上限による自動遮断がある
- [ ] キルスイッチの発動演習を実施したことがある
- [ ] トレースから影響セッションを抽出できる(構成バージョン・ツール呼び出しで絞れる)
- [ ] 書き込み系ツールの操作ログがあり、補償(巻き戻し・訂正)の手段が決まっている
- [ ] 取り消せない操作が平時から承認ゲートの対象になっている
- [ ] インシデント後に回帰ケース化・ランブック更新を行う運用がある
