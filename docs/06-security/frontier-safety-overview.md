# フロンティアセーフティの概観

## この記事の目的

最前線モデルの安全枠組み(危険能力評価・責任あるスケーリング)を読み解き、**モデル選定・調達時に「提供者の安全体制」を評価軸にできる**ようになります。通常のアプリ安全との違い・各社フレームワークの共通構造・第三者機関(各国 AISI 系)・実務者への含意を理解し、システムカードや安全方針の何を見るかを判断できる状態を目指します。個別のフレームワーク名・版は変化が速いため、本記事は**共通構造と読み方**に徹します。

## 対象読者

- モデル選定・調達で、提供者の安全体制を評価軸に加えたいテックリード・アーキテクト
- 「責任あるスケーリング」「危険能力評価」などの語を目にするが、実務で何を見ればよいか掴みたい人

## 前提知識

- [モデル選定ガイド](../03-implementation/model-selection.md) — モデルの選び方(本記事は「提供者の安全体制」という選定軸を足す)
- [Agent の脅威モデル概観](threat-model-overview.md) — 自社システムの脅威(本記事は「モデルそのものの能力」の安全)

## 本文

### 概要: 「アプリの安全」ではなく「能力の安全」

通常のセキュリティは、**動いているアプリ・Agent への攻撃**を守ります([Agent の脅威モデル概観](threat-model-overview.md))。**フロンティアセーフティ(frontier safety)**はそれとは層が違い、**最前線モデルの能力そのもの**が重大被害(サイバー・生物化学・AI の自己改善など)に使われうるリスクを、モデル提供者が管理する枠組みです。

- **違いは「何を評価するか」**: アプリ安全は「このシステムは攻撃に耐えるか」を、フロンティアセーフティは「**このモデルの能力は危険な水準に達したか**」を評価します
- **実務者の関わり方**: ほとんどの実務者はフレームワークを作る側ではありません。しかし**モデルを選ぶ・調達する側**として、「提供者がどんな安全体制を持ち、何を公表しているか」を評価軸にできます。本記事はその読み方を扱います

> **最終確認日: 2026-07。** 本記事で触れる各社フレームワーク名・版・各国機関の名称は**改版・改称のペースが速いです**(版番号は数か月単位で動き、機関名も改称された例があります)。代表例は「2026-07 時点」のスナップショットとして読み、採用時は各社・各機関の公式ページで必ず再確認してください(後述の「変わりやすい項目」)。

### 各社フレームワークの共通構造

主要な提供者は、それぞれ**フロンティア安全のフレームワーク**を公表しています(2026-07 時点の代表例として、責任あるスケーリング方針・準備フレームワーク・フロンティア安全枠組みなど、呼称は各社で異なります)。名称・粒度は違っても、**共通の 4 段構造**を持ちます。この構造こそが、実務者が押さえるべき骨格です。

| 段階 | 内容 |
| --- | --- |
| ①能力閾値の設定 | 重大被害につながりうる能力領域ごとに、危険とみなす「能力水準」を事前定義する |
| ②評価 | 開発・展開の過程で危険能力評価を行い、閾値に達したかを判定する(早期警告を含む) |
| ③対応措置 | 閾値に達したら、能力に応じたセキュリティ強化・展開制限・公開前審査を課す |
| ④公表 | フレームワークの版管理・モデルごとのカード・リスク報告で説明責任を果たす |

- **これは提供者の「自己統治」**: この 4 段は、多くが**提供者自身が定め・運用する自己統治(self-governance)**です。外部の強制力は、各国機関の評価連携や規制([コンプライアンスとガバナンス](compliance-and-governance.md))で補完されます
- **「安全だ」は自己申告**: フレームワークの**存在・版・内容**は公表された事実ですが、そこに書かれた「安全性の達成度」は**提供者の自己評価(自己申告)**です。鵜呑みにせず、評価軸として使います(後述)

### 危険能力評価の領域(概観)

フレームワークが評価対象とする「危険能力」の領域は、各社でおおむね共通します。**本記事は領域の概観に留め、具体的な危険能力の手法(攻撃方法)は一切扱いません。**

- 代表的な領域: サイバーセキュリティ、生物・化学(CBRN)、**AI の自己改善 / ML R&D の加速**、自律性・操作(ミスアラインメント)など
- 評価は「これらの能力が、緩和策なしに重大被害を生みうる水準に達したか」を測るもので、**能力の閾値判定の手段**として位置づけられます
- 実務者が見るのは「どの領域を・どの段階で評価しているか」という**枠組み**であって、危険能力そのものではありません

### 第三者機関と評価エコシステム

提供者の自己統治を補完するのが、**各国の AI 安全機関(AISI 系)**と評価エコシステムです。

- **各国の AI 安全機関**: 米・英・日などに、AI の安全/セキュリティを評価・標準化する政府系機関があります(2026-07 時点で、名称が「Safety」から「Security」「Standards」へ改称された例があり、名称・所在は動きます)。展開前後のテスト・評価ツールの提供・ガイドライン整備などを担います
- **国際的な連携**: これらの機関は国際ネットワークを組み、評価・基準策定を協調しています
- **認証ではない点に注意**: 各機関の評価は**網羅的な認証ではなく**、機関ごとに焦点(セキュリティ寄り・標準寄り・評価手法寄り)が違います。「ある機関が見た = 全面的に安全」ではありません
- **システムカード / モデルカード**: モデルの能力・限界・安全評価結果を公表する主要手段が、**システムカード(モデルカード)**です。実務者が調達時に最も具体的に読めるのがこれです

### 実務者への含意: 何を評価軸にするか

フロンティアセーフティを、モデル選定・調達の**評価軸**に落とし込みます。共通構造を、そのままチェック項目に変換できます。

1. **提供者のフレームワーク現行版**: 提供者がフロンティア安全のフレームワークを公表しているか、その現行版は何か
2. **モデルのシステムカード**: 当該モデルのシステムカード/モデルカードに、危険能力評価・閾値判定・緩和策が記載されているか
3. **公表の更新履歴**: 透明性・安全のページが更新され続けているか(一度きりでないか)
4. **第三者評価との連携**: 各国 AI 安全機関との評価連携があるか(補助的なシグナル)

- **複数を突き合わせる**: いずれも**提供者の自己申告が中心**です。1 つの主張を信じるのでなく、フレームワーク・カード・更新履歴・第三者連携を**突き合わせて**評価軸にします
- **選定の一要素**: フロンティアセーフティは、[モデル選定ガイド](../03-implementation/model-selection.md)の判断軸(品質・コスト・レイテンシ等)に加える**もう 1 つの軸**です。とくに機微な用途・規制産業では、提供者の安全体制の重みが増します

### 変わりやすい項目(定点観測)

本記事のうち、次は変化が非常に速く、四半期ごと(および改訂時)の再確認が必要です。

- 各社フロンティア安全**フレームワークの版番号**(数か月単位で改訂される)
- 各国 AI 安全機関の**名称・所在・役割**(改称・改組がある)
- 対象とする提供者の**顔ぶれ**(新たに公表する提供者の追加)
- 規制側の枠組み(汎用 AI モデルへの義務など)との接続

調査メモ `research/trust/frontier-safety.md`(2026-07-08 時点)に、この定点観測の詳細と出典を記録しています。

## 実務での注意点

### アンチパターン

- **提供者の「安全」宣言を鵜呑みにする** → 自己申告を検証なしに信頼する → フレームワーク・カード・更新履歴・第三者連携を突き合わせて評価軸にする
- **フレームワークの版番号を固定情報として扱う** → 数か月で改訂され陳腐化する → 「最新版を各社公式で確認」を前提に、版は都度確認する
- **AI 安全機関の評価を「認証」と誤解する** → 網羅的なお墨付きと誤認する → 機関ごとの焦点を理解し、網羅的認証ではないと踏まえる
- **フロンティアセーフティを選定軸に入れない** → 機微用途で提供者の安全体制を見落とす → 品質・コスト等に加え、安全体制を選定軸にする
- **危険能力の具体を追う** → 本来見るべき「枠組み・公表・評価連携」から逸れる → 領域の概観と公表の有無を見る

### チェックリスト

- [ ] フロンティアセーフティ(能力の安全)と、自社システムのアプリ安全を区別している
- [ ] 提供者がフロンティア安全のフレームワークを公表し、その現行版を確認した
- [ ] 当該モデルのシステムカード/モデルカードで、危険能力評価・緩和策を確認した
- [ ] 透明性・安全ページの更新が継続しているかを見た
- [ ] 各国 AI 安全機関との評価連携を補助シグナルとして確認した
- [ ] これらを「自己申告」と理解し、複数を突き合わせて評価軸にした
- [ ] 版番号・機関名は変わる前提で、都度公式を確認している
