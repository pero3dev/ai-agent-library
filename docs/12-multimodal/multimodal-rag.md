# マルチモーダル RAG

## この記事の目的

画像・図表・PDF を含む知識源で RAG を組む際の **3 つの構成**を使い分けられるようになります。テキスト化して索引する構成・マルチモーダル埋め込みで直接検索する構成・ページ画像を VLM で読みながら検索する構成のトレードオフを理解し、図表の根拠提示・ハイブリッド・評価の設計まで持ち帰れる状態を目指します。テキスト RAG の原則は [RAG 実装パターン](../03-implementation/rag-implementation-patterns.md) が正本で、本記事はその**マルチモーダル差分**に絞ります。

## 対象読者

- 図表・帳票・スライドを含む知識源で RAG を組む必要があり、テキスト RAG では取りこぼす実装担当
- 既存のテキスト RAG に画像・PDF を取り込む拡張を検討しているエンジニア

## 本文

### 概要: 差分は「画像をどう検索可能にするか」

**マルチモーダル RAG(multimodal RAG)**は、テキスト以外(画像・図表・PDF ページ)を含む知識源で検索拡張生成を行う RAG です。テキスト RAG の原則(チャンク・検索・生成・評価)は [RAG 実装パターン](../03-implementation/rag-implementation-patterns.md) が正本で、本記事の差分は 1 点 — **画像・図表をどう検索可能にし、どう根拠として返すか**です。以下、その 3 構成を扱います。

### 構成の 3 類型とトレードオフ

マルチモーダル RAG は、大きく 3 つの構成に分かれます。どれを選ぶかが設計の核心です。

| 構成 | 索引の作り方 | 強み | 弱み |
| --- | --- | --- | --- |
| ①テキスト化して索引 | ドキュメント AI で画像/PDF をテキスト化し、通常のテキスト RAG に載せる | 既存テキスト RAG を流用できる・安価 | テキスト化で失う情報(図・レイアウト)がある |
| ②マルチモーダル埋め込みで直接検索 | 画像とテキストを同じ空間の埋め込みにして直接検索 | テキスト化を介さず画像を検索できる | 埋め込みの質・運用が難しい |
| ③ページ画像を VLM で読みながら検索 | ページを画像のまま扱い、VLM で読解して回答 | レイアウト・図をそのまま活かせる | 画像トークンでコスト大・スケールしにくい |

- **①テキスト化して索引**: 最も手堅い構成です。[ドキュメント AI](document-ai.md)で画像・PDF をテキスト化し、既存のテキスト RAG に載せます。安価で流用が効く一方、**図・グラフ・レイアウト**の情報はテキスト化で失われます。「文書の文字情報が主」なら第一選択です
- **②マルチモーダル埋め込み**: 画像とテキストを**同じ表現空間**の埋め込みにし、テキストクエリで画像を直接検索します([埋め込み](../03-implementation/embeddings.md))。テキスト化を介さず図表を引ける一方、埋め込みの質・評価・運用の難易度が上がります
- **③ページ画像を VLM で読む**: ページを画像のまま保持し、検索で絞った候補ページを VLM に読ませて回答します([画像理解の実務パターン](vision-understanding-patterns.md))。レイアウト・図をそのまま活かせますが、画像トークンでコストが大きく、大規模にはスケールしにくいです
- **選び方**: 「文字情報が主 → ①」「図表そのものを検索したい → ②」「レイアウト・図の読解が要る少数の重要文書 → ③」が目安です。実務では次の**ハイブリッド**が多いです

### 図表の引用・根拠提示

マルチモーダル RAG の価値の 1 つは、**該当箇所を画像で示せる**ことです。

- **画像で根拠を返す**: テキストの引用に加え、「この図の・このページの・この領域」を画像で提示できると、利用者が根拠を確認しやすくなります。数値・図の解釈は特に、元画像を見せる価値が高いです
- **出典の粒度**: どのページ・どの図が根拠かを、回答と対応づけて返します。テキスト RAG の出典提示([RAG 実装パターン](../03-implementation/rag-implementation-patterns.md))の画像版です
- **幻覚対策になる**: 根拠画像を併せて返すと、VLM が図表を誤読した場合に利用者が気づけます([画像理解の実務パターン](vision-understanding-patterns.md)の幻覚対策)

### ハイブリッド構成

実務では、3 構成を**組み合わせる**のが現実解です。

- **テキスト + 画像の併用索引**: 文書をテキスト化して索引(①)しつつ、図表はページ画像・マルチモーダル埋め込み(②③)でも引けるようにします。「本文はテキスト検索、図表は画像検索」と役割分担します
- **2 段構え**: テキスト検索で候補を絞り(安価)、重要な図表を含む候補だけ VLM で読む(高価)、という段階設計でコストを抑えます
- **知識源の性質で配分**: 文字主体の文書は①、図表主体(スライド・図面)は②③、と知識源の性質に応じて構成を配分します

### 評価: 検索と読解を分離する

マルチモーダル RAG の評価は、**検索と読解を分けて**測ります。テキスト RAG の評価に、視覚の失敗モードが加わるためです。

- **検索の評価**: 正しいページ・図表が検索で上位に来るか(検索の再現率・精度)。テキスト RAG の検索評価([RAG 実装パターン](../03-implementation/rag-implementation-patterns.md))と同じ考え方です
- **読解の評価**: 検索できた画像を、VLM が正しく読めているか([画像理解の実務パターン](vision-understanding-patterns.md)の視覚タスク評価)。「正しい図を引いたのに数値を誤読」を切り分けます
- **分離の理由**: 検索と読解を混ぜて測ると、失敗の原因が分かりません。「検索が悪いのか、読解が悪いのか」を分けることで、どちらを直すかが決まります([Agent 評価の基礎](../04-evaluation/agent-evaluation-basics.md))

## 実務での注意点

### アンチパターン

- **すべてを③(ページ画像 VLM)で組む** → 画像トークンでコストが爆発し、大規模にスケールしない → 文字主体は①、図表主体のみ②③に配分する
- **①テキスト化だけで図表を扱う** → 図・グラフ・レイアウトの情報が失われ、図表の問いに答えられない → 図表はマルチモーダル埋め込み・ページ画像で引けるようにする
- **根拠画像を返さない** → 図表の誤読に利用者が気づけない → 該当ページ・図を画像で根拠提示する
- **検索と読解をまとめて評価する** → 失敗の原因(検索 or 読解)が切り分けられない → 検索の評価と読解の評価を分離する
- **テキスト RAG の原則を無視して作り直す** → チャンク・出典・評価の基礎を再発明する → テキスト RAG の原則を土台に、マルチモーダル差分だけ足す

### チェックリスト

- [ ] 知識源の性質(文字主体 / 図表主体)を見極めて構成①②③を配分した
- [ ] 図表そのものが問われるなら、テキスト化以外(②③)で引ける設計にした
- [ ] コストのため、テキスト検索で絞ってから VLM で読む段階設計を検討した
- [ ] 該当ページ・図を画像で根拠提示している
- [ ] 検索の評価と読解の評価を分離して測っている
- [ ] テキスト RAG の原則(チャンク・出典・評価)を土台にしている
