# エラー処理・リトライ・フォールバック設計

## この記事の目的

Agent システムで起きるエラーを 4 つの層に分類し、それぞれを「どこで・誰が(コードかモデルか)・どう」処理するかを設計できるようになります。Agent 特有の選択肢である「モデルに回復させる」を、コードによる処理と正しく使い分けられる状態がゴールです。

## 対象読者

- Agent を本番投入するにあたり、失敗時の挙動を設計するエンジニア
- 「Agent がたまに変な止まり方をする」問題を体系的に潰したいエンジニア

## 前提知識

- [Agent ループ](../01-concepts/agent-loop.md) — 停止条件と「失敗を観測として返す」原則
- [ツール使用](../01-concepts/tool-use.md) — エラーメッセージがモデルへの情報であること

## 本文

### 概要: 「モデルに回復させる」という選択肢

通常のシステムのエラー処理はコードがすべてを担いますが、Agent には**エラーをモデルに見せて回復させる**という選択肢が加わります。強力な反面、使いどころを誤ると「モデルが認証エラーに 5 回挑戦する」ような無駄が生まれます。層の分類が出発点です。

### 詳細: エラーの 4 層分類

| 層 | 例 | 処理の主体 | 方針 |
| --- | --- | --- | --- |
| インフラ層 | レート制限、タイムアウト、一時的なネットワーク障害 | コード | 指数バックオフでリトライ。**モデルに見せない** |
| ツール実行層 | 引数の形式不正、対象が存在しない、権限不足 | モデル | 観測として返す。モデルが引数修正・代替手段を判断できる |
| モデル出力層 | 出力形式の不正、存在しない対象への言及、指示違反 | コード + モデル | 検証で検出し、エラー内容を添えて再生成を求める |
| タスク層 | 目標未達のまま行き詰まり、堂々巡り | コード + 人間 | 上限で停止し、途中経過とともにエスカレーション |

判断基準はシンプルです — **決定論的に直せる失敗はコードで、判断が要る失敗はモデルで**。レート制限の待機に判断は要りません。引数の間違いはエラーメッセージを読んだモデルが直せます。

### 詳細: リトライ設計の 3 つの注意点

1. **冪等性** — 副作用のあるツール(送信・登録・決済)のリトライは二重実行の危険があります。冪等キーを付ける、実行前に状態を確認する、または自動リトライ対象から外して承認に回します([Human-in-the-Loop 設計](human-in-the-loop.md))
2. **上限** — コードのリトライにもモデルの再試行にも上限を置きます。上限は [Agent ループ](../01-concepts/agent-loop.md) の停止条件と連動させます
3. **同じ失敗の繰り返し検出** — モデルが同じツールを同じ引数で呼んで同じエラーを受け取り続けることがあります(堂々巡り)。「直近 N 回のツール呼び出しが同一・同エラー」を検出したら、ループを打ち切って人間に渡します

### 詳細: フォールバック — 失敗しても価値を残す

- **縮退運転** — Agent が失敗したら、定型応答・固定 Workflow・FAQ など、確実に動く簡易版に切り替えます。「Agent が使えないとき何も返せない」構成を避けます
- **部分結果の保全** — 20 ステップ中 15 ステップまで進んだ成果は、未完了でも価値があります。強制終了時に「どこまで終わり、何が残っているか」を報告します
- **人間への引き継ぎ** — エスカレーション時は、生ログではなく「試したこと・失敗した理由・残りの作業」を要約して渡します。引き継ぎの質がエスカレーションの運用コストを決めます

### 設計判断: 失敗時のユーザー体験を先に決める

エラー処理の設計は、技術的な分類より先に「**失敗したとき、ユーザーに何を見せるか**」を決めると迷いません。「黙って失敗する」だけは選択肢から除外します。静かな失敗は、ユーザーの信頼を最も深く損ないます。

## 実務での注意点

### アンチパターン

- **全エラーを一律リトライ** → 認証エラーへの無意味な再試行でコストと時間を浪費し、副作用ツールでは二重実行事故を起こす → 4 層分類で方針を分け、冪等でない操作を自動リトライから外す
- **モデルの堂々巡りを放置** → 同じ失敗の繰り返しに気づかず、上限までコストを消費して結局失敗する → 同一呼び出し・同一エラーの反復検出で早期に打ち切る
- **エラーを握りつぶして成功として報告** → 「完了しました」と報告されたタスクが実は失敗しており、発覚時に信頼を全て失う → 失敗の報告経路を成功の報告と同じ品質で設計する

### チェックリスト

- [ ] 想定エラーを 4 層(インフラ / ツール / モデル出力 / タスク)に分類し、各層の処理方針を決めた
- [ ] 副作用のあるツールについて冪等性(またはリトライ除外)を設計した
- [ ] コードのリトライとモデルの再試行の両方に上限がある
