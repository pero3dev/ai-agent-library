# Human-in-the-Loop 設計

## この記事の目的

人間の確認・介入(Human-in-the-Loop)を Agent の実行フローのどこに置くかを、リスクと運用負荷のバランスで設計できるようになります。「承認を増やせば安全」ではないこと — 承認疲れが安全装置を形骸化させること — を踏まえた設計がゴールです。

## 対象読者

- 副作用のある操作(送信・更新・削除・購入)を行う Agent を設計するエンジニア
- Agent の自動化範囲と人間の責任分界を決める立場のテックリード・プロダクトオーナー

## 本文

### 概要

Human-in-the-Loop(HITL)とは、人間の判断を Agent の実行フローに組み込む設計です。[ツール使用](../01-concepts/tool-use.md) で見たとおり、モデルは実行要求を出すだけで実行はアプリケーション側が握っています。HITL はこの構造を利用して、「実行の手前」「公開の手前」に人間の判断を差し込みます。

### 詳細: 介入の 4 形態

| 形態 | タイミング | 向く場面 |
| --- | --- | --- |
| 事前承認(approval) | 行動の実行前 | 不可逆・影響の大きい操作(送信・削除・支払い) |
| 事後レビュー(review) | 成果物の公開・反映前 | 品質責任が人間にある成果物(顧客向け文書・コード) |
| 例外エスカレーション(escalation) | 規定外・低信頼のときのみ | 大量処理の中の少数の難しいケース |
| 監視と中断(oversight) | 実行中いつでも | 長時間タスク。進捗の可視化と停止手段をセットで |

「全部承認」か「全自動」かの二択ではなく、操作ごとにこの 4 形態を使い分けます。

### 詳細: どこに承認を置くか — リスクベースで決める

承認の要否は「不可逆性 × 影響範囲」で整理します。

| | 影響が小さい | 影響が大きい |
| --- | --- | --- |
| **可逆な操作**(下書き作成・取り消し可能な更新) | 自動実行 | 事後レビュー |
| **不可逆な操作**(外部送信・削除・支払い) | 事後レビュー または 許可リスト化 | 事前承認(必須) |

「可逆にできないか」を先に検討する価値があります。直接送信ではなく下書き作成にする、削除ではなくアーカイブにする — 操作自体を可逆に設計できれば、承認そのものを減らせます。

### 詳細: 承認疲れ(approval fatigue)が安全装置を壊す

承認が多すぎると、人間は内容を読まずに承認するようになります(ラバースタンプ化)。こうなると承認は**安全装置として機能していないのに、機能している前提で運用される**ため、承認なしより危険な状態にすらなりえます。対策:

- **承認対象を絞る** — リスクの低い操作は許可リストに移し、自動実行にします
- **判断材料を要約して出す** — 「実行していいですか?」ではなく、何を・なぜ・影響範囲(宛先・件数・金額)を一目で判断できる形で提示します
- **実績に応じて自動化範囲を広げる** — 立ち上げ期は承認を厚くし、承認ログで拒否率が十分低い操作から自動化へ移します(信頼の段階的拡大)

### 詳細: 実装上の要点

- **承認待ちの永続化** — 承認は数時間〜数日かかりえます。Agent の作業状態をチェックポイントとして保存し、承認後に再開できる構造にします([メモリと状態管理](../01-concepts/memory-and-state.md))
- **タイムアウトはデフォルト拒否** — 応答がない場合に「実行して先へ進む」実装は、承認の意味を無効化します
- **拒否はモデルへのフィードバック** — 拒否の事実と理由を観測として返し、Agent が代替案を検討できるようにします。黙って握りつぶすと同じ要求を繰り返します
- **監査ログ** — 誰が・何を・いつ承認/拒否したかを記録します。事故調査と自動化範囲拡大の判断材料になります

## 実務での注意点

### アンチパターン

- **すべての操作に承認を要求する** → 承認疲れでラバースタンプ化し、本当に危険な操作が素通りする → リスクマトリクスで承認対象を不可逆 × 影響大に絞る
- **判断材料のない承認画面**(「ツール X を実行します。OK?」)→ 人間は判断できないまま承認し、責任だけを負う → 何を・なぜ・影響範囲を要約提示する
- **拒否後の挙動が未定義** → Agent が同じ操作を再試行し続ける、または黙って停止する → 拒否を理由付きでモデルに返し、代替案の提示または報告で終わるフローにする

### チェックリスト

- [ ] 不可逆な操作をすべて列挙し、承認要否をリスクマトリクスで決めた
- [ ] 操作を可逆化する設計(下書き化・アーカイブ化)を検討した
- [ ] 承認画面に判断材料(何を・なぜ・影響範囲)が表示される
- [ ] 承認待ち状態が永続化され、タイムアウトはデフォルト拒否になっている
- [ ] 拒否がモデルにフィードバックされる
- [ ] 承認・拒否の監査ログがあり、自動化範囲の見直しに使われている
