# コンピュータ操作型・マルチモーダル Agent

## この記事の目的

コンピュータ操作型 Agent の動作原理(画面観測ループ)を理解し、テキスト・API 中心の Agent との実務上の差分 — 速度・不確実性・安全性 — を踏まえて、採用の可否とリスク対策の当たりを付けられるようになります。

## 対象読者

- GUI しかない業務システム・Web サイトの自動化を検討しているエンジニア
- コンピュータ操作型 Agent の導入リスクを評価する立場のテックリード・セキュリティ担当者

## 前提知識

- [AI Agent とは何か](what-is-an-ai-agent.md) — 3 類型(会話型・ツール実行型・コンピュータ操作型)
- [Agent ループ](agent-loop.md) — 観測・思考・行動のサイクル
- [ツール使用](tool-use.md) — 行動がツール呼び出しとして表現される仕組み

## 本文

### 概要: 画面を見て、マウスとキーボードで操作する Agent

コンピュータ操作(computer use)型 Agent とは、画面のスクリーンショットを観測し、マウスクリック・キーボード入力を行動とする Agent です。API が用意されていない GUI — レガシー業務システム、社外の Web サイト — を自動化できる点が唯一無二の価値です。

前提として、モデルがマルチモーダル(multimodal)であること、つまり画像(スクリーンショット)を入力として理解できることが必要です。マルチモーダル対応は音声・PDF・動画などにも広がっていますが、本記事はコンピュータ操作に焦点を当てます。

### 詳細: 画面観測ループの特殊性

構造は通常の [Agent ループ](agent-loop.md) と同じですが、ツールが「スクリーンショット取得」「座標クリック」「文字入力」「キー送信」などに置き換わります。この置き換えが生む特性は次のとおりです。

- **1 アクションごとに画像 1 枚** — 行動のたびに画面を撮り直して確認するため、画像トークンの消費とレイテンシが操作数に比例して積み上がります
- **物理的な待ち時間** — 画面の描画完了・ページ遷移を待つ必要があり、LLM の速度以外の遅延が加わります
- **座標依存の脆さ** — 「(512, 384) をクリック」という行動は、解像度・ウィンドウサイズ・UI 変更で簡単に壊れます

### 詳細: ツール実行型との違い

| 観点 | ツール実行型(API) | コンピュータ操作型 |
| --- | --- | --- |
| 速度 | 1 操作あたり秒オーダー | 1 操作あたり数秒 × 操作数(タスク全体で数分〜) |
| 決定性 | API 契約に守られる | UI 変更・解像度・タイミングで壊れる |
| 成功判定 | 戻り値・ステータスコードで判定できる | 画面からの推測になりがちで、確実な判定が難しい |
| コスト | テキストトークン | 画像トークンが操作数に比例して累積 |
| 失敗時の影響 | エラーが返って止まる | 誤クリックがその場で実行される(実 UI に実害) |

### 詳細: この類型に固有の 3 つのリスク

1. **画面経由の間接プロンプトインジェクション** — Agent は画面に映るものすべてを「読み」ます。閲覧した Web ページや受信メールに埋め込まれた悪意ある指示を、タスクの指示と混同する攻撃面が構造的に存在します([プロンプトインジェクション](../06-security/prompt-injection.md))
2. **誤操作の実害** — 画面上の「送信」「購入」「削除」ボタンは本物です。テキスト生成の失敗と違い、誤操作はその場で現実の副作用になります
3. **認証情報との近接** — 操作対象はログイン済みセッションであることが多く、Agent の誤動作・乗っ取りがそのまま認証済み権限の悪用につながります

対策の基本形は、**専用のサンドボックス環境**(専用 VM・専用ブラウザプロファイル・最小権限の専用アカウント)で動かし、**不可逆操作の前に人間の承認**を挟むことです。詳細は [ツール権限設計とサンドボックス](../06-security/tool-permissions-and-sandboxing.md) と [Human-in-the-Loop 設計](../02-architecture/human-in-the-loop.md) で扱います。

### 設計判断: API があるなら API

コンピュータ操作は強力ですが「最後の手段」です。判断の順序:

1. **公式 API・MCP 連携がある** → それをツールにします([ツール使用](tool-use.md))。速く・安く・堅牢です
2. **大半は API で、一部だけ GUI 必須** → ハイブリッド構成にし、コンピュータ操作は最小の区間に限定します
3. **GUI しかない** → コンピュータ操作 + サンドボックス + 承認フローで設計します

「操作数を減らす」ことが品質・コスト・安全性のすべてに効きます。GUI 操作 20 手順のうち 15 手順を API 化できるなら、それだけで失敗確率とコストが大きく下がります。

## 実務での注意点

### アンチパターン

- **API が存在する操作を画面操作で自動化する** → 遅く・脆く・高い上に、API なら存在しなかったリスク(誤クリック・インジェクション)を抱え込む → 自動化対象の API・MCP 連携の有無を最初に調査する
- **開発者本人のアカウント・実環境で動かす** → 誤操作やインジェクションの実害が本物のデータ・本物の権限に直撃する → 専用アカウント・専用環境を用意してから開発を始める
- **成功判定を「最後のスクリーンショットがそれっぽい」で済ませる** → 静かな失敗(入力欄がずれていた等)が検出されず蓄積する → データ側の確認手段(照会 API・DB 確認)で完了を検証する

### チェックリスト

- [ ] 対象操作に API・MCP 連携の代替がないことを確認した
- [ ] 専用のサンドボックス環境(VM / ブラウザプロファイル / 最小権限アカウント)で動かしている
- [ ] 不可逆操作(送信・購入・削除)の前に人間の承認が入る
- [ ] 実行中に人間が即座に停止できる手段がある
- [ ] タスクの成功を画面以外の手段でも検証している
- [ ] 操作対象の UI 変更を検知できる(定期実行の場合)
