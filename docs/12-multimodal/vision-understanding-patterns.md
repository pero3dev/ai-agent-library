# 画像理解の実務パターン

## この記事の目的

視覚言語モデル(VLM)への画像入力を、**ユースケース類型別のプロンプトパターン**と**限界の知識**をもって設計できるようになります。図表読解・UI 解析・検品などの類型ごとの効くプロンプト、構造化出力との併用、視覚特有の限界(細かい文字・数え上げ・空間関係の弱さ)と対策、解像度・コストの構造、評価設計を持ち帰れる状態を目指します。画像を「操作」する側は [コンピュータ操作エージェントの実装](../03-implementation/computer-use-implementation.md) が正本で、本記事は「読解・理解」までを扱います。

## 対象読者

- 画像入力を使う機能(図表読解・検品・UI 解析)を作っていて、精度とコストを両立させたい実装担当
- VLM の「読めるが細部を外す」挙動に振り回されていて、限界を踏まえた設計をしたいエンジニア

## 前提知識

- [マルチモーダルモデルの仕組み(数式なしの直感)](../10-llm-foundations/multimodal-models.md) — 画像入力のトークン経済・精度の崖の由来(本記事の前提)
- [プロンプトエンジニアリングの基礎技法](../03-implementation/prompt-engineering-fundamentals.md) — プロンプト設計の土台(本記事は画像入力への適用)

## 本文

### 概要: VLM は「だいたい読める、細部で外す」

**VLM(視覚言語モデル / Vision-Language Model)**は、画像とテキストを一緒に扱えるモデルです。強力ですが、その挙動には癖があります — **大まかな内容は読めるが、細かい文字・数え上げ・精密な位置関係で急に外す**([マルチモーダルモデルの仕組み](../10-llm-foundations/multimodal-models.md)の精度の崖)。本記事は、この癖を踏まえてユースケース別にプロンプトを設計し、限界を検証で捕まえる実務を扱います。画面の「操作」は [コンピュータ操作エージェントの実装](../03-implementation/computer-use-implementation.md) が正本です。

### ユースケース類型

画像理解は、類型で効くプロンプト・注意点が変わります。

| 類型 | 内容 | 主な注意点 |
| --- | --- | --- |
| 図表・グラフ読解 | チャート・グラフから数値・傾向を読む | 細かい数値・軸の読み取りは外しやすい |
| スクリーンショット・UI 解析 | 画面の要素・状態を読む | 座標・要素の正確な位置は弱い |
| 写真の検品・分類 | 製品・状態の判定・分類 | 微細な欠陥・境界事例は苦手 |
| 手書き | 手書き文字・記入の読み取り | 崩れた字・自由記述で精度が落ちる |

- **読解系(図表・UI)**: 「何が書いてあるか」の大枠は得意です。ただし**正確な数値・座標**は外しやすく、そこを鵜呑みにしないことが肝心です
- **判定系(検品・分類)**: 明確なカテゴリ分けは得意ですが、**微細な差・境界事例**で崖が来ます。重要な判定は検証・人手と組み合わせます
- **手書き**: 崩れ・自由度が高く、精度が落ちやすい領域です。重要用途では実測して可否を判断します

### プロンプトパターン

画像入力でも、プロンプトの書き方で精度が変わります。

- **観点を指定する**: 「この画像を説明して」より「この請求書から金額・日付・宛名を抽出して」と、**見るべき観点を明示**します。漠然と見せると重要箇所を外します
- **領域に言及する**: 「左上の表」「グラフの凡例」など、**注目させたい領域を言葉で指す**と精度が上がります
- **複数画像の比較**: 「A と B の差分を挙げて」のように複数画像を渡して比較させます。順序・対応を明示すると混同を防げます
- **画像付き few-shot**: 「こういう画像はこう判定する」の例を数個見せると、判定基準が揃います([プロンプトエンジニアリングの基礎技法](../03-implementation/prompt-engineering-fundamentals.md)の few-shot の画像版)。ただし画像 few-shot はトークンコストが大きい点に注意します

### 構造化出力との併用

画像理解の結果を後続処理で使うには、**構造化**します。

- **スキーマで縛る**: 「検品結果は {判定: OK/NG, 理由: string, 欠陥箇所: []}」のように型付けします([構造化出力](../03-implementation/structured-output.md))。自由文で返させるより、後続システムが扱いやすくなります
- **形式保証と内容保証は別**: 構造化出力は**形式**を保証しますが、**中身が正しい**ことは保証しません。「NG と綺麗に構造化されたが、実際は OK」は起こります。内容は次項の限界・評価で担保します

### 限界と対策

VLM の限界は、[LLM の能力と限界の由来](../10-llm-foundations/capabilities-and-limits.md)の視覚版として構造的に予測できます。

- **細かい文字**: 小さい文字・低解像度で読み間違えます。対策は解像度確保・関心領域の切り出し・OCR との併用([ドキュメント AI](document-ai.md))
- **数え上げ**: 「何個あるか」は苦手で、多いほど外します。正確な数が要るなら検証・別手段を併用します
- **空間関係・座標**: 「A の右にある B」「この要素の座標」は弱いです。精密な位置が要る操作系は、座標を VLM に頼らない設計にします([コンピュータ操作エージェントの実装](../03-implementation/computer-use-implementation.md))
- **幻覚**: 見えない細部を「もっともらしく補完」します。画像にないものを、あるかのように答えることがあります。重要な読み取りは裏取りします
- **共通対策**: 「重要な読み取りは検証する・危ういタスクは人手や別手段と組み合わせる・解像度とコストのバランスを取る」に集約されます

### 解像度・トークンコストの構造

画像理解の実務判断の多くは、**解像度とコスト**に帰着します([マルチモーダルモデルの仕組み](../10-llm-foundations/multimodal-models.md)のトークン経済)。

- **解像度は精度とコストのつまみ**: 細部を読ませたいなら高解像度(=多トークン=高コスト)が要ります。低解像度で「読めない」のは、多くがモデルの知能でなく入力の粒度の問題です
- **関心領域を切り出す**: 全画面を高解像度で渡す前に、必要な領域を切り出す・縮小することでコストを抑えます
- **画像枚数に注意**: 複数画像・動画フレームは枚数分のトークンを食います。必要最小限の枚数にします

### 視覚タスクの評価設計

画像理解も、評価で品質を確かめます。

- **正解を用意して測る**: 判定・抽出のタスクは、正解データと突き合わせて精度を測ります([Agent 評価の基礎](../04-evaluation/agent-evaluation-basics.md))。「それらしく答える」ことと「正しい」ことは別です
- **崖のケースを評価に含める**: 細かい文字・数え上げ・境界事例・低品質画像など、崖が来るケースを評価セットに入れます。平均だけ見ると崖を見逃します
- **複数回実行のばらつき**: 同じ画像でも実行ごとに答えが揺れることがあります。重要タスクは複数回実行でばらつきを見ます

## 実務での注意点

### アンチパターン

- **「この画像を説明して」と漠然と渡す** → 重要箇所を外し、無駄な説明が返る → 見るべき観点・領域を明示する
- **VLM の数値・座標・個数を鵜呑みにする** → 細かい数字・数え上げ・位置で外したまま業務に使う → 重要な読み取りは検証・別手段と併用する
- **構造化出力の「綺麗な結果」を正しいと思う** → 形式は正しいが中身が誤っている → 内容は評価・検証で担保する
- **低解像度で細部を読ませてモデルのせいにする** → 入力の粒度の問題を知能の問題と誤認する → 解像度・関心領域を調整する
- **平均精度だけで評価する** → 崖のケース(細字・数え上げ・境界)を見逃す → 崖のケースを評価セットに含める

### チェックリスト

- [ ] ユースケース類型ごとの得意・苦手(数値・座標・数え上げ・手書き)を把握した
- [ ] 観点・領域を明示するプロンプトにした
- [ ] 結果を構造化出力で型付けし、内容は別途検証している
- [ ] 数値・座標・個数など苦手項目を鵜呑みにせず裏取りしている
- [ ] 解像度・関心領域・画像枚数をコストと精度のバランスで設計した
- [ ] 正解データで精度を測り、崖のケースを評価に含めている
