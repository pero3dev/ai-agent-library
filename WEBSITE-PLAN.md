- ステータス: **W4(インタラクティブ UI)完了**(2026-07-05)。/roadmap(React Flow 依存マップ)・/glossary(用語カード + フィルタ)・/tags(タグ別一覧)を追加。残りは W5(公開準備)のみ

## 2. 技術スタック 3 案の比較

### 案 A: Docusaurus

ドキュメントサイト特化のフレームワーク。サイドバー自動生成・検索連携・ダークモード・Mermaid 公式テーマ・i18n・バージョニングが標準装備。

- **メリット**: 導入即日で「普通に良いドキュメントサイト」になる。コンテンツ作業に集中でき、運用ノウハウ・事例が最多。MDX 対応で React コンポーネントも使える
- **デメリット**: 見た目とレイアウトの自由度はテーマ(Infima)の枠内。ヒーローの演出・カード UI・React Flow のような独自 UI を深く入れるには swizzle(テーマ内部の上書き)が必要で、**アップグレード時に壊れやすい負債**になりやすい。「先進的な独自デザイン」の天井が低い
- **保守性**: ◎(デフォルト近傍にいる限り)/ swizzle を重ねると △
- **拡張性**: ○(ドキュメント機能)/ △(独自 UI)
- **学習コスト**: 低(React の深い知識は不要)

### 案 B: Next.js + Nextra + MDX(推奨)

Next.js 上のドキュメントテーマ/フレームワーク。サイドバー・目次・検索・ダークモードなどのドキュメント基盤を持ちつつ、**通常の Next.js アプリとしてカスタムページ・カスタムコンポーネントを一級市民として書ける**。

- **メリット**: 「記事ページはテーマに任せ、トップ・ロードマップ・用語集だけフルカスタム」という本件の要求構造にちょうど一致する。MDX コンポーネント差し替え(`mdx-components`)で、原本を書き換えずに見た目を進化させられる。Tailwind CSS 等との併用も素直
- **デメリット**: Docusaurus よりエコシステムが小さく、メジャーバージョン間の破壊的変更の歴史がある。Next.js の基礎知識が前提。検索・ルーティング等の仕様がバージョンで動く
- **保守性**: ○(テーマ基盤あり + 独自部分を限定すれば)
- **拡張性**: ◎
- **学習コスト**: 中(Next.js / React の基礎が必要)

### 案 C: Next.js + 自前 MDX パイプライン(フルカスタム)

コンテンツ読み込み(remark/rehype パイプライン等)からサイドバー・目次・検索・ナビゲーションまですべて自作。

- **メリット**: デザイン・UX の自由度が最大。テーマと戦う場面がゼロ
- **デメリット**: ドキュメントサイトの「当たり前品質」(サイドバー、目次、アンカー、前後ナビ、モバイル、検索、a11y)を全部自作・自己保守することになる。コンテンツが主役のプロジェクトで、インフラ開発が主戦場化するリスクが高い。コンテンツレイヤのライブラリ選定にも不確実性がある
- **保守性**: △(すべて自前コード)
- **拡張性**: ◎(ただし対価が大きい)
- **学習コスト**: 高

### 比較まとめ

| 観点 | A: Docusaurus | B: Nextra | C: フルカスタム |
| --- | --- | --- | --- |
| 立ち上げ速度 | ◎ | ○ | △ |
| 独自 UI(ヒーロー・カード・React Flow) | △ | ◎ | ◎ |
| ドキュメント基盤の完成度 | ◎ | ○ | △(自作) |
| 保守性 | ◎→△(swizzle 次第) | ○ | △ |
| 学習コスト | 低 | 中 | 高 |
| 本件の要求(先進的 UI × 保守性)との適合 | △ | **◎** | ○ |

## 3. 推奨: 案 B(Next.js + Nextra + MDX)

理由は 3 つです。

1. **要求の構造に一致** — 本件は「37 本の記事ページ = 定型で高品質に」「トップ・ロードマップ・用語集 = 独自 UI で先進的に」という二層構造。Nextra は記事側の基盤を提供しつつ、独自ページを Next.js としてそのまま書ける唯一の中間解です
2. **原本無変更の変換戦略と相性が良い** — MDX コンポーネント差し替えと remark プラグインにより、`docs/` の Markdown を書き換えずに「TODO ブロック → コールアウト」「チェックリスト → インタラクティブ UI」等の見た目進化ができます
3. **過不足のなさ** — 案 A は独自 UI で必ずテーマと戦うことになり、案 C はドキュメント基盤の再発明に工数を吸われます

**W0 での確定(2026-07-05)**: Nextra 4.6.1 + Next.js 16.2.10 + React 19.2.7 の組み合わせで全 47 ページのビルド・検索(Pagefind 1.5.2、日本語 48 ページ索引)・Mermaid 描画(組み込みサポート)を検証済み。**案 B で確定**します。

**既知の問題(W0 で対処済み)**: nextra-theme-docs 4.6.1 は zod の新しいマイナー(4.4.x)と非互換(`Layout` の props 検証が全ページで失敗する)。`website/package.json` の `overrides` で **zod を 4.1.12 に固定**して回避しています。

**設計変更(W3)**: 当初「自動装飾は next.config の remarkPlugins で行う」計画だったが、**Turbopack はローダーオプションの直列化を要求するため関数プラグインを渡せない**ことが判明。装飾変換は sync スクリプト側(`lib/doc-decorations.mjs`)で実行し、remark-mdx で正しくエスケープした .mdx を出力する方式に変更した(バンドラー非依存で、「変換は 1 か所に集約」の原則にも合致)。

> **TODO(要確認):** Nextra のバージョン更新時に zod override(4.1.12 固定)が不要になったかを nextra-theme-docs のリリースノートで確認する(最終確認: 2026-07)

## 4. 提案ディレクトリ構成

**大原則: `docs/` は正本のまま一切動かさない。** 執筆ルール([CLAUDE.md](CLAUDE.md))・lint・レビューのパイプラインは現状のまま継続し、Web サイトは `website/` に分離します。

```text
ai-agent-library/
├── docs/                        # 【正本・変更なし】執筆は今後もここに Markdown で行う
├── examples/                    # 【変更なし】
├── GLOSSARY.md ほか既存ファイル   # 【変更なし】
├── WEBSITE-PLAN.md              # 本ファイル
└── website/                     # 新設: ドキュメントサイト(Next.js + Nextra)
    ├── app/                     # カスタムページ(トップ/ロードマップ、用語集、タグ一覧)
    ├── content/                 # 【生成物・.gitignore】sync スクリプトが docs/ から生成する MDX
    ├── components/              # ConceptCard / RouteStepper / TodoCallout / AntiPatternList /
    │                            # InteractiveChecklist / GlossaryTooltip / RoadmapFlow / SectionCard など
    ├── lib/                     # front matter・GLOSSARY のパーサ、リンク変換表
    ├── scripts/
    │   └── sync-content.mjs     # docs/*.md → content/**.mdx 変換(リンク書き換え・メタ変換)
    ├── mdx-components.tsx       # Markdown 要素 → React コンポーネントのマッピング
    └── package.json / next.config.mjs / tailwind.config など
```

**sync スクリプト方式を採る理由**: (1) 原本に手を入れない、(2) 変換ロジック(`.md` リンク → ルート、`NN-` 連番の除去、GLOSSARY の構造化)を 1 か所に集約できる、(3) Windows 環境でシンボリックリンクの権限問題を避けられる、(4) 変換結果が `content/` に実体化されるためデバッグが容易。

- URL 設計(W1 で確定): ドキュメントは `contentDirBasePath: '/docs'` で `/docs` 配下に配信 — `docs/01-concepts/agent-loop.md` → `/docs/concepts/agent-loop`(連番除去)。トップページ等のカスタムページは `app/` 直下(`/`)で、Next.js の「静的ルートとオプショナル catch-all の同一階層衝突」を回避します
- `content/` と `generated/`(sections.json / glossary.json)は生成物なので Git 管理外とし、ビルド時に毎回生成します

**W0 での確定(2026-07-05)**: sync スクリプト方式を採用し、`website/scripts/sync-content.mjs` として実装済み。docs 47 ファイル(本文 37 + セクション README 8 + GLOSSARY + examples README)を変換し、未解決リンク 0 件を確認しています。手書きページは `website/content-src/` に置き、sync が `content/` に重ねます。

## 5. MDX 化の移行方針(原本無変更・3 段階)

**原本は `.md` のまま維持し、MDX 化は変換層で行います。** 既存本文の書き換えは行いません。

| 段階 | 内容 | 原本への変更 |
| --- | --- | --- |
| **段階 1: そのまま全文表示** | sync スクリプトでコピー + 最小変換(front matter → ページメタ、相対リンク → ルート、README → セクション index)。Mermaid はクライアント描画。まず全 37 本が読める状態を作る | なし |
| **段階 2: 構造の自動コンポーネント化** | テンプレートの規則性を利用し、remark プラグイン/変換で自動装飾 — `> **TODO(要確認):**` → `<TodoCallout>`、`### アンチパターン` 配下 → 警告スタイルのリスト、`- [ ]` → インタラクティブなチェックリスト、front matter の `level` / `tags` / `last_updated` → ヘッダーバッジ、GLOSSARY 登録語 → ホバーツールチップ | なし |
| **段階 3: 選択的な手書き MDX** | 効果の大きいページ(トップ/学習ロードマップ等)に限り、`website/` 側に専用 MDX/TSX ページを作成。**原本 `.md` は残し**、「そのページは website 側が表示の正」とページ単位の対応表で明示(二重管理を最小化) | なし(表示の正のみ移動) |

- **GLOSSARY**: パースして JSON 化 → 用語カードグリッドページ + 全記事内ツールチップの単一ソースにします
- **Mermaid**: 原本のコードブロックが引き続き図の正本。React Flow によるインタラクティブ図(段階 3)は「学習ロードマップの依存グラフ」など少数の高価値ページに限定し、Mermaid との二重管理はその対応表で管理します
- **執筆フロー**: 今後の記事追加も従来どおり `docs/` に Markdown で書けば、sync により自動的にサイトへ反映されます(CLAUDE.md のルール変更は不要。サイト固有の注意が出てきた段階で CLAUDE.md に 1 節追記を検討)

## 6. PoC ページの選定

**PoC: [docs/00-overview/learning-roadmap.md](docs/00-overview/learning-roadmap.md) をサイトのトップページ(`/`)として再構成する。**

選定理由:

1. ライブラリの入口であり、サイトの顔(ヒーロー)と兼ねられる
2. 読者タイプ別ルート A〜E・依存グラフ(Mermaid)・全セクションへのリンクを既に持ち、要求されている UI 要素(ロードマップ・図解・カード・関連リンク)を 1 ページで全部検証できる
3. 「原本 Markdown を核に、GLOSSARY と common-anti-patterns.md から少量を引用して合成する」構成のため、**コンテンツパイプライン(段階 1〜2)と独自 UI(段階 3)の両方を 1 ページで検証できる**

次点(PoC 第 2 弾): [docs/01-concepts/agent-loop.md](docs/01-concepts/agent-loop.md) — 「標準記事レイアウト」の代表として、テンプレート 8 セクションの自動装飾(段階 2)を検証する題材に最適です。

## 11. 開発時の運用メモ

- **dev サーバーの起動**: `cd website && npm run dev`(sync が自動実行されます)。W1 時点で全ルート 200 を確認済み
- **「Jest worker …」「500 エラー」が dev で出た場合**: 過去の dev サーバーの残骸プロセスがポート 3000 を占有している可能性が高い(W1 で実際に発生。zod 修正前の古いプロセスが :3000 に残り、新しいサーバーは :3001/:3002 に逃げていた)。`netstat -ano | findstr :3000` で PID を特定し、`taskkill /F /PID <pid> /T` でプロセスツリーごと終了してから起動し直す
- **トップページの編集データ**(学習ルート・アンチパターン 3 根)は `website/components/home/data.js`。原本(learning-roadmap.md / common-anti-patterns.md)を更新したら同期すること
- **依存マップのノード・エッジ**は `website/components/roadmap/dependency-graph.jsx`。原本は learning-roadmap.md の Mermaid(変更時は両方を同期)
- dev の初回アクセスで稀に記事ページが 500 になることがある(コールドコンパイル中のレース)。リロードで解消し、本番ビルドには影響しない
