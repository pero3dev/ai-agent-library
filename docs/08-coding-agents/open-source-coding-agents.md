# オープンソースのコーディングエージェント

## この記事の目的

オープンソース(OSS)のコーディングエージェント — Aider / Cline / Continue / OpenHands / Goose / opencode — を俯瞰し、「OSS 系という選択肢」を評価する軸を持てるようになります。個別ツールの詳細な選定よりも、OSS を選ぶ動機・引き受ける責任・商用ツールとの違いの理解が主眼です。

## 対象読者

- データ主権・モデル選択の自由度を重視して OSS 系エージェントを検討しているエンジニア
- 商用 SaaS 型のデータ取り扱い制約により、代替を探しているチーム

## 本文

> **最終確認日:** 2026-07-05 — 本記事の情報はこの日付時点の各公式サイト・公式リポジトリに基づきます。OSS はプロジェクト体制の変化が特に速いため、参照時は必ず現況を確認してください。

### 概要

OSS 系コーディングエージェントに共通するアイデンティティは **BYOK(Bring Your Own Key: 自分の API キーでモデルに接続)とマルチプロバイダー対応**です。ベンダーのサブスクリプションではなく、任意のモデル(ローカルモデル含む)を自分の契約・自分の経路で使えます。これがもたらすのは、①データの経路を自分で決められる(データ主権)②モデルの進化に乗り換えで追従できる ③コードが読める・拡張できる、という自由です。

引き換えに、**運用責任(モデル契約・コスト管理・セキュリティ設定・プロジェクト存続性の監視)を自分で負う**のが OSS 系の本質的なトレードオフです。

### 主要ツールの一言サマリー(2026-07 時点)

| ツール | 形態 | ライセンス / 主体 | 特徴 |
| --- | --- | --- | --- |
| Aider | CLI | Apache-2.0 / コミュニティ | repo map と Git 自動コミットが特徴の草分け。エージェントが自律実行しない人間主導設計。**リリースは約 11 か月停止中** |
| Cline | VS Code 拡張中核(+ JetBrains / CLI / SDK) | Apache-2.0 / Cline Bot Inc. | Plan / Act モード、diff ビュー、都度承認が基本。非常に活発 |
| Continue | IDE 拡張 + CLI | Apache-2.0 / Continue Dev | **Cursor に買収され、リポジトリは read-only(final release)**。新規採用は事実上終了状態 |
| OpenHands(旧 OpenDevin) | セルフホスト可能なエージェント実行基盤 | MIT(オープンコア) / All Hands AI | Docker / VM のサンドボックス実行が第一級。LLM ベースのリスク評価承認。他社エージェントの実行基盤にもなる |
| Goose | CLI + デスクトップ | Apache-2.0 / **Linux Foundation 傘下 AAIF に移管** | 拡張機構が MCP そのもの。コーディング専用でない汎用エージェント。**既定モードが Autonomous(全自動)** |
| opencode | ターミナル TUI(+ デスクトップ / IDE) | MIT / Anomaly(旧 SST) | 75+ プロバイダー、既存サブスクリプション接続対応、`opencode.json` でコマンドパターン単位の権限制御。**既定はほぼ allow** |

### OSS 系を評価する 4 つの軸

個別機能の比較よりも、次の 4 軸で見ると OSS 系の判断がしやすくなります。

1. **ガバナンスと存続性(最重要)** — この 6 ツールだけでも、直近 1 年で「買収されて開発終了(Continue)」「財団へ移管(Goose)」「運営組織の移転(opencode)」「開発停滞(Aider)」の 4 パターンが実際に起きています。OSS を選ぶことは、**プロジェクトの健全性を継続監視する責任を負うこと**です。リリース頻度・ガバナンス形態(企業スポンサー / 財団 / 個人)・収益モデル(オープンコア / ホスティング)を選定時と定期見直しの両方で確認します
2. **既定の安全度** — 承認モデルの既定値がツール間で正反対です。最も保守的な Aider(人が明示的に実行コマンドを打つ)から、Cline(都度承認が基本)、OpenHands(リスク評価ベースの選択制)、Goose / opencode(**既定が全自動・ほぼ許可**)まで幅があります。「OSS だから安全」でも「同じ既定」でもありません。導入時に必ず権限設定を確認します([権限とセキュリティ](coding-agent-security.md))
3. **コスト構造** — ソフトウェアは無料でも、API 従量課金(使うほど増える)と自己運用の人件費がかかります。商用ツールのシート課金と比べる際は「無料 ≠ 運用コストゼロ」で総額を見積もります([コスト管理](../05-operations/cost-management.md))
4. **エコシステム互換性** — MCP 対応は 6 ツール中 5 つ(Aider は記載なし)と事実上の標準です。ルールファイルは AGENTS.md 対応(Goose / opencode)と独自形式(.clinerules / CONVENTIONS.md 等)が混在しており、マルチツール環境では互換性が保守コストに直結します([ルールファイルと設定](coding-agent-rules-and-config.md))

### サンドボックスと実行環境

ローカル直接実行が基本の中で、OpenHands は Docker / VM / クラウドでの隔離実行を第一級の設計にしている点が特徴です。逆に言えば、他のツールをローカルで使う場合の隔離(コンテナ・権限分離)は**利用者側の責任**になります。商用ツールが備える組織管理(SSO・ポリシー強制・監査ログ)も、OSS 単体には基本的にないため、チーム導入では自前の統制(設定ファイルのリポジトリ管理・CI での検査)で代替する設計が必要です。

## 実務での注意点

### アンチパターン

- **「OSS だから安心・無料」で導入を決める** — データはモデルプロバイダーに送られ(BYOK 先の規約次第)、運用・監視の責任は自分に来ます。→ データ経路・総コスト・存続性リスクを商用ツールと同じ基準で評価します
- **既定設定のまま使い始める** — Goose / opencode のように既定が全自動のツールがあります。→ 導入初日に承認モード・権限設定(deny リスト)を確認・変更します
- **存続性を監視しない** — 買収・停滞は 1 年以内に実例が複数あります。→ 四半期ごとの依存見直しにエージェント本体も含め、乗り換え先の当たりを常に持っておきます
- **チーム統制なしの野良運用** — 個人ごとにバラバラの設定・モデル契約で使うと、統制もコスト把握もできません。→ 設定ファイル(ルール・権限)をリポジトリで共有し、API キーは組織管理にします

### チェックリスト

- [ ] プロジェクトの健全性(直近リリース・ガバナンス・収益モデル)を確認したか
- [ ] 承認モデルの既定値を確認し、チーム標準の設定に変更したか
- [ ] BYOK 先(モデルプロバイダー)のデータ取り扱い・学習利用ポリシーを確認したか
- [ ] API 従量コストの上限・監視を設定したか
- [ ] 設定・ルール・権限ファイルをリポジトリ管理してチームで共有しているか
- [ ] 存続性リスクに備えた代替ツールの候補があるか
