# プロンプト資産の管理とバージョニング

## この記事の目的

プロンプトを「コードと同格の資産」として管理する仕組み — 置き場所・テンプレート化・意図の文書化・変更フロー・実験の記録 — を設計できるようになります。[バージョニング・デプロイ・モデル更新追従](../05-operations/versioning-and-model-updates.md)がリリースとロールバックの機構を扱うのに対し、本記事はその手前にある資産管理と変更ワークフローの日常運用を扱います。

## 対象読者

- プロンプトが増えてきて「どれが最新か・なぜこう書いてあるか」が分からなくなり始めたチームのエンジニア
- ドメイン担当者(非エンジニア)を含むチームでプロンプト改善を回したいテックリード

## 前提知識

- [Agent 向けプロンプト設計](agent-prompt-design.md) — プロンプトに何を書くか(本記事は書いたものをどう管理するか)
- [バージョニング・デプロイ・モデル更新追従](../05-operations/versioning-and-model-updates.md) — リポジトリ管理・段階的リリース・ロールバックの原則
- [回帰テストと CI 組み込み](../04-evaluation/regression-testing.md) — 変更フローのゲートになる評価

## 本文

### 概要: コードと同じ規律、コードと違う性質

プロンプトはコードと同じ規律(リポジトリ・レビュー・テスト・バージョン管理)に載せるべき資産です。これは [バージョニング・デプロイ・モデル更新追従](../05-operations/versioning-and-model-updates.md) で確立した原則です。一方で、プロンプトにはコードと違う性質が 3 つあり、管理の設計はこの差分に対応する必要があります。

1. **影響が非局所的**: 1 行の変更が、一見無関係なタスクの挙動を変えます。差分レビューだけでは影響を読み切れず、回帰テストが必須になります
2. **正しさが確率的**: 「動く / 動かない」ではなく「成功率が上がった / 下がった」で評価します。目視の 1 例では判断できません
3. **書き手がエンジニアに限らない**: 業務の言葉遣い・判断基準はドメイン担当者が最もよく知っており、その参加経路を設計する価値があります

### 資産としてのプロンプト

**置き場所と構成**。プロンプト・テンプレートはアプリケーションコードと同じリポジトリに置きます。ファイル構成の一例です。

```text
prompts/
├── triage/
│   ├── system.md          # システムプロンプト本体
│   ├── examples.md        # few-shot 例(分離して差し替え可能に)
│   └── meta.yaml          # 検証済みモデル・関連する評価セット・オーナー
└── shared/
    └── safety-policy.md   # 複数プロンプトで共有する部品
```

- **テンプレート化**: 実行時に変数(ユーザー情報・日付・検索結果)を注入する箇所をテンプレート変数として明示し、「固定部分」と「動的部分」を分離します。プロンプトキャッシュの効率(固定部分を先頭に)にも直結します([コスト管理](../05-operations/cost-management.md))
- **部品化と影響範囲**: 共通ポリシー(口調・禁止事項・安全指示)を部品として共有すると、直し漏れを防げます。ただし共有部品の変更は**それを使う全プロンプトの回帰テスト**を意味します。共有するもの・あえて複製するものを影響範囲で判断します
- **意図の文書化**: プロンプトの特定の 1 行が「過去のどの失敗への対策か」を、コメントやコミットメッセージに残します。意図が失われたプロンプトは「消してよいか誰にも分からない呪文」の集積になり、肥大化する一方になります
- **検証済みモデルの明記**: プロンプトは特定モデルに対してチューニングされた成果物です。「どのモデルで検証済みか」をメタ情報に持ち、モデル更新時の再検証対象を機械的に列挙できるようにします

### バージョニングと環境

- **コードと同時にバージョンされる単位を決める**: ツール定義の変更とプロンプトの変更は互いに依存することが多く(ツール名の変更はプロンプト内の言及にも影響)、原則として**同じリポジトリ・同じリリース単位**で扱います
- **環境分離**: 開発・ステージング・本番で参照するプロンプトのバージョンを分離し、本番参照は明示的な昇格操作(マージ・タグ・フラグ)でのみ変わるようにします
- **実行時の記録**: どのバージョンのプロンプトで応答が生成されたかをトレースに記録します([可観測性とトレーシング](../05-operations/observability-and-tracing.md))。これがないと本番の失敗を再現できません

### 変更フロー

プロンプト変更の標準フローを 1 本にします。緊急時も含めここを通します。

```mermaid
flowchart LR
    P["変更の提案<br/>(意図と期待効果を明記)"] --> R["レビュー<br/>(意図・副作用・部品への影響)"]
    R --> T["回帰テスト<br/>(評価セットで前回比較)"]
    T -->|合格| D["段階的リリース<br/>(フラグ / カナリア)"]
    T -->|悪化| P
    D --> M["本番の品質シグナル確認"]
```

- **レビューの観点**はコードレビューと少し違います: 変更の意図が書かれているか、既存の対策行を消していないか、共有部品への影響はないか、指示の追加がコンテキストを不必要に肥大させていないか
- **回帰テストをゲートにする**: 「良くなった気がする」は判断材料になりません。開発セットでの比較を通過した変更だけをリリース対象にします([回帰テストと CI 組み込み](../04-evaluation/regression-testing.md)、セットの分割は[評価データセットの構築と保守](../04-evaluation/evaluation-datasets.md))
- **ドメイン担当者の参加経路**: 業務側がプロンプトを直せると改善は速くなりますが、本番の直接編集は履歴・レビュー・テストのすべてを迂回します(このリスクは [バージョニング・デプロイ・モデル更新追従](../05-operations/versioning-and-model-updates.md) のアンチパターンのとおりです)。「編集の提案はだれでも・反映は同じフローで」を原則にし、非エンジニア向けには提案用の編集 UI やドキュメント経由の提案 → エンジニアが PR 化、という運用から始めるのが現実的です

なお、プロンプトの保管・バージョン・実験を専用に管理するレジストリ型の製品・機能も存在しますが、選択肢は変化が速いため本文では扱いません(下の TODO 参照)。どのツールを使う場合も、本記事の原則(単一の変更フロー・評価ゲート・トレースへのバージョン記録)は変わりません。

### 実験と計測

- **オフライン比較を先に**: 候補プロンプトの比較は、まず評価セットでのオフライン比較で行います。本番での A/B テストより桁違いに安く、大半の劣化案をここで落とせます
- **変更と結果の対応記録**: 「どの変更で・どの指標が・どれだけ動いたか」を変更ログとして残します。これが蓄積すると「このタスクには何が効くか」というチーム固有の知見になります
- **バリアントに寿命を設ける**: 実験用に作った複数バリアントは、結論が出たら勝者に統合して削除します。放置されたバリアントは「どれが本物か分からない」負債になります
- **本番での検証**: オフラインで有望な変更は、カナリアリリースで品質シグナルを新旧比較して確定します(仕組みは [バージョニング・デプロイ・モデル更新追従](../05-operations/versioning-and-model-updates.md))

## 実務での注意点

### アンチパターン

- **プロンプトのコピペ増殖** → 同じポリシー文が 5 か所にあり、修正が 4 か所で漏れる → 共有部品化するか、複製する場合は検索可能な形(共通の見出し・タグ)にして影響範囲を追えるようにする
- **意図を書かずにプロンプトを直す** → 数か月後、どの行が何の対策か分からず、誰も削除できない「呪文の地層」ができる → 変更の意図(どの失敗への対策か)をコミット・コメントに必ず残す
- **「良くなった気がする」でリリースする** → 確率的な出力の 1〜2 例では改善も劣化も判定できず、静かな劣化が本番に出る → 評価セットでの前回比較をリリースゲートにする
- **ドメイン担当者に本番プロンプトを直接編集させる** → 速い改善のつもりが、履歴もテストもない変更が本番に入り、障害調査が不可能になる → 提案は開かれた経路で受け、反映は単一の変更フローに通す
- **実験バリアントを放置する** → 「本番はどれを使っているのか」が分からなくなり、古いバリアントが誤って参照される → 実験には終了条件を付け、勝者統合と削除までを実験の完了とする

### チェックリスト

- [ ] プロンプト・テンプレートがコードと同じリポジトリでレビューを通っている
- [ ] 固定部分と動的部分(テンプレート変数)が分離されている
- [ ] 共有部品の一覧と、変更時に回帰テストすべき利用元が把握できる
- [ ] 各プロンプトに意図の記録(何の対策か)と検証済みモデルのメタ情報がある
- [ ] 本番参照のバージョンが明示的な昇格操作でのみ変わる
- [ ] 変更フローに回帰テストのゲートがあり、緊急時も迂回しない
- [ ] 非エンジニアの改善提案を受け取り、同じフローで反映する経路がある
- [ ] 実験の記録(変更 → 指標の動き)が残り、終了したバリアントは削除されている
