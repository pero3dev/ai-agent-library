# 日本語品質の評価

## この記事の目的

英語中心の評価では見えない**日本語固有の品質**(敬語・文体・表記)を測る評価を設計できるようになります。「意味は合っているが日本語として不自然」「敬語が崩れている」「文体が揺れる」といった、英語ベースの評価をすり抜ける品質問題を捕まえられる状態がゴールです。

## 対象読者

- 日本語で応対する Agent の品質を評価したいが、英語系の評価指標では不十分と感じているエンジニア
- 「意味の正しさ」は測れているが、日本語としての自然さ・適切さを評価に組み込みたい人

本記事は「日本語として適切か」の評価を扱います。翻訳品質そのもの(原文への忠実さ)は別領域で、日本語の**産出品質**に絞ります。英語で設計した評価をそのまま日本語に流用すると、日本語固有の崩れが見えません。

### 日本語固有の品質軸

英語評価には現れない、日本語特有の品質軸があります。

| 軸 | 見るもの | 崩れの例 |
| --- | --- | --- |
| 敬語・待遇表現 | 相手・場面に合った敬語か | 二重敬語、尊敬語と謙譲語の取り違え、タメ口の混入 |
| 文体一貫性 | 「です・ます」と「である」が混ざっていないか | 1 つの回答内で敬体と常体が揺れる |
| 表記ゆれ・全半角 | 用語・記号の表記が統一されているか | 「サーバ/サーバー」混在、全角半角の不統一 |
| 固有名詞・読み | 人名・地名・製品名の表記と読みが正しいか | 誤変換、読みの誤り |
| 不自然な直訳調 | 英語構文をなぞった不自然さがないか | 「〜することができます」の多用、冗長な受動態 |

これらは**意味は正しくても品質を損なう**軸です。英語ベースの「正確さ」評価では素通りするため、日本語の評価軸として明示的に立てます。

### 評価データの作り方

日本語品質の評価データは、**日本語の実タスク**から作ります。

- **実際の日本語入力から採取する**: 英語データセットの機械翻訳ではなく、実際に日本語で来る依頼・応対ログから作ります(翻訳データは日本語として不自然な入力になりがちです)
- **崩れやすいケースを含める**: 敬語が要る場面、長文で文体が揺れやすいケース、固有名詞の多いケースなど、日本語品質が崩れやすい入力を意図的に集めます
- **期待値は「満たすべき条件」で**: 「敬体で統一されている」「二重敬語がない」「指定表記に従っている」といった条件で書きます([Agent 評価の基礎](agent-evaluation-basics.md)の条件ベースの期待値)

### judge の日本語能力を検証する

日本語の品質を LLM に採点させる(日本語 judge)場合、**その judge が日本語の品質を正しく判定できるか**を検証する必要があります。

- **英語系 judge に日本語採点を任せる危険**: 日本語能力の低いモデルを judge にすると、敬語の崩れ・不自然さを見抜けず、甘い採点になります。judge の日本語能力は、被評価モデルと同等以上が要ります
- **人手ラベルとの一致を測る**: 日本語ネイティブが付けた品質ラベルと judge の判定の一致率を測り、judge を検証してから使います([LLM-as-a-Judge](llm-as-a-judge.md)の judge 検証)。この検証を日本語で行うことが肝心です
- **盲点の共有に注意**: judge と被評価モデルが同系なら、同じ日本語の癖(同じ不自然さ)を「自然」と判定しかねません。可能なら別系統を使います

### 自動チェックと judge の分担

日本語品質のすべてを judge に回す必要はありません。**機械的に測れるものはコードで**測ります。

- **自動チェックが向くもの**: 表記ゆれ(用語辞書との照合)、全半角、文体の敬体/常体の判定、禁止表記の検出。これらは決定的なルールで測れ、速く安く再現可能です([Agent 評価の基礎](agent-evaluation-basics.md)のコード採点)
- **judge が要るもの**: 敬語の適切さ、直訳調の不自然さ、文脈に照らした自然さ。ルール化しにくい開放的な品質は judge に回します
- **分担の原則**: コードで測れる表記・形式は自動チェックに、判断の要る自然さ・適切さは検証済みの日本語 judge に、と分けます。全部を judge に回すのはコストの無駄で、逆に全部を機械ルールにするのは自然さを測れません

### 公開日本語ベンチマークの読み方

日本語の能力・品質を測る公開ベンチマークもありますが、[エージェントベンチマークの全体像](agent-benchmarks-landscape.md)と同じ規律で読みます。

- 公開ベンチのスコアは**当たりを付ける**のに使い、自社タスクの品質は自前の日本語評価セットで測ります。ベンチの分布は自タスクと無関係です
- 具体的なベンチマーク名・リーダーボードの顔ぶれは変化が速いため、本文では帯・傾向にとどめ、選定時に最新を確認します(下記 TODO)

## 実務での注意点

### アンチパターン

- **英語で設計した評価をそのまま日本語に流用する** → 敬語・文体・表記の崩れが素通りする → 日本語固有の品質軸を明示的に立てる
- **英語データの機械翻訳を評価入力にする** → 日本語として不自然な入力になり、実態を測れない → 実際の日本語入力・ログから評価データを作る
- **日本語能力の低いモデルを judge にする** → 敬語の崩れ・不自然さを見抜けず甘く採点する → judge の日本語能力を人手ラベルとの一致で検証する
- **日本語品質をすべて judge に回す** → 表記・形式まで judge に投げてコストを浪費する → 機械的に測れる表記・文体は自動チェックに分ける
- **公開日本語ベンチのスコアで自タスクの品質を推定する** → 分布が違い順位も入れ替わる → 自前の日本語評価セットで測る
