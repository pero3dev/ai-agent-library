# 小型言語モデル(SLM)の活用戦略

## この記事の目的

「大きいモデルで全部やる」から「**小さいモデルを賢く配る**」設計へ移行する判断ができるようになります。小型言語モデル(SLM: Small Language Model)の性格を見極め、SLM ファースト・ルーティング・タスク特化で、品質を保ちながらコストとレイテンシを下げる設計を持ち帰れる状態がゴールです。

## 対象読者

- 上位モデルで作った Agent のコスト・レイテンシに悩み、小型化の余地を探しているエンジニア
- 分類・抽出・判定などの部品を、上位モデルから小型モデルへ移せないか検討しているテックリード

- [モデル選定ガイド](model-selection.md) — 判断軸・用途別の使い分け・ティア混在設計(本記事はその「小型を活かす」戦略の詳解)

### SLM の性格: 得意領域と落ちる崖

SLM を活かす第一歩は、**どこまでできて、どこで急に落ちるか**を見極めることです。

- **得意**: 定型的で範囲の狭いタスク(分類・抽出・整形・短い応答・ルールに沿った判定)。パターンが明確で、深い推論が要らない仕事
- **落ちる崖**: 多段の推論、広い世界知識、長い文脈の統合、曖昧な指示の解釈。ここで**急激に**品質が落ちます(緩やかにではなく、崖のように)
- **見極めが肝**: 「平均的にそこそこ」ではなく、「このタスクなら上位モデルと遜色ないか、それとも崖の向こうか」を評価で確かめます。SLM は得意領域では上位モデルに迫り、崖の向こうでは使い物になりません

### SLM ファースト設計

発想を逆転させ、**既定を小さく**します。

- **小さいモデルを既定にする**: まず SLM で試し、品質が足りないタスク・入力だけを上位モデルに**昇格**させます。「全部を上位モデルで」の逆です
- **昇格のトリガーを決める**: 「SLM が失敗した/確信が低い/検証を通らなかった」ときに上位へ回します(後述のルーティング)
- **上位モデルを最後の手段にする**: 上位モデルは高い・遅いので、SLM で届かない部分にだけ使います。これが効くほど、全体のコスト・レイテンシが下がります([コスト管理](../05-operations/cost-management.md)・[レイテンシ最適化](../05-operations/latency-optimization.md))

### ルーティング: 難易度で振り分ける

SLM ファーストの核が**ルーティング** — 入力の難易度に応じてモデルを振り分ける仕組みです([モデル選定ガイド](model-selection.md)のティア混在設計の実践)。

- **難易度推定で振り分ける**: 入力を見て「簡単そうなら SLM、難しそうなら上位」と振り分けます。難易度推定自体を軽量モデルやルールで行います
- **確信度による昇格**: SLM に答えさせ、確信が低ければ上位モデルに回します([信頼度と較正(calibration)](../04-evaluation/confidence-and-calibration.md)の選択的予測を、棄権でなく上位への昇格に使う形)
- **検証による昇格**: SLM の出力を検証器(スキーマ・テスト・照合)で確かめ、通らなければ上位に回します([ループ内フィードバックと検証器の設計](loop-feedback-and-verification.md))
- **ルーティングのコストを見る**: 難易度推定・確信度取得・検証にもコストがかかります。振り分けのオーバーヘッドが、SLM で得た節約を食い潰さないか確認します

### タスク特化: 「SLM で十分」な部品を切り出す

Agent の中には、**上位モデルが不要な部品**が多く隠れています。

- **切り出せる部品**: 分類(意図判定・カテゴリ分け)、抽出(構造化データの取り出し)、ガードレール判定(入力・出力のチェック)、短い整形・言い換え。これらは範囲が狭く、SLM で十分なことが多いです
- **ワークフロー化と相性が良い**: 手順が固定できる部品は、[Workflow 型](../02-architecture/workflow-vs-agent.md)の部品として SLM に割り当てられます。「Agent の頭脳は上位モデル、定型部品は SLM」という構成です
- **ガードレールは特に**: 入力・出力の安全チェック([ガードレール](../06-security/guardrails.md))は高頻度で走るため、SLM で回せるとコストが大きく下がります

### FT・蒸留との組み合わせ

SLM を「得意領域」で上位モデルに迫らせる手段が、ファインチューニングと蒸留です([ファインチューニングと蒸留](fine-tuning-and-distillation.md)が判断・手法の正本)。

- **タスク特化の FT**: 狭いタスクに特化させると、SLM が汎用の上位モデルに匹敵しうる領域があります
- **蒸留**: 上位モデル(教師)の出力で SLM(生徒)を訓練し、特定タスクの品質を移します。「品質が確認できたタスクを、安く速く量産する」手段です(学習データの作り方は [学習用合成データの実務](synthetic-data-for-training.md))
- **最後の手段として**: プロンプト・ルーティングで届くなら、FT・蒸留の前にそちらを尽くします。FT はデータ作成・維持のコストを伴います

### 評価: タスク別の品質ゲート

SLM 化の可否は、**タスクごとに評価**して決めます。

- **タスク別の品質ゲートを設ける**: 「このタスクを SLM に移して、品質が基準を割らないか」を評価セットで確かめます([Agent 評価の基礎](../04-evaluation/agent-evaluation-basics.md))。全体の平均でなく、移すタスク単位で見ます
- **崖を評価で見つける**: SLM が急に落ちる入力(難しいケース・エッジケース)を評価に含め、ルーティングの昇格条件を調整します
- **コストと品質を同時に見る**: SLM 化で下がったコスト・レイテンシと、品質の変化を並べて判断します。「コストは半分だが品質が基準割れ」なら移せません

## 実務での注意点

### アンチパターン

- **すべてを上位モデルで処理する** → 定型部品にも高コスト・高レイテンシを払う → SLM ファーストにし、SLM で届かない部分だけ上位に昇格する
- **SLM の平均品質だけで採否を決める** → 得意領域と崖の向こうを混同し、崖のタスクで失敗する → タスク別に評価し、崖の位置を見極める
- **ルーティングのオーバーヘッドを無視する** → 難易度推定・検証のコストが節約を食い潰す → 振り分けコスト込みで全体のコスト・レイテンシを測る
- **プロンプトで届くのに FT に飛びつく** → データ作成・維持コストを先に払ってしまう → プロンプト・ルーティングを尽くしてから FT・蒸留を検討する
- **移したタスクの品質を測らずに SLM 化する** → 品質が静かに落ちる → タスク別の品質ゲートで基準割れを確認する
