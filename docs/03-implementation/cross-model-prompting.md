# モデル間の違いと移行(横断比較)

## この記事の目的

1 つのプロンプトを複数モデルで使う、あるいはモデルを乗り換えるときに、**何が壊れ、何を書き直すか**を判断できるようになります。Claude / OpenAI / Gemini の各特化ガイドが個別の作法を扱うのに対し、本記事は**3 社の差分だけ**を横並びにし、移行とマルチプロバイダー設計の勘所を示します。

> **本記事は鮮度管理型です。** モデル世代・機能名は変化が速いため、本文冒頭の最終確認日と各特化ガイド・公式ページを必ず確認してください。

## 対象読者

- 複数のモデルプロバイダーを併用・切り替えする設計をしているエンジニア
- モデルを乗り換える際に、プロンプトの何を作り直すべきか見極めたいエンジニア

### 概要: 何が普遍で、何がモデル依存か

移行で書き直す量を最小化する鍵は、**普遍的な部分とモデル依存の部分を切り分ける**ことです。

| 層 | 普遍性 | 正本 |
| --- | --- | --- |
| 生成の原理(なぜ効くか) | 普遍(モデルに依存しない) | [10 章の基礎](../10-llm-foundations/how-llms-generate-text.md) |
| 汎用技法(明確な指示・区切り・例示・分解) | ほぼ普遍(効き方の強弱は差あり) | [基礎技法](prompt-engineering-fundamentals.md) / [上級パターン](prompt-engineering-patterns.md) |
| **モデル別の具体(記法・パラメータ・API)** | **モデル依存(本記事の対象)** | [Claude](claude-prompting.md) / [OpenAI](openai-prompting.md) / [Gemini](gemini-prompting.md) の各ガイド |
| 移行の運用(回帰評価・段階リリース) | プロセスは普遍 | [バージョニングとモデル更新追従](../05-operations/versioning-and-model-updates.md) |

普遍的な部分(役割を与える・指示とデータを区切る・例で仕様を示す)は乗り換えても生き残ります。壊れるのは主に**記法の癖・制御パラメータ・API の非互換**です。以下はその差分に絞ります。

### 相違の構造マップ(3 社横並び)

本記事の核です。プロンプト設計に効く差分を並べます(2026-07 時点。詳細と根拠は各特化ガイド)。

| 観点 | Claude | OpenAI(GPT 系) | Gemini |
| --- | --- | --- | --- |
| 構造化記法 | **XML タグが第一選択** | Markdown 見出し + XML タグ併用、既定は素のパラグラフ、入れ子箇条書きを避ける | XML タグ・Markdown 見出しの**どちらでも可** |
| システム指示 | `system`(役割)。会話途中の system は一部モデルのみ | **developer > user の指示階層**。推論モデルでは developer が system を置換 | `system_instruction` / 冒頭に重要指示 |
| 例示(few-shot) | Relevant / Diverse / Structured、`<example>` タグ | **zero-shot を先に、必要なら** few-shot | **常時推奨**・全例でフォーマット統一 |
| 思考制御の名前 | アダプティブ思考 + **effort**(low〜max) | **reasoning effort**(none〜xhigh) | **thinking_level**(minimal〜high) |
| 出力形式の強制 | 構造化出力(`output_config.format`) | Structured Outputs(strict `json_schema`) | JSON スキーマ強制 |
| 応答書き出し指定(prefill) | 最新世代で**非対応(400)** | 主要技法として非採用 | 未確認(前提にしない) |
| サンプリング温度 | 非既定値は最新世代で**400** | 下げてよい(決定性狙いで慣用) | **既定 1.0 維持を強く推奨**(下げると劣化) |
| 長文の配置 | データを上・クエリを末尾、文書を XML 構造化 | RAG / file search + コンパクション | **資料を先・質問を末尾** + 転換句 |
| 強調表現の効き方 | `CRITICAL` / `MUST` は過剰反応 | `ALWAYS` / `NEVER` は不変条件だけに | 過度に丁寧・説得的な前置きは不要 |

### 特に壊れやすい差分

上の表のうち、共通プロンプトを流用したときに**静かに品質を落とす**危険が高いのは次の 3 つです。

- **サンプリング温度が真逆**: Claude と GPT 系では決定性のために温度を下げる(または削除する)慣行がありますが、**Gemini 3.x では温度を下げると looping・性能劣化を招きうる**ため既定維持が公式推奨です。共通の「temperature=0」を全社に流すと Gemini だけ壊れます
- **思考制御のパラメータが三者三様**: effort / reasoning effort / thinking_level は名前も値域も既定も違います。しかも**同一ベンダー内でも世代で既定が変わります**(例: GPT の既定 effort、Gemini のモデル別既定)。ベンダー横断の共通値は作れません
- **強制系ハックの効き方が逆転している**: prefill・固定思考予算・強い強調(CRITICAL/ALWAYS)は、あるモデルでは非対応・別のモデルでは過剰反応になります。「効くから」で足したハックほど移行で壊れます

### 移行チェックリスト

モデルを乗り換える(または新世代へ上げる)ときにプロンプト面で確認する項目です。移行の運用手順(段階リリース・ロールバック)は [バージョニングとモデル更新追従](../05-operations/versioning-and-model-updates.md) が正本です。

- [ ] **API の非互換を洗う**: prefill・固定思考予算・非既定サンプリングパラメータが、移行先で拒否(エラー)されないか
- [ ] **思考制御を移行先の名前・値に置き換える**: effort ↔ reasoning effort ↔ thinking_level。既定値の違いも明示ピン留めする
- [ ] **出力形式を移行先の構造化出力機構へ**: プロンプト本文の出力スキーマ指示を、各社の構造化出力に載せ替える
- [ ] **強調表現を調整する**: 前モデル向けの強い強調(CRITICAL/ALWAYS)が移行先で過剰反応しないか
- [ ] **トークナイザ差を見積もり直す**: 同じ文章でもトークン数はモデルで変わります(例: Claude の新世代は従来比で最大約 35% 増)。`max_tokens`・コスト・コンテキスト予算を再計算する([トークナイザとトークン経済](../10-llm-foundations/tokenization.md))
- [ ] **回帰評価を必ず回す**: 言い換え・崩れた入力を含む評価セットで、移行前後を比較する([Agent 評価の基礎](../04-evaluation/agent-evaluation-basics.md))。「動いた気がする」で本番に出さない

### マルチプロバイダー設計

複数モデルを併用・切り替えする前提なら、プロンプトを**モデル別に差し替えられる構造**にします。

- **共通の骨格 + モデル別の薄い差分**: 「完全に共通のプロンプト」は各社の最大公約数に丸める分、どのモデルでも本来の性能を引き出せません。現実解は、タスクの意図・制約・例といった骨格を共通化し、記法(XML か Markdown か)・思考制御・出力形式といった**差分だけをモデル別バリアントに切り出す**ことです
- **バリアントを資産として管理する**: モデル別のプロンプト変種は、その場で分岐させず、テンプレートとして管理・バージョニングします([プロンプト資産の管理とバージョニング](prompt-management.md))
- **差し替え層で吸収する**: モデル選択とプロンプト差し替えを 1 つの層にまとめると、上位のアプリはモデルを意識せずに済みます(ゲートウェイ的な構成。マルチプロバイダーの運用基盤は今後の `llm-gateway` 計画で扱います)

「1 つのプロンプトで全社」を目指すより、「骨格は共通、差分は明示的に分ける」方が、品質と保守性の両立に現実的です。

## 実務での注意点

### アンチパターン

- **1 つのプロンプトを全モデルに無検証で流用する** → 記法・パラメータ・API の差で、あるモデルだけ静かに品質が落ちる → 骨格を共通化し、差分をモデル別バリアントに分けて各モデルで回帰評価する
- **「temperature=0」を全社共通で使う** → Gemini 3.x では劣化・looping を招く → サンプリングの扱いをモデル別に分ける
- **思考制御パラメータをベンダー横断で共通値にする** → 名前も値域も既定も違い、成立しない → 移行先の名前・既定に明示的に置き換える
- **前モデル向けの強調・prefill・固定思考予算を残して乗り換える** → 非対応エラーか過剰反応を招く → 移行先の作法に合わせて外す・置き換える
- **トークナイザ差を無視して `max_tokens`・コストを流用する** → 出力が途中で切れる・コスト見積りが外れる → 移行先で再計測する

### チェックリスト

- [ ] 普遍的な骨格とモデル依存の差分を切り分けている
- [ ] 移行先で拒否される API 機能(prefill・固定思考予算・非既定サンプリング)を洗い出した
- [ ] 思考制御を移行先の名前・既定に置き換えた
- [ ] 出力形式を移行先の構造化出力機構に載せ替えた
- [ ] 強調表現の効き方の差(過剰反応・非対応)を調整した
- [ ] トークナイザ差で `max_tokens`・コスト・コンテキスト予算を再計算した
- [ ] 言い換え・崩れた入力を含む回帰評価で移行前後を比較した
- [ ] マルチプロバイダーではモデル別バリアントを資産として管理している
