# チーム導入とレビュー体制

## この記事の目的

コーディングエージェントを個人の実験から組織の標準ツールへ広げる際の、導入ステップ・レビュー体制・利用ポリシー・コスト管理を設計できるようになります。「禁止か放任か」の二択を避け、統制と生産性を両立する導入計画を立てられることが目標です。

## 対象読者

- チーム・部門へのコーディングエージェント導入を計画するエンジニアリングマネージャー・テックリード
- AI 生成コードのレビュー方針を決める必要があるレビュアー

## 前提知識

- [AI コーディングエージェントの分類と全体像](coding-agents-overview.md)
- [コーディングエージェントの権限とセキュリティ](coding-agent-security.md) — ポリシーの技術的裏付け

## 本文

### 概要

コーディングエージェントの組織導入で最初に認識すべきことは、「導入するかどうか」を検討している間に、**メンバーは既に個人アカウントで使い始めている**という現実です。統制されないシャドー利用は、データ送信・ライセンス・品質のリスクを組織が把握できない形で蓄積します。導入計画の目的は、利用を承認された安全な経路に誘導することです。

### 導入ステップの設計

| 段階 | やること | 完了の目安 |
| --- | --- | --- |
| 1. 現状把握 | 誰が何を既に使っているかの調査(咎めない前提で) | 利用実態と要望の一覧 |
| 2. パイロット | 少数チームで承認ツールを試行。**測定設計を先に決める**([評価](coding-agent-evaluation.md)) | 評価データと利用ガイド草案 |
| 3. ポリシー策定 | 許可ツール・禁止データ・レビュー規約の文書化 | 全員が参照できる利用ポリシー |
| 4. 展開と教育 | 組織契約への移行、オンボーディング、社内チャンピオン設置 | 対象者の大半が承認経路で利用 |
| 5. 定着と改善 | 利用状況・コスト・品質指標の継続測定、ポリシー改訂 | 四半期ごとの見直しサイクル |

パイロットを飛ばして全社契約に進むと、効果の根拠も利用ガイドもないまま展開することになり、定着しません。

### レビュー体制と責任

**責任の所在は変わりません。** コードをコミット・マージした人間が、その内容に責任を持ちます。「AI が書いたから」は品質事故の免責になりません。この原則を明文化することが、レビュー体制の出発点です。

そのうえで、AI 生成コードに合わせてレビュー観点を調整します。

- **もっともらしい誤り** — 人間の書き間違いと異なり、AI の誤りは構文的に整っていて自信ありげです。「動いているように見えるが要件と微妙に違う」「存在しない API の呼び出し」を疑う観点が要ります
- **過剰な変更** — 依頼範囲外の「ついでの修正」が混入しやすく、差分の意図をレビューで問う必要があります
- **テストの質** — エージェントは「テストを通すこと」に最適化しがちです。テストが仕様を検証しているか(実装の写しになっていないか)を重点的に見ます

レビュー負荷への対策も導入設計の一部です。

- 依頼の粒度を「1 PR = 1 レビュー可能単位」に保つ([依頼設計](coding-agent-prompting.md))
- 提出者自身の自己レビュー(差分の全行説明ができる状態)を必須にする
- AI による PR レビュー機能は**人のレビューの補助**として使い、代替にしない([HITL](../02-architecture/human-in-the-loop.md) の承認設計)

また、AI の関与を追跡できるようにする**表示規約**(コミットメッセージの Co-Authored-By 表記、PR テンプレートでの利用ツール申告など)を決めておくと、事後の品質分析とインシデント調査が容易になります。

### 利用ポリシーの策定

最低限、次の 4 点を文書化します。

1. **許可ツールリスト** — セキュリティレビュー([権限とセキュリティ](coding-agent-security.md) のチェックリスト)を通過したツールと契約プラン。個人アカウントの業務利用は原則禁止し、組織契約(学習利用オプトアウト・監査ログ付き)へ誘導します
2. **禁止データ** — 顧客データ・秘密情報・未公開の事業情報をエージェントに渡さないこと、および技術的な強制手段(除外設定)
3. **ライセンス・知財** — 生成コードの著作権の扱い(契約条項の確認)、公開コードとの類似に対するフィルタ機能の利用方針、OSS ライセンス混入への対処
4. **インシデント時の連絡先** — 秘密情報を送信してしまった・不審な挙動を見た場合の報告経路([インシデント対応](../05-operations/incident-response.md))

### コストと利用状況の管理

- 課金モデルは大きく「シート型(定額/人)」と「従量型(トークン・クレジット)」に分かれます。従量型は利用が増えるほど青天井になるため、[コスト管理](../05-operations/cost-management.md) の予算アラート・上限設定の考え方をそのまま適用します
- シート型でも、上位プランの使用量制限(プレミアムリクエスト等)の超過課金があるプランが多く、単価表だけでなく超過時の挙動を確認します
- 利用状況ダッシュボード(誰が・どれだけ・何に使っているか)は、コスト管理と定着測定の両方に必要です

### 教育とオンボーディング

- ツールの操作方法よりも、**依頼設計とレビュー観点**の教育が効果に直結します(ツール操作は個人で習得できますが、依頼の粒度とレビュー基準はチームで揃える必要があります)
- 社内チャンピオン(先行して使いこなしているメンバー)による事例共有会は、ガイドライン文書より速く実践知を広めます
- 「うまくいかなかった依頼」の共有を歓迎する文化にします。失敗事例はルールファイルとガイドラインの改善材料です([ルールファイルの失敗駆動更新](coding-agent-rules-and-config.md))

## 実務での注意点

### アンチパターン

- **禁止一辺倒** — 需要がある限りシャドー利用は止まらず、統制の外で最もリスクの高い形(個人アカウント・無審査ツール)で使われます。→ 承認済みの経路を用意し、そちらを使う方が楽な状態を作ります
- **「AI が書いたから」とレビューを軽くする(または過剰に重くする)** — 軽くすれば品質事故、一律に重くすればスループットが失われます。→ 通常コードと同じ基準を維持しつつ、AI 特有の観点(もっともらしい誤り・過剰変更・テストの質)を追加します
- **効果測定なしの全社展開** — 更改・解約・乗り換えの判断材料がなくなります。→ パイロット段階で測定設計([評価](coding-agent-evaluation.md))を済ませます
- **個人アカウント任せの運用** — データ取り扱い設定・監査ログ・退職時のアクセス管理が効きません。→ 組織契約と SSO に移行します

### チェックリスト

- [ ] 承認済みツールと契約プラン・利用条件が文書化されているか
- [ ] コミット責任の所在(マージした人間)が明文化されているか
- [ ] AI 生成コードのレビュー観点(もっともらしい誤り・過剰変更・テストの質)がレビュアーに共有されているか
- [ ] AI 関与の表示規約(コミット・PR)が決まっているか
- [ ] 組織契約で学習利用オプトアウト・監査ログが有効になっているか
- [ ] 利用量とコストの可視化・予算アラートがあるか
- [ ] 秘密情報を送信してしまった場合の報告経路が周知されているか
