# OpenAI Codex 実践ガイド

## この記事の目的

OpenAI Codex の機能(ローカル / クラウドの各面・スキル・サブエージェント・自動化)を使いどころから理解し、消費(利用制限・クレジット)を抑えながら業務を効率化できるようになります。製品の選定判断は [OpenAI Codex](openai-codex.md) が正で、本記事は「導入済みの人が使いこなす」段階を扱います。

## 対象読者

- Codex を使い始めたが、対話以外の機能(クラウド委任・exec・スキル)を活用できていないエンジニア
- 利用制限・クレジット消費を管理しながらチームで運用したい人

## 前提知識

- [OpenAI Codex](openai-codex.md) — 製品構成・権限モデル・データ取り扱い
- [コーディングエージェントのコスト最適化](coding-agent-cost-optimization.md) / [自動化・業務効率化パターン](coding-agent-automation-patterns.md) — 本記事が前提とするツール非依存の原則

## 本文

> **最終確認日:** 2026-07-06 — 本記事の機能・消費の仕様はこの日付時点の公式ドキュメントに基づきます。制限値・レートは特に変わりやすいため、必ず公式ページ(参考資料)で最新値を確認してください。

### 面の使い分けとハンドオフ

公式ドキュメント自身が面(surface)ごとの使い分けを明示しています。

| 面 | 公式が推す用途 |
| --- | --- |
| IDE 拡張 | ローカル探索。開いているファイルが自動でコンテキストに入り、選択範囲でスコープを絞れる |
| CLI | 再現手順を回す「タイトなループ」。`@` パス補完でファイルを添付 |
| クラウド | 長時間の実装・並列実行・デバイス横断の委任 |
| デスクトップアプリ | スレッド開始時に Local / **Worktree**(隔離された git worktree)/ Cloud を選択。並列作業の管制 |

実践上の要点は**ハンドオフ(引き継ぎ)**です。ローカルで計画を詰めてからクラウドアイコンで委任すると、**既存スレッドの文脈を引き継いだクラウドスレッド**が作られます。「リファクタ計画はローカルで対話的に作り、実行はクラウドに委任する」が公式推奨の分担です。逆方向は、クラウドの結果 diff をローカルに適用して仕上げます。

### カスタマイズ機構の使い分け

Codex は拡張機構が多いため、公式の使い分け指針(Customization ページ)を軸に整理します。

| 機構 | 使うとき(公式指針) |
| --- | --- |
| AGENTS.md | そのリポジトリで毎回従わせたいルール(ビルドコマンド・規約・禁止事項) |
| スキル(`.agents/skills/`) | 反復するワークフロー(リリース手順・レビュー定型・移行手順)。「同じプロンプトを使い回し始めたらスキル化」が公式の判断基準 |
| MCP | リポジトリの外にある能力(Issue トラッカー・デザインツール・共有ドキュメント) |
| サブエージェント(`.codex/agents/*.toml`) | ノイズの多い・専門化したタスクの委任。明示的に指示したときだけ起動 |
| automations(デスクトップ) | 定期チェック。「スキル = 方法、automations = スケジュール」 |

- 公式の導入順序は「(1) AGENTS.md と基盤(lint・pre-commit)→ (2) スキル → (3) MCP → (4) サブエージェント」です。いきなり全部を導入しないでください
- AGENTS.md の更新トリガーも公式が明文化しています: **繰り返すミス・毎回読ませすぎているファイル・PR で繰り返される指摘**が出たら AGENTS.md に昇格します([ルールファイルの失敗駆動更新](coding-agent-rules-and-config.md) と同じ運用です)
- スキルは遅延読込(まず name / description だけ読み、採用時に全文ロード)のため、**description の書き方がトリガー精度を決めます**
- フック(`hooks.json`)はツール実行前後・コンパクション前後などのイベントにスクリプトを差し込めます。公式の用途例には「プロンプト内の API キー誤ペースト検知」があり、[権限とセキュリティ](coding-agent-security.md) の防御層として使えます

### コスト削減: 制限の構造とレバー

**構造の理解が先です**(2026-07 時点): ローカルのメッセージとクラウドタスクは**同一の 5 時間ウィンドウを共有**し、週次制限が追加で適用されます。表向きは「メッセージ数」ですが内部の計量はトークンベースのクレジットで、モデル・入出力・キャッシュ有無でレートが異なります。

そのうえで、公式が明記する節約レバーは次のとおりです。

1. **プロンプトと注入コンテキストの削減** — 不要なコンテキストを削る、AGENTS.md を階層にネストして「そのディレクトリで必要な分だけ」注入する
2. **MCP サーバーの絞り込み** — 接続した MCP はすべてコンテキストを太らせ、制限を消費します。使っていないものは外します
3. **軽量モデルへの切替** — ルーチンタスクは軽量系(mini 系)へ。サブエージェントにも軽量モデルを割り当てるのが公式の想定です
4. **推論レベルをタスクに合わせる** — 速いタスクは Low、複雑な変更だけ High 以上に
5. **スレッドはタスク単位** — 「プロジェクト単位の 1 スレッド運用」は公式の「よくある間違い」リストに載っています。肥大化したスレッドの継続は消費を膨らませます([コスト最適化](coding-agent-cost-optimization.md) の一般原則どおり)

相対関係として押さえる価値がある事実(2026-07 時点、絶対値は変動): **キャッシュ済み入力は通常入力の約 1/10 のレート**(同一スレッドの継続はキャッシュが効きやすい)、**Fast mode は速度 1.5 倍の代わりに消費 2〜2.5 倍**(急がないタスクでは切るのが明確な節約)、モデル間の入力レートは数倍差があります。

- 逆に、**サブエージェントの多用は消費を増やします**(「同等の単一エージェント実行よりトークンを消費する」と公式明記)。並列化の便益と消費のトレードオフで判断してください
- 残量確認は CLI の `/status` と Web の使用量ダッシュボードです。コンパクションは自動(閾値は `model_auto_compact_token_limit` で調整可)+ 手動 `/compact` があります

### 業務効率化・自動化

**非対話実行(`codex exec`)** は CI 組み込みの中核です。設計が CI 向けに整理されています:

- 進捗は stderr、**最終メッセージのみ stdout** に出る(パイプ前提)。`--json` で JSON Lines、`--output-schema` で構造化出力([構造化出力](../03-implementation/structured-output.md))を強制できます
- サンドボックスは **read-only が既定**。書き込みが要る場合だけ明示的に緩めます
- CI 認証は `CODEX_API_KEY` のインライン指定(exec のみ)。「リポジトリが制御するコードに API キーを晒すな」という公式警告どおり、信頼できないコードとの分離が前提です

**GitHub Action(`openai/codex-action@v1`)** は prompt-file 指定で PR レビュー等を組めます。`safety-strategy`(既定 `drop-sudo`)で権限を落とし、公式自身が「ユーザー入力のサニタイズによるプロンプトインジェクション対策」「ジョブの最終ステップに置く」「トリガーを信頼できるユーザーに制限」を指針にしています。

**コードレビュー自動化**: リポジトリ単位のトグルで全 PR の自動レビューを有効化でき、観点は AGENTS.md の `## Review guidelines` セクションで指定します(変更ファイルに最も近い AGENTS.md が適用されるため、パッケージ固有の観点は深い階層に置けます)。GitHub 経由のレビューのみ「Code Review 使用量」として別枠計上されます。

**チャットからの委任**: Slack はスレッド文脈を自動参照するため再説明が不要です。Linear は**トリアージルールで条件に合う Issue を Codex に自動アサイン**でき、人手ゼロの委任パターンが組めます([自動化パターン](coding-agent-automation-patterns.md) の Issue トリアージの実装例)。

**automations(デスクトップ)**: スレッド付随型(文脈を保持して定期的に起こす。長時間処理の監視向け)と独立型(毎回クリーン実行して Triage 受信箱に報告)の 2 種があります。公式ベストプラクティスは「スケジュール化する前に通常スレッドで手動テスト」です。

### 品質を上げる公式プラクティス

- **プロンプトの 4 要素**: Goal / Context / Constraints / **Done when(完了条件)**。[依頼設計](coding-agent-prompting.md) の基本構造と同じ枠組みが公式にも明文化されています
- **難しいタスクは計画から**: Plan mode(`/plan`)を使い、「Codex に自分をインタビューさせて前提を明確化する」手法も公式が案内しています
- **検証手段を渡す**: 「Codex は自分の作業を検証できるとき、より高品質な出力を出す」と公式明記。テスト・lint・型チェックの実行を依頼に含め、仕上げに `/review` を使います
- **公式の「よくある間違い」**: AGENTS.md を使わずプロンプトに毎回詰め込む / ビルド・テストコマンドを教えない / 計画を飛ばす / 理解前にフル権限を与える / **worktree なしでスレッドを並走させる** / 不安定なワークフローを自動化する / プロジェクト単位のスレッド運用 — いずれも本章の各記事の原則と一致します

## 実務での注意点

### アンチパターン

- **1 つのスレッドをプロジェクト全体で使い続ける** — 消費が毎ターン膨らみ、品質も落ちます(公式の「よくある間違い」筆頭)。→ タスク単位でスレッドを切り、`/compact`・新スレッドを使い分けます
- **MCP・サブエージェントを「便利そうだから」全部盛りにする** — 常時コンテキストと消費が増え、制限に早く当たります。→ 公式の導入順序(AGENTS.md → スキル → MCP → サブエージェント)に従い、使っていないものを外します
- **Fast mode を常時オンにする** — 消費が 2〜2.5 倍(2026-07 時点)になります。→ 対話的な反復など速度が効く場面に限定します
- **手動で安定していないワークフローを automations・自動レビューに載せる** — 失敗が定期的に量産されます。→ 通常スレッドで成功パターンを固めてから自動化します([自動化パターン](coding-agent-automation-patterns.md))
