# AI 機能の価格設計とパッケージング

## この記事の目的

原価が従量で変動する AI 機能を、持続可能な価格・パッケージとして設計できるようになります。価格モデルの類型と原価連動・無料枠とトライアルの設計・単価の見える化と予算統制・モデルティアとの整合・改定の設計を、事業判断として持ち帰れる状態を目指します。本記事は特定製品の価格には触れず、**類型と判断軸**を扱います。

## 対象読者

- AI 機能を持つプロダクトの価格・パッケージを設計するプロダクトマネージャー・事業責任者
- 「使われるほど赤字」を避け、持続可能な価格を組みたい担当者

## 本文

### 概要: 「使われるほど原価がかかる」という難しさ

従来のソフトウェアは、一度作れば追加利用の限界費用がほぼゼロでした。AI 機能はそうではありません。**利用のたびにトークンなどの原価がかかり、しかもその原価は従量で変動**します。定額で売って使われすぎれば赤字になり、原価をそのまま転嫁すれば利用者が予算を読めません。

価格設計の核心は、この「**変動する従量原価**」と「利用者が求める予測可能性」をどう両立させるかです。原価側の詳細は [コスト管理](../05-operations/cost-management.md)、契約の課金面(機械間)は [エージェントの API 設計](../02-architecture/agent-api-design.md) が扱い、本記事は**対人プロダクトの価格・パッケージ**を扱います。

### 価格モデルの類型と原価連動

AI 機能の価格には、いくつかの型があります。それぞれ原価との連動の仕方が違います。

| 類型 | 概要 | 原価連動と注意 |
| --- | --- | --- |
| 既存プランに同梱 | 既存の定額に AI 機能を含める | 原価が定額から出るため、ヘビーユーザーで赤字化。利用上限とセットに |
| アドオン定額 | AI 機能を月額アドオンで提供 | 予測可能だが、原価変動を吸収する上限設計が要る |
| クレジット制 | 利用量をクレジットに換算して消費 | 原価と連動しやすく、利用者も残量が見える。換算の分かりやすさが鍵 |
| 純従量 | 使った分だけ課金 | 原価に忠実だが、利用者が予算を読みにくい。上限設定とセットに |
| 成果課金 | 成果(件数・解決など)に対して課金 | 価値と連動するが、原価と成果がずれると赤字化。計測の定義が難しい |

- **原価と連動させる**: どの型でも、原価(従量)と売価が構造的に乖離しないようにします。定額で青天井に使わせると、ヘビーユーザーで赤字になります
- **予測可能性とのバランス**: 純従量は原価に忠実ですが、利用者は「いくらかかるか読めない」ことを嫌います。クレジット制や上限付き定額で、**予測可能性**を持たせます
- **1 つに固定しない**: 「定額 + 超過分は従量」「無料枠 + クレジット」など、型を組み合わせるのが実務的です

### 無料枠・トライアルの設計(コスト暴走ガードとセット)

無料枠やトライアルは獲得に有効ですが、AI では**原価がかかる**ため、設計を誤ると無料利用で赤字が膨らみます。

- **無料枠に上限を設ける**: 無料で使える量に明確な上限(回数・トークン・期間)を設けます。無料枠こそコスト暴走ガード([コスト管理](../05-operations/cost-management.md))とセットにします
- **濫用を防ぐ**: 無料枠の大量取得・自動化による濫用を、レート制限・本人確認などで防ぎます
- **有料への移行導線**: 無料枠を使い切ったときに、有料への自然な移行を設計します。無料で価値を感じさせ、上限で有料の必要性を示します

### 単価の見える化と予算統制(エンタープライズ要求)

特に法人利用では、**コストの見える化と統制**が購買の条件になります。

- **利用状況を見せる**: 誰が・どれだけ使ったか、コストがどう推移しているかを、管理者が見られるようにします。見えないコストは、法人には採用されません
- **予算統制の手段**: 部門・ユーザーごとの上限設定、超過時のアラート・停止を用意します。予算を超えない保証が、法人の安心材料です
- **請求の透明性**: 何にいくらかかったかを、後から説明できる粒度で示します

### 価格とモデルティアの整合

内部で複数のモデルティア([モデル選定ガイド](../03-implementation/model-selection.md))を使い分けるとき、価格設計と整合させます。

- **ティア混在を価格に反映する**: 安いモデルで足りるタスクと、上位モデルが要るタスクで原価が違います。これを価格・パッケージにどう反映するか(上位機能を上位プランに、など)を設計します
- **原価の内訳を把握する**: どの機能がどのティアを使い、原価がいくらかを把握しないと、価格が原価とずれます。価格設計は原価構造の把握が前提です

### 改定の設計(原価変動への追従)

AI の原価(モデルの価格)は変動します。価格を一度決めて放置せず、**改定できる設計**にします。

- **原価変動に追従する仕組み**: モデル価格の変動を、自社の価格に反映する運用を用意します。原価が下がったときに還元し、上がったときに吸収する幅を持たせます
- **クレジット制は改定しやすい**: 純粋な金額固定より、クレジット制のように「換算レートを調整できる」構造のほうが、原価変動に追従しやすいです
- **改定を利用者に納得してもらう**: 価格改定は不信を招きやすいため、根拠(原価変動・価値の向上)を説明し、既存利用者への配慮を設計します

## 実務での注意点

### アンチパターン

- **定額で青天井に使わせる** → ヘビーユーザーで赤字になる → 利用上限・クレジット制で原価と連動させる
- **純従量だけで売る** → 利用者が予算を読めず採用されない → 上限付き定額・クレジット制で予測可能性を持たせる
- **無料枠に上限もガードもない** → 無料利用で原価が膨らむ → 明確な上限とコスト暴走ガード・濫用対策をセットにする
- **コストが見えない・統制できない** → 法人に採用されない → 利用状況の可視化・部門別上限・アラートを用意する
- **原価構造を把握せず価格を決める** → 価格が原価とずれる → モデルティアの混在と原価内訳を把握する
- **価格を決めて放置する** → 原価変動に追従できない → 改定できる構造(クレジット制など)にし、変動への追従を運用する

### チェックリスト

- [ ] 価格モデルの型を選び、従量原価と構造的に乖離しないようにしたか
- [ ] 純従量の予測不能性を、上限やクレジット制で補ったか
- [ ] 無料枠に上限・コスト暴走ガード・濫用対策をセットにしたか
- [ ] 利用状況の可視化・部門別上限・アラートなど法人の予算統制を用意したか
- [ ] モデルティアの混在と原価内訳を把握し、価格に反映したか
- [ ] 原価変動に追従できる改定の仕組みを設計したか
