# ベクトルデータベースの選定と運用

## この記事の目的

ベクトル検索の基盤を、**過剰でも過小でもない構成**で選定・運用できるようになります。専用ベクトル DB を入れるべきか、既存 DB の拡張で足りるか、そもそも DB が要らないかを、規模・チーム・既存資産から判断できる状態がゴールです。

## 対象読者

- RAG や意味検索の基盤選定を担当し、「どのベクトル DB にするか(あるいは要るのか)」を決めるエンジニア
- 小さく始めた検索が育ってきて、基盤の作り直しを検討しているエンジニア

### 選択肢の類型

ベクトル検索の実現手段は、おおむね 4 類型です。

| 類型 | 内容 | 向く状況 |
| --- | --- | --- |
| 専用ベクトル DB | ベクトル検索に特化した製品 | 大規模・高スループット・高度なフィルタが要る |
| 既存 DB のベクトル拡張 | 使っている DB(リレーショナル DB 等)のベクトル機能 | 既存データと同居させたい・運用対象を増やしたくない |
| マネージド検索サービス | 検索基盤をサービスとして利用 | 運用負荷を外に出したい・スケールを任せたい |
| ライブラリ内蔵(組み込み) | アプリに埋め込む軽量な索引ライブラリ | 小〜中規模・単一プロセス・プロトタイプ |

判断軸は 3 つです。

- **規模**: チャンク件数・クエリ QPS・更新頻度。数万件と数億件では答えが変わります
- **チーム**: 運用できる人数・スキル。専用 DB は 1 つ運用対象が増えます
- **既存資産**: すでに使っている DB にベクトル機能があるなら、それで足りることが多い(運用対象を増やさない価値は大きい)

原則は、**既存資産で足りるならそれを使い、足りなくなってから専用 DB へ**です。最初から大規模前提の専用 DB を入れて持て余す、が最頻出の過剰設計です。

### 近似最近傍探索(ANN)の直感

ベクトル検索の中身は「クエリベクトルに最も近い上位 k 件を探す」ことです。全件と厳密に距離計算する(総当たり)と、件数が増えるほど遅くなります。そこで実務では**近似最近傍探索(ANN: Approximate Nearest Neighbor)**を使い、多少の取りこぼしと引き換えに高速化します。

- **再現率と速度のトレードオフ**: ANN は「本当の上位 k 件」を完全には保証しません。インデックスのパラメータで、**再現率(取りこぼさなさ)を上げると遅く・重くなり、速度を上げると再現率が下がる**という調整になります
- **インデックス方式は複数ある**: グラフ型・クラスタ型・量子化(ベクトルを圧縮してメモリを減らす)など、方式ごとに「速度・メモリ・再現率・更新のしやすさ」の得手不得手があります。細部より、**この 4 つのトレードオフで比較する**という見方を持つことが実務では有効です
- **既定値で始めて評価で詰める**: 多くの製品は妥当な既定値を持ちます。まず既定で動かし、再現率が足りなければパラメータを上げる、という順で調整します(再現率の測り方は [RAG 実装パターン](rag-implementation-patterns.md) の検索評価)

### メタデータフィルタとハイブリッド検索の対応力

選定で見落とされがちですが**最重要**なのが、ベクトル検索単体ではなく「**フィルタ付き・ハイブリッド検索**をどれだけ素直に実現できるか」です。

- **メタデータフィルタ**: 「この部署の・この期間の・閲覧権限のある文書だけ」で絞る機能。とくに**権限反映(ACL)検索**は検索段で必須で(生成後フィルタでは手遅れ。[RAG 実装パターン](rag-implementation-patterns.md)・[データ漏えい対策](../06-security/data-exfiltration.md))、基盤がフィルタ付き検索を効率的にできるかが選定を左右します
- **テナント分離**: マルチテナントでは、テナント境界をコレクション分離やメタデータフィルタで担保します。実装要件は [マルチテナント設計](../02-architecture/multi-tenancy-and-isolation.md) の「知識源の分離」が正本です
- **ハイブリッド検索**: ベクトルとキーワード(BM25 等)の併用を基盤側でどこまで支援するか(順位融合まで面倒を見るか、自前で組むか)。ハイブリッドが実務の初期値である以上([RAG 実装パターン](rag-implementation-patterns.md))、この対応力は効きます

「ベクトル検索が速い」だけで選ぶと、フィルタ・ハイブリッド・権限反映で苦労します。**実際のクエリ形(フィルタ込み)で評価**します。

### スケールとコスト構造

- **メモリ常駐 vs ディスク**: 高速な方式はベクトルをメモリに載せます。数百万〜数億ベクトルではメモリ量がそのままコストになります。ディスク中心の方式は安いが遅い、というトレードオフです。**次元数(埋め込み側の選定。[埋め込みの選定と運用](embeddings.md))がメモリ量に直結**します
- **量子化でメモリを下げる**: ベクトルを圧縮して常駐コストを下げる方式があります。再現率の劣化を評価で確認したうえで使います
- **コストは件数 × 次元 × 更新頻度で効く**: 静的な数万件と、高頻度更新の数億件では、コスト構造がまったく違います。規模の見積りを先に立てます

### 運用

- **再インデックス**: 埋め込みモデルやチャンキング設定を変えたら作り直しになります(埋め込みモデル更新 = 全再インデックスの詳細は [埋め込みの選定と運用](embeddings.md))。新インデックスを別に構築して切り替える手順にします
- **バックアップと復旧**: ベクトルインデックスは「埋め込みし直せば再生成できる」ものですが、再生成には時間とコストがかかります。原本(前処理済みデータ)とインデックスの両方の復旧手順を決めます
- **監視**: 検索レイテンシ・再現率の劣化・インデックスサイズの増加を監視します。「いつの間にか遅く・当たらなくなっていた」を防ぎます([可観測性とトレーシング](../05-operations/observability-and-tracing.md))

### 小規模なら単純解でよい

**数万件程度までなら、専用ベクトル DB は多くの場合不要**です。

- ライブラリ内蔵の索引や、既存 DB のベクトル拡張で十分な速度が出ます。運用対象を増やさない利点が大きいです
- 総当たり(厳密検索)でも、数万件なら実用的なレイテンシで動くことがあります。ANN のチューニングに悩む前に、まず単純解で要件を満たせないか確かめます
- 「将来数億件になるかもしれない」を理由に最初から大規模基盤を入れるより、**現在の規模の単純解で始め、成長シグナル(件数・QPS・レイテンシ)が出てから移行**するほうが、多くのプロジェクトで正解です

## 実務での注意点

### アンチパターン

- **最初から大規模前提の専用ベクトル DB を入れる** → 運用対象が増え、数万件のデータを持て余す → 既存資産・単純解で始め、成長シグナルが出てから移行する
- **ベクトル検索の速度だけで選ぶ** → フィルタ・ハイブリッド・権限反映で苦労する → 実際のクエリ形(フィルタ込み)で対応力を評価する
- **権限フィルタを基盤の外(生成後)で掛ける** → 権限外文書が混入し、検索が権限バイパス経路になる → 検索段のメタデータフィルタで担保する([マルチテナント設計](../02-architecture/multi-tenancy-and-isolation.md))
- **次元数を決めずに基盤を選ぶ** → メモリ量・コストの見積りが立たない → 埋め込みの次元をコスト構造の前提として先に固める
- **ANN の再現率を測らずに運用する** → 静かに取りこぼしが増える → 検索評価で再現率を監視し、パラメータを調整する

### チェックリスト

- [ ] 「専用ベクトル DB が本当に要るか」を規模から判断した(数万件なら単純解を検討)
- [ ] 既存資産(使っている DB のベクトル拡張)で足りないかを先に確認した
- [ ] メタデータフィルタ・権限反映・ハイブリッド検索の対応力を実クエリ形で評価した
- [ ] 次元数を前提に、メモリ常駐 vs ディスクのコスト構造を見積もった
- [ ] ANN の再現率と速度のトレードオフを評価で調整している
- [ ] 再インデックス・バックアップ・監視の運用手順を決めた
- [ ] マルチテナントの場合、テナント分離の実装要件を満たしている
