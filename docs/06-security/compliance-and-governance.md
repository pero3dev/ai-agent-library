# コンプライアンスとガバナンス

## この記事の目的

AI Agent を業務・プロダクトに導入する際のコンプライアンス対応を、(1) 規制の地図、(2) 会話ログのデータ保護、(3) 社内ガバナンス、(4) 監査への備え、(5) ベンダー契約の 5 つに分解して着手できるようになります。規制の内容を解説する記事ではなく、「何が存在し、いつ適用され、どの一次情報を見るべきか」の**入口の地図**を提供します。

> **免責:** 本記事は法的助言ではありません。規制の適用判断・契約の解釈は、必ず自社の法務部門・専門家に確認してください。

## 対象読者

- Agent の本番導入にあたり、法規制・社内統制の論点を洗い出す立場のエンジニア・テックリード
- 法務・コンプライアンス部門と会話するための共通地図が欲しい開発チーム

### 概要: 5 つの問いに分解する

AI のコンプライアンスは漠然と捉えると動けません。次の 5 つの問いに分解すると、それぞれ担当と作業が決まります。

| 問い | 主な相手 | 本記事の節 |
| --- | --- | --- |
| どの法域のどの規制が自分に適用されるか | 法務 | 規制の全体像 |
| 会話ログ(個人データ)をどう扱うか | 法務・セキュリティ | データ保護と会話ログ |
| 社内の AI 利用を誰がどう統制するか | 経営・情報システム | 社内ガバナンス |
| 「適切にやっている」ことをどう証明するか | 監査・顧客 | 監査への備え |
| ベンダーとの契約で何を確認するか | 法務・調達 | ベンダー契約チェック |

エージェントは「自動化された判断 + 実世界への行動」を持つため、AI 規制が重視する論点(透明性・人間による監視・ログ)と正面からぶつかります。後付けでの対応は設計変更を強いるため、[PoC から本番への進め方](../09-business/poc-to-production.md)の関門にデータ・コンプライアンス観点を含めるのが実務の型です。

### 規制の全体像(2026-07 時点)

規制は変化が速いため、ここでは「存在・時間軸・一次情報の場所」だけを押さえます。

**EU — AI Act(Regulation (EU) 2024/1689)**。世界で最も包括的な AI 規制で、リスクベースの階層構造を取ります: 禁止プラクティス / 高リスク(雇用・教育・重要サービス等。リスク管理・ログ・人間による監視などの義務)/ 透明性義務(AI との対話の開示・生成コンテンツの識別可能化)/ 最小リスク(義務なし)。加えて汎用 AI(GPAI)モデル提供者への横断的義務があります。適用は段階制で、2026-07 時点の時間軸は 3 層です。

1. **適用済み**: 禁止プラクティス(2025-02-02〜)、GPAI モデル義務(2025-08-02〜)
2. **2026-08-02 から**: 全般適用(AI との対話の開示などの透明性義務を含む)
3. **延期が確定**: 高リスクシステムの義務は「Digital Omnibus on AI」改正により 2027-12-02(単体システム)/ 2028-08-02(製品組み込み)へ延期。この改正は 2026-06 に立法プロセスを完了しており、2026-07-07 時点では官報公布待ちです

重要な概念として、義務は **提供者(provider)と導入者(deployer)で異なります**。LLM API を使って自社アプリを提供する企業は、構成によりどちらにもなり得るため、「自分はどちらの立場か」が最初の確認事項です。一次情報は欧州委員会の公式実装ポータル(AI Act Service Desk)が入口になります。

**日本 — 推進法 + ソフトロー + 個人情報保護法**。2025 年成立の AI 法(正式名称「人工知能関連技術の研究開発及び活用の推進に関する法律」)は**規制法ではなく推進法**で、事業者への義務は国の施策への協力責務にとどまります(公式の概要資料に罰則の記載はありません)。実務の参照点はむしろ、(1) 総務省・経産省の「AI 事業者ガイドライン」(法的拘束力のないソフトロー。開発者・提供者・利用者の 3 主体別に整理)、(2) **個人情報保護法**です。個人情報保護委員会は生成 AI について「個人データを含むプロンプトが応答生成以外の目的(学習など)に利用される場合、本人同意なしでは違反のおそれ」と注意喚起しており、これはエージェントの入力設計に直接効きます。なお課徴金制度を含む改正案が 2026-07 時点で国会審議中です。

**米国 — 包括連邦法なし・流動的なパッチワーク**。連邦の包括的 AI 規制法はなく、大統領令と既存法の執行、州法の組み合わせです。州法は California(学習データ透明性などが 2026 年から段階施行)や Colorado(縮小改正を経て 2027 年適用予定)が代表例ですが、連邦側が州 AI 法への対抗(訴訟・補助金の条件付け)を打ち出しており、**州法の適用見通し自体が流動的**です。米国向け展開では「執筆時点のスナップショットで判断しない」ことが最重要です。

**国際規格 — 説明可能な体制の道具**。NIST AI Risk Management Framework(任意のリスク管理フレームワーク。生成 AI プロファイルあり)と ISO/IEC 42001(AI マネジメントシステムの認証可能な規格)は、規制そのものではありませんが、「体系的に管理している」ことを対外的に説明する共通言語として使われます。顧客からの調達要件に現れることも増えています。

### データ保護と会話ログ

エージェントの会話ログは、ユーザー入力・ツールが取得した社内データ・生成応答を含む**個人データ・機密データの塊**です。技術面の扱い([可観測性とトレーシング](../05-operations/observability-and-tracing.md)のマスキング・保持・アクセス制御)に加え、制度面で次を決めます。

- **利用目的の特定と通知**: 会話ログを何に使うか(品質改善・評価データセット・監査)を利用目的として整理し、プライバシーポリシー・社内規程に反映します
- **学習利用の管理**: ベンダー側の学習利用の既定(後述)と、自社側での二次利用(評価・分析)の両方を管理します。本番ログを評価データセットへ還流するときのマスキングは[評価データセットの構築と保守](../04-evaluation/evaluation-datasets.md)の前提条件です
- **保持期間**: 「全文は短く、集計は長く」の原則で期間を決め、自動削除を実装します
- **越境移転**: ベンダーの処理リージョンと、自国の移転規制(EU の十分性・標準契約条項など)の対応を法務と確認します
- **削除要求への対応**: 削除要求は会話ログだけでなく、長期記憶・評価データセット・派生データまで波及します。削除経路の設計は[長期記憶の実装](../03-implementation/long-term-memory-implementation.md)で扱ったとおり、後付けできません

### 社内ガバナンス

- **AI 利用ポリシーを 1 枚作る**: 最低限、(1) 許可されるユースケースの類型と禁止事項、(2) 入力してよいデータの分類(公開情報 / 社内情報 / 個人データ・機密の別)、(3) 新規ユースケースの承認プロセス、(4) 責任者、を定めます。「何が許可されているか分からないから各自が勝手に使う」状態が最大のリスクです
- **AI インベントリ(台帳)**: どの部門で・何の AI/Agent が・どんなデータで動いているかの一覧を維持します。EU AI Act のような分類ベースの規制対応も、監査対応も、この台帳がなければ始まりません
- **リスクに応じたレビューゲート**: すべてを重厚に審査すると形骸化します。個人データ・自動実行・社外影響の有無でリスクを分類し、低リスクは軽く、高リスクは[脅威モデル](threat-model-overview.md)・データ保護・[人間の関与設計](../02-architecture/human-in-the-loop.md)のレビューを必須にします
- **既存プロセスに統合する**: 新しい「AI 委員会」を乱立させるより、既存のセキュリティレビュー・個人情報管理・調達プロセスに AI 観点を追加する方が回ります。[ユースケース発見と要件定義](../09-business/usecase-discovery.md)・[PoC から本番への進め方](../09-business/poc-to-production.md)の関門に組み込むのが自然な統合先です

### 監査への備え

監査対応の要諦は、「監査のために書類を作る」のではなく、**普段の運用の副産物として記録が残る**ようにすることです。エージェントの場合、求められる記録の多くは本ライブラリの他章で設計済みのものと一致します。

| 記録 | 普段の運用での出どころ |
| --- | --- |
| AI の利用一覧と用途 | AI インベントリ(前節) |
| リスク評価・レビューの記録 | レビューゲートの成果物(脅威モデル・承認記録) |
| いつ・どの構成で動いていたか | プロンプト・モデルのバージョン管理とトレース([バージョニング・デプロイ・モデル更新追従](../05-operations/versioning-and-model-updates.md)) |
| 人間による監視・承認の記録 | 承認ログと帰属([エージェントの認証・認可](agent-identity-and-auth.md)の actor / subject 記録) |
| 品質・安全性の検証記録 | 評価結果とリリース判定([Agent 評価の基礎](../04-evaluation/agent-evaluation-basics.md)) |
| インシデントと対応の記録 | [インシデント対応](../05-operations/incident-response.md)の記録・振り返り |

EU AI Act の高リスク義務(ログ保存・人間による監視・技術文書)は延期されましたが、上の表の運用はどの規制が来ても共通の土台になります。「規制が確定してから作る」のではなく、運用の質として先に整えておくのが結果的に安上がりです。

### ベンダー契約チェック

LLM ベンダー・エージェント製品との契約では、最低限次を確認します(2026-07 時点の状況を例示しますが、**規約は頻繁に変わるため必ず契約時に公式ページで確認し、確認日を記録**してください)。

- **学習利用の既定**: 主要ベンダー(Anthropic・OpenAI・Google)はいずれも「API・商用経由の入力は既定でモデル学習に使わない」ことを公式に明記しています。ただし**無償枠・個人向けプランでは既定が反転する例**があり(例: Gemini API の無償枠は提出内容を改善に利用すると明記)、「同じベンダーでも契約形態で既定が違う」のが落とし穴です
- **データ保持と ZDR**: 既定では不正利用監視などの目的で一定期間保持されるのが一般的です。保持ゼロ(ZDR: zero data retention)は**既定ではなく審査・承認制**であり、対象製品・エンドポイントも限定されます。「ZDR がある」ではなく「自分の構成が対象か」を確認します
- **データ処理契約(DPA)**: 個人データを扱うなら、管理者(controller)/ 処理者(processor)の役割分担、再委託先(サブプロセッサ)、侵害時の通知義務を含む DPA の締結を確認します
- **処理リージョンと越境**: データがどこで処理・保存されるか、リージョン指定が可能か
- **認証・監査資料**: SOC 2・ISO/IEC 27001・42001 などの第三者認証や監査レポートの提供有無
- **モデル・規約の変更通知**: モデルの廃止や規約変更がどう通知されるか([バージョニング・デプロイ・モデル更新追従](../05-operations/versioning-and-model-updates.md)の廃止スケジュール追跡と対になる契約面です)

## 実務での注意点

### アンチパターン

- **規制の適用時期・内容を解説記事の記憶で判断する** → 延期・改正が頻繁で(EU 高リスク義務の延期、米州法の縮小改正など)、古い情報での判断は過剰対応にも無対応にもなる → 一次情報(公式ポータル・官報・所管当局)で確認し、最終判断は法務に通す
- **「国内サービスだから海外規制は無関係」と決める** → EU 域内の利用者に影響が及ぶ構成では域外にも適用され得る → 適用範囲(地理・立場 = provider / deployer)の確認を法務と最初に行う
- **無償枠・個人アカウントで業務データを扱う** → 学習利用の既定が商用契約と反転している場合があり、入力した業務データが学習に使われる → 商用契約・API の規約を確認し、利用ポリシーで無償枠の業務利用を明示的に禁止する
- **会話ログを通常のアプリログと同じ扱いで無期限保持する** → 個人データの塊であり、保持・削除・越境の義務に抵触し得る → 利用目的・保持期間・削除経路を設計し、自動化する
- **監査書類を監査の直前に作る** → 過去の構成・判断を再現できず、記録の信頼性も低い → バージョン管理・トレース・承認ログの副産物として普段から残す
- **AI ガバナンスを新設の委員会だけで担う** → 現場のリリースフローと乖離して形骸化する → 既存のセキュリティ・個人情報・調達レビューに AI 観点を統合する

### チェックリスト

- [ ] 自社の Agent が「どの法域の・どの分類・どの立場(provider / deployer)」に当たるかを法務と確認した
- [ ] 規制の一次情報の定点観測先(公式ポータル・所管当局)と確認担当を決めた
- [ ] 会話ログの利用目的・保持期間・削除経路・越境の扱いが文書化されている
- [ ] AI 利用ポリシー(許可ユースケース・入力可能データ・承認プロセス・責任者)が周知されている
- [ ] AI インベントリ(どこで・何が・どんなデータで動くか)が維持されている
- [ ] リスクに応じたレビューゲートが PoC → 本番の関門に統合されている
- [ ] 監査で求められる記録(バージョン・承認・評価・インシデント)を普段の運用から出せる
- [ ] ベンダーの学習利用既定・保持 / ZDR・DPA・リージョンを契約時に確認し、確認日を記録した
- [ ] 規制・規約の変化を四半期ごとに見直す運用がある
