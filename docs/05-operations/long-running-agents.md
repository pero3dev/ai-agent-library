# 常駐エージェントのライフサイクル管理

## この記事の目的

数週間〜数か月動き続けるエージェント(常駐アシスタント・監視エージェント・継続タスク)の**緩やかな劣化と世代交代**を運用できるようになります。短時間のタスクでは表面化しない、長期運用に固有の劣化(記憶の肥大・指示の矛盾・環境とのずれ・品質ドリフト)を検知し、メンテナンス・引き継ぎ・退役まで設計できる状態がゴールです。

## 対象読者

- 立ち上げっぱなしで長期間動かすエージェントを運用する、または運用予定のエンジニア・SRE
- 「最初は良かったのに、動かし続けるうちに挙動が怪しくなる」を経験しているチーム

### 長期運用で起きる劣化の類型

短時間のタスクでは見えない、長期運用に固有の劣化があります。

| 劣化 | 内容 |
| --- | --- |
| 記憶・コンテキストの肥大と汚染 | 履歴・記憶が増え続け、古い決定・無関係な脱線が判断を狂わせる |
| 蓄積した設定・指示の矛盾 | 継ぎ足した指示・ルールが矛盾を抱える(「呪文の地層」の実行時版) |
| 外部環境とのずれ | ツール・API・知識源が変わり、個体の前提と現実がずれる |
| 品質の緩やかなドリフト | どこかで壊れたわけではないのに、少しずつ挙動・品質が想定から外れる |

要点は、これらが**急な故障ではなく緩やかな劣化**として来ることです。ある日落ちるのではなく、気づけば質が落ちている。だからこそ、後述の定期メンテナンスと計測(SLI・[AI 品質の SLO 設計](ai-slo-design.md))で**緩やかな下降を検知**します。

### 定期メンテナンスの設計

常駐個体は、放置すれば劣化します。**定期的な手入れ**を設計に組み込みます。

- **記憶の棚卸し**: 溜まった記憶・履歴を定期的に見直し、古い決定・無効になった情報を整理します([長期記憶の実装](../03-implementation/long-term-memory-implementation.md)の忘却の運用)。決定事項は残し、試行錯誤の生ログは落とす、という区別で行います
- **仕切り直し(コンテキストのリセット)**: 汚染が進んだら、外部化した状態(計画・決定・成果物への参照)から新しいコンテキストを作り直します([コンテキストの圧縮と隔離](../02-architecture/context-compaction-and-isolation.md)の仕切り直し)。平時から状態を外部化しておくことが、これを可能にします
- **ヘルスチェック**: 「個体が健全か」を定期的に確かめます。SLI の推移(品質・成功率・エスカレーション率)を監視し、緩やかな下降を早期に捉えます([可観測性とトレーシング](observability-and-tracing.md)・[AI 品質の SLO 設計](ai-slo-design.md))
- **設定・指示の整理**: 継ぎ足した指示の矛盾を定期的に棚卸しします(プロンプト資産の管理と同じ規律。[プロンプト資産の管理とバージョニング](../03-implementation/prompt-management.md))

### 世代交代: 新しい個体への引き継ぎ

劣化した個体は、手入れで戻らなければ**新しい個体に世代交代**します。人の引き継ぎと同じ設計が要ります。

- **何を移し、何を捨てるか**: 決定事項・確立した知識・進行中のタスクは引き継ぎ、汚染した履歴・無効な設定は捨てます。「全部コピー」は劣化ごと引き継ぐことになります
- **並行稼働で検証する**: 新個体をいきなり切り替えず、旧個体と並行して動かし、挙動・品質が同等以上かを確かめてから切り替えます(カナリア・シャドーランと同じ発想。[バージョニングとモデル更新追従](versioning-and-model-updates.md))
- **引き継ぎ材料を平時から用意する**: 世代交代を可能にするのは、外部化した状態(決定ログ・記憶ストア・タスクリスト)です。個体の頭の中だけに情報があると、引き継げません

### 個体差の管理

見落とされがちですが、**同じ設定でも履歴によって挙動が分かれます**。

- 常駐個体は、動いてきた履歴・蓄積した記憶が個体ごとに違うため、「同じプロンプト・同じモデルの個体 A と B が違う振る舞いをする」が起きます
- 複数の常駐個体を運用するなら、この個体差を前提にします。1 個体で観測した挙動が他個体でも同じとは限りません
- 再現性が要る場面では、個体の状態(記憶・設定)もバージョン管理・スナップショットの対象にします

### 退役の設計

常駐個体には**終わり**も設計します。

- 役目を終えた・劣化が進んだ個体は、状態を保全(または安全に破棄)して退役させます。動かしっぱなしの個体は、コスト・セキュリティ・劣化のリスクを溜め続けます
- 退役時に、記憶ストア・永続化した状態の削除・保持を決めます(データガバナンス・[会話データの管理基盤](conversation-data-management.md)の保持と削除)
- 「いつ退役させるか」の基準(SLI がこの水準を割ったら世代交代・退役)を、立ち上げ時に決めておきます

## 実務での注意点

### アンチパターン

- **常駐個体を立ち上げっぱなしで放置する** → 記憶肥大・指示矛盾・環境ずれで緩やかに劣化する → 定期メンテナンス(棚卸し・仕切り直し・ヘルスチェック)を組み込む
- **劣化を急な故障としてしか監視しない** → 緩やかな品質ドリフトを見逃す → SLI の推移で下降を早期に捉える
- **世代交代で全状態をコピーする** → 汚染した履歴・無効な設定ごと引き継ぐ → 決定・知識は移し、汚染・無効は捨てる
- **新個体をいきなり切り替える** → 品質低下に気づかず本番が劣化する → 並行稼働で同等以上を確認してから切り替える
- **状態を個体の中だけに持つ** → 世代交代も仕切り直しもできない → 決定・記憶・タスクを平時から外部化する
- **退役を設計しない** → 不要な個体がコスト・リスクを溜め続ける → 退役基準と状態の削除・保持を最初に決める

### チェックリスト

- [ ] 長期劣化の類型(記憶肥大・指示矛盾・環境ずれ・品質ドリフト)を監視対象にしている
- [ ] SLI の推移で緩やかな品質下降を検知している
- [ ] 記憶の棚卸し・仕切り直し・ヘルスチェックの定期メンテナンスがある
- [ ] 決定・記憶・タスクを外部化し、仕切り直し・引き継ぎを可能にしている
- [ ] 世代交代で何を移し何を捨てるかの基準がある
- [ ] 新個体を並行稼働で検証してから切り替えている
- [ ] 複数個体を運用する場合、個体差(同設定でも挙動が分かれる)を前提にしている
- [ ] 退役基準と、退役時の状態の削除・保持を決めている
