# エージェントの責任と説明責任

> **免責:** 本記事は責任・説明責任を**論点として整理**するもので、法的助言・特定の結論の提示ではありません。実際の責任分界・契約条項は、各案件の契約・準拠法・弁護士の判断に従ってください。

## この記事の目的

エージェントが事故を起こしたとき「**誰が・何に基づいて説明するか**」を、契約と技術の両面から事前に設計できるようになります。責任分界の論点・契約での手当の観点・説明責任を支える技術的裏付け・自律度と責任の関係・事故対応での説明の型を、**論点整理**として持ち帰れる状態を目指します。本記事は法的助言をせず、あくまで**設計・契約に落とすための論点整理**に徹します(法的判断は法務・弁護士の領分です)。

## 対象読者

- エージェントを導入・提供する立場で、事故時の責任と説明責任を事前に設計したいプロダクト責任者・テックリード
- 契約・ガバナンスの観点を技術設計に接続したい PM・法務連携担当

## 本文

### 概要: 「説明できる」を事前に設計する

エージェントは自律的に行動するため、事故が起きたとき「**誰の責任か・何に基づいて説明するか**」が曖昧になりがちです。本記事は、この曖昧さを事前に潰す論点整理です。規制の地図は [コンプライアンスとガバナンス](../06-security/compliance-and-governance.md) が、ログの実装は既存の運用記事が正本で、ここでは**論点の整理と設計への落とし込み**を扱います。法的な結論は出さず、「何を決めておくべきか」を示します。

### 責任分界の論点

エージェントの事故では、**複数の関係者**が関わります。「誰が責任を負うか」は契約・準拠法で決まるため断定できませんが、**論点として関係者を整理**しておくことが第一歩です。

| 関係者 | 関わり |
| --- | --- |
| モデル提供者 | 基盤モデルを提供する |
| システム提供者 | エージェントを構築・提供する |
| 導入企業 | エージェントを業務に組み込み運用する |
| 利用者 | エージェントを操作・利用する |

- **責任は一様でない**: 事故の原因(モデルの欠陥・システムの設計・運用の誤り・利用者の誤用)によって、責任の所在の論点は変わります。「AI がやったこと」で片付けず、**どの層に起因するか**を切り分けられるようにします
- **断定でなく論点として**: 本記事は「誰が悪い」を断定しません。関係者を洗い出し、**契約と設計で責任分界を事前に取り決める**ための論点を提供します

### 契約での手当(観点の列挙)

責任分界は、多くが**契約で手当て**されます。契約は法務の領分ですが、**設計者として押さえるべき観点**を列挙できます(条項の解釈・作成は法務・弁護士へ)。

- **責任制限・免責**: どの範囲まで責任を負うか・免責されるか
- **SLA(サービス品質保証)**: 品質・可用性の水準と、未達時の扱い
- **保険**: 事故時の補償をどう手当てするか
- **知的財産・データの取り扱い**: 生成物の権利・データの利用条件([AI と著作権・知的財産の入口マップ](ai-copyright-and-ip-map.md)・[コンプライアンスとガバナンス](../06-security/compliance-and-governance.md)のベンダー契約)
- **観点の提供に留める**: 本記事はこれらを**確認すべき観点**として挙げるまでです。具体の条項・可否は契約と法務で確定します

### 説明責任の技術的裏付け

「説明責任(accountability)を果たせる」は、精神論ではなく**技術的な裏付け**が要ります。事故時に「何が起きたか・誰の指示か・何を承認したか」を**実際に示せる**状態にしておきます。

- **行為の帰属**: 「誰(どのエージェント・どの権限)として行動したか」を記録・特定できること([エージェントの認証・認可](../06-security/agent-identity-and-auth.md))。帰属が曖昧だと、説明の起点が失われます
- **監査証跡**: 何を入力に・何を判断し・どのツールを・どう実行したかのトレース([可観測性とトレーシング](../05-operations/observability-and-tracing.md)・[コンプライアンスとガバナンス](../06-security/compliance-and-governance.md)の監査)。「あとから説明する」には、記録が残っている必要があります
- **承認記録**: 重要な操作に人の承認を挟み、その承認を記録すること([Human-in-the-Loop 設計](../02-architecture/human-in-the-loop.md))。「誰が承認したか」が説明責任の実体になります
- **設計に組み込む**: これらは事故後に作れません。**帰属・監査証跡・承認記録を事前に設計**しておくことが、「説明できる」の実体です

### 自律度と責任の関係

見落とされがちですが、**自律度を上げるほど、説明の難度が上がります**。これは構造的な関係です。

- **自律度が上がると説明が難しくなる**: エージェントが自分で多くを判断・実行するほど、「なぜその行動をしたか」を人が説明するのが難しくなります。人の関与点が減るほど、承認記録という裏付けも減ります
- **自律度は責任設計とセット**: 自律度を上げる判断は、「そのぶん説明責任をどう担保するか」とセットで行います。高い自律度には、より厚い監査証跡・帰属の記録が要ります([Workflow 型 vs Agent 型の使い分け](../02-architecture/workflow-vs-agent.md)の自律度の判断)
- **説明できない自律は避ける**: 「速いが何をしたか説明できない」エージェントは、事故時に責任を果たせません。自律度と説明可能性のバランスを設計します

### 事故対応での説明の型

事故が起きたとき、**対外的にどう説明するか**の型を用意しておきます([インシデント対応](../05-operations/incident-response.md)の対外説明面)。

- **何を説明できるかを事前に決める**: 事故時に「いつ・何が・なぜ起きたか、誰の指示・承認だったか」を、監査証跡・帰属・承認記録から示せるようにします
- **説明の相手を想定する**: 顧客・規制当局・利用者・社内など、説明の相手で必要な粒度が変わります。相手ごとに何を示すかを想定します
- **インシデント対応と接続**: 事故対応([インシデント対応](../05-operations/incident-response.md))の一部として、対外説明の準備を組み込みます。技術的な復旧だけでなく、「説明できる」ことも対応の要素です

## 実務での注意点

### アンチパターン

- **「AI がやったこと」で責任を曖昧にする** → 責任分界が決まらず事故時に紛糾する → 関係者と原因の層を切り分ける論点を事前に整理する
- **契約の責任・免責・IP・データ条項を確認しない** → 事故時に想定と違う責任を負う → 確認すべき観点を洗い出し法務につなぐ
- **監査証跡・帰属・承認記録を事前に設計しない** → 事故後に「説明できない」 → 帰属・トレース・承認記録を設計に組み込む
- **説明責任を担保せず自律度だけ上げる** → 速いが事故時に何をしたか説明できない → 自律度と説明可能性のバランスを設計する
- **事故時の対外説明を用意しない** → 復旧はできても説明で信頼を失う → インシデント対応に対外説明の準備を組み込む

### チェックリスト

- [ ] 関係者(モデル提供者・システム提供者・導入企業・利用者)と原因の層を切り分ける論点を整理した
- [ ] 契約で確認すべき観点(責任制限・免責・SLA・保険・IP・データ)を洗い出し、法務につないだ
- [ ] 行為の帰属(誰として行動したか)を記録・特定できる設計にした
- [ ] 監査証跡(入力・判断・ツール実行)を残している
- [ ] 重要操作の承認を記録している
- [ ] 自律度を上げる判断を、説明責任の担保とセットで行っている
- [ ] 事故時に「何を・誰に説明するか」の型を用意し、インシデント対応に組み込んだ
