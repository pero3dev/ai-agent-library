# ファインチューニングと蒸留

## この記事の目的

ファインチューニング(fine-tuning、FT)と蒸留(distillation)を「いつ選ぶべきで、いつ選ぶべきでないか」から判断できるようになります。プロンプト・RAG との使い分け、手法の概観(SFT・選好最適化・蒸留)、データ準備、そして FT 資産の運用(ベースモデル更新との共存)までを扱います。2026-07 時点で API 各社の FT 提供メニュー自体が縮小・限定される傾向にあり、「そもそも提供されているか」も判断材料になる、という現実も含めて整理します。

## 対象読者

- 「プロンプトでは品質が頭打ちなので FT したい」という要望の妥当性を判断するエンジニア・テックリード
- 小型モデルへの置き換え(コスト・レイテンシ削減)を蒸留で実現できないか検討しているエンジニア

## 前提知識

- [モデル選定ガイド](model-selection.md) — FT・蒸留は「モデル選定の延長」にある手段
- [Agent 向けプロンプト設計](agent-prompt-design.md) — FT の前に尽くすべき第一手段
- [評価データセットの構築と保守](../04-evaluation/evaluation-datasets.md) — FT の成否を判定するにも学習データを作るにも必要な規律

## 本文

### 概要: FT は「最後の手段」になりやすい

ファインチューニングとは、既存モデルの重みを自分のデータで追加調整することです。強力に聞こえますが、実務では**最後の手段**になりやすい手段です。理由は 4 つあります。

1. **知識の注入には向きません**: 「社内情報を教えたい」は FT ではなく RAG の仕事です。FT で覚えさせた知識は更新できず(再チューニングが必要)、幻覚も解決しません([RAG 実装パターン](rag-implementation-patterns.md))
2. **資産がモデル世代に固定されます**: FT 済みモデルはベースモデルの世代に紐づき、ベースの退役とともに作り直しになります。モデル更新の速い時代には維持費の高い資産です
3. **投資が大きい**: 質の高い学習データの準備・評価・再チューニングの運用は、プロンプト改善より桁違いに重い投資です
4. **提供の前提が変わりやすい**: 2026-07 時点では、API 経由の FT はフロンティア(最上位)モデルを対象にしないのが通例で、提供メニューの縮小・限定も起きています(後述)。FT 前提の設計はベンダー提供の継続というリスクを抱えます

まず押さえるべき使い分けの原則です。

| 直したいもの | 第一選択 |
| --- | --- |
| 指示への従い方・出力形式・判断基準 | プロンプト([Agent 向けプロンプト設計](agent-prompt-design.md)) |
| 知識の不足・情報の鮮度 | RAG([RAG 実装パターン](rag-implementation-patterns.md)) |
| タスク自体の能力不足 | 上位モデル・別モデルへの変更([モデル選定ガイド](model-selection.md)) |
| 形式・文体・ドメイン特化の挙動を小型モデルで安定させる | ここで初めて FT・蒸留が候補 |

### FT を選ぶ前のチェックリスト

FT の検討に入る前に、次をすべて確認します(主要ベンダーも「評価 → プロンプト改善を先に、FT は最後」という順序を公式に案内しています)。

- [ ] 評価データセットがあり、現状の品質を数値で説明できる(これがないと FT の成否も判定できません)
- [ ] プロンプト改善(指示の明確化・few-shot 例・構造化)を評価付きで試し、頭打ちの証拠がある
- [ ] 知識・鮮度の問題ではない(RAG で解決しない)ことを切り分けた
- [ ] 上位モデル・別モデルへの変更で足りないことを確認した(FT の開発・運用コストより、単価の高いモデルの方が安いことは珍しくありません)
- [ ] タスクが明確に定義され、良質なラベル付きデータを用意できる
- [ ] 使いたいモデルに FT の提供経路がそもそも存在する(2026-07 時点、後述)
- [ ] ベースモデル更新時の再チューニング(データ・評価・ジョブのやり直し)を運用として引き受けられる

### 手法の概観

| 手法 | 何をするか | 使いどころ |
| --- | --- | --- |
| 教師ありファインチューニング(SFT) | 入力と模範出力のペアで追加学習する | 形式・文体・ドメイン特化の挙動の安定化。FT の基本形 |
| 選好最適化(DPO など) | 「良い出力 / 悪い出力」のペアで、好ましい方に寄せる | SFT 後の仕上げ。良し悪しは言えるが模範解答を書きにくいタスク |
| 強化学習系(採点器を使う手法) | 出力を採点する仕組み(grader)を使って強化する | 検証可能なタスクの推論品質強化(提供・対象は限定的) |
| パラメータ効率型(LoRA など) | 重み全体でなく小さな追加パラメータだけ学習する | オープンウェイトの自前 FT で一般的(API 経由の FT では学習方式が公開されないことが多い) |

順序の定石は **SFT で基本の挙動を確立してから、必要なら選好最適化で仕上げる**です。いきなり選好最適化から始めても土台がぶれます。

**提供状況(2026-07 時点)** は流動的で、ここは「原則」ではなくスナップショットとして読んでください(一次情報の記録: `research/professional/fine-tuning.md`)。

- API 経由の FT は、提供されている場合でも**対象が旧世代・小型のモデルに限られる**のが通例です(フロンティアモデルは対象外)
- 提供メニューの縮小も現実に起きています。ある大手は FT プラットフォームの新規受付を停止し縮小を告知しており、別の大手は第一者 API で FT を提供していません(クラウド経由の限定的な旧世代対応のみ)。一方で、クラウド ML 基盤側は SFT・選好・蒸留を比較的広く提供しています
- **オープンウェイトモデルは自前 FT の自由度が最大**で、LoRA などの効率的な手法で自社インフラ・マネージド基盤の両方から実施できます([主要 LLM の全体像](llm-landscape.md))。「FT が本当に必要な要件」がある場合、モデル選定がオープンウェイトに寄る一因になります

### 蒸留によるコスト削減

蒸留とは、**上位モデル(教師)の出力を学習データにして、小型モデル(生徒)を SFT する**手法です。Agent の文脈では「ティア混在設計([モデル選定ガイド](model-selection.md))の発展形」と捉えると位置づけが明確になります。

- **使いどころ**: 上位モデルで品質が確認できた**特定のタスク**(分類・抽出・定型応答など)を、小型モデルで安く・速く量産したいとき。タスクが絞られているほど効きます
- **ワークフロー**: (1) 本番で上位モデルの入出力を収集する → (2) 評価で良い出力だけを選別する → (3) それを教師データに小型モデルを SFT する → (4) 同じ評価セットで教師とのギャップを測る。入出力の保存と評価がこのループの前提インフラです
- **注意点**: 教師の誤りもそのまま学びます(収集データの選別が品質を決めます)。また、入力の分布が変わると生徒の品質は静かに劣化するため、本番分布の監視と再蒸留の運用が要ります
- 蒸留も最終段は SFT なので、FT の提供状況・制約(前節)がそのまま適用されます

### データ準備

FT の成否はデータの質で決まります。ここは[評価データセットの構築と保守](../04-evaluation/evaluation-datasets.md)の規律がそのまま流用できます。

- **量より質・分布**: 公式の目安として「最低 10 例から動くが、意味のある改善は 50〜100 例あたりから」という規模感が示される例があります。数を稼ぐより、本番の入力分布を代表し、模範出力の品質が揃っていることが重要です
- **形式**: チャット形式(system / user / assistant のメッセージ列)の JSONL が一般的です。本番と同じプロンプト構造で作ります(学習時と推論時の形式がずれると効きません)
- **学習用と評価用を分離する**: 評価セットのケースを学習データに混ぜると、改善の測定が「カンニング済みの試験」になります(汚染防止)
- **本番ログから採るならマスキングを通す**: 個人情報・機密の除去は評価データセットと同じ前提条件です

### 評価と運用

- **FT 前後を同じ評価スイートで比較する**: FT の投入判断は「学習が収束した」ではなく、評価セットでの改善と、既存能力の劣化(意図しない挙動変化)がないことの両方で行います([回帰テストと CI 組み込み](../04-evaluation/regression-testing.md))
- **ベースモデル更新と共存する**: FT 済みモデルは「ベースが退役するまで使える」のが通例で、ベース更新時は新ベースでの再チューニング + 再評価が必要です。これは[バージョニング・デプロイ・モデル更新追従](../05-operations/versioning-and-model-updates.md)の「モデル更新は期限付きの計画タスク」に、再チューニング工程が加わるということです
- **再現可能性を保つ**: 学習データセット・ジョブ設定・ハイパーパラメータをバージョン管理し、「このモデルは何から作られたか」を追跡可能にします(プロンプトと同じ資産管理の規律です。[プロンプト資産の管理とバージョニング](prompt-management.md))
- **FT してもプロンプトは要る**: FT は挙動の傾向を調整するもので、指示・文脈が不要になるわけではありません。FT 後のプロンプトは軽くできることが多い一方、FT とプロンプトの役割分担を評価で確かめながら調整します

## 実務での注意点

### アンチパターン

- **知識を教えるために FT する** → 知識は更新できず陳腐化し、幻覚も解決しない → 知識・鮮度の問題は RAG で解き、FT は挙動・形式の安定化に限定する
- **プロンプト改善を評価なしに「試した」ことにして FT に進む** → FT の重い投資が、プロンプト 1 行で得られた改善に負ける → 評価付きでプロンプトの頭打ちを実証してから検討する
- **FT 済みモデルを塩漬けにする** → ベースモデルの退役で強制移行になり、再チューニングの準備がないまま品質が崩れる → ベース退役スケジュールの追跡と再チューニング運用をセットで持つ
- **教師データの質を見ずに量を積む** → 誤り・ばらつきをそのまま学習し、期待した安定性が出ない → 選別・品質基準・分布代表性を先に設計する(蒸留では特に教師出力の選別が品質を決める)
- **ベンダーの FT 提供が続く前提でアーキテクチャを組む** → 提供縮小・対象モデル変更で設計ごと見直しになる → 提供状況をスナップショットとして扱い、FT 依存を薄く保つ(必要ならオープンウェイトの自前 FT も含めて選定する)

### チェックリスト

- [ ] 「FT を選ぶ前のチェックリスト」(本文)をすべて通過した
- [ ] 直したいものが挙動・形式の安定化であり、知識の注入ではない
- [ ] 学習データが本番分布を代表し、模範出力の品質基準がある
- [ ] 学習用と評価用のデータが分離されている(汚染防止)
- [ ] FT 前後を同じ評価スイートで比較し、既存能力の劣化も確認した
- [ ] 学習データ・ジョブ設定がバージョン管理され、再現可能である
- [ ] ベース退役時の再チューニング計画(データ・評価・工数)がある
- [ ] 蒸留の場合、教師出力の選別基準と、本番分布変化の監視・再蒸留の運用がある
