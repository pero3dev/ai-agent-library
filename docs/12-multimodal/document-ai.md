# ドキュメント AI(帳票・PDF の構造化)

## この記事の目的

スキャン文書・PDF・帳票から、後続処理(RAG・業務システム)に使える**構造化データ**を取り出すパイプラインを設計できるようになります。入力の類型と前処理・OCR と VLM 直読の使い分け・レイアウトと表の扱い・抽出結果の検証・精度の測り方を、実務判断として持ち帰れる状態を目指します。取り込み後の検索は [RAG 実装パターン](../03-implementation/rag-implementation-patterns.md)、出力形式の強制は [構造化出力](../03-implementation/structured-output.md) が正本です。

## 対象読者

- 帳票・契約書・PDF を扱う業務システムで、文書からの情報抽出パイプラインを設計する実装担当
- RAG の前段として、非構造の文書を検索可能な形に取り込みたいエンジニア

## 前提知識

- [マルチモーダルモデルの仕組み(数式なしの直感)](../10-llm-foundations/multimodal-models.md) — 画像入力のトークン経済・精度の崖(本記事の前提)
- [構造化出力](../03-implementation/structured-output.md) — 抽出結果を型で保証する(本記事の出力側)

## 本文

### 概要: 「読める」から「使える」へ

**ドキュメント AI(document AI)**は、文書を「読む」だけでなく、後続処理が**使える構造化データ**にすることが目的です。単に本文テキストを起こすのではなく、「どの項目が・どの値か」を、検証可能な形で取り出します。本記事は取り出しのパイプラインを扱い、取り込み後の検索は [RAG 実装パターン](../03-implementation/rag-implementation-patterns.md)、形式の強制は [構造化出力](../03-implementation/structured-output.md) に委ねます。

### 入力の類型と前処理

まず、入力文書の**性質で処理が変わる**ことを押さえます。

| 入力 | 性質 | 主な前処理 |
| --- | --- | --- |
| ボーンデジタル PDF | テキスト情報を内部に持つ | テキスト抽出(OCR 不要なことが多い) |
| スキャン PDF・画像 | 見た目だけ(テキスト情報なし) | 傾き補正・ノイズ除去・OCR/VLM |
| 写真 | 歪み・影・低解像度が混じる | 台形補正・明度補正・解像度確認 |

- **ボーンデジタルは OCR を避ける**: PDF が内部にテキストを持つなら、そこから直接抽出するのが最も正確です。画像として読み直すと精度が落ちます。まず「テキストを持っているか」を確認します
- **スキャン・写真は前処理が効く**: 傾き・歪み・低解像度は読み取り精度を大きく下げます。補正・解像度確保([マルチモーダルモデルの仕組み](../10-llm-foundations/multimodal-models.md)の解像度とコスト)が効きます
- **混在を前提に振り分ける**: 実務では 1 つの取り込み口に複数の性質が混ざります。入力の性質を判定して処理を振り分ける設計にします

### OCR か VLM 直読か

文書からテキスト・項目を取り出す方式は、大きく **OCR** と **VLM 直読**の 2 系統です。使い分けが設計の要点です。

- **OCR(光学文字認識)**: 画像中の文字を検出してテキスト化する専用技術。文字起こしの精度・座標(位置情報)が得られ、大量処理のコストが読みやすい。レイアウトの復元は別途必要
- **VLM 直読**: 視覚言語モデル(VLM)に文書画像を直接読ませ、項目を抽出させる。レイアウトを踏まえた「意味的な読み取り」ができる一方、細かい文字・数字で誤りが出やすく([マルチモーダルモデルの仕組み](../10-llm-foundations/multimodal-models.md)の精度の崖)、画像トークンでコストが上がる
- **使い分けの軸**: 精度(細部の正確さ)・コスト(量)・レイアウト保持(構造の復元)で選びます。「文字起こしの正確さが要る大量処理」は OCR、「レイアウトを踏まえた項目抽出」は VLM、と役割で分け、**両者を組み合わせる**(OCR で文字と座標、VLM で意味的な項目対応)ことも多いです

### レイアウトと表の扱い

文書の価値は**レイアウト**に宿ります。段組・表・ヘッダーフッターを潰すと、意味が失われます。

- **レイアウト解析**: 段組・見出し・本文・ヘッダーフッター・図表領域を区別します。単純にテキストを連結すると、段組が混ざり・ヘッダーが本文に紛れます
- **表の抽出**: 表はドキュメント AI で最も難しい要素です。セルの結合・複数行・入れ子で崩れやすく、「もっともらしいが行列がずれた表」が生まれます。表は特に**抽出後の検証**が要ります
- **構造化出力へ接続**: 抽出した項目を、スキーマで型付けした構造化データにします([構造化出力](../03-implementation/structured-output.md))。「請求金額は数値・日付は日付型」と型で縛ると、後続システムが使えます。ただし**形式が正しいこと**と**値が正しいこと**は別で、後者は次項で検証します

### 帳票処理の実務: 抽出 + 検証 + 人手振り分け

帳票のような**正確さが要る**文書では、抽出だけで終わらせず、検証と人手を組み合わせます。

- **抽出 + 検証ルール**: 抽出した項目に、業務ルールでの検証を重ねます(合計 = 明細の和・日付の妥当性・必須項目の充足・コードの存在チェック)。モデルの抽出を業務ロジックで裏取りします
- **信頼度による人手振り分け**: モデル・OCR の確信度や検証結果に応じて、「自動確定 / 人手確認 / 却下」に振り分けます([信頼度と較正(calibration)](../04-evaluation/confidence-and-calibration.md)の選択的予測を、抽出の人手エスカレーションに使う形)。全件を人が見るのでも、全件を自動確定するのでもなく、**危ういものだけ人に回す**設計です
- **長文書の分割とメタデータ**: 長い文書は扱いやすい単位に分割し、文書単位のメタデータ(種別・日付・発行元)を付けます。分割は後続の RAG での検索単位にも影響します([RAG 実装パターン](../03-implementation/rag-implementation-patterns.md)のチャンク設計)

### 精度の測り方

ドキュメント AI は「動いた」では不十分で、**精度を測って**運用します。

- **グラウンドトゥルースを作る**: 正解データ(人が確認した抽出結果)を用意し、抽出結果と突き合わせます([評価データセットの構築と保守](../04-evaluation/evaluation-datasets.md))。作るのは手間ですが、精度を語るには不可欠です
- **項目ごとに測る**: 「文書全体の精度」でなく、項目ごと(金額・日付・宛名)に正確さを測ります。重要項目の精度が要件を満たすかで判断します
- **サンプリング検品**: 全件のグラウンドトゥルースが作れない場合、サンプリングして検品し、精度を推定します。とくに**表・細かい数字**は誤りが出やすいので重点的に見ます
- **失敗の型を知る**: どういう文書・項目で落ちるか(低解像度・特殊レイアウト・手書き)を把握し、前処理・人手振り分けの条件に反映します

## 実務での注意点

### アンチパターン

- **ボーンデジタル PDF を画像化して OCR にかける** → 内部テキストがあるのに精度を落とす → まずテキストを持っているか確認し、あれば直接抽出する
- **VLM 直読の抽出を検証なしで確定する** → 細かい数字・表の誤りをそのまま業務システムに流す → 業務ルールで検証し、危ういものは人手に回す
- **表を単純なテキスト連結で扱う** → 行列がずれて意味が壊れる → レイアウト解析で表構造を保持し、抽出後に検証する
- **全件を人手確認 or 全件を自動確定にする** → コスト過大か品質事故のどちらか → 信頼度・検証結果で人手振り分けする
- **精度を測らずに本番投入する** → どの項目がどれだけ正しいか分からないまま業務に使う → グラウンドトゥルース + 項目別・サンプリングで測る

### チェックリスト
