# ツール権限設計とサンドボックス

## この記事の目的

「モデルは騙されうる」を前提に、**被害の上限を決める側 — ツールの権限とサンドボックス(sandbox) — を設計**できるようになります。最小権限・ユーザー権限の継承・承認ゲートの配置、コード実行やコンピュータ操作型に必須の隔離環境、外部 MCP サーバーを信頼する条件を扱います。

## 対象読者

- Agent に書き込み系ツール・コード実行・外部サービス接続を持たせるエンジニア
- Agent の権限設計をレビューするテックリード・セキュリティ担当者

- [Agent の脅威モデル概観](threat-model-overview.md) — 「被害の上限は権限が決める」という前提
- [ツール接続標準(MCP とエコシステム)](../03-implementation/mcp-and-tool-protocols.md) — 外部ツールサーバーのリスクの概要

## 本文

### 概要: 権限は「渡した分だけ悪用されうる」と考える

プロンプトインジェクション([プロンプトインジェクション](prompt-injection.md))を完全には防げない以上、セキュリティ設計の主戦場は「騙されたときに何ができてしまうか」の側にあります。原則は 1 つです — **Agent の権限は、騙された・誤ったときに許容できる被害の範囲まで**。

出発点として、「この Agent が悪意ある操作者に乗っ取られたら何ができるか」を書き出してみてください。その答えがそのまま、権限設計の現状評価になります。

### 詳細: 権限設計の 4 原則

| 原則 | 内容 | 実装例 |
| --- | --- | --- |
| 最小権限 | タスクに必要なツールだけを渡す。「あると便利」は渡さない | タスク種別ごとにツールセットを分ける。照会タスクに書き込みツールを渡さない |
| スコープの限定 | ツールが触れるデータを、対象ユーザー・対象テナントの範囲に絞る | 検索・取得系ツールの実装側で権限フィルタを掛ける(モデルへの指示で絞らない) |
| 読み書きの分離 | 読み取り系と書き込み系を別ツールにし、書き込み系にだけ強い制御を掛ける | 書き込み系のみ承認ゲート・操作ログ・レート制限の対象にする |
| ユーザー権限の継承 | Agent は「呼び出したユーザーの権限」で動く。ユーザーを超える権限を持たせない | ユーザーの認可トークンをツール実行に引き回す(全権のサービスアカウントで代行しない) |

4 つ目は**混乱した代理人(confused deputy)問題**への対策です。Agent が全権のサービスアカウントで動いていると、権限の低いユーザーが「Agent 経由なら」他人のデータに届いてしまいます。認可の判定は Agent への指示ではなく、ツールの実装とその先のシステムで行います。

### 詳細: 承認ゲートの配置

すべての操作に承認を求めると承認疲れで形骸化します([Human-in-the-Loop 設計](../02-architecture/human-in-the-loop.md))。基準は 2 軸です。

- **取り消せるか** — 取り消せない操作(送信・削除・決済)は承認必須の第一候補
- **影響範囲** — 件数・金額・対象(社外に見えるか)で閾値を切る(例: 10 件超の一括更新、閾値以上の金額)

さらに、**信頼できないコンテンツを処理した直後の危険操作**は、通常なら自動承認できる操作でも承認対象に格上げする設計が有効です(間接インジェクションの典型経路を塞ぐため)。

### 詳細: サンドボックス — コード実行と画面操作は隔離が前提

モデルが生成したコードの実行と、コンピュータ操作型 Agent([コンピュータ操作型・マルチモーダル Agent](../01-concepts/computer-use-and-multimodal-agents.md))の動作環境は、**本番システム・開発者の端末から隔離された使い捨て環境**で行うことを前提にします。

| 制御 | 内容 |
| --- | --- |
| 実行環境の分離 | コンテナ・VM 等の使い捨て環境。ホストのファイルシステム・認証情報に触れない |
| ネットワーク制限 | 外向き通信(エグレス)を許可リストで絞る。任意の宛先への送信はデータ持ち出しの経路になる([データ漏えい対策](data-exfiltration.md)) |
| 資源制限 | CPU・メモリ・実行時間・ディスクの上限(暴走・DoS 対策) |
| 専用アカウント | コンピュータ操作型には専用の最小権限アカウント・専用ブラウザプロファイルを使い、開発者・ユーザー本人のセッションを操作させない |

「サンドボックス内で全部できる」ことは問題ではありません。サンドボックスの意味は、**中で何が起きても外に影響しない**ことにあります。逆に、サンドボックスから本番の認証情報が見える構成は、隔離の意味を失わせます。

### 詳細: 外部 MCP サーバーを信頼する条件

外部の MCP サーバーの接続は、第三者のコードとプロンプト(ツール説明文)の導入です([ツール接続標準(MCP とエコシステム)](../03-implementation/mcp-and-tool-protocols.md))。権限設計の観点から最低限の条件を挙げます。

- **提供元の検証とバージョン固定** — 自動更新は tool poisoning(ツール説明文の汚染)が「あとから」入る経路になります。更新時にツール定義の差分を確認します
- **ツール一覧の絞り込み** — サーバーが提供する全ツールを渡さず、必要なものだけを許可リストで通します
- **認証情報のスコープ最小化** — サーバーに渡すトークンの権限は、そのサーバーの用途に必要な範囲だけにします
- **信頼できないサーバーのツール結果は信頼できないコンテンツとして扱う** — 結果経由の間接インジェクションを前提に、処理後の危険操作へ承認を挟みます

### 設計判断: 権限は静的に決めず、状況で絞る

- **タスク単位の動的な絞り込み** — 同じ Agent でも、タスクの種類に応じて渡すツールセットを切り替えます(照会タスクの間は書き込みツールが存在しない状態にする)。「渡さない」は「使わせない指示」より確実です
- **縮退モードとの接続** — インシデント時にツール単位で無効化・承認必須化できる仕組み([インシデント対応](../05-operations/incident-response.md) のキルスイッチ)は、権限設計の動的な側面として同時に作っておきます

## 実務での注意点

### アンチパターン

- **全権のサービスアカウントで Agent を動かす** → 混乱した代理人問題により、ユーザーが自分の権限を超えるデータ・操作に Agent 経由で届く → ユーザーの認可を引き回し、認可判定をツール実装側で行う
- **「危険な操作はしないでください」とプロンプトで禁止して権限は渡す** → 指示は決定的な制御ではなく、インジェクションで上書きされうる → 使わせないツールはそもそも渡さない。危険な操作は承認ゲートで強制する
- **開発者の端末・本番環境でモデル生成コードを実行する** → 1 回の誤り・悪意で認証情報・ローカルファイルに届く → 使い捨てのサンドボックス(エグレス制限付き)を実行環境にする
- **MCP サーバーを自動更新で運用する** → 更新でツール説明文が汚染されても気づけない(サプライチェーン攻撃の窓口) → バージョン固定 + 更新時のツール定義差分レビューを手順にする

### チェックリスト

- [ ] 「この Agent が乗っ取られたら何ができるか」に答えられる(= 権限の棚卸しがある)
- [ ] タスクに不要なツールが渡っていない(特に書き込み系)
- [ ] データアクセスの認可がツール実装側で強制されている(モデルへの指示に依存していない)
- [ ] Agent がユーザーの権限を超えられない(全権サービスアカウントでない)
- [ ] 取り消せない・影響の大きい操作に承認ゲートがある
- [ ] コード実行・画面操作が隔離環境(エグレス・資源制限付き)で行われる
- [ ] サンドボックスから本番の認証情報・データに触れない
- [ ] 外部 MCP サーバーがバージョン固定され、ツールが許可リストで絞られている
