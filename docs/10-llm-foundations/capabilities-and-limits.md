# LLM の能力と限界の由来

## この記事の目的

「なぜ流暢な文章は得意なのに桁の大きい計算を間違えるのか」「なぜツールを持たせると解決するのか」を構造から理解し、タスクを任せる前に**能力の見積りを外さない**判断ができるようになります。10 章(LLM 基礎)の総仕上げとして、生成・トークン・学習の理解をタスク設計の実務に接続します。

- デモの流暢さと本番の正確さのギャップに驚いた経験がある人

## 本文

### 概要: 流暢さと正確さは別の能力

LLM の能力見積りで最初に外すのは、「こんなに流暢なら、これもできるだろう」という推論です。LLM は学習したパターンの補完・組み合わせとして動くため、**「それらしい出力を作る力」と「厳密に正しい出力を作る力」が乖離します**。流暢さは常に高く、正確さはタスクの構造次第 — このギャップを埋めるのが、ツール・検証・設計の仕事です。

### 得意と不得意の構造

個別の得手不得手を暗記するのではなく、由来で整理します。

| 傾向 | タスクの例 | 由来 |
| --- | --- | --- |
| 得意 | 要約・翻訳・言い換え・分類・トーン調整 | 「テキスト → テキスト」の変換パターンが学習データに大量にある |
| 得意 | コードの生成・説明・修正 | 学習データが豊富で、構文パターンが強く、実行・テストで検証もできる |
| 苦手 | 桁の大きい計算・厳密な集計 | 計算はパターン補完ではなくアルゴリズムの実行が必要。トークン列の操作でそれを模倣している |
| 苦手 | 文字数指定・特定文字の数え上げ | モデルはトークン単位で処理しており、文字を直接「見て」いない([トークナイザとトークン経済](tokenization.md)) |
| 苦手 | 最新情報・ニッチな事実 | 学習データの期限と頻度に依存する([LLM の学習パイプライン](llm-training-pipeline.md))。穴はもっともらしく補完される(幻覚) |
| 苦手 | 長い厳密な手順の暗算的な遂行 | 途中結果を保持・検証する仕組みが生成ループ自体にはなく、誤りが積み重なる |

このパターンから、見積りの経験則が出ます — **「もっともらしさで十分なタスク」は任せやすく、「1 文字・1 桁の厳密さが必要なタスク」は素の LLM に任せない**。

### ツールによる補完: Agent 設計の理論的根拠

苦手の多くは、外部の道具で正確に解ける種類の問題です。

- 計算・集計 → コード実行・電卓ツール
- 最新情報・ニッチな知識 → 検索・RAG([RAG と Agent の関係・使い分け](../01-concepts/rag-vs-agent.md))
- 正確な操作・状態変更 → API 呼び出し(判断はモデル、実行は決定的なコード)

つまり[ツール使用](../01-concepts/tool-use.md)は単なる機能追加ではなく、**LLM の構造的な弱点を、得意な仕事(何をすべきかの判断・道具の選択・結果の解釈)に置き換える再設計**です。「LLM に計算をさせない。計算をさせるコードを呼ばせる」という発想の転換が、Agent 設計の核にあります。ツールの結果はコンテキストに入って以降の生成の根拠になるため、幻覚の抑制にも直接効きます。

### 世代差とスケーリングの読み方

モデルの能力は世代とともに動きます。見積りを陳腐化させないための読み方を 3 つ挙げます。

- **「できなかった」には日付を付ける**: 能力の評価は世代で覆ります。「LLM には無理」ではなく「このモデルでは、この試し方では無理だった(いつ時点)」と記録し、新世代で再評価します([バージョニング・デプロイ・モデル更新追従](../05-operations/versioning-and-model-updates.md))
- **推論に時間を使うモデルの登場**: 応答前に長く考える(推論トークンを使う)ことで複雑な問題の正答率を上げるアプローチが、2026 年時点では広く使われています。難しい問題への正確さと、レイテンシ・コストのトレードオフとして扱います([モデル選定ガイド](../03-implementation/model-selection.md))
- **ベンチマークと実能力の乖離**: 公開ベンチマークのスコアは、ハーネス依存・汚染・飽和により実力を過大にも過小にも見せます。世代差の把握は参考にしつつ、自社タスクでの実測を正とします([エージェントベンチマークの全体像](../04-evaluation/agent-benchmarks-landscape.md))

### 能力見積りの実務

「できそうでできない」を企画段階で見抜くための、簡潔な手順です。

1. **厳密さの要求を見積もる**: 出力のどこに「1 文字・1 桁レベルの正しさ」が必要かを特定します。その部分は素の生成に任せず、ツール・検証・構造化で固めます
2. **検証可能性で選ぶ**: 結果の良し悪しを機械的・人間的に検証できるタスクから任せます(検証できないタスクは、失敗に気付けないタスクです)。[ユースケース発見と要件定義](../09-business/usecase-discovery.md)の「向き不向き」判断と同じ軸です
3. **少数の実ケースで実測する**: 能力の議論を抽象論で続けず、代表的な実データ 10〜20 件で試します。デモ用のきれいな入力ではなく、実務の「面倒なケース」を含めることが重要です([Agent 評価の基礎](../04-evaluation/agent-evaluation-basics.md))
4. **失敗コストで最終判断する**: 同じ正答率でも、失敗の損害が違えば判断は変わります。失敗時の検知・回復手段まで含めて「任せられるか」を決めます([モデル選定ガイド](../03-implementation/model-selection.md)、[Human-in-the-Loop 設計](../02-architecture/human-in-the-loop.md))

### この理解が効く場面

- **企画・要件定義**: 「AI で何ができるか」の議論を、流暢さの印象ではなく構造(厳密さ・検証可能性・ツール補完)で進められます
- **ツール設計**: 「モデルにさせること」と「ツールにさせること」の分担を原理から引けます([ツール定義の設計](../03-implementation/tool-definition-design.md))
- **モデル更新への追従**: 能力評価に日付を付け、世代交代のたびに安価に再判定できます

## 実務での注意点

### アンチパターン

- **流暢さから能力を外挿する**(「こんなに自然に話せるなら計算もできるはず」) → 流暢さと厳密さは別の能力で、乖離が大きい → 厳密さが必要な箇所を特定し、ツール・検証で固める
- **苦手なタスクをプロンプトの工夫だけで克服しようとする** → 構造由来の弱点(計算・文字数・最新情報)は指示では消えない → ツールによる補完へ設計を切り替える
- **一度の失敗で「LLM には無理」と結論を固定する** → 能力は世代で覆り、試し方(ツール・プロンプト)でも変わる → 日付と条件付きで記録し、定期的に再評価する
- **ベンチマーク上位モデルなら自社タスクもできると考える** → タスク分布・ハーネスが違えば序列は入れ替わる → 自社の実ケースで実測する

### チェックリスト

- [ ] タスクのうち「厳密さが必須の部分」を特定し、ツール・検証・構造化で固めた
- [ ] 結果を検証する手段(テスト・人のレビュー・突き合わせ)がある
- [ ] 実務の代表ケース(面倒なケースを含む)で実測してから判断した
- [ ] 「できなかった」記録に日付・モデル・試し方が付いている
- [ ] 失敗時の検知・回復まで含めて任せる範囲を決めた
