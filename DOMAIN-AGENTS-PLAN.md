## 1. 位置づけとギャップ分析

ライブラリの現状(92 本)は「Agent の作り方・運用の仕方」を**ドメイン非依存**に体系化しています。ドメイン別の知見は 07 章の構成事例 3 本(経費精算・サポート・ナレッジ)にあるだけで、これらは「物語で通す」形式であり、ドメインごとの設計判断を網羅する形式ではありません。

| 既存 | 役割 | 本計画との分担 |
| --- | --- | --- |
| 01〜06・10 章 | ドメイン非依存の作り方・運用(正本) | 各記事は共通論を再説明せずリンクで委譲する |
| [07-case-studies](docs/07-case-studies/) | 特定シナリオを物語で通す構成事例 | 07 = 縦の物語、本計画 = ドメインの設計ガイド(横の体系)。相互リンク |
| [09-business](docs/09-business/) | 案件推進(何をやるか) | usecase-discovery の「向き不向き」判断の、ドメイン別の具体化として機能 |

**本計画の各記事に共通する型**(セクションの一貫性の核): ①ドメインの特性分析(検証可能性・失敗コスト・データアクセス — usecase-discovery の軸で)②定番アーキテクチャの型 ③ドメイン固有の落とし穴 ④評価の設計 ⑤既存記事への接続。

## 3. 各ページの設計

### deep-research-agents.md — ディープリサーチ型エージェントの設計

- **目的**: 多段の調査(計画 → 検索 → 読解 → 統合 → レポート)を自律実行するエージェントを、出典の信頼性を保って設計できる
- **主要トピック**: 調査タスクの分解(調査計画の生成・サブクエスチョン化)/ 多段検索の制御(広げる・深める・打ち切るの判断 — ループ設計)/ 出典管理(全主張に出典を紐付ける・引用の検証・出典なし主張の排除)/ 相反する情報源の扱い(確度の表現)/ 社内資料 × Web の統合 / レポート形式の設計(構成・確度表記・未確認事項の明示)/ 評価(網羅性・出典正確性・再現性)
- **分担**: 検索基盤 = rag-implementation-patterns、ループ制御 = DEEP-DIVE の loop-engineering(着手済みなら)、本ライブラリ自身の調査メモ方式(research/)を実例として引用可能

### data-analysis-agents.md — データ分析エージェント

- **目的**: 自然言語からデータ分析(SQL・ノートブック・グラフ)を行うエージェントを、**もっともらしく間違った分析**を防ぐ構造で設計できる
- **主要トピック**: Text-to-SQL の設計(スキーマ・用語の与え方・少数例・分析用ビューの整備)/ セマンティックレイヤー(指標定義の一元化)が精度を決める構造 / 実行 + 検証のループ(構文エラーは自動回復・意味の誤りは検証が必要という非対称)/ ノートブック型(コード実行)の設計とサンドボックス / 「もっともらしい嘘のグラフ」対策(数字の突き合わせ・集計の再現・信頼度の提示)/ 権限(行・列レベルの制御 — 権限反映の分析版)/ 評価(正解のある質問セット・集計一致)
- **分担**: サンドボックス = tool-permissions-and-sandboxing、権限 = agent-identity-and-auth / multi-tenancy

### rpa-and-agents.md — RPA と Agent の使い分け・移行戦略

- **目的**: 既存 RPA 資産を持つ組織が、Agent との使い分け・共存・段階的移行を判断できる
- **主要トピック**: 特性比較(決定的スクリプト vs 適応的判断・画面操作 vs API・保守コストの構造 — workflow-vs-agent の商流版)/ RPA が壊れる場所と Agent が揺れる場所(相補性)/ 共存パターン(RPA の中の判断ステップだけ LLM 化・Agent のツールとして RPA を呼ぶ)/ 移行の優先順位付け(保守負債の大きい画面操作から)/ RPA ベンダーの Agent 統合動向(2026-07 時点の類型 — DA-R1)/ 統制の引き継ぎ(RPA の運用統制・監査を Agent にどう対応させるか)
- **分担**: 画面操作の実装 = computer-use-implementation、業務選定 = usecase-discovery

### personal-assistant-design.md — パーソナルアシスタントの設計

- **目的**: メール・カレンダー・タスクを横断する個人向けアシスタントを、権限とプライバシーの段階設計で安全に作れる
- **主要トピック**: 横断アクセスの設計(ツール群と権限の最小化)/ 自律度の段階(読み取り → 下書き → 承認付き送信 → 限定自動化)/ 記憶とパーソナライズ(好み・文体の学習 — long-term-memory の応用)/ プロアクティブ性の節度(通知・先回りの設計)/ 間接プロンプトインジェクション(受信メール経由)への防御 — [ケーススタディ: メールアシスタントの情報漏えいインシデント](docs/07-case-studies/case-study-email-assistant-incident.md) を教材に / 個人データの扱い
- **分担**: HITL = human-in-the-loop、漏えい対策 = data-exfiltration(正本)

### search-experience-redesign.md — 検索体験の再設計

- **目的**: 「検索して読む」体験を「聞いて答えを得る」体験へ再設計する際の構成と、従来検索との共存を判断できる
- **主要トピック**: 構成の類型(検索結果 + 要約 / 会話型検索 / エージェント型検索(多段))/ 社内検索の作り直し(既存検索基盤を活かすか置き換えるか)/ 回答型 UX の設計(出典必須・確度表示・「見つからない」の言い方)/ 検索ログという資産(クエリ分析 → 知識源の改善)/ ゼロ結果・低品質知識源の問題(検索を直す前に知識源を直す — knowledge-agent 事例と接続)/ 評価(検索指標と回答品質の 2 層)
- **分担**: RAG の実装 = rag-implementation-patterns(あちらは Agent の知識源、本記事は人間向け検索体験)

### writing-and-translation-workflows.md — 執筆・翻訳ワークフローの設計

- **目的**: 長文の執筆・翻訳を、品質を管理しながら量産できるワークフローとして設計できる
- **主要トピック**: 長文生成の分割統治(アウトライン駆動・セクション生成・一貫性の維持)/ スタイル・用語の統制(スタイルガイド・用語集の適用と検証)/ 翻訳ワークフロー(文脈の保持・用語統一・機械翻訳後編集(MTPE)の工程設計)/ レビュー工程の設計(人が見る箇所の絞り込み)/ 評価(judge + 人手・スタイル準拠の自動チェック)/ 本ライブラリ自身の執筆パイプライン(テンプレート・レビュー・lint)を実例として参照
- **分担**: プロンプト技法 = prompt-engineering 系、資産管理 = prompt-management

### education-agents.md — 教育・学習支援エージェントの設計

- **目的**: 「答えを渡す」のではなく「学習を進める」エージェントを、教育設計と安全の両面から作れる
- **主要トピック**: 教育設計の原則(段階的ヒント・ソクラテス式・生産的なもがきを守る)/ 学習者モデル(理解度の推定と適応 — 記憶の応用)/ 誤答への向き合い(訂正の仕方・自信の扱い)/ 安全(未成年への配慮・誤情報・依存)/ 評価(正答率ではなく学習効果をどう測るかの難しさ)/ 教育現場の規制・ガイドライン([industry-regulations-map](docs/09-business/industry-regulations-map.md) の教育の項へ接続)
- **分担**: ガードレール = guardrails、会話設計の一般論は将来の UX 計画候補

### emerging-agent-domains.md — 先端応用の概観(鮮度管理型)

- **目的**: 実務適用がまだ限定的な 3 領域 — 科学研究支援・シミュレーション/NPC・エージェント経済 — の現在地を、過度な期待も無視もせずに掴む
- **主要トピック**: 科学研究(文献調査・仮説生成・実験計画支援・ラボ自動化の現在地)/ シミュレーション・NPC(ゲーム・社会シミュレーション・合成ユーザーによるテスト)/ エージェント経済(エージェント間の取引・交渉・決済プロトコルの構想と現実)/ 各領域の「読み方」(何が実証済みで何が構想か)
- **鮮度**: 3 点セット(最終確認日 + アクセス日 + 定点観測)必須。調査 DA-R2 を執筆前に実施

### spreadsheet-agents.md — 表計算・構造化データ操作エージェント(第 2 期)

- **目的**: 「業務の共通言語」である表計算(セル・数式・複数シート)を読み書きするエージェントを、壊さない・黙って間違えない構造で設計できる
- **主要トピック**: 表の理解(レイアウトの多様性・結合セル・暗黙の意味 — ドキュメント AI の表とは別の「生きた計算モデル」問題)/ 操作の設計(数式を書くか値を書くかの原則・変更のプレビューと差分承認)/ 検証(再計算の突き合わせ・既存数式の破壊検知)/ 大規模シートの分割戦略 / データ分析エージェント(#2)との分担(BI・SQL の世界 vs 現場の Excel の世界)
- **分担**: 帳票からの抽出 = MULTIMODAL 計画の document-ai(採用時)

### time-series-and-forecasting.md — 予測・時系列タスクと LLM(第 2 期)

- **目的**: 需要予測・異常検知などの時系列タスクに対する LLM の適用範囲と限界を判断し、従来手法との分担を設計できる
- **主要トピック**: LLM が効く場所(要因の言語化・異常の説明・レポート生成・特徴量のアイデア出し)と効かない場所(数値予測そのものは専用手法が既定)/ ハイブリッド構成(予測モデル + LLM の解釈層)/ 「もっともらしい後付け説明」の危険(相関と因果の混同を増幅する)/ 評価(予測精度と説明品質の分離)
- **分担**: データ分析全般 = data-analysis-agents(#2)。本記事は時系列固有の判断

### hr-and-recruitment-ai.md — 人事・採用領域の AI(第 2 期)

- **目的**: 高リスク用途の代表例である人事・採用で、公平性・透明性・規制要求を満たす設計を判断できる
- **主要トピック**: 用途の類型と自律度の上限(スクリーニング支援・面接支援・評価支援 — 「最終判断は人」が規制・倫理の両面で強制される構造)/ 公平性の評価(EVAL 計画の fairness の適用先 — 属性による差の測定を運用に組み込む)/ 規制の確認先(EU AI Act の高リスク分類・国内の職業安定法系 — 入口マップ方式。`research/professional/compliance.md` を再利用)/ 候補者・従業員への透明性(AI 利用の開示・異議申し立て経路)/ 導入の合意形成(労使・法務)
- **方式**: 免責を冒頭に明記。規制の内容解説はせず確認先の提示に徹する

### legal-review-agents.md — 契約レビュー・法務ドメイン Agent(第 2 期)

- **目的**: 契約書レビュー・法務調査という「検証可能性が高く失敗コストも高い」ドメインで、弁護士・法務部門の判断を支援する(代替しない)エージェントを設計できる
- **主要トピック**: 向く業務の切り出し(定型条項のチェックリスト照合・変更点の抽出・過去契約との突き合わせ・ドラフトの下書き)/ 「見落とし」の非対称リスク(検出漏れが致命的 — 網羅性の担保と人の最終レビュー)/ 根拠の提示(該当条文・過去事例への参照必須 — 出典設計)/ 秘匿特権・機密の扱い(データ経路の統制)/ 弁護士法等の業際問題への配慮(確認先の提示まで)/ 評価(判例・契約の評価セットの作り方)
- **方式**: 免責を冒頭に明記。TRUST 計画の ai-copyright-and-ip-map・SE 計画の法務協働と相互リンク
