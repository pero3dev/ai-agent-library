# ツール接続標準(MCP とエコシステム)

## この記事の目的

MCP(Model Context Protocol)がどんな問題を解決する標準なのかを理解し、「自前でツールを実装するか、既存の MCP サーバーに接続するか」を判断できるようになります。外部ツールサーバーを使うことのセキュリティ上の含意も把握します。

## 対象読者

- Agent に外部サービス(コード管理・チケット管理・DB 等)を接続したいエンジニア
- 自前ツールと MCP サーバーの使い分け基準を持ちたいテックリード

## 前提知識

- [ツール使用](../01-concepts/tool-use.md) — ツールの仕組み(定義・要求・実行・結果)
- [ツール定義の設計](tool-definition-design.md) — 自前で定義する場合の設計

## 本文

### 概要: N × M 問題を解く標準

MCP(Model Context Protocol)は、ツールやデータソースを LLM アプリケーションに接続するためのオープンな標準プロトコルです。解決する問題はシンプルで、**N 個のアプリケーション × M 個のサービスの接続を、それぞれ個別に実装しなくて済むようにする**ことです。サービス側が一度 MCP サーバーを提供すれば、MCP 対応のあらゆるアプリケーション(ホスト)から利用できます。

概念上は [ツール使用](../01-concepts/tool-use.md) で見た仕組みと同じです — ツールの定義一覧がモデルに渡り、モデルが呼び出し要求を出し、実行結果が返ります。違いは「ツールの実装がどこにあるか」だけです。

### 詳細: 構成要素

| 要素 | 役割 |
| --- | --- |
| MCP ホスト(クライアント) | LLM アプリケーション側。サーバーに接続し、ツール一覧を取得してモデルに渡す |
| MCP サーバー | ツール・リソースを提供する側。サービス公式のものと、コミュニティ製がある |
| ツール / リソース / プロンプト | サーバーが公開するプリミティブ。ツール(実行できる操作)が中心 |

### 設計判断: 自前ツールか、MCP サーバーか

| 状況 | 推奨 |
| --- | --- |
| 自社固有のビジネスロジック・社内 API | 自前ツール([ツール定義の設計](tool-definition-design.md)) |
| 汎用サービスへの接続(コード管理・チケット・チャット・DB 等) | 公式の MCP サーバーがあればそれを使う |
| 汎用サービスだが、操作を強く絞りたい | 自前ツールでラップする(MCP サーバーの全ツールを渡さない)という選択も有効 |
| 社内ツールを複数の Agent アプリから使いたい | 社内 MCP サーバーとして実装する価値が出てくる |

判断の軸は「**そのツールの語彙を自分で設計する必要があるか**」です。MCP サーバーのツール定義(名前・説明文・粒度)はサーバー提供者の設計であり、自分のタスクに最適とは限りません。ツール数が多すぎる・説明が曖昧といった問題があれば、必要なものだけ自前でラップする方が品質が安定します。

### 詳細: セキュリティ — 外部サーバーはサプライチェーン

外部の MCP サーバーを接続することは、**第三者のコードとプロンプト(ツール説明文)を自分の Agent に組み込む**ことです。次のリスクを認識してください。

1. **ツール説明文の汚染(tool poisoning)** — ツールの説明文はモデルへのプロンプトの一部です。悪意ある(または改ざんされた)サーバーは、説明文経由でモデルの挙動を操作しえます
2. **過剰な権限** — サーバーに渡す認証情報のスコープが広いと、Agent の誤動作・乗っ取り時の被害がそのまま広がります
3. **結果経由の間接プロンプトインジェクション** — ツールが返す内容(外部サイトの本文など)に埋め込まれた指示をモデルが実行してしまう構造的リスク([プロンプトインジェクション](../06-security/prompt-injection.md))

基本対策は、信頼できる提供元のサーバーに限定する・バージョンを固定する・認証情報のスコープを最小にする・危険な操作に承認を挟む([Human-in-the-Loop 設計](../02-architecture/human-in-the-loop.md))ことです。詳細は [ツール権限設計とサンドボックス](../06-security/tool-permissions-and-sandboxing.md) で扱います。

## 実務での注意点

### アンチパターン

- **見つけた MCP サーバーを無検証で接続する** → 第三者のコードとプロンプトを本番 Agent に注入することと同義 → 提供元の信頼性・権限スコープ・ツール一覧を確認してから接続する
- **サーバーの全ツールをそのまま渡す** → 数十ツールがコンテキストを圧迫し、選択ミスが増える → 必要なツールだけに絞る(絞る機構がなければ自前ラップを検討)
- **MCP か自前かを二者択一で考える** → 実務は混成が普通 → 汎用接続は MCP、コアロジックは自前、と役割で分ける

### チェックリスト

- [ ] 接続する MCP サーバーの提供元・メンテナンス状況を確認した
- [ ] サーバーに渡す認証情報のスコープが必要最小限になっている
- [ ] モデルに渡るツール一覧を実際に確認した(数・説明文の品質)
- [ ] 副作用のあるツールに承認・権限制御がかかっている
- [ ] サーバーのバージョン固定・更新手順が決まっている
