# CLAUDE.md — このリポジトリでの執筆ルール

このリポジトリは、エンジニア向けの AI Agent 学習ドキュメントライブラリ(日本語)です。
Claude がこのリポジトリで作業するときは、以下のルールに **必ず** 従ってください。

## プロジェクトの性質

- 成果物は Markdown ドキュメント。コードは `examples/` 配下のサンプルのみ
- 読者は日本語話者のソフトウェアエンジニア。LLM API の基本的な利用経験は前提としてよい
- 「用語の説明」ではなく「実務での設計判断に使える知識」を目指す

## 最重要ルール

1. **本文は日本語、ファイル名・ディレクトリ名は英語ケバブケース**(例: `tool-use.md`, `prompt-injection.md`)
2. **新規ドキュメントは必ず [templates/doc-template.md](templates/doc-template.md) をベースに作成する**。固定 H2 セクションの削除・改名は不可。固定 H2 のあとに独自の H2 を追加するのは可
3. **1 回の作業セッションで新規作成するドキュメントは原則 3 ファイルまで**。量より一貫性と品質を優先する
4. **不確かな情報・変化の速い情報(モデル名、料金、フレームワークの最新 API、ベンチマーク数値など)は本文に断定で書かず、`TODO(要確認)` を残す**(書式は後述)
5. **同期更新の義務**。次のイベントが起きたら、同じセッション内で対応する更新を行う:
   - ドキュメントを新規作成・改名した → 所属セクション `README.md` の表を更新(執筆済みになったらファイル名をリンク化)、[ROADMAP.md](ROADMAP.md) のタスクステータス、新しい用語を導入していれば [GLOSSARY.md](GLOSSARY.md)
   - ドキュメントの内容(事実・推奨・コード)を実質的に変更した → front matter の `last_updated` を更新(誤字・リンク修正のみでは更新しない)
   - ドキュメントのステータスが変わった → front matter の `status` と ROADMAP のタスクステータスを対応表(後述)に従って両方更新

## ステータス管理(正本の一元化)

ファイル単位のステータスの **正本は front matter の `status` のみ** です(値は `draft` | `published` の 2 つ)。

- `draft`: 執筆済みだが、フェーズレビュー(ROADMAP の X-R タスク)を通過していない
- `published`: フェーズレビューを通過した

ROADMAP.md は **タスク単位** の進捗だけを管理します。対応関係:

| ROADMAP のステータス | 意味 | front matter |
| --- | --- | --- |
| 未着手 | 成果物ファイルが存在しない | — |
| 執筆中 | セッション中断などの書きかけ(原則発生させない) | — |
| レビュー待ち | タスクの成果物がすべて `draft` で存在する | `draft` |
| 完了 | フェーズレビューを通過した | `published` |

各セクション README の収録予定表には **ステータス列を置きません**。執筆済みドキュメントはファイル名をリンク化することで示します(リンクになっている = ファイルが存在する。バッククォートのままの名前 = 計画のみ)。

## 命名規約

| 対象 | 規約 | 例 |
| --- | --- | --- |
| セクションディレクトリ | `NN-section-name`(2 桁連番 + 英語) | `01-concepts/` |
| ドキュメントファイル | 英語ケバブケースの名詞句。連番は付けない。**docs/ 全体(セクション横断)で一意にする** | `agent-loop.md` |
| サンプルコードディレクトリ | 対応する docs のファイル名と揃える | `examples/python/tool-use/` |
| 図版のエクスポート画像 | `<セクション番号>-<トピック>-<内容>.<拡張子>` | `01-agent-loop-overview.svg` |
| H1 見出し | ファイルに 1 つだけ。日本語のトピック名(front matter の `title` と一致させる) | `# Agent ループ` |
| H2 見出し | テンプレート指定の固定セクション名を変更しない | `## 実務での注意点` |
| front matter の `tags` | 英語ケバブケース。原則としてファイル名や GLOSSARY 見出しに使われている語を再利用する。新規タグを導入する前に既存タグを grep し、同義語の揺れ(例: `tools` と `tool-use`)を作らない | `["tool-use", "function-calling"]` |

- ファイル名の英語は、業界で一般的な用語をそのまま使う(`rag` を `retrieval-augmented-generation` と過剰展開しない。慣用に従う: `rag-vs-agent.md` は可)
- 略語も小文字にする(正: `llm-as-a-judge.md` / 誤: `LLM-as-a-Judge.md`)

## ドキュメント構造(テンプレート準拠)

すべてのドキュメントは次の構造を持つこと。詳細は [templates/doc-template.md](templates/doc-template.md) を参照。

1. YAML front matter — **機械可読メタデータのみ**: `title` / `category` / `level` / `status` / `last_updated` / `tags`(対象読者・前提知識・関連トピックは本文セクションが正であり、front matter に重複させない)
2. `# タイトル`(H1、日本語)
3. `## この記事の目的`
4. `## 対象読者`
5. `## 前提知識`(このライブラリ内の前提ドキュメントへ相対リンク)
6. `## 本文`(内部の H3 構成は自由。推奨: 概要 → 詳細 → 設計判断 → 例)
7. `## 実務での注意点`(`### アンチパターン` と `### チェックリスト` を含む。記事の性質上どうしても該当しない場合は「該当なし」と理由を 1 行書く。空欄・削除は不可)
8. `## 関連トピック`(相対リンクの箇条書き。リンク切れ禁止)
9. `## 参考資料`(外部リンク。アクセス日を併記)
10. `## TODO・未確認事項`(なければ「なし」と明記)

## 文体・表記ルール

- 文体: である調は使わず「〜です・〜ます」調。箇条書き内は体言止め可
- 英語の技術用語は初出時に「日本語訳(英語)」の形で書く。例: 「ツール使用(tool use)」。以降は定着している方を使う
- コードブロックには必ず言語指定を付ける(`python`、`typescript`、`mermaid` など)
- 図は Mermaid を第一選択とし、**本文への埋め込みが正本**。`assets/diagrams/` にはエクスポート画像が必要になった図の生成物のみを置く(ソースの二重管理はしない)
- 「最近」「現在」などの相対的な時間表現を避け、「2026 年時点では」のように絶対表現を使う

## TODO(要確認)の書式

変化が速い・裏取りできていない情報は、次の書式で本文中に残します:

```markdown
> **TODO(要確認):** ○○ の最新仕様を公式ドキュメントで確認する(最終確認: 2026-07)
```

- タグ文字列は必ず `TODO(要確認)` に統一する(全文検索で一覧化するため)
- 「何を」「どこで」確認するかを書く。「あとで調べる」だけの TODO は禁止

## サンプルコードのルール(examples/)

- Python は 3.11 以降、TypeScript は 5.x 以降を前提とする
- 各サンプルは `examples/<言語>/<トピック名>/` に自己完結で置き、`README.md`(実行方法)を必ず含める
- **依存バージョンを固定するファイル(`requirements.txt` / `package.json` + ロックファイル等)を必ず含める**
- サンプルの `README.md` に **動作確認日** を記載し、実行確認したセッションで更新する
- docs 側からサンプルへ相対リンクし、サンプル側からも対応する docs へリンクする(双方向)
- API キーなどの秘密情報はコードに書かず、環境変数参照とする
- 特定ベンダーの API を使う場合、ドキュメント本文はベンダー中立に書き、コード例で具体化する

## 作業の進め方

- 作業を依頼されたら、まず [ROADMAP.md](ROADMAP.md) で現在のフェーズと担当タスクを確認する
- 大量のドキュメントを一括生成しない。ROADMAP のタスク分割単位(1 セッション 1〜3 ファイル)を守る
- 既存ドキュメントと矛盾する内容を書く前に、関連ドキュメントを読み、矛盾があれば指摘してユーザーに確認する
- レビュー観点(フェーズレビュー X-R でも同じものを使う。1〜3 は下記「検証ツールと自動化」のスクリプトで機械化済み。4 と内容の質は人手 / doc-reviewer サブエージェントで見る):
  1. テンプレート準拠(固定 H2 がすべて存在するか)
  2. リンク切れなし(docs ↔ examples の双方向リンクを含む)
  3. `TODO(要確認)` の書式統一
  4. 同期漏れなし(セクション README のリンク化 / ROADMAP ステータス / GLOSSARY / `status` / `last_updated`)

## 検証ツールと自動化

- レビュー観点 1〜3 は `scripts/` で機械チェックできる(CI の docs ジョブでも同じものが走る):
  - `node scripts/validate-docs.mjs <file|--all>` — front matter / 固定 H2 / `TODO(要確認)` 書式
  - `node scripts/check-links.mjs` — 相対リンク切れ(大文字小文字の不一致を含む)+ セクション README 収録表の漏れ
  - `node scripts/todo-report.mjs` — `TODO(要確認)` の棚卸し(「最終確認」月ごとの集計。`--json` あり)
- docs/ の `.md` を編集するとフックが validate-docs を自動実行し、問題があればその場で指摘される
- `website/content/` `website/generated/` `website/out/` は sync-content.mjs / next build の生成物のため、フックが書き込みをブロックする(正本は docs/。手書き上書きページのみ `website/content-src/`)
- 定型作業はスキルを使う: `/quarterly-maintenance`(四半期定点観測)/ `/new-doc`(新規作成 + 同期更新)/ `/publish-review`(draft → published のフェーズレビュー)/ `/examples-check`(サンプルの実行確認)
