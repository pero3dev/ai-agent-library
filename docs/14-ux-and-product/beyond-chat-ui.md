# チャットを超える UI(プロンプトレス設計)

## この記事の目的

「テキストボックスに書かせる」以外の選択肢 — 構造化入力・成果物 UI・インライン支援・バックグラウンド実行 — を使い分けられるようになります。チャットが向く場面と向かない場面を見極め、ユーザーに書かせずに AI の価値を届ける設計判断を持ち帰れる状態を目指します。

## 対象読者

- AI 機能の UI を「チャットありき」で考えず、最適な形を選びたいデザイナー・PM・エンジニア
- 「テキストボックスを置いたが使われない」を、UI の形から見直したい実装者

## 前提知識

- [AI プロダクトの UX パターン](ai-ux-patterns.md) — 不確実性・待ち時間・信頼のパターン
- [会話設計](conversation-design.md) — チャットを選ぶ場合の会話体験

## 本文

### 概要: チャットは選択肢の 1 つにすぎない

AI プロダクトというと、まずチャットの UI を思い浮かべます。しかしチャット、すなわち「テキストボックスに自由に書かせる」形は、万能ではありません。**白紙のテキストボックスは、多くの利用者にとって難しい**——何をどう書けばよいか分からず、価値に辿り着く前に離脱します。

チャットは強力ですが、それは数ある選択肢の 1 つです。本記事は「テキストボックス以外の形」を並べ、いつどれを使うかを整理します。狙いは、**ユーザーにプロンプトを書かせずに AI の価値を届ける**ことです。なお、機械間のジョブ型インターフェースは [エージェントの API 設計](../02-architecture/agent-api-design.md) が扱い、本記事は対人の UI 形態を扱います。

### チャットが向く場面・向かない場面

まず、チャットが適切かを見極めます。

- **チャットが向く**: 依頼が多様で事前に選択肢を用意できない、対話で絞り込む価値がある、探索的なやり取りが中心、といった場面です
- **チャットが向かない**: やることが定型的で選択肢を出せる、入力に決まった構造がある、既存の作業フローに埋め込みたい、初心者が多い、といった場面です。ここでチャットを使うと、かえって使いにくくなります
- **「とりあえずチャット」を疑う**: 流行だからチャットにする、ではなく、タスクの性質から UI を選びます

### プロンプトレス設計(書かせない)

ユーザーに文章を書かせず、**選ぶ・押す・文脈から起動する**形で AI を動かす設計です。

- **ボタン・アクションで起動**: 「要約する」「言い換える」「返信案を作る」をボタンにします。ユーザーはプロンプトを考えず、意図をワンクリックで伝えられます
- **テンプレートと選択肢**: よくある依頼をテンプレート化し、穴埋めや選択で使えるようにします。白紙より、埋めるべき枠があるほうが易しいです
- **文脈からの自動起動**: ユーザーの操作・状況から、AI が支援を先回りで提示します(過剰にならない範囲で — [プロアクティブエージェントの UX](proactive-agent-ux.md))。「今この画面でよくある操作」を提案する形です
- **構造化入力とチャットの併用**: 骨格は構造化入力(フォーム・選択)で受け、自由記述が要る部分だけテキストにする、という組み合わせも有効です

### 成果物中心 UI(会話は脇役)

ドキュメント・画像・コードといった**成果物**が主役で、AI との対話は脇役、という形です。

- **成果物を中心に置く**: 編集中のドキュメントやキャンバスを画面の主役にし、AI はその横で編集・提案を手伝います。会話ログを追うのではなく、**成果物そのもの**を見ながら作業します
- **AI の変更を成果物上で見せる**: AI の提案を、成果物への差分・ハイライトとして示し、採否を選べるようにします。チャットで説明されるより、現物で見えるほうが速いです
- **会話は補助線**: 「なぜそう直したか」を聞ける対話は残しつつ、主導権は成果物に置きます

### インライン支援(埋め込み型)

独立した AI 画面を持たず、既存のエディタ・フォーム・ツールに**埋め込む**形です。

- **作業の場に溶け込ませる**: 文章を書いているエディタ、入力しているフォームの、その場で支援します。別画面へ移動させないことで、作業を中断させません
- **邪魔しない既定**: インライン支援は、求められたときに出て、普段は引っ込んでいる設計にします。常に割り込むと作業の邪魔になります
- **受け入れ/却下を軽く**: 提案の採否を、キー操作一つで選べるようにします(コード補完のような体験)

### バックグラウンドエージェントの UI(投げて後で受け取る)

時間のかかる処理を、その場で待たせず**投げて後で受け取る**形です。

- **待たせない**: 数十秒〜数分かかる処理は、同期的に待たせず「実行中」にして、終わったら通知します([非同期・長時間タスクの設計](../02-architecture/async-and-durable-agents.md)の UX 面)
- **進捗と結果の置き場**: 実行中のタスク一覧・完了通知・結果の受け取り場所を用意します。「投げたものがどこに行ったか」が分かる導線が要ります
- **割り込みと確認**: バックグラウンド処理でも、途中で承認が要る場面([Human-in-the-Loop 設計](../02-architecture/human-in-the-loop.md))は通知して確認を挟みます

### 複合(ハイブリッド)

実際のプロダクトは、単一の形ではなく**組み合わせ**になることが多いです。

- **タスクごとに最適な形を選ぶ**: 探索はチャット、定型はボタン、制作は成果物中心、長時間処理はバックグラウンド、と使い分けます
- **一貫した枠組みで束ねる**: 複数の形を使っても、AI の関与の見せ方・信頼の枠付け([AI プロダクトの UX パターン](ai-ux-patterns.md))は一貫させます

## 実務での注意点

### アンチパターン

- **流行だからチャットにする** → 定型タスク・初心者に使いにくい → タスクの性質から UI を選ぶ
- **白紙のテキストボックスで放り出す** → 何を書けばよいか分からず離脱 → ボタン・テンプレート・選択肢でプロンプトレスに導く
- **会話ログを主役にして成果物を埋もれさせる** → 現物が見えず作業しにくい → 成果物を中心に置き、AI の変更を差分で見せる
- **インライン支援が常に割り込む** → 作業の邪魔になる → 求められたときに出て普段は引っ込む
- **長時間処理をその場で待たせる** → 体感が悪く離脱する → バックグラウンド化し、進捗と結果の置き場を用意する
- **複数の UI 形態で信頼の見せ方がバラバラ** → 混乱する → 関与の見せ方・信頼の枠付けを一貫させる

### チェックリスト

- [ ] タスクの性質から、チャットが適切かを見極めたか
- [ ] 定型・構造のある入力を、ボタン・テンプレート・選択肢でプロンプトレスにしたか
- [ ] 制作系は成果物を中心に置き、AI の変更を差分・ハイライトで見せているか
- [ ] インライン支援を、邪魔しない既定(求められたら出る)にしたか
- [ ] 長時間処理をバックグラウンド化し、進捗・完了通知・結果の置き場を用意したか
- [ ] 複数の UI 形態を使う場合、信頼の見せ方を一貫させているか
