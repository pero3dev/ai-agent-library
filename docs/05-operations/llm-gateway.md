# LLM ゲートウェイの設計

## この記事の目的

複数モデル・複数プロバイダー・複数チームの LLM 利用を一元化する**ゲートウェイ層**を設計できるようになります。ゲートウェイが解く問題(キー管理・モデル抽象化・コスト集計・監査・レート制御)を理解し、構成の類型(自作 / OSS / マネージド)を選び、ルーティング・フォールバック・マルチテナント制御の実装点を押さえ、落とし穴(単一障害点化・モデル固有機能の喪失)を避けられる状態を目指します。フォールバック・分離の**原則**は各正本に譲り、本記事は「それらを 1 つの部品に集約する」設計論です。

## 対象読者

- 社内で複数チーム・複数プロバイダーの LLM 利用が広がり、統制・コスト把握に困っているプラットフォーム/運用担当
- モデルの切り替え・フォールバック・使用量集計をアプリごとに実装していて、共通化したいテックリード

## 本文

### 概要: ゲートウェイは「共通の交差点」

**LLM ゲートウェイ(LLM gateway)**は、アプリと LLM プロバイダーの間に置く共通の中継層です。各アプリが個別にプロバイダーを直接叩く代わりに、ゲートウェイを経由させることで、キー管理・モデル抽象化・コスト集計・統制を**1 か所に集約**します。本記事はこの集約の設計を扱い、フォールバック・分離・リトライの原則は [デプロイとスケーリング](deployment-and-scaling.md) / [マルチテナント設計とテナント分離](../02-architecture/multi-tenancy-and-isolation.md) / [エラーハンドリングとリトライ設計](../02-architecture/error-handling-and-retries.md) が正本です。

### ゲートウェイが解く問題

ゲートウェイの価値は、**各アプリで重複していた面倒を 1 か所に集める**ことです。

- **キー管理の一元化**: プロバイダーの API キーをゲートウェイが保持し、各アプリには**仮想キー**を配る。キーの漏えい範囲を絞り、失効・ローテーションを一元化できます
- **モデル抽象化**: 各アプリは統一 API(多くは OpenAI 互換形式)で呼び、裏側でどのプロバイダー/モデルを使うかをゲートウェイが決める。アプリ改修なしにモデルを差し替えられます
- **使用量・コストの集計**: チーム・アプリ・モデル別のトークン消費とコストを一元集計([コスト管理](cost-management.md))。「誰がいくら使ったか」が見えます
- **監査ログ**: 誰が何を呼んだかを記録。統制・トレーサビリティの土台になります
- **レート制御・予算**: チーム・キー単位のレート制限と予算上限を課し、暴走・超過を防ぎます

### 構成の類型

ゲートウェイの実現方法は 3 類型で、規模と統制要件で選びます(具体の製品/OSS は変化が速いため 2026-07 時点の代表例)。

| 類型 | 内容 | 向く場面 | 代表例(2026-07 時点) |
| --- | --- | --- | --- |
| 薄い自作プロキシ | 最小限の中継を自作 | ごく小規模・要件が単純 | 自前実装 |
| OSS ゲートウェイ | OSS を自社ホスト | 統制・データ管理を自社に置きたい | LiteLLM・Portkey Gateway・Kong AI Gateway 等(OSS) |
| マネージド | SaaS のゲートウェイ | 運用を任せたい | Cloudflare AI Gateway・各 OSS のマネージド版等 |

- **薄い自作**: 小さく始められますが、フォールバック・集計・監査を作り込むと結局重くなります。「本当に薄いまま済むか」を見極めます
- **OSS ゲートウェイ**: 統一 API・仮想キー・使用量集計・フォールバックなどを備えたものが複数あります。自社ホストで統制・データ経路を握れます。ライセンスは OSS コアと有償エンタープライズ機能の**境界**を確認します(コアは寛容ライセンスでも一部機能が有償のことがある)
- **マネージド**: 運用負担を肩代わりしますが、データがベンダーを経由します。閉域網・データ主権の要件とは相性が悪いことがあります

### ルーティング設計

ゲートウェイの中核機能が、**用途に応じてモデルを振り分ける**ルーティングです。

- **用途別の振り分け**: 簡単なタスクは小型・安価なモデル、難しいタスクは上位モデル、という振り分けを一元的に実装します([モデル選定ガイド](../03-implementation/model-selection.md)のティア混在の実装点。小型活用は [小型言語モデル(SLM)の活用戦略](../03-implementation/slm-strategy.md))
- **抽象化の粒度**: どこまで抽象化するかを決めます。抽象化しすぎると後述の「モデル固有機能の喪失」が起き、しすぎないとアプリごとの分岐が増えます
- **自ホストモデルも束ねる**: セルフホストした OpenAI 互換エンドポイント([セルフホスト推論の実務](self-hosted-inference.md))も、API プロバイダーと同じルーティングに載せられます。「借りる」と「自ホスト」を 1 つの入口に統一できます

### フォールバック・リトライ・サーキットブレーカの実装点

信頼性機能は**原則が別記事**にあり、ゲートウェイはその**実装を集約する場所**です。

- **フォールバック**: あるプロバイダー/モデルが失敗・過負荷のとき、別へ切り替える(原則は [デプロイとスケーリング](deployment-and-scaling.md))。ゲートウェイに集約すると、全アプリが自動でフォールバックの恩恵を受けます
- **リトライ・サーキットブレーカ**: 一時失敗のリトライと、繰り返し失敗する先を一時遮断するサーキットブレーカ(原則は [エラーハンドリングとリトライ設計](../02-architecture/error-handling-and-retries.md))。これもゲートウェイ層で共通化できます
- **アプリから消える利点**: これらをアプリ個別に書かず、ゲートウェイに寄せることで、アプリはビジネスロジックに集中できます

### マルチテナント制御との統合

複数チーム・複数テナントが使うゲートウェイは、**分離とクォータ**を統合します。

- **テナント別のクォータ**: チーム・テナント単位のレート制限・予算上限を課します([マルチテナント設計とテナント分離](../02-architecture/multi-tenancy-and-isolation.md)のクォータの実装点)。1 テナントの暴走が他に波及しないようにします
- **仮想キーで境界を引く**: テナント/アプリごとの仮想キーに権限・上限を紐づけ、分離の境界を明確にします
- **集計の単位**: コスト・使用量をテナント単位で集計し、課金・配賦の根拠にします

### 落とし穴

ゲートウェイは便利な一方、**集約ゆえのリスク**を抱えます。

- **単一障害点化**: 全 LLM トラフィックがゲートウェイを通るため、ここが落ちると**全アプリが止まります**。冗長化・高可用性を最優先で設計します。「共通化したら共通で落ちる」を避けます
- **モデル固有機能の喪失**: 統一 API に抽象化すると、特定モデルの独自機能(固有のパラメータ・思考制御・構造化出力の作法など [モデル間の違いと移行(横断比較)](../03-implementation/cross-model-prompting.md))が使えなくなることがあります。抽象化と固有機能アクセスのバランスを取ります
- **ゲートウェイ自体の運用負担**: ゲートウェイも運用対象です。可観測性・アップデート・スケールの手間が増えます。「アプリの運用を減らすためにゲートウェイの運用が増える」ことを織り込みます

## 実務での注意点

### アンチパターン

- **ゲートウェイを冗長化せず単一構成にする** → ここが落ちると全アプリが止まる → 高可用性を最優先で設計する
- **過度に抽象化してモデル固有機能を潰す** → 思考制御・構造化出力などが使えず品質が落ちる → 固有機能へのアクセス経路を残す
- **薄い自作のまま機能を足し続ける** → フォールバック・集計・監査で結局重くなり保守不能に → 要件が増えるなら OSS/マネージドへ切り替える
- **マネージドを閉域網・データ主権要件で採用する** → データがベンダー経由になり要件違反 → 要件に応じて自社ホストの OSS を選ぶ
- **OSS のライセンス境界を確認しない** → 必要機能が有償エンタープライズ限定で想定が崩れる → OSS コアと有償機能の境界を採用前に確認する

### チェックリスト

- [ ] ゲートウェイに集約する機能(キー管理・抽象化・集計・監査・レート制御)を決めた
- [ ] 構成類型(自作 / OSS / マネージド)を規模・統制・データ要件で選んだ
- [ ] OSS を使う場合、OSS コアと有償エンタープライズ機能の境界を確認した
- [ ] 用途別ルーティングを設計し、自ホストモデルも同じ入口に束ねた
- [ ] フォールバック・リトライ・サーキットブレーカをゲートウェイ層に集約した
- [ ] テナント別のクォータ・予算・仮想キーで分離と集計を設計した
- [ ] ゲートウェイを冗長化し、単一障害点にしない設計にした
- [ ] 抽象化でモデル固有機能を失わない経路を残した
