# フィジカル AI とロボティクスの概観

## この記事の目的

ソフトウェア Agent の知識が、物理世界で動く Agent(ロボット)にどこまで通用し、何が本質的に違うかを説明できるようになります。フィジカル AI(physical AI)の全体像 — VLA モデル・デュアルシステム構成・世界モデル・2026 年時点の成熟度 — を、ソフトウェアエンジニアの語彙で 1 本に圧縮した概観です。ロボットの実装(ROS・制御理論・ハードウェア)には踏み込みません。

## 対象読者

- ソフトウェア Agent を学んだうえで、隣接するフィジカル AI の動向を素早く把握したいエンジニア
- 「ヒューマノイドのニュース」を、デモの印象ではなく構造で評価できるようになりたい人

## 前提知識

- [AI Agent とは何か](what-is-an-ai-agent.md) — 観測・思考・行動という基本構造
- [コンピュータ操作型・マルチモーダル Agent](computer-use-and-multimodal-agents.md) — 画面を「見て操作する」Agent(フィジカル AI に最も近い隣接領域)

## 本文

### 概要: なぜソフトウェアエンジニアが知っておくべきか

フィジカル AI とは、物理世界を知覚し、物理世界に対して行動する AI の総称です。2023 年以降、この分野の中心は「ロボット工学の延長」から「**基盤モデルの延長**」へ移りました — 視覚・言語の基盤モデルをロボットの行動出力に拡張するアプローチが主流になり、ソフトウェア Agent とよく似た構造(計画層と実行層、ループ、ツール)が現れています。

ソフトウェアエンジニアにとっての要点は 2 つです。第一に、**あなたが本ライブラリで学んだ構造の多くがそのまま通用します**(次節)。第二に、**通用しない 3 点(リアルタイム・不可逆性・データ)が、この分野の難しさの本体**です(次々節)。この 2 つを押さえれば、フィジカル AI のニュースを構造で読めるようになります。

### ソフトウェア Agent との共通点

2026 年時点の主要なフィジカル AI は、ソフトウェア Agent の設計語彙でかなり正確に説明できます。

- **観測 → 思考 → 行動のループ**: カメラ画像等を観測し、状況を解釈し、行動を出す、という [Agent ループ](agent-loop.md)と同じ骨格です
- **オーケストレータ + ツールの同型性**: 代表的な構成では、具身推論(embodied reasoning)モデルが「高レベルの脳」としてタスクを分解・計画し、各ステップを自然言語の指示として行動モデルに渡します。検索などのデジタルツールも呼び出せます。つまり[オーケストレーションパターン](../02-architecture/orchestration-patterns.md)の構造で、**ツールの 1 つが「ロボットの身体」になっている**形です。提供元自身がこれを「agentic framework」と呼んでいます
- **遅い計画層 + 速い制御層(デュアルシステム)**: 大きな視覚・言語モデル(VLM)が状況理解・計画を担い(毎秒数回)、小さな制御モデルが連続制御を担う(毎秒数十〜数百回)、という分離が 2025〜2026 年の各社共通パターンです。オーケストレータとサブエージェント・部品の分離と同型の発想です
- **行動前に考える・経験から改善する**: 行動前に思考過程を生成する(think before acting)設計や、実演 → 人の修正 → 自律試行からの強化学習という改善ループも公式に採用されており、[プランニングと推論](planning-and-reasoning.md)や[フィードバックループの運用](../05-operations/feedback-loops.md)の物理版と読めます

### 本質的に違う点: 3 つの制約

同型性の先にある違いが、この分野の固有の難しさです。

1. **リアルタイム制約**: 物理の制御は毎秒数十〜数百回の頻度で途切れず続ける必要があり、LLM の推論レイテンシでは間に合いません。だからこそ計画層と制御層の分離が必須で、制御側は小さなモデルをロボット搭載の計算機でローカル実行します。「全部を 1 つの大きいモデルで」が成立しない構造的理由です
2. **失敗の不可逆性と物理的安全**: ソフトウェア Agent の「サンドボックスで試して失敗したらやり直す」([ツール権限設計とサンドボックス](../06-security/tool-permissions-and-sandboxing.md))という前提が成立しません。転倒・衝突・人への危害は取り消せないためです。対策は、意味的安全(「危険な依頼に従わない」といった常識的制約)と物理的安全(衝突回避・力制限などの下位コントローラ)を**別の層として分離**する多層防御として体系化されつつあります。[ガードレール](../06-security/guardrails.md)の「モデルの判断の外側で強制する」という思想の、賭け金が桁違いに大きい版です
3. **データを「製造」する必要**: Web テキストに相当する大規模な行動データは自然には存在しません。テレオペ(遠隔操作)による実演収集、人間の動画からの転移、そしてシミュレーションによる合成データ生成を組み合わせて、学習データを作るところから始まります。シミュレーション中心の路線と実機データ中心の路線が 2026 年時点で並立しています

### VLA モデルの概観

この分野の中核となるモデル類型が VLA(Vision-Language-Action)モデルです。VLM をロボットの実演データでさらに学習させ、**視覚入力と言語指示から直接ロボットの行動を出力**します。「行動をもう 1 つの言語(トークン列)として扱う」という発想で、2023 年の研究発表(RT-2)がこのパラダイムの起点とされます。従来のロボティクスが知覚・計画・制御を別モジュールで組んでいたのに対し、Web 規模の知識と意味理解がそのまま行動に転移することが転換点でした。

2026-07 時点の主要な系統と提供形態は次のとおりです(**世代交代が数か月単位で起きる領域**のため、系統名レベルで記します。版数・詳細はリポジトリ内 `research/supplementary/physical-ai.md` の調査メモと各社公式ページを参照)。

| 系統(開発元) | 特徴 | 提供形態(2026-07 時点) |
| --- | --- | --- |
| Gemini Robotics 系(Google DeepMind) | 具身推論モデル + VLA の二層構成。ロボットは作らずモデルを供給 | 推論層は一般の API で公開。VLA 本体は限定パートナーのみ |
| GR00T 系(NVIDIA) | モデル + シミュレーション + エッジ計算機の 3 層プラットフォーム戦略 | オープンウェイト(モデルは公開、ハードとシミュで収益化) |
| π 系(Physical Intelligence) | 「単一モデルであらゆるロボットを制御する」ことを掲げるモデル専業 | 初期モデルはオープンソース化。最新世代の提供形態は要確認 |
| Helix(Figure) | ヒューマノイド上半身(35 自由度)の高頻度連続制御。デュアルシステムの代表例 | 自社ロボット専用(外部提供なし) |

読み方の注意が 2 つあります。第一に、**VLA には共通ベンチマーク・リーダーボードが 2026-07 時点で確立していません**。各社の性能数値は条件の異なる自己報告であり、横並び比較はできません([エージェントベンチマークの全体像](../04-evaluation/agent-benchmarks-landscape.md)で扱った「実行主体と条件を確認する」という規律が、より強い形で必要な領域です)。第二に、**世界モデル(world model)**は「ロボットを直接動かすモデル」としてではなく、合成データの生成とポリシー評価を支える**学習インフラ(データ工場・シミュレータ)**として実用化が先行しています。

### 現在地と展望(2026-07 時点)

成熟度は領域ごとにはっきり分かれています。

- **商用初期**: 構造化された倉庫環境 × 限定タスク(荷物の移載など)のヒューマノイドは、常時稼働の商用契約(サービスとしてのロボット提供)に到達した事例があります。ただし規模は世界で数十台の帯です
- **量産立ち上げ期**: 複数社が 2026 年のヒューマノイド量産開始・消費者向け出荷計画を発表しています。台数・時期の数値はいずれもベンダー自己報告で、第三者検証はありません
- **研究段階**: 「任意の環境で任意のタスクをこなす」汎用ヒューマノイドは研究発表・限定デモの域です。デモが示す汎用性と、商用の品質保証を伴う汎用性の間には大きな距離があります — ソフトウェア Agent で学んだ「デモの罠」([PoC から本番への進め方](../09-business/poc-to-production.md))と同じ構図です

コンピュータ操作型 Agent([コンピュータ操作型・マルチモーダル Agent](computer-use-and-multimodal-agents.md))との関係も押さえておくと見通しが良くなります。画面を観測して GUI を操作する Agent と、世界を観測して身体を動かす Agent は、観測の次元・行動の不可逆性・リアルタイム性が段階的に増していく**連続線上**にあり、「観測 → 行動のループを、どこまで危険な作用域で回すか」という同じ問いの異なる地点です。

## 実務での注意点

### アンチパターン

- **デモ動画から汎用性を外挿する** → デモは選ばれた環境・選ばれた成功例であり、商用の汎用性とは別物 → 「誰が・どの環境で・常時稼働しているか」という商用実績で現在地を判断する
- **ベンダー自己報告の数値を検証済みの事実として引用する** → 共通ベンチマークが存在せず、条件の異なる自己報告しかない → 数値には出典・時点・確度(自己報告か第三者か)を必ず付ける
- **ソフトウェアの安全の直感(再試行・ロールバック可能)を持ち込む** → 物理世界の失敗は不可逆で、サンドボックスの前提が成立しない → 意味的安全と物理的安全の層分離を前提に評価・設計する
- **特定バージョンのモデル知識を長期間使い回す** → この領域は数か月単位で世代交代する → 系統名 + 時点で把握し、引用前に公式ページで現行世代を確認する

### チェックリスト

フィジカル AI の情報を評価・引用するときのチェックです。

- [ ] 計画層と制御層の分離(デュアルシステム)を前提に構成を読んだ
- [ ] モデルの提供形態(一般 API / オープンウェイト / 限定・専用)を確認した
- [ ] 性能数値の確度(ベンダー自己報告か第三者検証か)と時点を確認した
- [ ] 「デモの汎用性」と「商用実績」を区別して現在地を判断した
- [ ] 安全の議論で意味的安全と物理的安全を区別した
