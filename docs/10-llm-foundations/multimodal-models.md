# マルチモーダルモデルの仕組み(数式なしの直感)

## この記事の目的

画像や音声を扱うマルチモーダルモデル(multimodal model)が、**内部で何をしてテキストと同じように扱っているのか**の直感を持てるようになります。10 章の方針どおり数式には踏み込まず、「なぜ画像がトークンとして消費されるのか」「なぜ細部の読み取りで急に外すのか」といった、実務の設計・デバッグに効く挙動の由来を理解するのがゴールです。

## 対象読者

- 画像・音声を入力に使う Agent を作っていて、コスト・精度の挙動の理由を掴みたいエンジニア
- 「マルチモーダル対応」の内部を、実装の詳細抜きで直感的に理解したい人

- [トークナイザとトークン経済](tokenization.md) — トークンが課金・文脈の単位(画像もトークンを消費する理由の土台)

マルチモーダルモデルの核心は、**テキスト以外の入力を、テキストと同じ「土俵」に載せる**ことです。本記事はその内部の直感を扱い、それを使ってエージェントを作る側は [コンピュータ操作とマルチモーダルエージェント](../01-concepts/computer-use-and-multimodal-agents.md) が扱います。

### 直感①: すべてを「表現ベクトルの列」に変換する

LLM は本来、テキストのトークンを**表現ベクトル(意味を表す数の並び)**に変換して処理します([埋め込み(embeddings)](../03-implementation/embeddings.md)と同じ発想)。マルチモーダルモデルは、この入り口を拡張します。

- テキストは、これまでどおりトークン → 表現ベクトルに変換します
- 画像や音声は、専用の変換器(エンコーダ)を通して、**同じ形の表現ベクトルの列**に変換します
- こうして、入力の種類が違っても、モデルの内部では**同じ形式のベクトル列**になります。あとはテキストと同じように処理できます

要点は、モデルが「画像そのもの」を見ているのではなく、**画像を変換した表現ベクトルの列**を見ている、ということです。この変換の質が、読み取りの精度を左右します。

### 直感②: 画像は「パッチ」に分けてトークン相当にする

では画像はどうやってベクトルの列になるのでしょうか。直感は、**画像を格子状の小片(パッチ)に分割する**ことです。

- 画像を小さなタイルに分け、各タイルを 1 つの表現ベクトルに変換します。テキストが単語片(トークン)の列になるように、画像は**パッチの列**になります
- だから、画像は**トークンを消費します**([トークナイザとトークン経済](tokenization.md))。1 枚の画像が何十〜何百トークン分にもなり、**解像度が高い・大きい画像ほど多くのトークン**を食います
- これが「画像を入れたら急にコンテキストとコストが増えた」の正体です。画像は「1 個の入力」ではなく「多数のパッチ=多数のトークン」として扱われます

### 直感③: 統一された系列を注意機構で混ぜ合わせる

テキストのトークン列と画像のパッチ列が同じ形式になったら、あとは**1 つの系列に並べて**、いつもの仕組みで処理します。

- テキストと画像のベクトルを 1 本の系列に連ね、[注意機構](attention-and-context.md)で相互に関連づけます。これにより「画像のこの部分と、質問のこの語」を結びつけられます
- 出力は、これまでどおり**テキストのトークンを 1 つずつ生成**します([LLM はどうやってテキストを生成するか](how-llms-generate-text.md))。入力はマルチモーダルでも、多くのモデルの出力はテキストです
- つまりマルチモーダル理解は、「入力の入り口を増やし、あとは共通の系列処理に流し込む」構図です。生成の仕組みそのものは変わりません

### なぜ効くのか、どこで崖が来るのか

この構図は強力ですが、**限界も構図から予測できます**。

- **効く**: 画像・音声を言語と結びつけた理解(説明・要約・質問応答・大まかなレイアウト把握)。パッチ表現が捉えられる範囲の情報
- **崖が来る**: パッチに分割・圧縮する過程で**細部が失われる**ため、小さな文字・密な表・精密な位置関係・数え上げなどで急に外します。「だいたい合っているが細部が違う」が典型です
- **解像度がトレードオフ**: 細部を読ませたいなら解像度(=トークン数=コスト)を上げる必要があり、精度とコストが直結します。低解像度で渡して「読めない」のは、多くの場合モデルの知能ではなく入力の粒度の問題です
- **幻覚は同じ構造で起きる**: モデルは見えない細部を「もっともらしく補完」します。画像でも幻覚(hallucination)は起き([LLM の能力と限界の由来](capabilities-and-limits.md)の補完傾向)、しかも「画像に書いてあるように見せて」出るため気づきにくいです

### トークン経済とコスト・レイテンシ設計

マルチモーダルの実務判断は、多くが**トークン経済**に帰着します([トークナイザとトークン経済](tokenization.md))。

- 画像・音声・動画はトークンを大量に消費します。とくに動画は多数のフレーム=多数の画像で、コストとレイテンシが跳ねます
- **解像度・枚数を必要最小限にする**: 読み取りに要る粒度だけ渡します。全画面の高解像度を丸ごと渡す前に、関心領域の切り出し・縮小を検討します([コスト管理](../05-operations/cost-management.md)・[レイテンシ最適化](../05-operations/latency-optimization.md))
- **テキストで足りるなら渡さない**: OCR 済みテキストや構造化データがあるなら、画像より安く確実です。マルチモーダルは「テキスト化できない情報」にこそ使います

### 理解と生成は別物: 非対称に注意

「マルチモーダル」は、入力を理解する側と、画像・音声を**生成**する側で仕組みが異なります。混同しないことが実務上重要です。

- 本記事が扱ったのは**入力理解**(画像を読む)です。多くの LLM 系マルチモーダルモデルは、理解はできても出力はテキストです
- 画像・音声そのものの**生成**は、別系統の技術(拡散モデルなど)が担うことが多く、同じモデルが両方を高品質にこなすとは限りません
- 「この画像を説明して」(理解)と「こういう画像を作って」(生成)は、別の能力・別の課金・別の限界だと捉えます

## 実務での注意点

### アンチパターン

- **画像を「1 個の入力」だと思う** → トークン消費とコンテキスト圧迫を見落とす → 画像=多数のパッチ=多数のトークンとして見積もる
- **細部の読み取りを低解像度で渡す** → 小さい文字・表・位置関係で外し、モデルの知能のせいにする → 読み取りに要る解像度(=コスト)を確保するか、関心領域を切り出す
- **画像の幻覚を素通しする** → 「画像に書いてあるように見える」補完を検証しない → 重要な読み取りは別手段(OCR・構造化データ)と突き合わせる
- **テキストで足りるのに画像を渡す** → 高コスト・低確実性を招く → テキスト化できるものはテキストで渡し、画像は最後の手段にする
- **理解と生成を混同する** → 「読める=作れる」と誤認する → 入力理解と画像生成は別能力・別課金として扱う

### チェックリスト

- [ ] 画像・音声がトークンを消費し、解像度・枚数でコストが増えることを見積もりに入れた
- [ ] 読み取りに必要な解像度・関心領域を検討し、無駄に高解像度を渡していない
- [ ] 細部(小さい文字・表・位置・数え上げ)の読み取りは崖が来ると理解し、重要箇所を検証している
- [ ] テキスト化できる情報は画像でなくテキストで渡している
- [ ] 「入力理解」と「画像・音声生成」を別の能力・課金として区別している
