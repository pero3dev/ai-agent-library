# 軌跡(trajectory)評価

## この記事の目的

最終出力だけでは見えない Agent の品質 — 経路の正しさ・効率・安全性 — を、軌跡(実行ステップの列)に対する評価として設計できるようになります。ルールベースのアサーションと LLM-as-a-Judge をどう使い分けるか、そして「経路をどこまで縛るべきか」を判断できるようになります。

## 対象読者

- 「最終出力は合っているのに挙動が不安」「失敗の原因がどのステップか分からない」という状態にあるエンジニア
- ツール呼び出しの正しさ・効率・安全性を定量的に検証したいチーム

## 前提知識

- [Agent ループ](../01-concepts/agent-loop.md) — 軌跡はこのループの実行記録
- [Agent 評価の基礎](agent-evaluation-basics.md) — 評価対象 3 層(最終成果・軌跡・部品)の中での位置づけ

## 本文

### 概要: 最終成果だけでは足りない 2 つの理由

軌跡(trajectory)とは、Agent ループの 1 回の実行が残すステップの列 — モデルの思考・ツール呼び出し(名前と引数)・ツール結果・最終応答 — のことです。

最終成果の評価だけでは不足する理由は 2 つあります。

1. **正解でも悪い経路がありえます** — 偶然の正解(検索に失敗したのにモデルの学習知識で答えた)、非効率な正解(同じ検索を 5 回繰り返した)、危険な正解(不要な書き込み操作を経由した)は、最終出力の採点ではすべて「合格」になります
2. **不正解の原因が分かりません** — 失敗がツール選択・引数・結果の解釈・統合のどこで起きたのかは最終出力からは見えず、分からなければ直せません。軌跡は失敗の局所化(どのステップから壊れたか)の一次情報です

### 詳細: 軌跡の何を見るか

| 観点 | 問い | 例 |
| --- | --- | --- |
| ツール選択 | 呼ぶべきツールを呼ぶべきときに呼んだか | 検索が必要な質問に、検索せず学習知識で答えていないか |
| 引数の質 | 引数が正しく妥当か | ID の取り違え、検索クエリが的を外していないか |
| 効率 | 無駄なステップがないか | 総ステップ数、同一ツールの同一引数での繰り返し |
| エラー回復 | ツール失敗後の行動が適切か | エラーを無視して進む・同じ呼び出しを盲目的に繰り返すことがないか |
| 安全性 | 禁止・不要な操作をしていないか | 照会タスクの途中で書き込み系ツールを呼んでいないか |

### 詳細: 評価方法は 2 系統の併用

| 方式 | 仕組み | 向くもの | 限界 |
| --- | --- | --- | --- |
| ルールベースのアサーション | 軌跡の構造化ログに対してコードで条件を検査する | 必須ツールの呼び出し有無、順序の制約、回数上限、禁止操作、引数の形式 | 「引数の質」「回復の適切さ」のような判断を要する観点は書けない |
| LLM-as-a-Judge による軌跡レビュー | 整形した軌跡を judge に渡し、観点別に判定させる | 経路の合理性・エラー回復の適切さなど判断を要する観点 | コスト・非決定性。judge 自体の検証が必要([LLM-as-a-Judge](llm-as-a-judge.md)) |

**機械的に判定できる条件、特に安全性はアサーション**で固めます。たとえば「禁止ツールを呼んでいない」「承認なしに書き込みをしていない」「ステップ数が上限以内」は数行のコードで検査でき、決定的で速く、judge のような検証も要りません。判断を要する観点だけを judge に回します。

なお、安全性の条件は評価時の検査だけでなく**実行時のガードにもする**べきものです([Human-in-the-Loop 設計](../02-architecture/human-in-the-loop.md)、`guardrails.md`(執筆予定))。評価は「ガードが効いていること」の確認として機能します。

### 詳細: 前提は軌跡の構造化された記録

軌跡評価は、ループの各ステップ(種別・ツール名・引数・結果・トークン量・所要時間)が**構造化された形式で記録されている**ことが前提です。この記録は本番の可観測性([可観測性とトレーシング](../05-operations/observability-and-tracing.md))で必要になるものと同じです。評価専用の後付けログを作らず、**評価と本番で共通のトレース形式**に最初から投資すると、本番で起きた失敗をそのまま評価ケースへ変換できるようになります。

### 設計判断: 経路をどこまで縛るか

軌跡評価には「正しい経路」を固定しすぎる罠があります。Agent の価値は経路を実行時に決める自由度にあるため([Workflow 型 vs Agent 型の使い分け](../02-architecture/workflow-vs-agent.md))、期待する軌跡との完全一致を要求すると、**正しい別解を不合格にし、モデルやプロンプトの変更のたびに大量に壊れる脆いテスト**になります。

- 縛るのは**不変条件**だけにします — 安全性(禁止操作の不在)、必須のマイルストーン(「この情報を取得してから回答する」)、資源上限(ステップ数・コスト)
- 「このツールをこの順でこの引数で」まで指定したくなったら、その部分はそもそも Agent ではなく Workflow として固定すべきだったサインです

長いタスクには**マイルストーン単位の部分点**を設計します。全体の合格/不合格だけでは「どこまで進めたか」の改善が観測できません。「候補を提示できた」「確認を取ってから実行した」のような中間到達点を採点対象にすると、改善が数値に現れるようになります。

## 実務での注意点

### アンチパターン

- **期待する軌跡との完全一致を要求する** → 正しい別解を不合格にし、モデル・プロンプト変更のたびに大量に壊れて保守されなくなる → 不変条件(安全性・マイルストーン・上限)だけをアサーションにする
- **生の軌跡ログをそのまま judge に流し込む** → 長大な JSON がコンテキストを圧迫し、判定の質が下がる → ステップごとの要約(ツール名 + 引数の要点 + 結果の要点)に整形してから渡す
- **軌跡の記録を評価のときだけ後付けする** → 本番と評価で見えるものが食い違い、本番の失敗を再現・ケース化できない → 最初から評価と本番で共通のトレース形式で記録する

### チェックリスト

- [ ] 軌跡が構造化された形式で記録されている(評価と本番で共通)
- [ ] 安全性の条件(禁止操作・承認必須の操作)がアサーションで検査されている
- [ ] 効率の上限(ステップ数・同一呼び出しの繰り返し)が検査されている
- [ ] 経路の完全一致ではなく不変条件で縛っている
- [ ] 判断を要する観点に使う judge は人手ラベルで検証済みである
- [ ] 長いタスクにマイルストーン単位の部分点が設計されている
