# 埋め込み(embeddings)の選定と運用

## この記事の目的

RAG・長期記憶・分類の土台である**埋め込み(embeddings)モデル**を、根拠を持って選定・評価・運用できるようになります。[RAG 実装パターン](rag-implementation-patterns.md)が検索パイプライン全体を扱うのに対し、本記事はその一段下 — 「何が『近い』のか」「どのモデルを選ぶか」「更新でどう壊れるか」— に踏み込みます。

## 対象読者

- RAG や意味検索を実装し、埋め込みモデルの選定・乗り換え・評価を担当するエンジニア
- 「検索の精度が上がらない」原因を、埋め込みの層から切り分けたいエンジニア

## 前提知識

- [RAG 実装パターン](rag-implementation-patterns.md) — 検索パイプライン全体(本記事はその埋め込み層の詳解)
- ライブラリ外の前提: ベクトル(数値の並び)と類似度のごく基本的な考え方

### 埋め込みとは: 何が「近い」のか

埋め込みとは、テキスト(や画像)を**意味的な近さが距離に対応する**数値ベクトルに変換したものです。似た意味の文が近いベクトルになるため、クエリと文書のベクトル距離で「意味の近い文書」を検索できます。

要点は、**「近い」の中身はモデルの学習データとタスクで決まる**ことです。あるモデルにとっての「近さ」は、別のモデルでは違います。だからこそ、埋め込みは「どれでも同じ」ではなく、**自社のデータと質問で評価して選ぶ**対象になります。類似度の指標(コサイン類似度など)自体より、モデルがその指標で何を「近い」と見なすかが本質です。

### 選定軸

埋め込みモデルは次の軸で選びます。製品名ではなく、この軸で候補を絞ります。

| 軸 | 見るポイント |
| --- | --- |
| 対応言語 | 日本語(や多言語混在)の検索品質。英語中心のモデルは日本語で落ちることがある |
| ドメイン適合 | 対象領域(法務・医療・コードなど特殊語彙)での近さの妥当性 |
| 最大入力長 | 1 チャンクに入れたい長さを埋め込めるか(超過分は切り捨てられる) |
| 次元数 | ベクトルの長さ。保存・検索コストに直結(後述) |
| コスト・提供形態 | API 従量 / セルフホスト。大量文書では埋め込み生成コストも効く |

選定は勘ではなく評価で決めます(後述)。とくに**日本語**は、英語ベンチマークで上位でも日本語で振るわないモデルがあるため、自社データでの確認が要ります。

### 次元のトレードオフ

次元数(ベクトルの長さ)は、精度と保存・検索コストのトレードオフです。

- **高次元**: 表現力が高く精度が上がりやすいが、**保存容量・検索計算・メモリが増える**。数百万チャンク規模では次元数がそのままインフラコストに響きます
- **低次元**: 安く速いが、細かい意味差を捉えにくい
- **次元削減という選択肢**: 一部のモデルは、生成したベクトルの**先頭 N 次元だけを使っても品質が保たれる**設計になっています(切り詰めて次元を下げられる)。精度の劣化を評価で確かめたうえで、コストとのバランスで次元を選べます

「とにかく高次元」ではなく、**評価セットで精度がほぼ変わらない最小次元**を探すのが実務です。

### 非対称検索: クエリと文書は非対称

検索では、短い質問(クエリ)と長い文書(チャンク)を突き合わせます。この 2 つは性質が違うため、**同じ扱いをしないほうが精度が上がる**ことがあります。

- **クエリと文書で埋め込みを変える**: モデルによっては、クエリ用と文書用で異なる前処理(プレフィックスや指示文の付与)を推奨します。「これは検索クエリです」「これは検索対象の文書です」と役割を伝えると、非対称性を活かせます
- **指示付き埋め込み**: 一部のモデルは、埋め込み時に「何のための埋め込みか」を指示として渡せます(タスク条件付き)。同じ文でも用途で近さを変えられます
- **公式の推奨に従う**: この前処理はモデルごとに作法が異なるため、利用するモデルの公式ドキュメントの指示に従います(自己流のプレフィックスは逆効果になりえます)

### チャンクサイズとの相互作用

埋め込みの品質は、**渡すチャンクの大きさ**と切り離せません(チャンキング設計は [RAG 実装パターン](rag-implementation-patterns.md) が正本)。

- 長すぎるチャンクは、1 つのベクトルに複数の話題が混ざり、「近さ」がぼやけます
- モデルの最大入力長を超えたチャンクは、超過分が切り捨てられ、末尾の情報が埋め込みに反映されません
- したがって埋め込みモデルの選定とチャンクサイズの決定は**セットで評価**します。「モデルを変えたらチャンクサイズも再評価」が原則です

### 微調整はいつ効くか

汎用の埋め込みモデルで足りないときの手段が微調整(ファインチューニング)です。ただし**最後の手段**です。

- **効きやすいのは、語彙が特殊なドメイン**(社内固有の略語・型番体系・専門用語)で、汎用モデルが「近さ」を誤るケースです
- 微調整には、そのドメインの「近い/近くないペア」の教師データが要ります。作成・維持のコストと、汎用モデルの乗り換え(後述の再インデックス)を天秤にかけます
- まず汎用モデルの選定・非対称検索・チャンクサイズ調整を尽くし、それでも評価で足りないと確認できてから検討します

### 評価: 自社の検索評価セットで比べる

埋め込みの良し悪しは、**自社の質問と正解チャンクの対応表**で測ります(公開ベンチマークの読み方の規律は [エージェントベンチマークの全体像](../04-evaluation/agent-benchmarks-landscape.md) と同じ — 参考にはするが最終判断は自社評価)。

- 質問と「その質問に答えるチャンク」の対応表を作り、候補モデルごとに再現率(recall@k)や正解チャンクの順位を比べます([評価データセットの構築と保守](../04-evaluation/evaluation-datasets.md)、検索評価の位置づけは [RAG 実装パターン](rag-implementation-patterns.md))
- 公開ベンチマーク(MTEB など)は候補の当たりを付けるのに使えますが、**自社ドメイン・自社言語では順位が入れ替わりえます**。カタログの順位を鵜呑みにしないこと
- 評価はコスト(次元・API 単価)と**同時に**見ます。精度がわずかに高いだけで大幅に高い・遅いモデルは、実務では割に合わないことがあります

### 運用の最重要問題: モデル更新 = 全再インデックス

埋め込み運用で最も高くつくのが、**モデルを変えると全チャンクを埋め込み直す(全再インデックス)必要がある**ことです。旧モデルのベクトルと新モデルのベクトルは同じ空間にないため、混ぜて検索できません。

- **バージョンを固定・記録する**: インデックスに埋め込みモデル名とバージョンを必ず記録します。「いつの間にかモデルが変わって検索が劣化」を防ぎます
- **更新は「デプロイ」として扱う**: モデル変更は、新インデックスを別に構築 → 評価で新旧比較 → 切り替え(ロールバック可能に)、という手順を踏みます([バージョニングとモデル更新追従](../05-operations/versioning-and-model-updates.md) と同じ考え方)
- **API モデルの暗黙変更に注意**: 提供側がモデルを更新・退役すると、意図せず全再インデックスが必要になります。バージョン固定できる提供形態を選ぶか、退役告知を監視します
- **移行コストが乗り換えの障壁になる**: 数百万チャンクの再埋め込みは時間もコストもかかります。だからこそ**最初の選定を評価で固める**価値があります

## 実務での注意点

### アンチパターン

- **英語ベンチマークの順位だけで日本語の検索に採用する** → 日本語や自社ドメインで静かに精度が落ちる → 自社データの評価セットで候補を比較してから決める
- **とにかく高次元のモデルを選ぶ** → 保存・検索コストが膨らむのに精度はほぼ変わらないことがある → 評価で精度が保てる最小次元を探す
- **クエリと文書を同じ前処理で埋め込む** → モデルが推奨する非対称の作法を使わず精度を取りこぼす → 公式の推奨(プレフィックス・指示)に従う
- **埋め込みモデルを気軽に乗り換える** → 全再インデックスと評価やり直しのコストを見落とす → バージョンを固定し、変更を評価付きのデプロイとして扱う
- **チャンクサイズを固定したままモデルだけ変える** → 最大入力長・話題混在で埋め込みが劣化する → モデルとチャンクサイズをセットで再評価する

### チェックリスト

- [ ] 候補モデルを対応言語・ドメイン適合・入力長・次元・コストの軸で絞った
- [ ] 自社の質問と正解チャンクの対応表で候補を比較した(公開ベンチマークを鵜呑みにしていない)
- [ ] 精度が保てる最小次元を評価で確認した
- [ ] クエリ・文書の非対称な前処理(公式推奨)を使っている
- [ ] チャンクサイズと埋め込みモデルをセットで評価した
- [ ] インデックスに埋め込みモデル名・バージョンを記録している
- [ ] モデル変更を、別インデックス構築 + 新旧比較 + 切り替えのデプロイ手順にしている
- [ ] API モデルの暗黙更新・退役を監視する仕組みがある
