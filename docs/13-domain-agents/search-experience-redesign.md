# 検索体験の再設計

## この記事の目的

「検索して読む」体験を「**聞いて答えを得る**」体験へ再設計する際の構成と、従来検索との共存を判断できるようになります。回答型 UX の構成類型・社内検索を作り直すか活かすかの判断・出典と「見つからない」の扱い・検索ログという資産・評価の 2 層構造を、実務の設計判断として持ち帰れる状態を目指します。

## 対象読者

- 社内検索やヘルプ検索を「回答を返す」体験(アンサーエンジン)へ作り替えたいプロダクト責任者・エンジニア
- RAG は動かせるが、それを「人間向けの検索プロダクト」として設計する視点がほしい実装者

## 前提知識

- [RAG 実装パターン](../03-implementation/rag-implementation-patterns.md) — 検索と生成の実装(本記事はその上に載る検索体験の設計)
- [評価データセットの構築と保守](../04-evaluation/evaluation-datasets.md) — 検索・回答の評価セット
- [ナレッジ Agent のケーススタディ](../07-case-studies/case-study-knowledge-agent.md) — 知識源の品質が体験を決めた実例

## 本文

### 概要: 体験の主語が「探す」から「答える」へ変わる

従来の検索は「クエリを入れる → 結果一覧を見る → 自分で読んで判断する」体験です。これを「**質問する → 答えが返る**」体験に変えるのが、いわゆるアンサーエンジン(answer engine)です。RAG はその技術的な土台ですが、[RAG 実装パターン](../03-implementation/rag-implementation-patterns.md)が「Agent の知識源」を扱うのに対し、本記事は「**人間向けの検索プロダクト**」としての設計を扱います。

主語が「探す」から「答える」に変わると、責任も変わります。結果一覧なら、選ぶ責任は利用者にありました。答えを返すなら、**その答えが正しい根拠を示す責任**がプロダクト側に生じます。この責任の移動が、以下のすべての設計判断の起点です。

### 構成の類型

回答型の検索体験には、複雑さの異なる 3 つの類型があります。すべてを一足飛びに目指さず、知識源と用途に合う段階を選びます。

| 類型 | 構成 | 向く場面 |
| --- | --- | --- |
| 検索結果 + 要約 | 従来の検索結果に、上位数件の要約を添える | 既存検索を活かしつつ体験を底上げしたい |
| 会話型検索 | 対話で聞き返し・絞り込みをしながら答える | 問いが曖昧・多段の絞り込みが要る |
| エージェント型検索 | 多段の検索・読解を自律実行して答える | 1 回の検索では答えが出ない調査的な問い |

- **検索結果 + 要約**: 最も導入が軽い型です。既存の検索基盤を残したまま、上位結果の要約を足すだけで体験が上がります。出典は結果一覧がそのまま担うため、根拠提示の設計も楽です
- **会話型検索**: 曖昧な問いに聞き返し、条件を絞りながら答えます。問いの意図が定まっていない社内検索に向きます
- **エージェント型検索**: 多段で調べ切る型で、実態は[ディープリサーチ型エージェント](deep-research-agents.md)に近づきます。コストとレイテンシが上がるため、「1 回で答えが出ない問い」に用途を絞ります

### 社内検索の作り直し(活かすか置き換えるか)

社内検索を回答型にするとき、「既存の検索基盤を活かすか、置き換えるか」で迷います。多くの場合、**活かすほうが現実的**です。

- **既存の検索資産を活かす**: 全文検索・アクセス権・メタデータといった既存基盤は、そのまま RAG の検索段(一次検索)として使えることが多いです。特に**アクセス権の反映**は既存基盤が持つ強みで、これをゼロから作り直すのは大きなリスクです([RAG 実装パターン](../03-implementation/rag-implementation-patterns.md)の権限反映検索)
- **置き換えの前に不足を見極める**: 既存検索の不満が「検索精度」なのか「知識源の質」なのかを切り分けます。後者(古い・重複した・欠けた文書)なら、検索エンジンを替えても解決しません(後述)
- **段階的に載せる**: いきなり全面刷新せず、「検索結果 + 要約」から始めて効果を測り、必要なら会話型・エージェント型へ進みます

### 回答型 UX の設計(出典・確度・「見つからない」)

答えを返す以上、UX の設計は「正しく見せる」ことより「**検証できるように見せる**」ことを優先します。

- **出典を必須にする**: 回答の各主張に、根拠となった文書へのリンクを添えます。出典のない断定は、利用者が確かめられず、幻覚と区別がつきません。出典は「飾り」ではなく「回答の一部」です
- **確度を表示する**: 根拠が薄い・情報源が食い違うときは、**確度を下げて示す**か、断定を避けます([信頼度と較正](../04-evaluation/confidence-and-calibration.md))。自信満々に間違えるより、控えめに正しいほうが信頼されます
- **「見つからない」を正しく言う**: 知識源に答えがないとき、**もっともらしい作り話をせず**「見つかりませんでした」と言えることが、回答型検索の生命線です。ゼロ結果を無理に埋めない設計([幻覚](../10-llm-foundations/capabilities-and-limits.md)対策)を、UX とプロンプトの両方で担保します
- **答えと一次資料を往復できる**: 答えだけで完結させず、元の文書へ辿れる導線を残します。重要な判断ほど、利用者は一次資料を自分で読みたがります

### 検索ログという資産

回答型に変えると、利用者は「キーワード」ではなく「**自然な問い**」を入力します。この問いのログは、知識の欠落を映す鏡であり、改善の最大の資産です。

- **クエリを知識源の改善に還流する**: よく聞かれるのに答えられない問いは、**知識源に足りない情報**を教えてくれます。検索ログを分析し、不足している文書・古い文書を特定して知識源を直します([フィードバックループの運用](../05-operations/feedback-loops.md))
- **ゼロ結果・低評価を追う**: 答えられなかった問い・低評価の回答を継続的に追い、優先度の高い欠落から埋めます
- **プライバシーに配慮する**: 問いには機微な情報が含まれます。ログの保存・分析は[データガバナンス](../05-operations/data-governance-for-ai.md)の対象として扱います

### ゼロ結果・低品質知識源の問題(検索を直す前に知識源を直す)

回答型検索でうまくいかないとき、原因の多くは**検索アルゴリズムではなく知識源**です。ここを取り違えると、労力を無駄にします。[ナレッジ Agent のケーススタディ](../07-case-studies/case-study-knowledge-agent.md)がこの構造を示しています。

- **知識源の品質が回答の上限を決める**: 古い・重複した・矛盾した文書からは、正しい答えは出ません。検索エンジンをチューニングする前に、**知識源の棚卸し**(重複排除・鮮度・欠落の補充)をします([AI のためのデータガバナンス](../05-operations/data-governance-for-ai.md))
- **「答えがない」を放置しない**: ゼロ結果の問いは、知識源に情報を足すべきサインです。検索側で無理に何かを返すのではなく、知識源側を直します
- **オーナーシップを決める**: 知識源を誰が保守するかが曖昧だと、品質は下がり続けます。文書のオーナーと更新プロセスを決めるのが、検索体験を支える土台です

### 評価(検索指標と回答品質の 2 層)

回答型検索の評価は、**2 層**に分けます。片方だけでは品質を見誤ります([評価データセット](../04-evaluation/evaluation-datasets.md))。

- **検索層の評価**: 正しい根拠文書を候補に取れているか(再現率・上位の適合率)。ここが取れていないと、生成がどれだけ賢くても正しい答えは出ません
- **回答層の評価**: 取れた根拠から、正しく・出典付きで・過不足なく答えているか([LLM-as-a-Judge](../04-evaluation/llm-as-a-judge.md)と人手)。「見つからない」を正しく言えるか(答えるべきでない問いに答えていないか)も測ります
- **2 層を分けて診断する**: 悪い回答が「検索の失敗」か「生成の失敗」かを切り分けられると、直すべき場所が分かります。合算した 1 指標だけでは、原因が見えなくなります

## 実務での注意点

### アンチパターン

- **出典なしで答えを断定する** → 利用者が検証できず幻覚と区別がつかない → 各主張に根拠文書を必須で添える
- **ゼロ結果をもっともらしい作り話で埋める** → 静かに間違えて信頼を失う → 「見つからない」と正しく言える設計を UX とプロンプトで担保する
- **検索精度の問題だと思って知識源の劣化を放置する** → エンジンを替えても改善しない → まず知識源の棚卸し(重複・鮮度・欠落)をする
- **既存の検索基盤(特にアクセス権)を捨てて作り直す** → 権限反映を作り直す大きなリスク → 既存基盤を一次検索として活かす
- **検索指標と回答品質を 1 指標に混ぜる** → 悪化の原因が検索か生成か分からない → 2 層に分けて診断する
- **検索ログを改善に還流しない** → 知識の欠落が埋まらない → ゼロ結果・低評価クエリを知識源の改善に回す

### チェックリスト

- [ ] 用途と知識源に合う構成類型(結果+要約 / 会話型 / エージェント型)を選んだか
- [ ] 既存検索基盤(特にアクセス権反映)を活かす前提で設計したか
- [ ] 回答の各主張に出典を必須で添えているか
- [ ] 根拠が薄いときに確度を下げ、「見つからない」を正しく言えるか
- [ ] 検索ログ(ゼロ結果・低評価クエリ)を知識源の改善に還流しているか
- [ ] 検索を直す前に、知識源の品質(重複・鮮度・欠落・オーナー)を点検したか
- [ ] 検索層と回答層を分けて評価し、原因を切り分けられるか
