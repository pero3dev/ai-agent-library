# MODEL-PROMPTING-PLAN — 主要モデル特化プロンプティング 追加計画

> **ステータス: 完了(2026-07-07 作成・2026-07-08 実施 = 全 4 本 published)。PE-R1〜R3 の執筆前調査を `research/prompting/` に記録済み。**
> Anthropic(Claude)・OpenAI(GPT 系)・Google(Gemini)の主要モデルそれぞれに特化したプロンプトエンジニアリング手法を、**公式一次情報に基づいて網羅的・具体的に**体系化する追加計画です。進捗の正本は着手後 [ROADMAP.md](ROADMAP.md) に置きます。

## 1. 位置づけ: ベンダー中立原則の「例外類型」として設計する

本ライブラリの執筆ルールは「本文はベンダー中立に書く」ですが、**具体名が本質的な価値を持つ領域は、鮮度リスクごと専用ページに閉じ込める**という例外類型を既に確立しています — [llm-landscape](docs/03-implementation/llm-landscape.md)(モデルカタログ)と 08 章のツール別ページ(claude-code.md 等)がその前例です。本計画はこの類型の 3 例目です。

| 層 | 正本 | 性質 |
| --- | --- | --- |
| 原理(なぜ効くか) | 10 章(how-llms-generate-text ほか) | 安定・中立 |
| 汎用技法 | [prompt-engineering-fundamentals](docs/03-implementation/prompt-engineering-fundamentals.md)、DEEP-DIVE 計画の patterns / optimization | 安定・中立 |
| **モデル別の具体**(本計画) | ベンダー別ガイド 3 本 + 横断比較 1 本 | **変化が速い・具体名を扱う(鮮度管理型)** |

fundamentals は現在「利用中のモデルの公式ガイドを参照」と外部へ委ねています。本計画はその参照先の**ライブラリ内の橋**を作るものであり、汎用記事の中立性は保ったまま、具体を必要とする読者に答えます。

### ギャップ分析

既存・計画済みのどこにも「このモデルではこう書く」がありません。実務では「Claude では XML タグ、GPT 系では別の作法」といったモデル差がプロンプト品質に直結し、モデル移行時の書き直し判断([versioning-and-model-updates](docs/05-operations/versioning-and-model-updates.md))にも必要な知識です。

## 3. 各ページの設計

### ベンダー別ガイド 3 本の共通構成(#1〜#3)

3 本は同じ骨格で書き、比較可能性を持たせます(08 章のツール別テンプレート方式)。内容はすべて**各社公式のプロンプトガイド・モデルカード・クックブック(PE-R1〜R3)を根拠**とし、俗説・非公式テクニックは採用しません。

- **モデルファミリーの前提**(顔ぶれ・選び方は llm-landscape / model-selection へ委譲し、プロンプトに効く差分のみ)
- **メッセージ構造の作法**(システム/開発者メッセージの位置づけ・優先順位の扱い)
- **構造化の推奨記法**(区切り・タグ・セクションの公式推奨 — 例: XML タグ系か markdown 系か)
- **例示(few-shot)の作法**(推奨形式・配置)
- **思考の制御**(拡張思考・推論努力・思考予算 — 各社の呼称と制御方法、思考モデルで変わる書き方)
- **出力の制御**(応答の書き出し指定(prefill)・構造化出力・停止指定 — 各社の対応機能との併用)
- **長文書・長コンテキストの作法**(資料の配置・引用の作法・キャッシュとの相互作用)
- **ツール使用・エージェント文脈のプロンプト**(ツール説明との分担・並列呼び出し等の挙動制御)
- **世代間の移行注意**(同一ベンダー内での世代交代時に見直す点)
- **そのモデルで「効かない・不要になった」俗説**(公式が明示的に否定・非推奨とするもの)
- **鮮度管理**: 最終確認日 blockquote + 参考資料全件アクセス日 + 「変わりやすい項目(定点観測)」の 3 点セット必須

### cross-model-prompting.md — モデル間の違いと移行(#4)

- **目的**: 「1 つのプロンプトを複数モデルで使う」「モデルを乗り換える」ときに何が壊れ、何を書き直すかを判断できる
- **主要トピック**: 共通原理と差分の切り分け(原理は 10 章・汎用技法は fundamentals が正本 — 差分だけを本記事が持つ)/ 相違の構造マップ(構造化記法・システムメッセージの強さ・思考制御・出力制御・パラメータの互換性を 3 社横並びで)/ 移行チェックリスト(トークナイザ差([tokenization](docs/10-llm-foundations/tokenization.md))・強調表現の効き方の差・回帰評価必須 — versioning 接続)/ マルチプロバイダー設計(プロンプトのベンダー別差し替え層 — LLMOPS 計画の llm-gateway と接続)/ 「完全共通プロンプト」の限界と現実解
- **分担**: 移行の運用 = versioning-and-model-updates(正本)。本記事はプロンプト面の具体

## 4. スコープ外(検討のうえ除外)

- **オープンウェイト系(Llama・Qwen 等)の個別ガイド**: チューニング自由度が高く「公式推奨」の形が異なるため本計画では扱わない(需要が確認できたら第 2 期で検討 — §8)
- **API パラメータのリファレンス**(全引数の解説): プロンプトに効くものまで(仕様は公式が正)
- **ジェイルブレイク・制約回避のテクニック**: 扱わない(TRUST 系の防御方針と整合)
- **各社の料金・モデル選定**: llm-landscape / model-selection(正本)へ
