# プランニングと推論

## この記事の目的

Agent の「考える」部分を構成する主要パターン — 逐次判断・事前計画・リフレクション — を区別し、タスクの長さと探索性に合わせて選択・組み合わせできるようになります。

## 対象読者

- 「長いタスクで Agent が迷子になる」「手戻りが多い」問題に取り組むエンジニア
- Agent の思考の質をプロンプトと構造の両面から改善したいエンジニア

## 前提知識

- [AI Agent とは何か](what-is-an-ai-agent.md) — 自律性のスペクトラム
- [Agent ループ](agent-loop.md) — 「思考」がループのどこに位置するか

## 本文

### 概要: プランニングと推論は別の働き

- **プランニング(planning)** — タスクをステップに分解し、実行順序を決め、進捗に応じて見直す働き。複数ステップの**構造**を扱います
- **推論(reasoning)** — 個々の判断(次の一手・分析・検証)の質を上げる働き。**1 回の思考の深さ**を扱います

長いタスクの失敗は多くの場合プランニングの問題(全体を見失う)、個別ステップの失敗は推論の問題(判断が浅い)です。切り分けて対処します。

### 詳細: プランニングの 3 つの基本パターン

| パターン | 動き方 | 向くタスク | 弱点 |
| --- | --- | --- | --- |
| 逐次判断 | 1 手ずつ考えて動く(ReAct として知られる形) | 探索的で次の手が状況依存のタスク | 視野が短く、長いタスクで迷子になる |
| 事前計画 + 実行(plan-then-execute) | 最初に計画(TODO リスト)を立て、順に実行し、必要に応じて更新 | ステップ数が多い・進捗管理が要るタスク | 計画が古びる。見直しの仕組みが必要 |
| リフレクション(reflection) | 成果物を自己評価・批判し、修正を繰り返す | 品質基準が明確な成果物の仕上げ | コスト増。回数上限がないと回り続ける |

これらは排他ではなく組み合わせます。実務でよく使う形は「事前計画で全体を管理し、各ステップ内は逐次判断、最後にリフレクションで検品」です。

### 詳細: 計画は「作る」より「維持する」が難しい

事前計画の失敗の大半は、計画を立てたのに使われなくなることです。

- 計画がテキストとして履歴の彼方に流れ、モデルが参照しなくなる → 計画を**構造化された作業状態**として保持し、毎ターン見える位置に置きます([メモリと状態管理](memory-and-state.md) の作業状態)
- 前提が変わったのに計画が固定のまま → **見直しトリガー**(ステップ完了ごと / 失敗時 / 新情報の発見時)を決めておきます

### 詳細: モデル側の推論機能との関係

2026 年時点では、主要な LLM の多くが、応答の前に内部で長い思考を行う推論モード(拡張思考(extended thinking)などと呼ばれます)を備えています。これは**個々の判断の質**を上げる機能であり、複数ステップの構造を管理するプランニングを置き換えるものではありません。

- 難しい単発の判断(設計の比較、バグの分析)→ 推論モードの強化が効く
- 数十ステップのタスク管理 → 明示的な計画と状態管理が効く

### 設計判断: 計画をどこに持たせるか

1. **モデル任せ(計画なし)** — 数ステップで終わるタスクに計画は過剰装備です。逐次判断で十分です
2. **プロンプトで誘導** — 「まず手順を列挙してから着手する」という指示だけでも、中規模タスクでは効果があります
3. **構造として強制** — 計画の作成・更新を専用ステップやツール(TODO 管理)として分離し、計画を構造化データとして持ちます。長時間・高信頼が必要なタスク向けです

タスクの規模に対して 1 段強い程度を選び、足りない証拠が出てから強めるのが原則です。

## 実務での注意点

### アンチパターン

- **最初の計画を絶対視して見直さない** → 実行中に前提が崩れても計画どおり突き進み、無意味な作業を積む → 見直しトリガー(ステップ完了・失敗・新情報)を計画に組み込む
- **計画を立てさせるが参照させない** → 履歴に流れて形骸化し、逐次判断と同じ挙動に戻る → 計画を作業状態として保持し、毎ターンの入力に含める
- **リフレクションに上限を置かない** → 「もう少し良くできる」で回り続けてコストが数倍になる → 回数上限と「改善幅がしきい値未満なら終了」の条件を置く
