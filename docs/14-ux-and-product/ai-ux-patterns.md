# AI プロダクトの UX パターン

## この記事の目的

非決定的な AI システムに固有の UX 課題 — 不確実性・誤り・待ち時間・信頼 — に、名前の付いたパターンで対処できるようになります。不確実性の提示・誤りと訂正の導線・待ち時間の意味付け・信頼の較正・自動化レベルの提示・初回体験の期待値設定を、実装以前の設計判断として持ち帰れる状態を目指します。

## 対象読者

- AI 機能を持つプロダクトの UX を設計するデザイナー・プロダクトマネージャー・フロントエンドエンジニア
- 「動くのに使われない/信用されない」を、体験の設計で解きたい実装者

## 前提知識

- [ストリーミングと Agent の UX 実装パターン](../03-implementation/streaming-and-agent-ux.md) — 進捗・ストリーミング・中断の実装(本記事はその上位のパターン言語)
- [信頼度と較正](../04-evaluation/confidence-and-calibration.md) — 確信度と正解率の一致(不確実性提示の裏付け)

## 本文

### 概要: 非決定的なシステムの UX は「別物」

従来の UI は、同じ操作に同じ結果を返す**決定的**なシステムを前提に発展してきました。AI プロダクトはそうではありません。同じ入力でも出力がぶれ、時に自信満々に間違え、応答に時間がかかります。この非決定性ゆえに、従来のデザイン原則がそのまま通用しない、独立した設計領域になります。

本記事は、この領域で繰り返し現れる課題に**名前の付いたパターン**で対処するためのカタログです。実装(ストリーミング・進捗の出し方)は [ストリーミングと Agent の UX 実装パターン](../03-implementation/streaming-and-agent-ux.md) が正本で、ここでは「何を・なぜ見せるか」の設計を扱います。共通する狙いは、**適切な信頼**を築くこと — 過信も不信も避けることです。

### 不確実性の提示(確度・出典・「わからない」)

AI の出力には必ず不確実性があります。それを**隠さず、しかし過剰に不安にさせず**見せるのが第一のパターンです。

- **確度を示す**: 根拠が薄い・情報源が食い違う出力は、断定を避けるか確度を下げて示します([信頼度と較正](../04-evaluation/confidence-and-calibration.md))。ただし数値の確信度(「87%」)は、較正されていないと誤解を招くため、質的な表現(「参考程度」「確度が低い」)のほうが安全なことが多いです
- **出典を添える**: 事実を述べるなら、根拠へ辿れるようにします。出典は「飾り」ではなく、利用者が自分で検証するための導線です
- **「わからない」を言える UI にする**: AI が答えられないとき、もっともらしい作り話をさせず「わかりません/確認が必要です」と言える設計にします。この選択肢を UI とプロンプトの両方で用意することが、信頼の土台です

### 誤りと訂正の UX(修正のしやすさが信頼を作る)

AI は必ず間違えます。**間違えない**ことより、**間違えても直しやすい**ことが、信頼できるプロダクトの条件です。

- **訂正を安く・可逆にする**: 出力を修正・やり直し・取り消しできる導線を、目立つ形で用意します。「間違えたら簡単に直せる」と分かると、利用者は安心して使えます
- **フィードバック導線を体験に組み込む**: 良い/悪いの評価、修正内容の記録を、邪魔にならない形で置きます。これは体験であると同時に、[フィードバックループ](../05-operations/feedback-loops.md)の入口として改善データになります
- **誤りを責任転嫁しない**: 「AI が言ったこと」と突き放さず、プロダクトとして誤りに向き合う姿勢(訂正のしやすさ・報告への応答)を見せます

### 待ち時間の設計(進捗の意味付け)

AI の応答には時間がかかります。待ち時間そのものを消せないなら、**待ち時間に意味を持たせます**。

- **何をしているかを見せる**: 「考えています」より「資料を検索しています → 3 件見つかりました → まとめています」のように、進捗に意味を持たせると、同じ待ち時間でも体感が変わります(実装は [ストリーミングと Agent の UX 実装パターン](../03-implementation/streaming-and-agent-ux.md))
- **早く一部を返す**: 全部が終わるのを待たせず、確定した部分から順に見せます(ストリーミング)。最初の反応が早いほど、体感の待ち時間は短くなります
- **長い処理は投げて待たせない**: 数十秒以上かかる処理は、その場で待たせず「終わったら知らせる」形([チャットを超える UI](beyond-chat-ui.md)のバックグラウンド実行)に逃がすことを検討します

### 信頼の較正(過信も過小評価もさせない)

UX の最終目標は、利用者に**適切な信頼**を持ってもらうことです。過信は事故を、不信は不使用を招きます。

- **過信させない**: 流暢で自信ありげな出力は、実力以上に信頼されがちです。重要な判断では「確認してください」「これは下書きです」と、鵜呑みを防ぐ枠付けをします
- **過小評価もさせない**: 逆に、免責を過剰に並べると「使えないもの」に見え、有用な機能が使われません。能力と限界を**正直に**示し、できることは自信を持って提示します
- **一貫性で信頼を積む**: 同じ質の体験を繰り返すことで信頼は積み上がり、1 回の派手な失敗で崩れます。派手さより一貫性を優先します

### 自動化レベルの提示と切替

AI の関与度(提案するだけか、実行までするか)を、利用者が**見て・選べる**ようにするパターンです。

- **段階を明示する**: 「提案 → 承認して実行 → 自動実行」という自動化の段階を、利用者に分かる形で示します([Human-in-the-Loop 設計](../02-architecture/human-in-the-loop.md)の承認 UX 面)。今どの段階かが曖昧だと、勝手に実行されたと感じさせます
- **利用者が段階を選べる**: 慣れるにつれて自動化を上げたい人、常に確認したい人がいます。自動化レベルを利用者が調整できると、信頼の度合いに合わせて使えます
- **不可逆操作は承認を厚く**: 取り消しにくい操作ほど、承認を挟み、何が起きるかを事前に見せます

### 初回体験(能力と限界の期待値設定)

最初の体験が、その後の使い方と信頼を決めます。**期待値を正しく設定**するパターンです。

- **できること・できないことを最初に伝える**: 万能に見せると、できないことに当たったとき失望が大きくなります。得意な使い方を具体例で示し、限界も正直に伝えます
- **成功体験へ最短で導く**: 最初の一手で価値を感じてもらえるよう、例・テンプレート・サンプルを用意します(白紙のテキストボックスは初心者に厳しい — [チャットを超える UI](beyond-chat-ui.md))
- **失敗しても立て直せる導線**: 初回で外しても、やり直し方・別の使い方へ導く経路を用意します

## 実務での注意点

### アンチパターン

- **不確実性を隠して断定的に見せる** → 過信され、誤りが実害になる → 確度・出典・「わからない」を見せ、鵜呑みを防ぐ枠付けをする
- **数値の確信度をそのまま出す** → 較正されておらず誤解を招く → 質的な表現を優先し、較正できる場合のみ数値を使う
- **訂正・やり直しの導線が貧弱** → 間違いを直せず信頼を失う → 修正・取り消しを安く可逆にし、目立つ形で置く
- **待ち時間に意味がない(ただ回るスピナー)** → 体感が長く不安になる → 進捗に意味を持たせ、早く一部を返す
- **免責を過剰に並べる** → 使えないものに見えて機能が使われない → 能力と限界を正直に、できることは自信を持って示す
- **自動化レベルが不透明** → 勝手に実行されたと感じさせる → 段階を明示し、利用者が選べるようにする
- **白紙のテキストボックスで放り出す** → 初回で価値に辿り着けず離脱する → 例・テンプレートで成功体験へ最短で導く

### チェックリスト

- [ ] 不確実性(確度・出典・「わからない」)を、過剰に不安にさせず見せているか
- [ ] 数値の確信度は較正できているか(できないなら質的表現にしているか)
- [ ] 訂正・やり直し・取り消しを安く・可逆にし、フィードバック導線を組み込んだか
- [ ] 待ち時間に意味を持たせ、確定部分から早く返しているか
- [ ] 過信も過小評価もさせない枠付け(鵜呑み防止と正直な能力提示)をしているか
- [ ] 自動化レベルの段階を明示し、利用者が選べるようにしたか
- [ ] 初回体験で能力と限界を伝え、成功体験へ最短で導いているか
