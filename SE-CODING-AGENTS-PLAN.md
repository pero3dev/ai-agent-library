# SE-CODING-AGENTS-PLAN — システムエンジニア向け AI コーディングエージェント活用計画

> **ステータス: 設計案(2026-07-07 作成、ユーザーの追加要望に基づく。着手指示待ち)。**
> 日本の企業システム開発における**システムエンジニア(SE)**に特化して、AI コーディングエージェントの活用方法を工程横断で網羅する追加計画です。進捗の正本は着手後 [ROADMAP.md](ROADMAP.md) に置きます。

## 1. 位置づけ: 08 章(開発者視点)の上に「工程と商流の層」を足す

### 対象読者の定義

本計画の「システムエンジニア(SE)」は、日本の企業システム開発(SIer・ユーザー企業の情報システム部門)で、**実装だけでなく要件定義・設計・テスト・保守運用までの工程を担う**エンジニアを指します。特徴:

- 業務の多くがコーディング以外(設計書・レビュー・テスト・顧客調整・保守)
- 成果物にドキュメント(設計書・テスト仕様書・エビデンス)が含まれ、品質責任を契約で負う
- 環境に強い制約がある(顧客データ・秘密保持契約・持ち込みツール承認・閉域網・監査)
- 多重請負・多社協業の商流の中で権限と責任が分かれている

### 既存 08 章との関係

[docs/08-coding-agents/](docs/08-coding-agents/)(21 本)は「エージェントを使う開発者」の視点で、ツール選定・設定・依頼設計・セキュリティ・チーム導入・ツール別リファレンスを完成させています。**これらは正本のまま変更しません**。本計画は次の 2 軸で不足している応用層を足します。

| 軸 | 既存 08 章のカバー | ギャップ |
| --- | --- | --- |
| **工程** | 実装(コーディング)中心。テスト・自動化は開発フロー文脈 | **V 字モデルの各工程**(要件定義・基本/詳細設計・テスト設計・総合テスト・保守運用)での活用と限界が体系化されていない |
| **商流・環境** | [coding-agent-security](docs/08-coding-agents/coding-agent-security.md)(技術的防御)・[coding-agent-team-adoption](docs/08-coding-agents/coding-agent-team-adoption.md)(自社チーム導入) | **顧客資産を扱う制約**(秘密保持・持ち込み承認・閉域網/オンプレ・監査証跡・多重請負の責任分界)と**顧客との合意形成**が未カバー |

このほか SE 業務の中核である**レガシーコード理解・仕様復元**、**ドキュメント成果物**(Excel 設計書文化への現実解)も独立した価値があります。

### 配置

**08-coding-agents 内のサブシリーズ**(ファイル名接頭辞 `se-`)として追加することを推奨します(新セクションは作らない)。理由: 主題は同じ「コーディングエージェントの活用」であり、選定・設定・セキュリティの基礎は既存記事を前提にできるため。08 章 README の収録表に「SE 実践シリーズ」の区切り行(小見出し)を設けて視認性を保ちます。

## 3. 各ページの設計

### se-process-map.md — SE 工程別活用マップ(シリーズの背骨)

- **目的**: 要件定義から保守までの各工程で「何に使えるか・何に使えないか・何を人が握るか」を 1 枚で見渡し、自分の工程の詳細記事へ進める
- **主要トピック**: V 字モデル × 活用マップ(工程ごとの向く作業・向かない作業・リスク)/ 「成果物責任は人」という全工程共通の原則 / 工程ごとの詳細記事への導線 / ウォーターフォールとアジャイルの読み替え
- **H3 案**: 概要 / 全工程に共通する原則 / 工程別マップ(上流 → 実装 → テスト → 保守) / 自分の工程から始める
- **分担**: 実装工程の詳細 = 既存 08 章(依頼設計・実践ガイド)へ委譲

### se-requirements-and-design.md — 要件定義・設計工程での活用

- **主要トピック**: 要件の壁打ち・抜け漏れ観点出し(発散はエージェント・確定は人)/ 設計書ドラフト生成(既存コード・過去案件からの下書き)/ 設計レビュー支援(整合性チェック・観点レビュー)/ 図の生成(Mermaid での構成図・シーケンス図)/ **ドキュメント形式の壁への現実解**(Excel 設計書文化: Markdown で作って変換する・スクリーンショットからの読み取り・「正本をテキストに寄せる」段階論)/ 要件定義での機微情報の扱い(顧客名・業務データのマスキング)
- **分担**: 要件定義の方法論そのもの = [usecase-discovery](docs/09-business/usecase-discovery.md)(09 章)

### se-test-process.md — テスト工程での活用

- **主要トピック**: テスト観点出し(観点表のドラフト・境界値の列挙)/ テストケース生成(仕様書からの生成と、その検証責任)/ テストコード・テストデータ生成 / エビデンス整理の自動化 / 「エージェントが書いたコードをエージェントがテストする」自己検証の罠 / 品質保証の責任は人(生成されたテストの妥当性レビュー)
- **分担**: 開発フローの回帰テスト = [regression-testing](docs/04-evaluation/regression-testing.md)、エージェント自体の評価 = [coding-agent-evaluation](docs/08-coding-agents/coding-agent-evaluation.md)

### se-legacy-code-analysis.md — レガシーコード理解と仕様復元

- **主要トピック**: 大規模・古いコードベースの理解(構造把握・処理フロー追跡・「なぜこうなっているか」の仮説出し)/ 仕様書起こし(コード → 日本語仕様のドラフトと、その検証方法)/ 影響調査(改修範囲の洗い出し支援)/ デッドコード・重複の発見 / 古い言語・環境(COBOL・VB 等)への対応度の考え方(モデル・ツールの対応は変化が速いため確認方法を示す)/ モダナイゼーション支援(段階的移行での使いどころ)
- **注意**: 「復元した仕様」はドラフトであり正としない(現行システムの挙動が正)という原則を貫く

### se-maintenance-and-operations.md — 保守・運用フェーズでの活用

- **主要トピック**: 障害調査(ログ・スタックトレースの解析支援・再現手順の推定)/ 改修の影響調査と修正案 / 過去経緯の調査(コミット履歴・チケットからの背景復元)/ ドキュメントの追従(改修とセットで設計書を更新させる)/ 定型保守作業の自動化([coding-agent-automation-patterns](docs/08-coding-agents/coding-agent-automation-patterns.md) の保守文脈への適用)/ 本番データ・本番環境を扱う際の境界線
- **分担**: Agent システム自体の運用 = 05 章(本記事は「保守対象システムの保守に使う」話)

### se-enterprise-constraints.md — 企業システム環境の制約と対応(鮮度管理型)

- **主要トピック**: 制約の類型(顧客データ・秘密保持契約 / 持ち込みツールの承認プロセス / 閉域網・オンプレ環境 / 監査証跡・ログ保全 / 多重請負での責任分界)/ 提供形態の選択肢マップ(SaaS 型 + データ保護契約・学習オフ / クラウド経由(ゲートウェイ・リージョン固定)/ 自社ホスト(OSS エージェント + ローカル/専有モデル)— 2026-07 時点の一般類型)/ 顧客コード・データを渡してよいかの判断フレーム(契約 → 分類 → 経路)/ 承認を取るための説明資料の型(データフロー・残留・学習利用の 3 点)
- **鮮度**: 各ツールのエンタープライズ提供形態・データ残留ポリシーは変化が速いため、最終確認日 + 定点観測の 3 点セットを必須とし、調査 SE-R1 を執筆前に実施
- **分担**: 技術的防御(権限・サンドボックス)= [coding-agent-security](docs/08-coding-agents/coding-agent-security.md)、規制の地図 = [industry-regulations-map](docs/09-business/industry-regulations-map.md)

### se-client-adoption.md — 顧客・プロジェクトへの導入合意形成

- **主要トピック**: 誰の合意が要るか(顧客・元請・自社・監査)と何を示すか / 契約観点(成果物の品質責任・知的財産・データの取り扱い条項)/ 見積り・工数モデルへの影響(効率化を単価にどう反映するかの論点整理 — 結論は出さず観点を示す)/ 小さく始める型(社内検証 → 顧客非依存作業 → 顧客合意のうえ本適用)/ 効果の示し方([roi-and-business-case](docs/09-business/roi-and-business-case.md) の SE 文脈への適用)
- **分担**: 自社チームへの展開 = [coding-agent-team-adoption](docs/08-coding-agents/coding-agent-team-adoption.md)(本記事は顧客・商流側)

## 4. スコープ外(検討のうえ除外)

- **特定 SIer・特定顧客の社内規程への適合手順**: 一般類型と判断フレームまで(個別規程は各社の正本に従う)
- **法務・契約文言の雛形**: 観点の列挙まで(契約は法務の領分。[industry-regulations-map](docs/09-business/industry-regulations-map.md) と同じ「入口マップ」原則)
- **Excel/Word 操作の自動化ツール個別解説**: 「正本をテキストに寄せる」原則と変換の型まで
- **プロジェクト管理手法(WBS・進捗管理)への AI 適用一般論**: コーディングエージェントの活用に限定する
