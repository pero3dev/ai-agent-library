# よくあるアンチパターン集(横断)

## この記事の目的

本ライブラリの各記事に分散しているアンチパターンのうち、**特に頻度と被害が大きいものをライフサイクル順に一覧化**し、設計レビュー・実装レビュー・振り返りの場でそのまま使えるようにします。各項目の詳細と処方箋は、リンク先の記事が正です。

## 対象読者

- Agent プロジェクトの設計・コードレビューを行うエンジニア・テックリード
- 「うちのプロジェクトが今どの罠にはまっているか」を短時間で棚卸ししたい人

## 本文

### 概要: アンチパターンの 3 つの共通根

個々の失敗は多様に見えますが、根をたどると多くが次の 3 つに行き着きます。

1. **自律性をタダだと思う** — 自律性はコスト・レイテンシ・不確実性・攻撃面で支払う「買い物」です。必要最小限に絞るという判断が抜けると、企画から運用まで全段階で罠が生まれます([Workflow 型 vs Agent 型の使い分け](../02-architecture/workflow-vs-agent.md))
2. **従来ソフトウェアの直感を持ち込む** — 決定的なコードの直感(1 回テストすれば OK、変更の影響は局所的、エラーは例外で現れる)は、非決定的で非局所的な LLM には通用しません
3. **モデルへの指示を強制と混同する** — プロンプトに書いた禁止・注意は「お願い」であり、コードで強制される制御([ガードレール](../06-security/guardrails.md))とは別物です

以下の一覧で自プロジェクトを点検するときも、個別の症状より「どの根から生えているか」を見ると、対策がまとまります。

### 詳細: 企画・設計段階

| アンチパターン | なぜ問題か | 処方箋 |
| --- | --- | --- |
| とりあえず Agent 化 | 固定パイプラインで足りるタスクに自律性のコスト(不安定・高価・遅い)だけを支払う | 設計判断フローを通す([Workflow 型 vs Agent 型の使い分け](../02-architecture/workflow-vs-agent.md)) |
| 自律性の上限を決めずに作り始める | 暴走時の影響範囲が見積もれない | 停止条件・ツール権限を最初に設計([AI Agent とは何か](../01-concepts/what-is-an-ai-agent.md)) |
| デモの成功率で本番投入を判断 | デモは成功しやすい入力に偏っており、実運用の分布とずれる | 投入前に評価設計([Agent 評価の基礎](../04-evaluation/agent-evaluation-basics.md)) |
| 初手からマルチエージェント | 複雑さ・コスト・デバッグ困難が先に来る。多くは単一 Agent で足りる | 単一で限界を確認してから分割([シングルエージェントとマルチエージェント](../01-concepts/single-vs-multi-agent.md)) |

### 詳細: 実装段階

| アンチパターン | なぜ問題か | 処方箋 |
| --- | --- | --- |
| ツール説明文が名前の言い換えだけ | いつ使う・使わないが伝わらず、誤用と不使用の両方が起きる | 用途・制約・形式例まで書く([ツール定義の設計](../03-implementation/tool-definition-design.md)) |
| 「念のため」全ツールを常時渡す | 選択ミスとトークンコストが増え、攻撃面も広がる | タスク種別ごとにツールセットを絞る([ツール使用](../01-concepts/tool-use.md)) |
| 停止条件がモデル任せだけ | 「完了」を出し損ねると無限ループ・コスト暴走 | 最大ステップ数・タイムアウト・コスト上限をコード側に置く([Agent ループ](../01-concepts/agent-loop.md)) |
| プロンプトで直せない問題をプロンプトで直し続ける | ツール設計・アーキテクチャ起因の問題は指示の追加では直らず、プロンプトが肥大化する | 問題の切り分け表で原因の層を特定([Agent 向けプロンプト設計](../03-implementation/agent-prompt-design.md)) |
| 生のツールログをユーザーに流す | 読めない上に、内部データ・秘密情報の画面流出になる | ユーザー語彙への変換層を挟む([ストリーミングと Agent の UX 実装パターン](../03-implementation/streaming-and-agent-ux.md)) |
| フレームワークをデモの豪華さで選ぶ | 可観測性・再開・保守というつらい部分はデモに映らない | 選定軸チェックリストで要件と照合([フレームワーク選定ガイド](../03-implementation/framework-selection.md)) |

### 詳細: 評価段階

| アンチパターン | なぜ問題か | 処方箋 |
| --- | --- | --- |
| 平均スコア 1 本で品質管理 | 数値が動いても理由が分からず、行動につながらない | ケース単位の合格/不合格と失敗一覧を一次情報に([Agent 評価の基礎](../04-evaluation/agent-evaluation-basics.md)) |
| 検証なしで judge を信頼 | judge の誤判定に向かって本体が「最適化」されていく | 人手ラベルとの一致率を測ってから使う([LLM-as-a-Judge](../04-evaluation/llm-as-a-judge.md)) |
| 期待する軌跡との完全一致を要求 | 正しい別解を不合格にし、変更のたびに大量に壊れる | 不変条件(安全性・マイルストーン・上限)だけを縛る([軌跡(trajectory)評価](../04-evaluation/trajectory-evaluation.md)) |
| 全ケースを E2E で毎 PR 実行 | コストと時間が破綻し、やがて誰も回さなくなる | レイヤ分けとトリガ設計([回帰テストと CI 組み込み](../04-evaluation/regression-testing.md)) |

### 詳細: 運用段階

| アンチパターン | なぜ問題か | 処方箋 |
| --- | --- | --- |
| 最終応答だけをログする | 「たまに変」を再現も分析もできない | ループの全ステップを構造化記録([可観測性とトレーシング](../05-operations/observability-and-tracing.md)) |
| 月末の請求書でコストに気づく | 原因特定も対策もできない | タスク単位の計測と日次アラート([コスト管理](../05-operations/cost-management.md)) |
| モデルをエイリアス参照で本番運用 | ベンダー側の更新で無告知に挙動が変わる | スナップショットにピン留めし、更新は計画タスクに([バージョニング・デプロイ・モデル更新追従](../05-operations/versioning-and-model-updates.md)) |
| 止める手段がデプロイしかない | 封じ込めまでの間、被害(コスト・誤操作)が進行し続ける | 粒度別のキルスイッチを事前に用意・演習([インシデント対応](../05-operations/incident-response.md)) |

### 詳細: セキュリティ

| アンチパターン | なぜ問題か | 処方箋 |
| --- | --- | --- |
| モデルの拒否を防御として当てにする | 安全学習は迂回されうる。保証ではない | 決定的な制御(権限・承認・ガード)を主防御に([Agent の脅威モデル概観](../06-security/threat-model-overview.md)) |
| プロンプトの禁止指示で「対策済み」 | 指示はインジェクションで上書きされうる「お願い」 | コードで強制されるガードレールを実装([ガードレール](../06-security/guardrails.md)、[プロンプトインジェクション](../06-security/prompt-injection.md)) |
| 全権のサービスアカウントで Agent を動かす | 混乱した代理人問題。ユーザーが権限外のデータに Agent 経由で届く | ユーザー権限の継承と実装側の認可([ツール権限設計とサンドボックス](../06-security/tool-permissions-and-sandboxing.md)) |
| 致命的三重奏を 1 つの Agent に揃える | 間接インジェクション 1 発でデータ持ち出しが構造的に可能になる | どれか 1 つを外すか、送信前承認で断つ([データ漏えい対策](../06-security/data-exfiltration.md)) |

### 設計判断: この一覧の使い方

- **設計レビューの冒頭 10 分**で、企画・設計段階とセキュリティの表を通します。この 2 つは後から直すコストが最も高い層です
- **実装・評価・運用の表はフェーズゲート**(実装完了時・投入前・運用開始 1 か月後)で見直します
- 各項目は「発見したら該当記事のチェックリストへ飛ぶ」入口として使います。この一覧自体は診断用で、処方の詳細は各記事が正です

## 実務での注意点

### アンチパターン

この記事自体がアンチパターン集のため、ここでは**この一覧の使い方**に関するメタなアンチパターンを挙げます。

- **チェックを儀式化する** → 「全部確認済み」のスタンプだけが目的になり、実態と乖離する → 各項目で「該当しない理由」を一言書ける状態をもって確認済みとする
- **一覧を網羅と誤解する** → ここにあるのは頻度と被害で選んだ代表例であり、すべてではない → 各セクションの記事のアンチパターン・チェックリストまで降りる
- **「うちは大丈夫」を口頭で済ませる** → 判断の根拠が残らず、構成変更後に再点検されない → 脅威モデルや評価と同様、点検結果を文書化し構成変更時に見直す
