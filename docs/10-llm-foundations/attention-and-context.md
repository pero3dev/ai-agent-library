# 注意機構とコンテキストウィンドウの仕組み

## この記事の目的

「なぜ長いコンテキストは遅く・高く・品質が落ちるのか」「なぜプロンプトキャッシュは前方一致なのか」を、注意機構(attention)と KV キャッシュ(KV cache)の仕組みから説明できるようになります。[コンテキストエンジニアリング](../02-architecture/context-engineering.md)が示す実務原則(厳選する・固定部分を先頭に置く)の「なぜ」を埋める記事です。

## 対象読者

- コンテキスト設計・プロンプトキャッシュ・レイテンシ最適化を、根拠を持って行いたいエンジニア
- 「コンテキストに入れれば読んでくれる」という直感が実際と合わない理由を知りたい人

## 前提知識

- [LLM はどうやってテキストを生成するか](how-llms-generate-text.md) — 自己回帰生成のループ
- [トークナイザとトークン経済](tokenization.md) — トークンという単位

## 本文

### 概要: 「全部が全部を見る」仕組みとその請求書

Transformer 系 LLM の中核である注意機構は、**各トークンが、文脈中の他のすべてのトークンとの関連度を計算し、関連の強い情報を集めて自分の表現を作る**仕組みです。文脈全体を一度に「見渡せる」ことが LLM の言語能力の源泉ですが、「全部が全部を見る」計算は文脈が長くなるほど急速に重くなります。コンテキストウィンドウの上限も、長文入力の割増価格も、長文での品質劣化も、この構造の請求書です。

### 注意機構の直感

文中の「それ」が何を指すかを理解するには、離れた位置の名詞を参照する必要があります。注意機構はこれを、すべてのトークンペアの間で行います — 各トークンが「自分に関係するのはどのトークンか」を重み付けし、重みに応じて情報を混ぜ合わせます。この処理を何層も重ねることで、語 → 句 → 文 → 文書という抽象度の積み上げが起きます。

実務者が持ち帰るべき直感は 2 つです。

1. **参照は「機会」であって「保証」ではない**: すべてのトークンを見られることと、その情報を正しく使うことは別です。重み付けは学習された傾向であり、長い文脈では重要な情報への重みが薄まりえます
2. **計算量は文脈長に対して急速に増える**: 素朴には「トークン数の 2 乗」に比例するペア計算が発生します(実装上はさまざまな工夫で緩和されますが、長いほど重い、という基本構造は変わりません)

### コンテキスト長のコスト構造

自己回帰生成([LLM はどうやってテキストを生成するか](how-llms-generate-text.md))と組み合わせると、レイテンシの構造が見えてきます。

| 段階 | 何が起きるか | 効く指標 |
| --- | --- | --- |
| プリフィル(prefill) | 入力全体を一括処理し、各トークンの中間表現を計算する | 最初のトークンが出るまでの時間(TTFT)。**入力長に依存** |
| デコード(decode) | 1 トークンずつ生成する。各ステップで過去全トークンの中間表現を参照 | トークン毎秒。**文脈が長いほど 1 トークンが重い** |

「入力を短くすると応答開始が速くなる」「長い会話ほど生成がもたつく」という体感は、この 2 段階の構造そのものです([レイテンシ最適化](../05-operations/latency-optimization.md))。また、長コンテキストは計算だけでなくメモリ(後述の KV キャッシュ)も比例して消費するため、プロバイダーが長文入力に割増価格を設定する背景にもなっています。

### KV キャッシュとプロンプトキャッシュ

デコードの各ステップで過去全トークンの中間表現を毎回計算し直すと、生成は非現実的に遅くなります。そこで、各トークンについて一度計算した中間表現(Key と Value)を保存して再利用します。これが KV キャッシュで、**プロンプトキャッシュの実体**でもあります。

- **なぜ「前方一致」なのか**: あるトークンの中間表現は、それより前の全トークンに依存して計算されます。つまり**先頭からの並びが 1 トークンでも変われば、それ以降の中間表現はすべて無効**です。プロンプトキャッシュが「前方一致した部分だけ」再利用できるのは、この依存構造の直接の帰結です
- **設計原則の由来**: 「固定部分(システムプロンプト・ツール定義)を先頭に、可変部分(ユーザー入力・検索結果)を後ろに」というキャッシュ設計の定石([コスト管理](../05-operations/cost-management.md))は、前方一致を最大化するための配置です。タイムスタンプや乱数を先頭部分に埋め込むと、毎回キャッシュを壊します
- **エージェントとの相性**: Agent ループでは「システムプロンプト + ツール定義 + 積み上がる履歴」という前方安定・後方追記の構造が自然に成立するため、キャッシュが特に効きます([コーディングエージェントのコスト最適化](../08-coding-agents/coding-agent-cost-optimization.md)はこの応用です)

### 長文での品質劣化

コンテキストに「入る」ことと「使われる」ことは別です。長い文脈では、次の劣化が広く観測されています。

- **位置による想起の偏り**: 文脈の先頭と末尾に比べ、中間に置かれた情報の想起が弱くなる傾向(いわゆる lost in the middle)が知られています。重要な指示・情報を大量の資料の真ん中に埋めない、という配置原則の根拠です
- **希釈と干渉**: 無関係な情報が増えるほど、注意の重みが分散し、似た情報同士が干渉します。「念のため全部入れる」は品質に対して中立ではなく、マイナスに働きえます
- **「入る」と「解ける」のギャップ**: 単純な検索(文脈中の一節を見つける)は長文でも高精度でも、複数箇所の情報を統合する推論は同じ長さで劣化する、という非対称があります。長文ベンチマークの好成績を「長文なら何でも大丈夫」と読まないことです

これらが、[コンテキストエンジニアリング](../02-architecture/context-engineering.md)の中心原則 — 入れられるだけ入れるのではなく、必要な情報を選んで渡す — の理論的な裏付けです。長期記憶を「全部コンテキストに置く」のではなく「検索して都度取り出す」設計([長期記憶の実装](../03-implementation/long-term-memory-implementation.md))も同じ根拠に立っています。

### この理解が効く場面

- **コンテキスト予算の判断**: 「この資料を丸ごと入れるか、要約・検索に変えるか」を、コスト(プリフィル・キャッシュ)と品質(希釈・中間の弱さ)の両面から判断できます
- **キャッシュ設計**: プロンプト構造を前方一致最大化の観点でレビューできます(固定部を先頭に・可変要素を混ぜない)
- **レイテンシの切り分け**: 「応答開始が遅い(プリフィル・入力側)」と「生成が遅い(デコード・出力側)」を区別して対策できます([レイテンシ最適化](../05-operations/latency-optimization.md))

## 実務での注意点

### アンチパターン

- **「コンテキストに入れたのだから読んでいるはず」と考える** → 中間位置の希釈・干渉で重要情報が使われないことがある → 重要情報は先頭か末尾に置き、量を厳選する
- **タイムスタンプ・セッション ID などの可変値をプロンプト先頭部に埋め込む** → 前方一致が毎回壊れ、キャッシュが一切効かなくなる → 可変値は可変部分(後方)へ移す
- **長文ベンチマークの好成績を「長文処理は解決済み」と読む** → 単純検索と統合推論では劣化の仕方が違う → 自社タスクの長さ・構造で実測する
- **レイテンシ対策を出力側だけで行う** → 応答開始の遅さは入力長(プリフィル)由来のことが多い → TTFT と生成速度を分けて計測・対策する

### チェックリスト

- [ ] プロンプトが「固定部分が先頭・可変部分が後方」の構造になっている
- [ ] キャッシュの効き(ヒット率・課金上の割引)を計測している
- [ ] 重要な指示・情報を長い資料の中間に埋めていない
- [ ] コンテキストに入れる情報の取捨選択基準がある(「全部入れる」をやめた)
- [ ] TTFT(入力側)とトークン毎秒(出力側)を分けて監視している
