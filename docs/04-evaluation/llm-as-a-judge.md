# LLM-as-a-Judge

## この記事の目的

コードでは採点できない開放的な出力を LLM に採点させる手法(LLM-as-a-Judge)を、**判定形式の設計 → プロンプト構成 → バイアス対策 → judge 自体の検証** の順で信頼できる形に組み立てられるようになります。「judge のスコアを信じてよいか」を判断する基準を持てるようになります。

## 対象読者

- 要約・回答・提案文など、正解が一意でない出力の品質を自動評価したいエンジニア
- judge の導入を検討しており、その信頼性の担保方法を知りたいテックリード

## 前提知識

- [Agent 評価の基礎](agent-evaluation-basics.md) — 採点方法 3 分類(コード・judge・人手)の中での位置づけ
- [構造化出力](../03-implementation/structured-output.md) — 判定結果を機械可読に受け取る方法

## 本文

### 概要: judge は「もう 1 つの LLM アプリケーション」

LLM-as-a-Judge は、評価対象の出力(と判定に必要な文脈)を別の LLM 呼び出し(judge)に渡し、あらかじめ定義した基準で判定させる手法です。人手評価をスケールさせる代替として、2026 年時点で開放的な出力の自動評価の主流になっています。

最初に持つべき心構えは、**judge はそれ自体が 1 つの LLM アプリケーションである**ということです。プロンプト設計を誤れば誤った判定を量産し、その誤った判定に向かって本体が「最適化」されていきます。「作って信じる」のではなく「検証してから使う」が本記事の主題です。

### 詳細: 判定形式の設計

| 形式 | 内容 | 向く場面 |
| --- | --- | --- |
| 二値判定(合格/不合格) | 基準を満たすか否かだけを判定する | ほとんどの場合の推奨。基準が明確になり、集計がそのまま成功率になる |
| ペア比較(A/B) | 2 つの出力のどちらが良いかを判定する | プロンプト・モデル変更の比較。絶対基準を書き下しにくい品質軸 |
| 数値スコア(1〜5 点等) | 程度を数値で出させる | 非推奨寄り。各点数の基準が曖昧になり、判定が特定の値に偏りやすい |

実務の出発点は **観点ごとの二値判定** です。「総合品質を 10 点満点で」のような 1 本のスコアではなく、「事実性(入力にない情報を追加していないか)」「指示遵守」「網羅性」のように観点を分解し、**1 judge 1 観点**で判定させます。観点が分かれていれば、不合格の理由がそのまま改善対象になります。

### 詳細: judge プロンプトの構成

1. **基準を具体的に書きます** — 「良い応答か」ではなく「入力に含まれない事実を追加していないか」。合格・不合格の境界例を少数含めると判定が安定します
2. **判定に必要な文脈を渡します** — 元のタスク・参照すべき資料・会話履歴。「出力だけを見てこの観点を判定できるか」を自問し、できないなら文脈が不足しています
3. **理由 → 判定の順で出力させます** — 先に判定理由を書かせると判定の質が上がり、不一致の原因分析もできるようになります
4. **結果は構造化出力で受けます** — 例: `{"reasoning": "...", "verdict": "pass"}`([構造化出力](../03-implementation/structured-output.md))

### 詳細: 既知のバイアスと対策

| バイアス | 現象 | 対策 |
| --- | --- | --- |
| 位置バイアス | ペア比較で提示順の一方(多くは先)を優遇する | 順序を入れ替えて 2 回判定し、一致した場合のみ結果を採用する |
| 長さバイアス | 長く詳しい出力を過大評価する | 「簡潔さ」を独立の観点に分離する。基準に長さへの言及を明示する |
| 自己選好バイアス | 自分(同系統のモデル)の出力を優遇する | 生成側と judge に別系統のモデルを使う。特にペア比較で注意する |
| 寛大さバイアス | 迷ったら合格に倒す | 不合格の基準と境界例を明文化して与える |

### 設計判断: judge を検証してから使う

judge の判定が人間の判断と一致することを、運用に載せる前に確認します。

1. 評価対象の出力 20〜50 件に、人手で合格/不合格のラベルを付けます(基準の曖昧さの多くはこの作業中に発覚します)
2. 同じ 20〜50 件を judge に判定させ、人手ラベルとの一致率を測ります。一致率だけでなく**不一致の内訳**(不合格の見逃しか、過剰検出か)を見ます
3. 不一致ケースを読み、judge プロンプト、または基準そのものを修正します
4. 一致率が実用水準(目安として 9 割前後。観点の重要度で調整)に達してから、評価スイートに組み込みます

judge プロンプトはチームの資産です。バージョン管理し、変更したら同じ人手ラベルセットで再検証します。judge 自体が [回帰テストと CI 組み込み](regression-testing.md) の対象です。

### 設計判断: judge を使わない場所を残す

judge は遅く、高く、非決定です。コードで判定できるもの — 形式・スキーマ準拠・必須要素の有無・最終状態の検証 — を judge に回してはいけません([Agent 評価の基礎](agent-evaluation-basics.md) の採点方法 3 分類)。「コードで採れる部分を削ぎ落とし、判断が必要な観点だけを judge に残す」のが正しい分担です。

## 実務での注意点

### アンチパターン

- **検証なしで judge のスコアを信頼する** → judge の誤判定が評価全体を汚染し、誤った方向にプロンプトが「改善」されていく → 人手ラベルとの一致率を測ってから使う
- **「総合品質を 10 点満点で」と聞く** → 各点数の基準がなく再現性も低い。7 点と 8 点の差は行動につながらない → 観点別の二値判定に分解する
- **判定だけ出させて理由を出させない** → 人手ラベルと不一致だったときに原因分析ができず、judge を改善できない → 理由 → 判定の順で構造化出力に含める
- **コードで測れるものまで judge に回す** → コスト・時間・不確実性が不要に増える → まずコード採点で削ぎ落とし、残りだけを judge に渡す

### チェックリスト

- [ ] 判定形式が観点ごとの二値判定(または手順を決めたペア比較)になっている
- [ ] judge に判定に必要な文脈(元タスク・参照資料)が渡っている
- [ ] 理由 → 判定の順で出力させ、構造化出力で受けている
- [ ] 人手ラベル 20〜50 件との一致率を測定し、不一致の内訳を確認した
- [ ] ペア比較では順序を入れ替えて判定している
- [ ] 生成側と judge のモデル系統(自己選好)を意識して選んでいる
- [ ] judge プロンプトがバージョン管理され、変更時の再検証手順が決まっている
