# レガシーコード理解と仕様復元

## この記事の目的

大規模・古いコードベースの保守や再構築で、AI コーディングエージェントを**構造把握・処理フロー追跡・仕様書起こし・影響調査・モダナイゼーション支援**に使い分けられるようになります。あわせて、この領域で最も重要な原則である「**現行システムの挙動が正、復元した仕様はドラフト**」を保ったまま作業を進める方法と、古い言語・環境への対応度を見極める確認手順を持ち帰れる状態を目指します。

## 対象読者

- ドキュメントが失われた・古いコードベースの保守や改修を担当する SE
- レガシーシステムのモダナイゼーション(段階的移行)を検討していて、AI をどこまで頼れるか見極めたいエンジニア

## 本文

### 概要: 何に使えて、何を正とするか

レガシーコードの理解は、コードから設計意図を**逆向きに読み解く**作業(リバースエンジニアリング(reverse engineering))です。エージェントは読解の速度を上げますが、**何が正しいか**を決めるのはコードとその実挙動です。

| 作業 | エージェントの役割(発散・下書き) | 何が正か・人が握ること |
| --- | --- | --- |
| 構造把握・フロー追跡 | 構造の要約・処理フローの説明 | 実コードと実挙動が正(要約は仮説) |
| 仕様書起こし | コード → 仕様ドラフト | 現行システムの挙動が正(仕様は復元案) |
| 影響調査 | 改修範囲の候補洗い出し | 影響有無の最終判断 |
| デッドコード・重複発見 | 候補の指摘 | 削除可否の判断(呼び出し実態の確認) |
| モダナイゼーション | 移行方針・変換の下書き | 移行判断・等価性の検証 |

この表を貫く原則は 1 つ、**「現行の挙動が正、生成物は仮説・ドラフト」**です。以下、各作業を見ます。

### 大規模・古いコードベースの理解

まず効くのは、**全体像をつかむ**発散の使い方です。

- **構造把握**: ディレクトリ構成・モジュール依存・主要な処理の入口を要約させ、コードベースの地図を速く作ります。人が一から読むより、当たりをつけてから深掘りできます
- **処理フローの追跡**: 「この入力がどこを通ってどう処理されるか」を説明させ、フローの下書きを得ます。分岐・例外系の見落としを拾う補助になります
- **「なぜこうなっているか」の仮説出し**: 不可解な実装について、背景の仮説を挙げさせます。ただしこれは**仮説**であり、実際の経緯(過去の障害対応・業務都合)は履歴や関係者に当たって確かめます
- **要約は仮説として扱う**: エージェントの構造要約は、読み飛ばした分岐や隠れた副作用を含みます。重要な理解は、必ず実コードで裏を取ります

### 仕様書起こし(コード → 仕様)

ドキュメントが失われたシステムで需要が高いのが、**コードから日本語仕様のドラフトを起こす**使い方です。

- **仕様ドラフトの生成**: 対象コードを渡し、処理内容・入出力・分岐条件を日本語仕様のドラフトにさせます。白紙から書き起こすより速く進みます
- **復元仕様は正としない**: 生成された仕様は、コードの**もっともらしい説明**であって、正解ではありません。エージェントは複雑な条件分岐を単純化したり、例外系を落としたりします。「復元仕様=正」とした瞬間に、実挙動とのズレが仕様として固定されます
- **検証は現行挙動と突き合わせる**: 復元仕様は、実際のシステムの入出力・テスト結果と突き合わせて検証します。仕様とコードが食い違ったら、**コードの挙動が正**です(その挙動が「バグか仕様か」の判断は別途、関係者と行います)

### 影響調査

改修時の**影響調査(impact analysis)** — 変更が波及する範囲の洗い出し — は、レガシーで特に骨の折れる作業です。

- **改修範囲の候補洗い出し**: 「この関数/テーブルを変えたら影響しそうな箇所」を列挙させ、調査の当たりをつけます。呼び出し関係・データの流れの追跡補助になります
- **候補であって結論ではない**: エージェントの洗い出しは**見落としも過剰検出もあります**。動的な呼び出し(リフレクション・文字列によるディスパッチ・設定駆動)や、コード外の連携(バッチ・DB トリガ・外部システム)は追い切れないことがあります
- **影響なしの断定を信じない**: 「影響ありそうな箇所」の提示には使えますが、「影響なし」の保証には使いません。影響有無の最終判断は、実コードの確認とテストで人が行います

### デッドコード・重複の発見

- **候補の指摘**: 使われていなさそうなコード・重複した処理を指摘させ、整理の候補を得ます
- **「使われていない」の確認が要る**: レガシーでは、動的呼び出し・条件付き実行・年次バッチのように**低頻度だが生きている**コードがあります。エージェントが「デッド」と判断しても、削除前に呼び出し実態(ログ・呼び出し元・運用スケジュール)を人が確認します
- **重複は統合方針とセットで**: 重複の指摘は有用ですが、統合してよいか(片方に固有の分岐がないか)は人が判断します

### 古い言語・環境への対応度

COBOL・VB・古いメインフレーム資産などでは、**モデル・ツールの対応度**が成否を分けます。ここは変化が速いため、断定せず確認手順で扱います。

- **対応度は事前に測る**: 対象の言語・方言・環境について、小さな実コードで「正しく読解・変換できるか」を試してから本格投入します。メジャーな言語ほど得意で、方言・独自拡張・環境依存が強いほど精度が落ちます
- **学習データの薄い領域に注意**: 世に出回るコード量が少ない言語・環境は、モデルの理解が浅く、もっともらしい誤りが増えます。「読めているように見えて要点を外す」リスクが高い領域です
- **確認方法を持つ**: 特定の言語対応が必要なら、使うツール・モデルの対応状況を公式情報と実測で確認します(下記 TODO)

### モダナイゼーション支援

**モダナイゼーション(modernization)** — 古い技術資産を新しい基盤へ移す取り組み — でも、エージェントは下書きの道具として効きます。

- **段階的移行での使いどころ**: 一括書き換えではなく、モジュール単位の変換・テスト整備・ドキュメント化を段階的に支援させます。リスクの高い全面置換より、小さく移して検証する進め方に合います
- **等価性の検証が核**: 移行の成否は「**新旧で挙動が等価か**」で決まります。エージェントの変換は下書きであり、等価性は現行挙動を正としたテスト(現行の入出力を期待値にする)で人が確かめます
- **移行判断は人**: 何をどの順で移すか、そもそも移すべきか(塩漬け・再構築・現状維持の選択)は、業務リスクと投資判断であり人が決めます。エージェントは判断材料の整理を助けるに留めます

## 実務での注意点

### アンチパターン

- **復元した仕様を「正」として扱う** → 単純化・例外系の欠落が仕様として固定される → 現行システムの挙動を正とし、復元仕様はドラフトとして検証する
- **影響調査の「影響なし」を信じる** → 動的呼び出し・コード外連携を見落として本番障害になる → 「影響ありそう」の当たり付けに使い、影響なしの判断は人がテストで行う
- **「デッドコード」を確認せず削除する** → 低頻度だが生きているコードを消す → 呼び出し実態(ログ・運用スケジュール)を確認してから削除する
- **レガシー言語の読解を無条件に信頼する** → 学習データの薄い言語で要点を外す → 小さな実コードで対応度を実測してから投入する
- **モダナイゼーションで等価性検証を省く** → 新旧で挙動がズレたまま移行する → 現行挙動を期待値にしたテストで等価性を人が確かめる

### チェックリスト

- [ ] 「現行システムの挙動が正、生成物は仮説・ドラフト」を全作業で徹底した
- [ ] 復元仕様を実際の入出力・テストで検証している
- [ ] 影響調査を「当たり付け」に使い、影響なしの判断を人がテストで行っている
- [ ] デッドコード削除の前に呼び出し実態を確認している
- [ ] 対象のレガシー言語・環境での読解/変換精度を、小さな実コードで実測した
- [ ] モダナイゼーションで新旧の等価性を、現行挙動を正としたテストで検証している
