# コード実行サンドボックスの実装

## この記事の目的

モデルが生成したコードの実行環境を、**隔離強度・起動速度・運用コスト**のトレードオフで選定・構築できるようになります。守るべき対象の整理・実装技術の類型と比較・セッションとライフサイクル・エグレス制御・リソース制限・依存の扱い・マルチテナントでの分離水準を、実装の判断として持ち帰れる状態を目指します。

## 対象読者

- データ分析・コーディングなど、モデル生成コードを実行するエージェントの実行基盤を設計・構築するエンジニア
- 「サンドボックスで隔離する」の原則の先、具体的な実装技術の選択に進みたい実装者

- [データ漏えい対策](../06-security/data-exfiltration.md) — エグレス制御が守る対象

## 本文

### 概要: 「サンドボックスを使え」の先へ

エージェントに任意のコードを実行させるなら、隔離は必須です。この「サンドボックスを使え」という原則は [ツール権限設計とサンドボックス](../06-security/tool-permissions-and-sandboxing.md) が正本です。本記事はその先——**どの実装技術で隔離を実現するか**——を扱います。

実装技術には、プロセス分離からコンテナ・軽量 VM・WebAssembly まで幅があり、それぞれ**隔離の強さ・起動の速さ・運用の重さ**が違います。唯一の正解はなく、「何から何を守るか」の要件で選びます。まず守るべき対象を整理し、次に技術を選ぶ、という順序で設計します。

### 要件の整理(何から何を守るか)

技術を選ぶ前に、守るべき対象を明確にします。守る対象で必要な隔離強度が変わります。

- **ホストを守る**: 生成コードがホスト OS を侵害・破壊しないこと。最も基本的な守りです
- **ネットワークを守る**: サンドボックスから内部ネットワーク・外部へ勝手に出られないこと(エグレス制御)。データ持ち出しの経路を塞ぎます([データ漏えい対策](../06-security/data-exfiltration.md))
- **データを守る**: 実行対象以外のデータ・秘密情報にアクセスできないこと
- **他セッション/他テナントを守る**: あるセッションのコードが、別のセッション・別の顧客のデータや処理に干渉しないこと(後述のマルチテナント)
- **生成コードは信頼しない前提**: モデルが生成したコードも、外部から注入された指示で悪意あるものになりえます。「信頼できない入力」として扱うのが出発点です

### 実装技術の類型と比較

隔離を実現する技術には、代表的な類型があります。隔離強度・起動レイテンシ・状態管理のトレードオフで比較します。

| 技術 | 隔離強度 | 起動レイテンシ | 特徴 |
| --- | --- | --- | --- |
| プロセス分離(namespace / seccomp 等) | 弱〜中 | 速い | OS 機能で権限・資源を絞る。カーネルを共有するため境界は相対的に薄い |
| コンテナ | 中 | 速い | 広く普及。標準のコンテナはカーネル共有で、任意コード実行には隔離が不足しがち |
| 強化コンテナ(ユーザー空間カーネル等) | 中〜強 | 中 | コンテナとカーネルの間に隔離層を挟み、境界を厚くする |
| microVM(軽量仮想マシン) | 強 | 中(数百 ms 級) | VM の強い隔離を、軽量・高速起動で得る。任意コード実行の定番の 1 つ |
| WebAssembly(Wasm) | 中〜強(能力ベース) | 速い | 既定で何もできず、許可した能力だけ与える。軽量だが実行できる処理に制約 |
| ブラウザ内/クライアント実行 | 用途依存 | — | 利用者の端末で実行。サーバーにコードを持ち込ませない選択肢 |
| マネージドサンドボックスサービス | 中〜強 | サービス依存 | 隔離・スケール・ライフサイクルを外部に委ねる。運用が軽い |

- **カーネル共有の弱さを理解する**: 標準的なコンテナはホストのカーネルを共有します。任意コード実行では、カーネルの脆弱性を突く攻撃(コンテナからの脱出)を想定し、**より強い隔離**(強化コンテナ・microVM)を検討します
- **強い隔離は起動が重くなりがち**: 一般に、隔離を強くするほど起動が重くなります。ただし microVM は「強い隔離を高速起動で」を狙った技術で、この両立が進んでいます
- **WebAssembly は既定拒否**: Wasm は「既定で何もできず、許可した能力だけ与える」能力ベースの隔離が特徴です。軽量で起動も速い一方、実行できる処理(ネイティブ依存・システムコール)に制約があります
- **運用コストで選ぶ**: 自前で隔離基盤を作る運用コストが重いなら、マネージドサンドボックスサービスに委ねる選択肢もあります(顔ぶれは変化が速いため後述の TODO)

### セッションとライフサイクル

サンドボックスを「使い捨て」にするか「永続ワークスペース」にするかは、重要な設計判断です。

- **使い捨て(ephemeral)**: 1 回の実行ごとに作って捨てます。前の実行の状態が残らないため、汚染や情報の持ち越しが起きにくく、隔離の観点で安全です
- **永続ワークスペース**: セッションをまたいで状態(ファイル・インストール済みパッケージ)を保持します。反復的な作業(データ分析の続き・コーディングの継続)に便利ですが、**状態が汚染・悪用されるリスク**が加わります([常駐エージェントのライフサイクル管理](../05-operations/long-running-agents.md))
- **状態の持ち越しを設計する**: 永続化するなら、何を残し・何をリセットするかを決めます。秘密情報や一時データを不用意に残さない設計にします
- **確実に破棄する**: 使い捨てでも永続でも、不要になった環境を確実に破棄し、資源とデータを残さないようにします

### ネットワークとエグレス制御

サンドボックスからの通信は、**既定で遮断**し、必要なものだけ許可します。

- **既定でネットワークを遮断する**: 実行に外部通信が不要なら、ネットワークを完全に遮断します。これがデータ持ち出しへの最も強い防御です
- **必要な宛先だけ許可する**: 通信が必要な場合も、許可リストで宛先を限定します。任意の外部への接続を許すと、致命的三重奏([Agent の脅威モデル概観](../06-security/threat-model-overview.md))が成立し、データ持ち出しが可能になります
- **内部ネットワークへの到達を防ぐ**: サンドボックスから社内ネットワーク・メタデータエンドポイントへ到達できないようにします。外向きだけでなく、内向きの経路も塞ぎます

### リソース制限と暴走対策

生成コードは、無限ループ・メモリ枯渇・大量ファイル生成などで暴走しえます。**資源に上限**をかけます。

- **CPU・メモリ・実行時間の上限**: 各実行に CPU・メモリ・時間の上限を課し、超えたら強制終了します。1 つの暴走が基盤全体を巻き込まないようにします
- **ディスク・ファイル数の制限**: ディスク使用量・生成ファイル数にも上限をかけます
- **リトライと打ち切り**: エラー時の自動リトライには回数上限を設け、直らないときは打ち切ります([ループのフィードバックと検証](loop-feedback-and-verification.md))

### パッケージ・依存の扱い

コード実行では、外部パッケージのインストールを許すかが判断点です。

- **インストールを制御する**: 任意のパッケージインストールを許すと、悪意あるパッケージ(サプライチェーン攻撃)の経路になります([AI サプライチェーンセキュリティ](../06-security/ai-supply-chain-security.md))。許可リスト・事前準備済みイメージ・プロキシ経由に絞ります
- **事前準備済み環境**: よく使う依存を含んだイメージを事前に用意すると、実行のたびのインストールを避けられ、速度と安全性の両方に効きます
- **オフライン実行**: ネットワーク遮断とセットで、依存を事前に固めたオフライン環境にすると、持ち出しとサプライチェーンの両リスクを下げられます

### マルチテナントでの分離水準

複数の顧客・部門が同じ実行基盤を共有するなら、**テナント間の分離**を設計します([マルチテナント設計](../02-architecture/multi-tenancy-and-isolation.md))。

- **テナント間を跨がせない**: あるテナントのコードが、別テナントのデータ・処理に触れないよう、隔離の境界をテナント単位でも引きます。カーネル共有のコンテナでは、テナント間の分離に不安が残ります
- **分離水準を要件で決める**: 顧客データの機密性が高いなら、テナントごとに microVM・専用環境を割り当てるなど、分離水準を上げます。共有度と分離強度はトレードオフです
- **ノイジーネイバーを防ぐ**: リソース制限を、テナント単位でもかけ、1 テナントの暴走が他テナントを巻き込まないようにします

## 実務での注意点

### アンチパターン

- **標準的なコンテナで任意コードを実行して安心する** → カーネル共有でコンテナ脱出のリスク → 任意コード実行には強化コンテナ・microVM など強い隔離を検討する
- **隔離強度だけで選び起動速度・運用を無視する** → 使い物にならない/運用が破綻 → 隔離・起動・運用コストのトレードオフで選ぶ
- **永続ワークスペースに状態を無制限に残す** → 汚染・情報の持ち越し → 何を残し何をリセットするかを設計し、確実に破棄する
- **サンドボックスからのネットワークを既定で許可する** → データ持ち出しの経路になる → 既定で遮断し、必要な宛先だけ許可する
- **リソース上限をかけない** → 暴走が基盤全体を巻き込む → CPU・メモリ・時間・ディスクに上限をかける
- **任意のパッケージインストールを許す** → サプライチェーン攻撃の経路 → 許可リスト・事前準備済みイメージ・オフラインに絞る
- **マルチテナントでカーネル共有のまま分離する** → テナント間の干渉・漏えい → 機密度に応じてテナント単位の隔離水準を上げる
