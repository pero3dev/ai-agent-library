# AI システムのカオスエンジニアリング

## この記事の目的

「フォールバックが本当に動くか」を、事故が起きてからではなく**意図的な障害注入で事前に検証する**演習を設計・運用できるようになります。プロバイダー障害・モデルの劣化応答・ツール障害を計画的に起こし、「壊れても守られる」ことを確かめられる状態がゴールです。

## 対象読者

- フォールバック・リトライを実装したが、それが本番で本当に発動するか確かめていないエンジニア・SRE
- 外部プロバイダー依存の Agent で、障害時の挙動を事前に把握しておきたいテックリード

カオスエンジニアリング(chaos engineering)は、**システムに意図的に障害を注入し、回復力を検証する**実践です。備え自体の設計は [エラー処理・リトライ・フォールバック設計](../02-architecture/error-handling-and-retries.md) が、攻撃者視点の**セキュリティ**演習は [エージェントのレッドチーミング](../06-security/red-teaming-agents.md) が正本です。本記事は「信頼性(壊れても動くか)」を意図的な障害で検証する演習を扱います。SRE のカオス工学一般論は確立領域として、AI 固有の面に絞ります。

### AI 固有の注入対象

一般的なインフラ障害(ノード停止・ネットワーク遅延)に加え、AI システムには**固有の注入対象**があります。

| カテゴリ | 注入する障害 |
| --- | --- |
| プロバイダー | API 障害・レート制限(429)・レイテンシ急増・タイムアウト |
| モデル出力 | 劣化した応答・不当な拒否・形式崩れ(壊れた JSON)・空応答 |
| ツール | ツール障害・タイムアウト・想定外のエラー返却 |
| 知識源 | 検索の空振り・知識源の欠落・古い情報の返却 |

これらは AI システムで実際に起きるのに、正常系のテストでは再現されません。「モデルが形式崩れを返したらパースが落ちる」「プロバイダーが 429 を返したらフォールバックが発動するはずが実は落ちる」を、**注入して確かめます**。

### 仮説駆動の演習設計

カオス演習は「とりあえず壊す」ではなく、**仮説を立てて検証**します(セキュリティのレッドチーミングと同型の演習運営。[エージェントのレッドチーミング](../06-security/red-teaming-agents.md))。

- **仮説の形**: 「**X が落ちても Y が守られる**」。例: 「主モデルが 429 を返しても、フォールバックモデルで応答が返り、成功率は 5% 以上落ちない」
- **定常状態を定義する**: 障害注入の前に「正常なら SLI がこの水準」を決めます([AI 品質の SLO 設計](ai-slo-design.md))。注入で SLI がどう動くかを見ます
- **影響範囲を絞る**: 最初は 1 つの障害を、小さな範囲で注入します。複数を同時に注入すると、何が原因か切り分けられません
- **成功と失敗の基準を先に決める**: 「フォールバックが発動し SLI が閾値内に収まれば合格、そうでなければ弱点発見」を演習前に定義します

### 実施環境の選択

障害注入は、いきなり本番でやりません。**段階的に忠実度を上げます**。

- **評価環境で始める**: モック・状態付きサンドボックス([エージェント評価環境の構築](../04-evaluation/evaluation-environments.md))で障害を注入すれば、ユーザー影響ゼロでフォールバックの発動を確かめられます。多くの弱点はここで見つかります
- **ステージングで確かめる**: 本番相当の構成で、より現実的な障害(実プロバイダーのレート制限相当)を注入します
- **本番の限定実験**: 副作用のない範囲・小さなトラフィックで、本番の実際の依存関係を試します。本番でやるのは「評価環境では再現できない実依存の挙動」に限ります。副作用のあるツールを本番注入しない設計が前提です

### 定常化: ゲームデーと自動注入

一度きりの演習は忘れられます。**定常的に回す**ことで、劣化した回復力を検知し続けます。

- **ゲームデー**: 定期的に、チームで計画的な障害注入演習を行います。手順書どおりに縮退・復旧できるかを、人も含めて訓練します(キルスイッチの発動演習と同じ発想。[インシデント対応](incident-response.md))
- **自動注入**: 成熟したら、CI や本番で継続的に小さな障害を注入し、フォールバックが常に動くことを自動で確かめます
- **変更のたびに回す**: フォールバック・リトライのコードを変えたら、それが本当に発動するかをカオス演習で確認します(実装したフォールバックが、いつの間にか壊れているのはよくあります)

### 発見の還流

演習の価値は、**見つけた弱点を手順・設計の改善につなげる**ことにあります。

- 「フォールバックが発動しなかった」「縮退はしたが復旧手順が曖昧だった」といった発見を、[エラー処理・リトライ・フォールバック設計](../02-architecture/error-handling-and-retries.md) の設計改善と [インシデント対応](incident-response.md) の手順改善に還流します
- 見つけた障害シナリオを、再現可能な評価ケース + 環境に変換して回帰スイートに常駐させます([エージェント評価環境の構築](../04-evaluation/evaluation-environments.md)の失敗の再現環境化)。同じ弱点が二度出ないようにします

## 実務での注意点

### アンチパターン

- **フォールバックを実装したまま検証しない** → 本番で初めて「実は発動しない」と分かり、障害が素通りする → 障害を注入して発動を事前に確かめる
- **AI 固有の障害(形式崩れ・拒否・空応答)を注入しない** → 正常系テストをすり抜ける弱点が残る → プロバイダー・モデル・ツール・知識源の固有障害を注入対象にする
- **いきなり本番で障害注入する** → ユーザーに実害が出る → 評価環境 → ステージング → 本番限定と段階的に上げる
- **仮説なしで手当たり次第に壊す** → 何を確かめたのか分からず、改善につながらない → 「X が落ちても Y が守られる」を仮説にして検証する
- **一度演習して終わりにする** → 回復力が劣化しても気づけない → ゲームデー・自動注入で定常化し、変更のたびに回す

### チェックリスト

- [ ] AI 固有の障害(プロバイダー・モデル出力・ツール・知識源)を注入対象にしている
- [ ] 「X が落ちても Y が守られる」という仮説と成否基準を演習前に定義している
- [ ] 定常状態(正常時の SLI 水準)を基準に、注入の影響を測っている
- [ ] 評価環境から始め、段階的に忠実度を上げている(いきなり本番にしない)
- [ ] 本番注入で副作用のあるツールを実行しない設計になっている
- [ ] ゲームデー・自動注入で演習を定常化している
- [ ] フォールバック・リトライの変更後に、発動をカオス演習で確認している
- [ ] 見つけた弱点を手順改善と回帰スイートに還流している
