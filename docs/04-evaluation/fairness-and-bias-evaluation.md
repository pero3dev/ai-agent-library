# 公平性・バイアスの評価

## この記事の目的

属性・グループによる**品質の差**と**表現の偏り**を、感覚論ではなく**測定可能な形**で評価に組み込めるようになります。「完全に公平なモデル」を目指すのではなく、偏りを測って改善に回す仕組みを作れる状態がゴールです。

## 対象読者

- 不特定多数のユーザーが使う Agent で、特定の属性・グループに不利益が出ていないか確かめたいエンジニア
- 「バイアスに配慮している」を、言葉ではなく測定で示したいテックリード

公平性は「完璧な状態」を一度で達成するものではありません。**偏りを測れる形にし、許容できない差を見つけて改善に回す**という、評価の営みとして扱います。規制上「何が禁止か」は [コンプライアンスとガバナンス](../06-security/compliance-and-governance.md) が、本記事は「どう測るか」を扱います。

### バイアスの類型

まず「何をバイアスと呼ぶか」を分けます。対処法が違うためです。

| 類型 | 内容 | 例 |
| --- | --- | --- |
| グループ間の品質差 | 属性によって、正確さ・成功率・有用性に差が出る | ある方言・名前・業種の入力で回答品質が落ちる |
| 表現の固定観念 | 生成内容にステレオタイプが混じる | 職業と性別、国籍と性格を紋切り型に結びつける |
| 拒否率の偏り | 同種の依頼でも、属性によって拒否・警告の出方が違う | 特定の名前・話題でだけ過剰に警告する |

「バイアス」と一括りにせず、**どの類型を問題にしているか**を特定します。品質差は層別評価で、固定観念は生成内容の分析で、拒否率の偏りは同種依頼での挙動比較で測る、と測り方が変わります。

### 評価設計: 対照ペアテストと層別評価

公平性を測る中心的な手法が 2 つあります。

- **対照ペアテスト(counterfactual test)**: 入力のうち**属性だけを入れ替え**、それ以外は同じにしたペアを作り、出力が変わるかを見ます。「山田さん」と別の名前、同じ内容の依頼で、品質・トーン・拒否が変わるなら、属性が不当に効いています。属性だけを変数にできるため、原因の切り分けが明快です
- **層別評価**: 評価セットを属性・グループで層に分け、層ごとに品質指標(成功率・正確さ)を出して差を見ます。全体の平均では、少数グループの品質低下が多数派に埋もれて見えません([オンライン評価と A/B テスト](online-evaluation-and-ab-testing.md)のセグメント別分析の公平性版)。本番でも主要セグメントで指標を分けて監視します
- **どちらも評価セットの設計が肝**: 対照ペアは「変えるべき属性」を、層別は「分けるべきグループ」を、事前に決めます。何を属性・グループと見なすかの選定自体が設計判断です

### 日本語・日本文化圏の観点

英語圏中心のバイアス評価では見落とされる、日本語・日本文化圏固有の観点があります([日本語品質の評価](japanese-quality-evaluation.md)と隣接)。

- **名前**: 日本の姓名(読みの多様性・外国由来の名前)で品質・トーンが変わらないか
- **方言・地域差**: 標準語以外の入力(関西弁など)で品質が落ちないか
- **敬語・待遇表現**: 相手や属性によって敬語の扱いが不当に変わらないか
- これらは英語系のバイアス評価データには含まれないため、**日本語の対照ペア・層を自前で設計**します

### 対応の選択肢と限界

偏りを見つけたときの手段は複数あり、それぞれ限界があります。

- **プロンプト**: 「属性に基づく差別的な扱いをしない」と指示します。手軽ですが「指示はお願い」であり確実ではありません
- **ガードレール**: 出力側で問題表現を検知・ブロックします([ガードレール](../06-security/guardrails.md))。コードで強制できますが、検知の網羅性が課題です
- **データ**: 偏りの少ない例で調整します(FT・few-shot)。効きますが、データ作成コストと新たな偏りの混入リスクがあります
- **限界を認める**: どの手段も偏りをゼロにはできません。**残る偏りを測り続け、許容範囲を明示し、改善を回す**のが現実的な姿勢です。「対策したから公平」と宣言して測定をやめるのが最も危険です

### 「完全な公平」ではなく「測って改善する」

公平性の定義は 1 つではなく(グループ間で何を等しくするかで複数の指標があり、同時に満たせないこともあります)、「完全な公平」を技術だけで達成することはできません。

- 目指すのは**測定可能な形にすること**です。「どの属性で・どの指標に・どれだけ差があるか」を数字にできれば、議論と改善ができます
- 許容できる差の水準は、タスクの影響度と関係者の合意で決めます(技術者だけで決めない領域です)
- 規制・法務が関わる領域(採用・与信・医療など)では、測定結果を [コンプライアンスとガバナンス](../06-security/compliance-and-governance.md) の統制につなぎます

## 実務での注意点

### アンチパターン

- **全体の平均品質だけを見る** → 少数グループの品質低下が多数派に埋もれて見えない → 属性・グループで層別し、層ごとの指標差を見る
- **「バイアス」を一括りにして対策する** → 品質差・固定観念・拒否率の偏りは対処法が違い、的外れになる → 類型を特定してから測り方・対策を選ぶ
- **プロンプトで「差別しない」と書いて対策完了とする** → 指示は確率的に破られ、偏りが残る → 対照ペア・層別で残る偏りを測り続ける
- **対策後に測定をやめる** → 「対応済み」の宣言だけが残り、実際の偏りが放置される → 偏りを継続的に測り、許容範囲を明示する
- **英語圏のバイアス評価データだけで済ませる** → 日本語・日本文化圏固有の偏り(名前・方言・敬語)が見えない → 日本語の対照ペア・層を自前で設計する

### チェックリスト

- [ ] 問題にしているバイアスの類型(品質差・固定観念・拒否率の偏り)を特定した
- [ ] 属性だけを入れ替える対照ペアテストを設計した
- [ ] 評価セットを属性・グループで層別し、層ごとの指標差を見ている
- [ ] 日本語・日本文化圏固有の観点(名前・方言・敬語)を評価に含めた
- [ ] 対応手段(プロンプト・ガードレール・データ)の限界を理解している
- [ ] 対策後も偏りを継続的に測り、許容範囲を明示している
