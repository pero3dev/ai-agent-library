# エージェント間連携プロトコル

> **最終確認日: 2026-07-09。** 本記事はエージェント間連携の標準が動いている領域を扱います。標準の仕様バージョン・ガバナンス・採用状況は変化が速いため、具体の仕様値や版数は本文に固定せず、公式仕様とリポジトリの `research/infra/agent-protocols.md`(調査メモ)の最新を確認してください。ここでは**構造と判断軸**に重心を置きます。

## この記事の目的

組織・システムの境界を越えてエージェント同士が連携する際の、**発見・認証・委譲・信頼**の設計を判断できるようになります。ツール接続との非対称・プロトコルが定める要素の構造・組織間の信頼の扱い・信頼の段階設計・新しい攻撃面・標準の現在地を、実装以前の設計判断として持ち帰れる状態を目指します。

## 対象読者

- 組織やシステムをまたいでエージェントを連携させる基盤を設計するアーキテクト・エンジニア
- MCP でのツール接続は理解しており、「エージェントに任せる」連携の設計に進みたい実装者

## 本文

### 概要: 「部品を呼ぶ」と「相手に任せる」は違う

エージェントの連携には、性質の異なる 2 つがあります。1 つは**ツールを呼ぶ**——データベースや API といった部品を、決まった入出力で使います。もう 1 つは**他のエージェントに任せる**——自律的に考え・計画し・状態を持つ相手に、タスクを委ねます。前者を担うのがツール接続の標準([MCP](mcp-and-tool-protocols.md))、後者を担うのがエージェント間連携の標準です。

この 2 つは対立ではなく**相補的**です。あるエージェントが他のエージェントに連携で任せつつ、各エージェントは内部でツール接続を使って自分の道具を操る、という重ね方が標準の公式な位置づけです。本記事は「相手に任せる」側の設計を扱います。核心は、**相手を「部品」ではなく「自律した主体」として扱う**ことで、渡す情報・受け取る結果の形・信頼モデルが変わる、という点です。標準の具体(A2A・MCP・関連提案の版数やガバナンス)は変化が速いため、本文は構造に絞り、詳細は調査メモと公式に委ねます。

### ツール接続と連携の非対称

「ツールを呼ぶ」と「エージェントに任せる」は、責任と会話の形が非対称です。

- **ツール = 部品(多くは状態を持たない)**: 決まった入力に決まった出力を返す機能です。呼ぶ側が主導権を持ち、結果の解釈も呼ぶ側の責任です
- **エージェント = 自律した主体**: 自分で考え・計画し・複数の道具を使い、長いやり取りにわたって状態を保ち、多ターンの対話をします。任せる以上、**中身のやり方は相手に委ねます**
- **相手の内部を共有しない(不透明さ)**: 連携の標準は、エージェントを「中身の見えない相手(opaque)」として扱うのが原則です。互いのメモリ・ツール・コンテキストを共有せず、外部インターフェースだけでやり取りします。これは「相手を tool に閉じ込めない」という設計思想の表れです
- **非対称が信頼モデルを変える**: 部品なら「正しく動くか」を気にすれば済みますが、自律した相手には「何を開示するか」「結果を信頼してよいか」という、より重い判断が加わります

### プロトコルが定める 5 要素の構造

エージェント間連携のプロトコルが定めるものは、**5 つの要素**に整理できます。この枠は「プロトコルが何を決めるか」の骨格として使えます。

| 要素 | 何を定めるか | 代表的な形(A2A の例) |
| --- | --- | --- |
| 発見(discovery) | 相手をどう見つけるか | エージェントカード(機械可読メタデータ)を既知の場所で公開 |
| 能力広告 | 相手が何をできるか | できることの一覧(スキル・入出力の形式)を宣言 |
| 委譲(delegation) | タスクをどう渡すか | メッセージを送り、非同期作業はタスクとして扱う |
| 進捗/結果 | 長い処理の途中と結果をどう受け取るか | ポーリング/ストリーミング/通知 + タスク状態 + 成果物 |
| 認証・認可 | 誰と・どこまで | 対応する認証方式を広告し、標準ヘッダで搬送 |

- **発見**: 相手のエージェントカード(何者で・何ができ・どう認証するかを書いた機械可読の名刺)を公開・取得します。これにより、連携相手を機械的に見つけ・選べます
- **能力広告**: 相手が提供する能力(スキル)と、その入出力の形式を宣言します。任せる側は、これを見てタスクに合う相手を選びます
- **委譲**: タスクを渡します。即答できるものはその場で、時間のかかるものは「タスク」として非同期に扱い、会話の文脈を紐づけて継続します
- **進捗/結果(長時間タスク)**: 連携タスクは、数秒で終わるものから数時間・数日に及ぶ調査まで幅があります。途中経過をポーリング・ストリーミング・通知で受け取り、タスクの状態(実行中・要入力・完了・失敗など)と成果物を扱います。**長時間タスクを前提とした状態管理**が、単発の API 呼び出しとの大きな違いです
- **認証・認可**: 誰からの委譲かを確かめ、何を許すかを決めます(次節)

### 組織間の信頼(認証・認可)

組織をまたぐ連携で最も難しいのが、**相手を信頼してよいか**です。標準の設計は「**認証はプロトコルに埋め込まず、既存の標準に委ねる**」方向です。

- **アイデンティティは搬送層で確立する**: 連携の標準は、プロトコルの中身(ペイロード)に利用者やクライアントの ID を載せず、**通信(HTTP)の層で身元を確立**します。対応する認証方式を広告し、標準的なヘッダで認証情報を運びます
- **認可は実装固有**: 「誰に何を許すか」は、プロトコルではなく**各エージェントの実装・扱うデータ・組織のポリシー**が決めます。エージェント固有の認可標準は、2026 年時点でまだ確立していません。既存の認可基盤([エージェントの認証・認可](../06-security/agent-identity-and-auth.md))で吸収します
- **最小権限・能力単位で許す**: 相手に許す範囲を、能力(スキル)単位で最小限に絞ります。「連携できる」ことと「何でも任せてよい」ことは別です
- **決済のような重い委譲は証跡を暗号的に**: 金銭のように失敗コストの高い委譲では、署名付きの同意記録と検証可能なクレデンシャルで、**認可・真正性・説明責任**を暗号的に担保する上位の仕組み(決済プロトコル)が現れ始めています。「エージェントに重要なことを委ねる」領域は、最初から証跡と同意の検証を前提に設計されつつあります

### 信頼の段階設計(社内 → パートナー → 公開)

すべての相手を同じ信頼度で扱うのは危険です。**信頼を段階**で設計します。

- **段階を分ける**: 社内のエージェント・契約したパートナーのエージェント・公開の未知のエージェントで、許す委譲の範囲を変えます。社内には広く、公開には狭く、が基本です
- **委譲してよいタスクを分類する**: どんなタスクなら委ねてよいか(読み取りのみ・低リスク・可逆)を、相手の信頼段階ごとに分類します。高リスクな委譲は、信頼できる相手にのみ許します
- **動的発見に賭けない**: 「未知の相手を動的に発見して任せる」設計は、まだ時期尚早です。発見のうちレジストリの仕組みは標準化が進んでおらず、当面は**既知の相手か、自前のカタログ**に閉じるのが安全です

### 相手エージェントという新しい攻撃面

連携相手は、**新しい攻撃面**です。標準の公式ガイダンスも「相手を信頼できない外部アプリとして扱う」ことを基本にしています。

- **相手を信頼しきらない**: 発見できた相手・連携する相手を、無条件に信頼しません。公式ガイダンスも、リモートのエージェントを「信頼された内部システムではなく、信頼できない外部の HTTP アプリケーション」として扱うよう求めています
- **通信を保護する**: 本番の連携通信は暗号化(HTTPS/TLS)を必須にします。相手のなりすまし・盗聴・改ざんを防ぎます
- **開示を最小化する**: エージェントカードに静的な秘密情報を埋め込まない、機微な能力情報は認証で保護する、開示する能力・データを最小限にする、といった**過剰な情報開示**を避けます
- **悪意ある応答を想定する**: 相手からの応答に、プロンプトインジェクションや誤誘導が仕込まれうると想定します([新興攻撃パターンの体系](../06-security/advanced-attack-patterns.md)の攻撃伝播)。この領域の包括的な公式仕様はまだ薄いため、相手の応答を検証する設計を自前で持ちます

### 標準の現在地と「まだ標準に賭けない」設計

エージェント間連携の標準は、**骨格は固まりつつあるが周辺は流動的**です。この現在地を踏まえて設計します。

- **安定している部分**: 発見・能力広告・メッセージとタスクの構造は、比較的固まってきています。ガバナンスも、中立な標準化団体への移管と、乱立していた提案の収斂が進んでいます
- **まだ動く部分**: 認可・組織間 ID・決済・レジストリの仕組みは活発に動いています。土台となる認証の標準自体もまだ草案段階のものがあります
- **「まだ標準に賭けきらない」設計**: 中核(発見・委譲・タスク)は使いつつ、動きの速い部分(認可・ID・レジストリ)は**実装固有の層で吸収**し、標準の変化に引きずられない逃げ道を残します。特定の標準・版に密結合せず、抽象化層を挟んで最小結合にします
- **ツール接続と混同しない**: 「ツールを呼ぶ(MCP)」と「エージェントに任せる(連携)」は層が違います。両者を混同せず、それぞれの標準を役割で使い分けます

## 実務での注意点

### アンチパターン

- **連携相手を「部品」と同じ感覚で扱う** → 内部を共有させる・結果を無検証で信頼するなど信頼モデルを誤る → 相手を自律した不透明な主体として扱い、開示と信頼を設計する
- **認可をプロトコルが決めてくれると期待する** → エージェント固有の認可標準はまだない → 認可は既存の認可基盤で、能力単位・最小権限で自前に設計する
- **すべての相手を同じ信頼度で扱う** → 公開の未知の相手に広い委譲を許して事故 → 社内/パートナー/公開で信頼を段階化し、委譲してよいタスクを分類する
- **未知の相手を動的に発見して任せる設計に賭ける** → レジストリの仕組みは未標準で時期尚早 → 既知の相手・自前カタログに閉じる
- **連携相手の応答を無条件に信頼する** → 悪意ある応答・過剰開示・インジェクションの経路 → 相手を信頼できない外部アプリとして扱い、応答を検証し開示を最小化する
- **特定の標準・版に密結合する** → 動きの速い周辺仕様の変化に引きずられる → 中核だけ使い、認可・ID・レジストリは抽象化層で吸収する
- **ツール接続と連携を混同する** → 層の違う標準を取り違える → 「呼ぶ(MCP)」と「任せる(連携)」を役割で使い分ける
