# コーディングエージェントの選定基準と使い分け

## この記事の目的

自チームの制約(データ取り扱い・セキュリティ要件・開発フロー・予算)からコーディングエージェントの候補を絞り込む判断軸を持ち、選定プロセス自体を設計できるようになります。「どれが一番良いか」ではなく「自分たちの条件で何が残るか」で考えられることが目標です。

## 対象読者

- チーム・組織で使うコーディングエージェントの選定責任を持つテックリード・EM
- 個人利用から組織標準化への移行を検討しているエンジニア

## 前提知識

- [AI コーディングエージェントの分類と全体像](coding-agents-overview.md) — 提供形態の 5 分類と 2 軸(トリガー面 × 実行場所)

## 本文

### 概要

コーディングエージェントの選定で最初にやるべきことは、機能比較表を眺めることではなく、**落とせない制約を列挙すること**です。2026 年時点の主要ツールは機能面で急速に収斂しており(エージェント実行・MCP 対応・ルールファイルはほぼ共通装備)、差が出るのはデータの流れ・権限モデル・開発フローへの適合・コスト構造です。制約で候補を 2〜3 に絞り、最後は自社タスクでの試用([評価](coding-agent-evaluation.md))で決めます。

### 選定の判断軸

| 軸 | 問うこと | 判断への影響 |
| --- | --- | --- |
| データ取り扱い | コードはどこへ送られ、どれだけ保持され、学習に使われるか(既定とオプトアウト)。準拠すべき規制はあるか | 制約次第で SaaS 型が全滅し、OSS + 自社 API キー(BYOK: Bring Your Own Key)や自己ホスト型に絞られます |
| 実行場所 | ローカル / ベンダークラウド / 自社 CI のどれを許容するか | クラウド実行はコードのアップロードを意味します。逆にローカル実行は開発機の権限管理が課題になります([権限とセキュリティ](coding-agent-security.md)) |
| 権限モデル | 承認制御の粒度(都度・許可リスト・サンドボックス)は要件を満たすか | 統制要件が厳しいほど、細かい承認制御・監査ログが必須になります |
| 開発フロー適合 | 既存 IDE を変えられるか。SCM は GitHub か。モノレポの規模は | 専用 IDE 型は移行コストが、GitHub 連携型は GitHub 前提が制約になります |
| 拡張性 | MCP・フック・SDK で社内ツールと接続できるか | 社内システム連携が必要なら、拡張機構の有無が効きます([MCP とツール接続標準](../03-implementation/mcp-and-tool-protocols.md)) |
| チーム管理 | SSO・ポリシー強制・監査ログ・シート管理は組織要件を満たすか | 多くのツールで上位プラン限定です。個人プランの寄せ集めでは統制できません([チーム導入](coding-agent-team-adoption.md)) |
| コスト構造 | シート型か従量型か。超過時の挙動は予算として読めるか | 従量型は使うほど青天井、シート型は含有枠の超過条件の確認が必要です |
| モデル選択 | 特定ベンダーのモデルに固定されるか、選択・持ち込みできるか | モデル固定型はモデルの進化と心中します。BYOK 型はモデル更改の自由度と引き換えに運用責任を持ちます |

### 用途から形態を選ぶ

[全体像](coding-agents-overview.md) の 2 軸(トリガー面 × 実行場所)を使うと、用途から形態を素直に導けます。

| 用途 | 向く形態 | 理由 |
| --- | --- | --- |
| 対話しながらの設計・実装(ペアプログラミング) | ターミナル型 / IDE 統合型(ローカル) | 往復の速さと文脈共有。人がそばで舵を切れます |
| 独立性の高い定型タスクの大量処理(依存更新、一括修正) | クラウド実行型 | 並列実行と隔離。手元の環境を塞ぎません |
| Issue からの修正 PR 自動作成 | GitHub / Issue / PR 連携型 | 開発フローのイベントがそのままトリガーになります |
| PR レビューの補助 | 連携型(レビュー機能) | 人のレビューの前段として網羅的な一次チェック |
| 機密性の高いコードベース(データ主権重視) | OSS + BYOK / 自己ホスト型 | データの経路を自分で決められます。運用負担と引き換えです |
| 既存 IDE を変えられないチーム | IDE 拡張型 / ターミナル型 | 専用 IDE 型は乗り換えが前提になります |

### 併用パターン

実際の現場は 1 ツールに統一されるより、**補完(IDE)+ 対話型エージェント(ターミナルまたは IDE)+ PR レビュー(連携型)** のような併用に落ち着くことが多いです。役割が違うため排他ではありません。併用時の注意点は次の 3 つです。

1. **ルールファイルの正本を 1 つに決める** — ツールごとに別のルールを書くと二重管理になります([ルールファイルと設定](coding-agent-rules-and-config.md))
2. **課金の重複を確認する** — 同一ベンダーのサブスクリプションと API 従量の二重払い、シートの重複を棚卸しします
3. **ポリシーの一貫性** — データ取り扱い・権限設定の基準はツール横断で揃えます。最も緩いツールが組織の実効的なセキュリティレベルになります

### 選定プロセスの設計

1. **制約で足切りする** — データ取り扱い・実行場所・SCM・IDE の制約を先に文書化し、候補を 2〜3 に絞ります。この段階で [主要コーディングエージェント比較](coding-agents-comparison.md) と各ツール別記事、そして必ず各ツールの一次情報(公式ドキュメント・データポリシー)を使います
2. **試用を設計する** — 同一の実タスク・同一の依頼文・同等の設定で比較します。測定項目(成功率・介入回数・時間・コスト)は [評価](coding-agent-evaluation.md) に従います。**評価は組織プラン相当の条件で行います**(無料版・個人版はデータ取り扱いの既定が異なることが多いためです)
3. **決定と見直し周期を決める** — 「何が変わったら再評価するか」(メジャー更新・料金改定・社内評価セットでの成績変化)を決定時に文書化します。このジャンルは 1 年で勢力図が変わるため、決定の固定化こそがリスクです

## 実務での注意点

### アンチパターン

- **話題性・ベンチマークで即決する** — ベンチマークのスコアは自社タスクでの成績を保証しません([評価](coding-agent-evaluation.md) の限界の議論)。→ 制約での足切り + 社内タスクでの試用に置き換えます
- **機能表の○×だけで比較する** — 「MCP 対応○」でも対応トランスポートや承認制御の深さはツールごとに違います。○の中身は一次情報と試用で確認します
- **全員に同じ 1 ツールを強制する** — 役割(アプリ開発・インフラ・データ)で最適解は分かれます。→ 審査済みツールの承認リストを用途別に持ちます
- **無料版・個人版で組織向けの評価をする** — データ取り扱い・管理機能・レート制限が組織プランと異なります。→ 導入予定のプラン条件で評価します

### チェックリスト

- [ ] 落とせない制約(データ・実行場所・SCM・IDE)を文書化したか
- [ ] 候補ツールの学習利用の既定とオプトアウト手段を、導入予定プランの条件で確認したか
- [ ] 2〜3 候補で同一タスク・同一条件の試用を行ったか
- [ ] 併用構成のルール正本・課金重複・ポリシー一貫性を確認したか
- [ ] 再評価のトリガーと乗り換え基準を決定時に文書化したか
