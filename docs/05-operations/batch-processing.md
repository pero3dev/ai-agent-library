# バッチ処理の設計(バッチ API の活用)

## この記事の目的

「すぐには要らないが量が多い」LLM 処理を、バッチ API・オフライン実行で**安く・確実に**流す設計ができるようになります。バッチ処理の性質(納期と引き換えの割引)を理解し、向くワークロードの見極め・ジョブ設計(分割・部分再投入・冪等性)・リアルタイム処理との振り分けを判断できる状態を目指します。

## 対象読者

- 大量の LLM 処理(評価・埋め込み再計算・一括分類など)のコストと確実性に悩む運用・データ基盤担当
- リアルタイム API で大量処理を回してコスト・レート制限に苦しんでいる実装担当

## 前提知識

- [コスト管理](cost-management.md) — トークン経済とコスト構造(バッチ割引の前提)
- [非同期・長時間タスクの設計(耐久実行)](../02-architecture/async-and-durable-agents.md) — 分割・再開・冪等性の原則(本記事はそのバッチ処理への適用)

## 本文

### 概要: 「急がない大量処理」を別経路に逃がす

LLM 処理には、即時応答が要るもの(対話・その場の判定)と、**すぐには要らないが量が多いもの**(夜間の一括処理・バックフィル)があります。後者をリアルタイム API で回すと、割高でレート制限にも当たります。バッチ処理は、この「急がない大量処理」を別経路に逃がす設計です。

### バッチ API の性質

多くの LLM プロバイダーが提供する**バッチ API** は、リアルタイム API とは性質が異なります(具体的な割引率・上限は変化するため下記 TODO 参照)。

- **納期保証と引き換えの割引**: 「一定時間内に完了」という緩い納期を受け入れる代わりに、単価が割引されるのが一般的です。即時性を手放してコストを下げる取引です
- **容量の別枠**: バッチはリアルタイムとは別のキュー・容量枠で処理されることが多く、リアルタイムのレート制限を消費せずに大量投入できます
- **非同期の投げっぱなし**: ジョブを投入し、後で結果を回収する非同期モデルです。同期的に待つ処理には向きません

### 向くワークロード

バッチが効くのは、**即時性が要らず、まとめて流せる**処理です。

- **評価スイートの一括実行**: 回帰テスト・評価データセットの一括採点([回帰テストと評価データセットの管理](../04-evaluation/regression-testing.md))。夜間にまとめて回せます
- **埋め込みの再計算**: モデル変更に伴う全チャンクの再インデックス([埋め込み(embeddings)の選定と運用](../03-implementation/embeddings.md))。大量かつ急がない典型例です
- **バックフィル**: 過去データへの要約・タグ付け・分類の一括付与。新機能を既存データに遡って適用する処理
- **一括分類・抽出**: 大量のドキュメント・レコードへの分類・構造化抽出。件数が多く、締切が緩いもの

共通するのは「**件数が多い + 締切が緩い + 1 件ずつの即時性が不要**」です。逆に、ユーザーを待たせる処理・その場の判断はバッチに向きません。

### ジョブ設計: 分割・部分再投入・冪等性

大量ジョブは**失敗を前提**に設計します。原則は [非同期・長時間タスクの設計(耐久実行)](../02-architecture/async-and-durable-agents.md) と同じで、それをバッチに適用します。

- **分割する**: 巨大な 1 ジョブにせず、扱いやすい単位に分割します。一部が失敗しても全体をやり直さずに済み、進捗も見えます
- **部分再投入**: 失敗した項目だけを再投入できるようにします。「1 万件中 30 件が失敗」で全 1 万件を再実行するのは無駄です。どの項目が成功/失敗したかを記録します
- **冪等性を持たせる**: 同じ項目を二度処理しても結果が壊れないようにします([冪等性](../02-architecture/async-and-durable-agents.md))。再投入・重複実行が前提のバッチでは必須です
- **結果とキーを対応づける**: どの入力に対する結果かを一意キーで対応づけ、順序に依存しない回収にします。バッチは順序保証が弱いことがあります

### リアルタイム処理との振り分け

同じ種類の処理を、**リアルタイムとバッチの 2 経路**で持つ設計がよく効きます。

- **2 経路化**: 「その場で要るときはリアルタイム、まとめてでよいときはバッチ」と、同じ処理ロジックを 2 つの経路で呼べるようにします。例: 新規ドキュメントは即時に埋め込み、過去分の再計算はバッチ
- **振り分けの基準**: 締切(いつまでに要るか)とコスト感度で振り分けます。急ぐ少量はリアルタイム、急がない大量はバッチ
- **ロジックを共有する**: 2 経路で処理内容(プロンプト・後処理)がずれると、結果に一貫性がなくなります。中身のロジックは共有し、実行経路だけを変えます

### コスト・納期の見積りと監視

バッチも投げっぱなしにはできません。見積りと監視が要ります。

- **コストの見積り**: 件数 × 1 件あたりトークン × バッチ単価で概算します([コスト管理](cost-management.md)のトークン経済)。リアルタイム比の削減幅を把握します
- **納期の見積り**: 「いつまでに結果が要るか」に対し、バッチの完了目安が間に合うかを確認します。締切が厳しいならバッチは選べません
- **監視**: ジョブの進捗・成功/失敗率・完了時間を監視します([可観測性とトレーシング](observability-and-tracing.md))。失敗の偏り(特定の入力で落ちる)を早く捕まえます

## 実務での注意点

### アンチパターン

- **急ぐ処理をバッチに載せる** → 納期の緩さがユーザー体験を壊す → 締切で振り分け、急ぐものはリアルタイムにする
- **巨大な 1 ジョブで投入する** → 一部失敗で全体をやり直す羽目になる → 扱いやすい単位に分割し、部分再投入できるようにする
- **冪等性なしで再投入する** → 重複処理で結果が壊れる・二重課金になる → 一意キーで冪等にする
- **リアルタイムとバッチでロジックがずれる** → 経路によって結果が変わり一貫性が崩れる → 処理ロジックを共有し実行経路だけ変える
- **投げっぱなしで監視しない** → 大量失敗・納期超過に気づかない → 進捗・成功率・完了時間を監視する

### チェックリスト

- [ ] バッチ対象を「件数が多い + 締切が緩い + 即時性不要」で選んでいる
- [ ] ジョブを扱いやすい単位に分割している
- [ ] 失敗した項目だけを部分再投入できる(成功/失敗を記録している)
- [ ] 同じ項目の二重処理でも壊れない冪等性を持たせた
- [ ] リアルタイムとバッチの 2 経路で処理ロジックを共有している
- [ ] コスト削減幅と納期の見積りをし、締切に間に合うことを確認した
- [ ] ジョブの進捗・成功率・完了時間を監視している
