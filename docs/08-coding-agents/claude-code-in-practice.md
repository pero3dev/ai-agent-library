# Claude Code 実践ガイド

## この記事の目的

Claude Code の拡張機能(スキル・サブエージェント・フック・worktree 並列・headless)を使いどころから理解し、消費(トークン・使用量上限)を抑えながら業務を効率化できるようになります。製品の選定判断は [Claude Code](claude-code.md) が正で、本記事は「導入済みの人が使いこなす」段階を扱います。

## 対象読者

- Claude Code を対話中心に使っており、拡張機能・自動化・コスト管理を体系的に押さえたいエンジニア
- チームに Claude Code の運用標準(スキル・設定・計測)を整備したい人

### 機能の使いどころ

公式ドキュメントは各機構の「使いどき」を明文化しています。判断基準ごと押さえるのが早道です。

| 機構 | 使いどき(公式基準) |
| --- | --- |
| スキル(`.claude/skills/`) | 「同じ指示・チェックリストをチャットに繰り返し貼っているとき」「CLAUDE.md の一節が手順書に育ったとき」。**旧カスタムスラッシュコマンドはスキルに統合**されており(`.claude/commands/` も継続動作)、本文は使用時のみロードされます |
| サブエージェント(`.claude/agents/`) | 「副次タスクがメイン会話を検索結果・ログで溢れさせるとき」。コンテキスト温存に加え、**安いモデルへのルーティングによるコスト制御**が公式の効用に挙げられています |
| fork(`/fork`) | 同じ文脈を引き継ぐ分岐作業。親のプロンプトキャッシュを初回から再利用できるため、**新規サブエージェントより安い**選択肢です |
| フック | 「LLM の判断に頼らず、必ず起きてほしい決定論的な処理」(編集後の自動フォーマット、保護ファイルのブロック、コンパクション後のコンテキスト再注入など) |
| プラグイン | チーム・複数プロジェクトへの配布。「まず `.claude/` で試作し、共有段階でプラグイン化」が公式推奨です |
| plan モード | アプローチが不確か・複数ファイル変更・不慣れなコードのとき。**「diff を一文で説明できるならプランは省け」**という不使用基準まで公式が明記しています |
| チェックポイント(`/rewind`) | 「リスクのある試行をやらせて、ダメなら巻き戻す」。bash 経由のファイル操作は追跡されない点に注意(恒久履歴は git) |
| worktree(`claude -w <name>`) | 同一リポジトリでの並列セッション。ブランチと作業ツリーを自動作成し、無変更なら終了時に自動削除されます |
| auto memory | **CLAUDE.md = 人が書く指示 / auto memory = Claude が書く学習**という分担。「これを覚えて」は auto memory へ、「CLAUDE.md に追加して」は規約へ、と発話で使い分けられます |

スキルの設計では、起動制御(手動専用 `disable-model-invocation` / Claude 専用 `user-invocable: false`)と、**起動後は本文がセッションに残り続ける**(継続コスト)ことを踏まえて、SKILL.md は 500 行以下・詳細は補助ファイルに分ける公式推奨に従います。

### コスト削減: 公式テクニックの要点

公式の大原則は 1 つです: **「トークンコストはコンテキストサイズに比例する」**。[コスト最適化](coding-agent-cost-optimization.md) の一般原則が、Claude Code では次の操作に対応します。

1. **コンテキストの能動管理** — 無関係な作業に移るときは `/clear`(古い文脈は以後の全メッセージで課金され続けます)。後で戻るなら `/rename` してから `/clear` し、`/resume` で復帰します
2. **コンパクションはタスクの区切りで** — 自動発火をタスク途中に食らうのではなく、区切りの良いところで `/compact`(残すものの指示も付けられます)。**捨てたい経路に入り込んだときは `/compact` ではなく `/rewind`** — キャッシュ済みの位置へ戻るため安上がりです
3. **CLAUDE.md は 200 行以下を目標に、手順はスキルへ** — CLAUDE.md は毎セッション全ターンで課金されます。特定ワークフローの詳細手順はスキル(使用時のみロード)に移すのが公式推奨です
4. **MCP より CLI ツール** — `gh`・`aws` 等の CLI はツール定義のコンテキストを追加しないため、MCP サーバーよりコンテキスト効率が良いと公式が明記しています。`/context` で何が消費しているかを確認し、未使用の MCP は外します
5. **フック・スキルへの前処理オフロード** — 公式例: 1 万行のログを読ませる代わりに、フックが `ERROR` 行だけを grep して返す(数万→数百トークン)
6. **モデルと思考量の使い分け** — 大半のコーディングは中位モデルで足り、上位モデルは複雑な設計判断に温存します。サブエージェントには軽量モデルを割り当てます。**thinking トークンは出力として課金される**ため、単純作業では effort を下げ、単発の深掘りはプロンプト内の `ultrathink` で済ませます
7. **Agent teams は高コスト** — 公式が「標準セッションの約 7 倍」(2026-07 時点)と明記する構成です。使うなら小さなチーム・軽量モデルで

**プロンプトキャッシュの「踏み抜きどころ」**も公式がリスト化しています。キャッシュ読み取りは通常入力の約 1 割の課金(2026-07 時点)なので、無効化する操作を知ることが実質的な節約です。日常で踏みやすいのは次の 3 つです:

- **セッション途中のモデル・effort 切替**(モデルごとに別キャッシュ。全履歴を非キャッシュで再読)→ モデルと effort は**セッション冒頭に決める**
- **`opusplan` 設定での plan モード出入り**(そのたびにモデルが切り替わる)
- **アップグレード直後に長いセッションを resume**(公式が「最も高価なリクエストになり得る」と明記)

**測定**: `/usage`(スキル / サブエージェント / MCP 別の内訳)、`/context`、ステータスラインの常時表示が個人向け、組織は OpenTelemetry(コスト・トークンメトリクスに `skill.name`・`agent.name`・`mcp_server.name` 属性が付き、**スキル別・サブエージェント別のコスト帰属**が可能)です。headless の `--output-format json` は `total_cost_usd` を返すため、自動化の呼び出し単位でコストを集計できます。

### 業務効率化・自動化

- **headless(`claude -p`)**: 全 CLI オプション併用可・stdin パイプ対応で、`--output-format json`(+ `--json-schema` で構造化出力)が CI の基本形です。CI では **`--bare`**(CLAUDE.md・スキル・MCP 等の自動発見をスキップし、必要なものだけ明示注入)が公式推奨で、「どのマシンでも同じ結果」を得られます
- **ファンアウト**: タスクリストを生成し、`claude -p` をループで回す一括処理パターンです。公式手順は「**2〜3 ファイルで試してプロンプトを磨いてから全量実行**」— [自動化パターン](coding-agent-automation-patterns.md) の「対話で成功してから自動化」の実装形です
- **GitHub Actions**: `@claude` メンション応答と、`prompt` 指定の自動実行(cron・PR イベント)の 2 モード。コスト制御は **`--max-turns`(既定 10)+ ワークフローのタイムアウト + concurrency 制限**の 3 点セットが公式推奨です。API キーは Secrets、Bedrock 等では OpenID Connect(OIDC)によるキーレス認証が推奨されます。リポジトリのスキルを `prompt: /skill-name` で起動でき、PR ごとのコードレビュー等を定型化できます
- **GitLab CI/CD**: beta で、メンテナンスは GitLab 社です。実体は headless 実行のジョブで、`max_turns` と `timeout_minutes` で上限を設けます
- **Routines**(research preview): 「プロンプト + リポジトリ + コネクタ」を Anthropic 管理クラウドで定期・API・GitHub イベント起動する仕組みです。**承認プロンプトなしの完全自律実行**のため、既定の防御(プッシュ先は `claude/` プレフィックスブランチのみ、ネットワークは許可リスト制)を理解して使います。日次実行数の上限があり、**実行一覧のグリーン表示は「インフラエラーなし」の意味でタスク成功ではない**、という公式の注意はスケジュール自動化一般の教訓です([自動化パターン](coding-agent-automation-patterns.md) の通知・検証設計)

### 品質を上げる公式プラクティス

公式ベストプラクティスの中核は「**コンテキストウィンドウが最重要の管理対象リソース**」という原則と、次の運用です。

- **検証手段を渡す(最上位のプラクティス)** — 「Claude が実行できるチェック(テスト・ビルド・スクリーンショット比較)を渡せ。**見守るセッションと離席できるセッションの違いはそこにある**」。成功を主張させるのではなく、証拠(テスト出力・実行結果)を出させます。バグ修正は「再現する失敗テストを先に書いてから直す」依頼が公式の型です
- **Explore → Plan → Implement → Commit の 4 フェーズ** — 不確かなタスクの標準形。小さな修正はスキップします(前述の「diff 一文」基準)
- **早めのコース修正** — `Esc` で即中断 → 巻き戻し → やり直し。**「同じ問題で 2 回以上訂正したら、コンテキストは失敗アプローチで汚染されている。`/clear` して学びを織り込んだプロンプトで再開する方がほぼ常に勝る」**([依頼設計](coding-agent-prompting.md) の仕切り直しと同じ判断です)
- **インタビューパターン** — 大きめの機能は「Claude に自分をインタビューさせて SPEC.md に仕様を書かせ、**新しいセッションで実装**」する 2 段構えが公式に案内されています
- **Writer / Reviewer 分離** — 別セッションの新鮮なコンテキストでレビューさせるとバイアスがかからない、というマルチセッション活用。敵対的レビューでは「正しさと要件に影響する gap だけ挙げさせる」(全部追うと過剰設計になる)という注意書きまで公式にあります
- **5 つの失敗パターン**(公式)— 何でも 1 セッションに詰める / 訂正の繰り返し / 肥大した CLAUDE.md / 検証なしの信頼 / 際限ない調査。本章の各記事の原則と一致します

## 実務での注意点

### アンチパターン

- **1 セッションに何でも詰め込む(kitchen sink)** — 消費が毎ターン膨らみ、性能も落ちます。→ タスク間で `/clear`、長期タスクは区切りで `/compact`、戻る可能性があれば `/rename` + `/resume` を使います
- **セッション途中でモデル・effort を頻繁に切り替える** — キャッシュが無効化され、全履歴を非キャッシュ単価で再読します。→ セッション冒頭に決め、途中で変えたくなったらセッションを分けます
- **CLAUDE.md に手順書を書き続ける** — 毎セッション・毎ターンの固定費になります。→ 「消したら Claude はミスするか?」で選別し、手順はスキルへ移します
- **Routines・Actions に検証なしのタスクを載せる** — 完全自律実行では「成功したように見える失敗」を誰も見ていません。→ 機械検証(テスト・チェックスクリプト)と結果の通知先を先に設計します

### チェックリスト

- [ ] タスク間の `/clear`・区切りでの `/compact`・失敗時の `/rewind` を使い分けているか
- [ ] モデル・effort をセッション冒頭に固定し、キャッシュ無効化の操作を把握しているか
- [ ] CLAUDE.md がスリムで(目安 200 行以下)、手順はスキルに移してあるか
- [ ] サブエージェント・定型タスクに軽量モデルを割り当てたか
- [ ] `/usage`・`/context`(組織は OpenTelemetry)で消費の内訳を確認できるか
- [ ] 委任・自動化するタスクに検証手段(テスト・期待出力)を渡しているか
- [ ] CI・Routines で上限(`--max-turns`・タイムアウト・日次キャップ)を理解して設定したか
