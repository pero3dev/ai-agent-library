# 執筆・翻訳ワークフローの設計

## この記事の目的

長文の執筆・翻訳を、**品質を管理しながら量産できるワークフロー**として設計できるようになります。長文生成の分割統治・スタイルと用語の統制・翻訳工程(機械翻訳後編集)の設計・人が見る箇所の絞り込み・スタイル準拠の自動評価を、実務の判断として持ち帰れる状態を目指します。

## 対象読者

- ドキュメント・記事・マニュアル・翻訳を、品質を保ちながら量産する仕組みを作るエンジニア・コンテンツ責任者
- 単発の生成はできるが、それを「レビューと統制のあるパイプライン」に組み上げたい実装者

## 本文

### 概要: 品質は「一発生成」ではなく「工程」で作る

長文の執筆・翻訳を LLM で量産しようとすると、「1 回の生成で全部書かせる」やり方はすぐ頭打ちになります。長くなるほど一貫性が崩れ、用語がぶれ、後半で失速します。品質は一発の賢さではなく、**分割統治と工程設計**で作ります。

このライブラリ自身が、その一例です。テンプレート(`templates/doc-template.md`)で構造を固定し、執筆規約(`CLAUDE.md`)でスタイルを統制し、独立レビューと lint で検証する、という工程で品質を保っています。以下では、この「工程で品質を作る」考え方を、執筆と翻訳のワークフローとして一般化します。

### 長文生成の分割統治(アウトライン駆動)

長文は、一度に書かせず、**アウトライン → セクション → 統合**の順で分割して作ります。

- **アウトラインを先に確定する**: 全体の構成(見出しと各節の要点)を先に生成・確定し、それを骨格にします。骨格があると、各節を独立に生成でき、抜けや重複が見えます
- **セクション単位で生成する**: 各節を、アウトラインと「隣接する節の要約」を文脈として個別に生成します。全文を毎回コンテキストに載せるより、必要な文脈だけを渡すほうが安定します([コンテキストの圧縮と隔離](../02-architecture/context-compaction-and-isolation.md))
- **一貫性を後段で担保する**: 分割生成は、節ごとに用語・トーン・主張がぶれる弱点があります。統合の段で、用語の統一・重複の除去・つなぎの調整をします。**分割で速く書き、統合で揃える**、が基本形です

### スタイル・用語の統制(スタイルガイド・用語集)

品質の量産には、「良い文章とは何か」を**明文化して機械に渡す**ことが要ります。人の頭の中の基準では、量産で必ずぶれます。

- **スタイルガイドを資産として持つ**: 文体(敬体か常体か)・表記ルール・禁止事項を明文化し、[プロンプト資産](../03-implementation/prompt-management.md)として一元管理します。各生成はこれを参照するので、基準を変えると全体に反映されます
- **用語集を適用し、検証する**: 訳語・製品名・専門用語の対応表を用意し、生成後に**用語集どおりか自動チェック**します。「適用を指示する」だけでは守られないことがあるため、生成と検証の両方に用語集を効かせます
- **例が仕様として働く**: 望ましい文章の実例を数個渡すと、スタイルガイドの文字だけより伝わります([プロンプトエンジニアリングの基礎技法](../03-implementation/prompt-engineering-fundamentals.md)の少数例)。ただし例の偏りは出力の偏りになるため、代表的な例を選びます

### 翻訳ワークフロー(文脈保持・用語統一・MTPE)

翻訳は、執筆と共通点が多い一方、**原文への忠実さ**という固有の制約があります。品質を管理する定番が、機械翻訳後編集(MTPE / Machine Translation Post-Editing)です。

- **文脈を保持して訳す**: 文を 1 文ずつ孤立して訳すと、指示語・敬語・専門用語の一貫性が崩れます。段落・文書単位の文脈を渡し、**前後関係を保った訳**にします
- **用語を統一する**: 用語集(グロッサリ)と、過去訳の対訳メモリを適用し、同じ語が文書内・文書間でぶれないようにします。これは翻訳品質の体感を最も左右する要素です
- **MTPE の工程を設計する**: 「機械翻訳 → 人による後編集」の工程で、**人がどこをどこまで直すか**を決めます。全文を人が精査するなら量産になりません。重要度・リスクで、軽い後編集(意味が通ればよい)と重い後編集(公開品質)を使い分けます
- **後編集の負担を減らす前処理**: 訳しにくい原文(曖昧・長すぎる文)は、前処理で整えると後編集が軽くなります。工程全体で人の手を減らす設計をします

### レビュー工程の設計(人が見る箇所の絞り込み)

量産の律速は、多くの場合**人のレビュー**です。全部を人が精査すると量産になりません。人の目を、**効く場所に絞る**のが工程設計の肝です。

- **機械で落とせるものは機械で落とす**: 用語違反・表記ゆれ・リンク切れ・スタイル違反は、自動チェック(lint)で先に落とします。人はそこに時間を使いません
- **人はリスクの高い箇所に集中する**: 事実の正しさ・機微な表現・重要な結論など、**間違えると被害が大きい箇所**に人のレビューを寄せます。確度の低い生成箇所を機械が印付けし、そこを人が見る、という振り分けも有効です
- **レビュー結果を還流する**: 人が直した箇所は、スタイルガイド・用語集・プロンプトの改善材料です。同じ指摘を繰り返さないよう、工程に戻します([フィードバックループの運用](../05-operations/feedback-loops.md))

### 評価(judge + 人手・スタイル準拠の自動チェック)

執筆・翻訳の品質は、正解が 1 つでないため評価が難しいドメインです。**自動チェックと judge と人手**を組み合わせます。

- **決定的チェックを先に**: 用語集準拠・表記ルール・禁止語・リンク切れは、コードで確実に落とせます。ここは自動化を厚くします
- **開放的な品質は judge + 人手**: 自然さ・一貫性・原文への忠実さは、基準を与えた[LLM-as-a-Judge](../04-evaluation/llm-as-a-judge.md)で一次判定し、judge 自体の信頼性を人手ラベルとの一致で検証します。日本語の文体・自然さは[日本語品質の評価](../04-evaluation/japanese-quality-evaluation.md)の観点を使います
- **人手評価を賢く使う**: 人手は最も信頼できるが高コストなので、judge が低確度と判定した箇所・リスクの高い箇所に集中させます。全数評価ではなく、**サンプリングと重点評価**を設計します

## 実務での注意点

### アンチパターン

- **長文を一度に全部生成させる** → 後半で一貫性が崩れ用語がぶれる → アウトライン → セクション → 統合の分割統治にする
- **スタイル基準を人の頭の中に置く** → 量産で必ずぶれる → スタイルガイド・用語集を明文化しプロンプト資産として管理する
- **用語集を「指示するだけ」で守らせる** → 守られないことがある → 生成後に用語集準拠を自動チェックする
- **翻訳を 1 文ずつ孤立して訳す** → 指示語・敬語・用語の一貫性が崩れる → 段落・文書単位の文脈を保持して訳す
- **全文を人が精査する** → 量産にならない → 機械で落とせるものは lint で落とし、人はリスクの高い箇所に集中する
- **人の修正を工程に還流しない** → 同じ指摘を繰り返す → 修正をスタイルガイド・用語集・プロンプトの改善に回す

### チェックリスト

- [ ] 長文をアウトライン → セクション → 統合の分割統治で生成しているか
- [ ] スタイルガイド・用語集を明文化し、プロンプト資産として一元管理しているか
- [ ] 生成後に用語集準拠・表記ルールを自動チェックしているか
- [ ] 翻訳で文脈を保持し、用語集・対訳メモリで用語を統一しているか
- [ ] MTPE の後編集レベル(軽い/重い)を重要度・リスクで使い分けているか
- [ ] 機械で落とせる欠陥を lint で落とし、人のレビューをリスクの高い箇所に絞っているか
- [ ] 決定的チェック + judge + 人手サンプリングの多層で品質を評価しているか
