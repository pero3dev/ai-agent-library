# プロンプト最適化(評価駆動の改善と自動化)

## この記事の目的

プロンプトの改善を「変えてみたら良くなった気がする」という感想戦から、**失敗分析 → 仮説 → 最小変更 → 回帰評価**という再現可能な工学に変えます。あわせて、LLM 自身にプロンプトを改善させる方法と、自動プロンプト最適化の原理・限界を理解し、導入の判断ができるようになります。

## 対象読者

- プロンプト変更のたびに「直ったか分からない・別の何かが壊れたかもしれない」状態にあるエンジニア
- 自動プロンプト最適化ツールの導入を検討していて、前提条件と限界を知りたいテックリード

## 前提知識

- [プロンプトエンジニアリングの上級パターン](prompt-engineering-patterns.md) — 改善の手札(何に変えるか)
- [Agent 評価の基礎](../04-evaluation/agent-evaluation-basics.md) / [評価データセットの構築と保守](../04-evaluation/evaluation-datasets.md) — 改善を測る土台(本記事はこれがある前提で書きます)

## 本文

### 概要: 最適化は評価の従属変数

プロンプト最適化の成否は、プロンプトを書く腕前より**評価の質**で決まります。信頼できる評価セットと判定基準がなければ、手動でも自動でも「良くなった」を確認する手段がなく、改善は感想に退化します。本記事の全手法は、評価セット([評価データセットの構築と保守](../04-evaluation/evaluation-datasets.md))と変更管理([プロンプト資産の管理とバージョニング](prompt-management.md))が存在する前提で動きます。

```mermaid
flowchart LR
    F["失敗を集める<br/>(本番ログ・評価落ち)"] --> A["失敗分析<br/>(類型に分類)"]
    A --> H["仮説<br/>(原因 → 変更案)"]
    H --> C["最小変更<br/>(1 回 1 変更)"]
    C --> E["回帰評価<br/>(改善 + 非劣化)"]
    E -->|採用| R["記録して資産化"]
    E -->|棄却| A
    R --> F
```

### 評価駆動の改善サイクル

1. **失敗を集める**: 材料は本番の失敗([フィードバックループの運用](../05-operations/feedback-loops.md))と評価セットの不合格ケースです。「もっと良くしたい」という漠然とした動機からは、検証可能な変更は出てきません
2. **失敗を分類する**: 個々のケースを深掘りする前に、失敗を類型に分けて分布を見ます(形式崩れ・指示の無視・事実誤り・判断の揺れ・冗長・頑健性)。最頻の類型から着手するのが投資対効果の基本です
3. **仮説を立てる**: 「この類型は◯◯が原因なので、△△を変えれば直るはず」という反証可能な形式にします。原因の説明に 10 章の原理を使えると、仮説の精度が上がります
4. **最小変更**: 1 回の実験で変えるのは 1 箇所です。複数同時に変えると、効果の帰属が永久に分からなくなります(交絡)
5. **回帰評価**: 狙った類型の改善と、**他の類型・既存ケースの非劣化**を同時に確認します([回帰テストと CI 組み込み](../04-evaluation/regression-testing.md))。プロンプトの変更は非局所的に波及する、が大前提です
6. **記録**: 採否と理由・評価結果を残します。棄却した変更の記録は「二度と試さないリスト」として同じ価値があります

### 失敗モード別の対策マップ

「症状 → まず試す変更」の対応表です。**プロンプト以外が原因の症状**を先に除外することが重要です([Agent 向けプロンプト設計](agent-prompt-design.md)の「プロンプトで直すべきでない問題」と同じ発想です)。

| 症状 | 原因の仮説 | まず試す変更(→ 上級パターンの該当節) |
| --- | --- | --- |
| 形式が崩れる | 形式指定が「お願い」のまま | 構造化出力での強制・応答の書き出し指定(→ 出力の制御) |
| 指示の一部を無視する | 指示の埋没・優先順位の不在 | レイヤ構造化・重要指示の再配置(先頭 / 末尾)・優先順位の明示(→ 構造化) |
| 判断が入力ごとに揺れる | 基準が暗黙 | 判断基準の明文化・境界例と否定例の追加(→ 例示) |
| 難しい問題で浅い答え | 考える前に答えている | 考察 → 結論の順序強制・推論モデルの検討(→ 思考の制御) |
| 冗長・トーン不一致 | 形式の手本がない | 出力仕様の具体化・例による形式の伝染(→ 例示) |
| 事実が間違う | プロンプトでは直らない | 根拠の外部供給(RAG)・検証の追加へ(プロンプト外) |
| 実入力でだけ壊れる | きれいな入力で開発した | 頑健性テストの追加(→ 頑健性) |

### LLM による改善(メタプロンプティング)

LLM をプロンプト改善の道具として使う方法は 3 つあります。

- **書き換え**: 現行プロンプトと改善意図を渡し、改善案を出させる。叩き台の生成としては速い
- **批評**: 失敗ケース(入力・出力・期待)を見せ、原因仮説と修正案を出させる。人間の失敗分析の補助として有効
- **メタプロンプティング(meta-prompting)**: タスク記述からプロンプト自体を生成させる。ゼロから書く初速に効く

共通の規律は 1 つです — **LLM の提案は仮説であり、評価を通すまで改善ではありません**。もっともらしい提案ほど無検証で採用したくなりますが、それは感想戦への逆戻りです。また、改善案の良し悪しを同じ LLM に聞く構成は自己選好バイアスの罠になります([LLM-as-a-Judge](../04-evaluation/llm-as-a-judge.md)の注意がそのまま適用されます)。

### 自動プロンプト最適化の原理と限界

自動最適化ツール・フレームワークは多数ありますが、内部の構造はほぼ共通です。

> **候補生成**(現行プロンプトの変異・書き換え・例の選択)× **評価**(データセット + 自動判定)× **選択**(スコア上位を残して反復)

この構造から、導入判断に必要な結論が直接導けます。

1. **前提条件は「信頼できる自動評価」**: 最適化の上限は評価器の質で決まります。自動判定できない品質(ニュアンス・安全性)は最適化できず、判定器が甘ければ「判定器を騙すプロンプト」が選ばれます。つまり自動最適化の導入判断とは、実質的に**評価ハーネスの成熟度の判断**です
2. **データが要る**: 候補を選別するには十分な数の評価ケースが必要です。数十ケースでの自動最適化は過適合の量産になります
3. **成果物の保守性**: 自動最適化されたプロンプトは「なぜ効くか」が人間に読めない形になりがちで、以後の手動改善・監査が難しくなります。資産管理([プロンプト資産の管理とバージョニング](prompt-management.md))に「生成元の設定とデータ」を残すことが実質の可読性になります

個別のフレームワークの動向は変化が速いため本記事では扱いません(下記 TODO)。原理が共通である以上、選定は「評価との接続・記録の残り方・コスト」で判断できます。

### 過適合の罠: 3 つの向き

最適化が進むほど、3 種類の過適合が忍び寄ります。

- **評価セットへの過適合**: 開発用セットでだけ強いプロンプトができる。→ 開発用と判定用の分離・定期的な新ケース追加([評価データセットの構築と保守](../04-evaluation/evaluation-datasets.md)の原則)
- **判定器への過適合**: judge の癖(長い回答を好む等)に最適化される。→ 判定器の検証・複数観点の判定・人手サンプリングとの突き合わせ
- **時点への過適合**: 特定モデル版の癖に深く最適化するほど、モデル更新での崩れ幅が大きくなる。→ 「なぜ効くか」を原理で説明できる変更を優先し、更新時の再評価を前提にする([バージョニング・デプロイ・モデル更新追従](../05-operations/versioning-and-model-updates.md))

### この先: 最適化を運用に組み込む

単発の最適化で終わらせず、変更フロー(レビュー + 回帰)への組み込みは [プロンプト資産の管理とバージョニング](prompt-management.md)、本番での効果確認は [オンライン評価と A/B テスト](../04-evaluation/online-evaluation-and-ab-testing.md)、失敗の継続的な還流は [フィードバックループの運用](../05-operations/feedback-loops.md) へ接続します。

## 実務での注意点

### アンチパターン

- **感想で変更を積み重ねる** → 効果不明の変更が地層化し、触れないプロンプトになる → 失敗分析 → 仮説 → 最小変更 → 回帰評価のサイクルに載せる
- **複数の変更を一度に入れる** → 何が効いたか分からず、悪化時の切り戻しもできない → 1 回 1 変更
- **開発用セットだけで「改善した」と判断する** → 本番・判定用セットで劣化していることに気付けない → セット分離と非劣化確認を必須にする
- **LLM の改善提案を無検証で採用する** → もっともらしい仮説を事実扱いする感想戦に戻る → 提案はすべて評価を通す
- **評価器が未成熟なまま自動最適化を回す** → 判定器を騙すプロンプトが「最良」として選ばれる → 先に評価ハーネスへ投資する

### チェックリスト

- [ ] 改善の起点が具体的な失敗ケース(ログ・評価落ち)になっている
- [ ] 失敗が類型に分類され、最頻のものから着手している
- [ ] 1 回の実験の変更が 1 箇所で、仮説が書かれている
- [ ] 回帰評価で「狙いの改善」と「他の非劣化」を両方確認している
- [ ] 棄却した変更も理由付きで記録している
- [ ] 自動最適化を使う場合、評価器の検証とデータ量の確認が先に済んでいる
- [ ] モデル更新時に最適化済みプロンプトを再評価する運用がある
