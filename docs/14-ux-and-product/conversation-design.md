# 会話設計(ペルソナ・トーン・境界)

## この記事の目的

「何でも答える汎用チャット」ではなく、役割・トーン・境界が**設計された**会話体験を作れるようになります。ペルソナとトーンの設計・話題の境界と断り方・会話の構造・記憶の見せ方・多言語への配慮・ログからの改善を、設計判断として持ち帰れる状態を目指します。

## 対象読者

- 会話型 AI(アシスタント・サポート・対話 UI)の体験を設計するデザイナー・PM・エンジニア
- 「汎用チャットを置いただけ」から、ブランドと目的に沿った会話へ進めたい実装者

## 前提知識

- [AI プロダクトの UX パターン](ai-ux-patterns.md) — 不確実性・信頼の提示(会話体験の土台)
- [Agent 向けプロンプト設計](../03-implementation/agent-prompt-design.md) — ペルソナ・トーンをプロンプトへ落とし込む

## 本文

### 概要: 「何でも答える」は設計ではない

テキストボックスに汎用 LLM を繋ぐと、確かに何でも答えます。しかしそれは「設計された会話」ではありません。会話設計とは、**この製品の中でエージェントがどう振る舞うか**——どんな役割で、どんなトーンで、何に答え何を断るか——を意図的に決めることです。

設計がないと、ブランドと合わない口調、答えるべきでない話題への応答、一貫性のなさが生じます。会話設計は、その振る舞いを設計し、[プロンプト](../03-implementation/agent-prompt-design.md)に落とし込む営みです。以下の各節が、その構成要素です。

### ペルソナとトーンの設計

エージェントの「人格」は、明示的に設計しないと、モデルの既定の口調がそのまま出ます。ブランドと目的に合わせて設計します。

- **役割を定義する**: 「フレンドリーな案内役」「簡潔な専門家」など、役割を一言で決めます。役割が決まると、トーン・語彙・回答の長さが自ずと定まります
- **トーンを具体化する**: 一人称・敬語レベル・絵文字の可否・回答の長さを具体的に決めます。「丁寧に」だけでは曖昧なので、良い例・悪い例で示します([Agent 向けプロンプト設計](../03-implementation/agent-prompt-design.md)の少数例)
- **ブランドと整合させる**: 会話のトーンはブランド体験の一部です。他のチャネル(サポート文書・UI 文言)と口調を揃えます
- **人間のふりをさせない**: 親しみやすさのために「私は人間です」と偽らせるのは、信頼を損ねます。AI であることを隠さない設計にします

### 話題の境界と断り方

「何でも答える」は、答えるべきでないことにも答えてしまうことです。**境界**を設計します。

- **スコープを決める**: この製品が答える範囲・答えない範囲を明確にします。医療・法律・投資などの助言、競合の話題、業務外の雑談などをどう扱うかを決めます
- **断り方を設計する**: スコープ外の依頼を、冷たく拒否せず、**次の一手へ導く**断り方にします(「その件はお答えできませんが、○○ならお手伝いできます」)。断り方の質が体験を左右します
- **境界はコードでも守る**: 重要な境界(危険な助言・情報漏えい)は、プロンプトのお願いだけでなく[ガードレール](../06-security/guardrails.md)でも強制します。会話設計とガードレールは別レイヤーです

### 会話の構造(オンボーディング・提案・確認・完了)

良い会話には型があります。放任せず、**構造**を設計します。

- **オンボーディング**: 最初の応答で、何ができるか・どう話しかければよいかを示します。白紙から始めさせず、例や選択肢を提示します([チャットを超える UI](beyond-chat-ui.md))
- **提案と確認**: エージェントから選択肢を提案し、曖昧な依頼には聞き返して意図を確認します。一問一答で終わらせず、対話で絞り込みます
- **完了の合図**: タスクが終わったら「完了しました」「他にありますか」と区切りを示します。終わったのか続くのか曖昧な会話は、利用者を迷わせます

### 記憶の見せ方(覚えていることの透明性)

パーソナライズのためにエージェントは記憶を持ちますが、**何を覚えているかが見えない**と不気味さや不信を生みます。

- **覚えていることを透明にする**: 「以前○○とおっしゃっていたので」と、記憶を使うときはその出所を分かるようにします([長期記憶の実装](../03-implementation/long-term-memory-implementation.md)の UX 面)
- **忘れる手段を用意する**: 「これは覚えないで」「記憶を消して」と言える経路を用意します。記憶の制御が利用者にあることが、安心につながります
- **記憶違いに備える**: 記憶は間違うことがあります。記憶に基づく発言を断定せず、訂正できる形にします

### 多言語・文化差

言語が変わると、敬語・丁寧さ・距離感の適切さも変わります。多言語対応は翻訳だけでは足りません。

- **言語ごとにトーンを調整する**: 直訳では不自然な敬語・距離感になります。言語・文化ごとに適切なトーンを設計します(日本語の敬語・文体は [日本語品質の評価](../04-evaluation/japanese-quality-evaluation.md) の観点)
- **文化差に配慮する**: 婉曲表現・直接性・フォーマルさの好みは文化で異なります。ターゲットに合わせて調整します

### 会話ログからの改善

会話設計は一度で完成しません。実際の会話ログが、改善の最大の材料です。

- **つまずきを見つける**: 誤解された依頼・繰り返された質問・途中離脱を分析し、ペルソナ・境界・オンボーディングを直します
- **プライバシーに配慮して扱う**: 会話には機微な情報が含まれます。ログの保存・分析は[会話データの管理基盤](../05-operations/conversation-data-management.md)の対象として扱います
- **改善を還流する**: 見つかった問題を、プロンプト・境界・例に反映し、同じつまずきを繰り返さないようにします

## 実務での注意点

### アンチパターン

- **汎用チャットを置いただけで会話設計とみなす** → ブランドと合わない・答えるべきでない話題に答える → 役割・トーン・境界を明示的に設計する
- **「丁寧に」などの曖昧な指示でトーンを決める** → 一貫しない → 一人称・敬語・長さを具体化し、良い例・悪い例で示す
- **親しみやすさのため人間のふりをさせる** → 発覚時に信頼を失う → AI であることを隠さない
- **スコープ外を冷たく拒否する** → 体験を損ねる → 次の一手へ導く断り方を設計する
- **境界をプロンプトのお願いだけで守る** → 回避される → 重要な境界はガードレールでも強制する
- **記憶を不透明に使う** → 不気味さ・不信を生む → 覚えていることを透明にし、忘れる手段を用意する
- **多言語対応を翻訳で済ませる** → 不自然な敬語・距離感になる → 言語・文化ごとにトーンを調整する

### チェックリスト

- [ ] 役割・トーン(一人称・敬語・長さ)を具体的に設計し、プロンプトに落としたか
- [ ] AI であることを隠していないか
- [ ] 話題の境界と、次へ導く断り方を設計したか(重要な境界はガードレールでも守っているか)
- [ ] オンボーディング・提案・確認・完了の会話構造があるか
- [ ] 記憶を透明に使い、忘れる手段を用意しているか
- [ ] 多言語で、言語・文化に合わせてトーンを調整しているか
- [ ] 会話ログのつまずきを分析し、設計へ還流しているか
