# Devin

## この記事の目的

クラウド完全自律型コーディングエージェントの代表である Devin(Cognition)の、委任モデル・実行環境・権限とデータ取り扱いを理解し、自分のチーム・用途に合うかを判断できるようになります。「操作するツール」ではなく「タスクを委任する相手」という運用モデルの評価が主眼です。

なお本記事はクラウドの Devin(Devin Cloud)を主軸とします。専用 IDE(Devin Desktop、旧 Windsurf)は [Windsurf(現: Devin Desktop)](windsurf.md) を参照してください。

## 対象読者

- バックログの定型タスク(バグ修正・依存更新・移行)を非同期で処理させたいチーム
- クラウド自律型という形態のセキュリティ・ガバナンスを評価する担当者

## 本文

### 概要

Devin は Cognition が提供する自律型コーディングエージェントです。公式の位置づけは「AI ソフトウェアエンジニア」で、タスク規模の目安として「人間なら 3 時間以内で終わるタスクならおそらく実行できる」と公式が示しています。5 分類([全体像](coding-agents-overview.md))ではクラウド実行型・GitHub / Issue 連携型の代表例です。

2025 年の Windsurf 買収を経て、2026-07 時点の Cognition 製品は **Devin Cloud(本記事)/ Devin Desktop(旧 Windsurf)/ Devin CLI / Devin Review** の 4 本柱に統一されています。モデルは独自の SWE 系モデルを中心に、上位プランでは他社フロンティアモデルも選択できます。

他ツールとの最大の違いは承認モデルです: **コマンド単位の事前承認は存在せず、「委任 → 自律実行 → 事後レビュー」型**で運用します。

### 提供形態と実行環境

- **実行環境**: セッションごとに隔離された Linux VM でリポジトリをクローンして実行します(Windows VM は限定提供)。環境は検証済みスナップショットまたは宣言的 YAML(**Blueprints**、Git 管理可)で定義し、**毎セッションがクリーンなコピーから起動**します(セッション内の変更は環境に残りません)
- **入口(トリガー面)**: Web アプリ / Slack・Microsoft Teams(@Devin メンション)/ Linear・Jira(チケットのアサイン・`!implement` 等のラベル・コメントでのメンション)/ GitHub(App として PR コメントに自動応答)/ REST API(CI からのセッション作成)/ Devin CLI
- **並列実行**: 独立したタスクを複数セッションで同時に走らせる運用を公式が推奨しています。cron 式のスケジュールセッション(依存更新の定期実行など)もあります
- **SCM**: GitHub のほか GitLab / Bitbucket / Azure DevOps(セルフホスト含む)に対応します
- **Enterprise のデプロイ形態**: マルチテナント SaaS に加え、実行環境(Devbox)を顧客専用 VPC に置く Customer Dedicated 構成(AWS PrivateLink 接続)があります

### リポジトリ理解・編集・実行の仕組み

- **リポジトリ理解**: 組織設定でリポジトリを**事前インデックス**し、その上に **DeepWiki**(アーキテクチャ図・出典リンク付きのドキュメントを自動生成)と **Ask Devin**(コードベース Q&A)が乗る構成です。セッション実行時は VM 内のクローンに対するオンデマンド探索も行います
- **ファイル編集**: セッション内に組み込みの VS Code 環境があり、人間は編集をリアルタイムに確認し、**セッションを停止して IDE を直接引き継ぐ**ことができます。成果物はブランチ + PR として提出され、Git の通常手段でレビュー・差し戻しします
- **コマンド実行**: VM 内でフルのシェルアクセスとブラウザ操作(computer use)を持ち、テスト・ビルドを自律実行します。全シェルコマンド・編集・ブラウザ操作は Progress Tab に記録されます

### 設定ファイルとカスタマイズ

Devin への「教え込み」は複数の機構に分かれています。

| 機構 | 役割 |
| --- | --- |
| **AGENTS.md** | リポジトリ内の標準ルールファイル(コーディング前に読み込み) |
| **Knowledge** | 全セッション横断のヒント集。チャットのフィードバックから**Devin 自身が追加を提案**し、エントリごとに適用条件(トリガー)を持つ |
| **Playbooks** | 再利用可能な手順書テンプレート(`*.devin.md`)。Linear / Jira のラベルと対応付け可能 |
| **Blueprints** | 実行環境(VM)の宣言的定義 |
| **Secrets** | 認証情報の管理(保存時暗号化、セッションには環境変数として注入) |

Knowledge / Playbook は API で組織横断的に管理でき、チーム標準の配布が可能です。なお Devin CLI は 2026-07 時点で Knowledge / Playbooks / Secrets に未対応という機能差があります。

### 権限管理とセキュリティ

- **承認モデル**: クラウドセッションにはコマンド単位の事前承認がありません。人間の関与点は ①事前(Ask Devin での方針確認、明確な完了基準の付与)②実行中(Progress Tab の追跡、停止、IDE 引き継ぎ)③事後(PR レビュー)です。[権限とセキュリティ](coding-agent-security.md) の分類では「サンドボックス内自動 + 境界で確認」型の純粋形であり、**PR レビューとブランチ保護が実質の防御境界**になります(公式も main ブランチ保護の有効化を推奨)
- **Enterprise の AI Guardrails**: Devin へのメッセージをリアルタイム解析し、プロンプトインジェクション・データ持ち出し・ポリシー違反を検知して log / warn / block / kill_session の 4 段階で対応する追加レイヤーがあります。違反は監査ログと API に記録されます
- **データ学習の既定(選定上の重要事実)**: **有償のセルフサーブプランでも、既定ではデータが学習改善に使われ得ます**(Data Controls からオプトアウト可能。オプトアウト後はモデルプロバイダーとの ZDR が有効化)。**Enterprise のみ既定で学習不使用**(書面同意なしに学習しないと明記)です
- コンプライアンスは SOC 2 Type II、政府向け提供(Cognition for Government)、Customer Managed Keys(Enterprise)などが公表されています。GitHub App の要求権限(contents / pull requests 等の read & write)も公開されています

### 外部連携(MCP・CI・API)

- **MCP クライアント**: STDIO / SSE(非推奨化中)/ Streamable HTTP に対応。30 以上の事前設定済みサーバー(Linear / Notion / Sentry / Stripe 等)を持つ **MCP Marketplace** があり、カスタムサーバーの追加は**組織管理者のみ**というガバナンス設計です
- **Devin 自身が MCP サーバー**(Devin MCP / DeepWiki MCP)としても公開されており、他のエージェントから Devin を呼ぶ構成が可能です
- **API(v3)**: セッション作成・Knowledge / Playbook / Secret 管理・監査ログ・利用メトリクス(DAU / セッション数 / 消費量)・組織横断管理(Enterprise)を提供します

### チーム導入と提供プラン

- プランは Free / Pro / Max / Teams / Enterprise です(旧 Windsurf と統合された単一体系で、クォータはクラウドセッション・CLI・Desktop を横断します)。具体額は公式料金ページ(参考資料)で確認してください
- 課金は「含有クォータ + オンデマンドクレジット(繰越可)」、**Enterprise のみ ACU(Agent Compute Units)** という消費単位で課金されます。ACU の厳密な定義は公式に記載がなく、アクションの数と複雑さ・VM 時間・帯域で加算されます。非活動時は自動スリープし消費が止まります。Devin Review には PR 単位の支出上限があります
- 組織統制: SAML / OIDC SSO・SCIM・カスタムロール(RBAC)・監査ログ API・IP アクセスリスト・リポジトリ権限管理・複数組織の階層管理(Enterprise)が提供されます

### 代表的なユースケースと向き不向き

**公式が想定する用途**: バックログ解消(バグ修正・チケット処理・テスト作成)、コード移行・モダナイゼーション(COBOL・Java アップグレード・JS→TS 移行など公式事例が厚い領域)、PR レビュー、ドキュメント保守、定期タスクの自動化です。公式ガイドは「検証可能な完了基準(テスト・CI)があるタスクが最適」「大きな仕事は複数セッションに分割」と明言しています。

**向き不向き(特性として)**:

- 向く: 独立性が高く完了基準が機械検証できる定型タスクの大量処理、Issue トラッカー(Linear / Jira)起点の開発フロー、開発者の手元環境を塞がない並列委任。「同僚に依頼する」運用モデルを受け入れられるチーム
- 注意が要る: **事前承認型の統制を求める組織には合いません**(防御は環境隔離 + ブランチ保護 + 事後レビューで設計する必要があります)。対話的なペアプログラミングは主用途ではありません(その領域は Devin Desktop / CLI が担当)。セルフサーブプランの学習利用の既定はオプトアウト方式である点の確認が必須です。ACU / クォータの消費は依頼の質(スコープの明確さ)に大きく依存し、曖昧な依頼はコストに直結します

## 実務での注意点

### アンチパターン

- **曖昧なタスクをそのまま投げる** — 自律実行型では途中修正の機会が少なく、誤った方向の作業がそのまま消費(クォータ / ACU)になります。→ 完了基準・制約を明確にし、大きな仕事は分割します([依頼設計](coding-agent-prompting.md))
- **ブランチ保護なしで運用する** — 事前承認がないモデルでは、マージ前の人手レビューが唯一の品質ゲートです。→ main のブランチ保護と必須チェックを先に整備します
- **セルフサーブプランのまま組織展開する** — 学習利用の既定・ガードレール・監査は Enterprise と大きく異なります。→ 組織利用は Data Controls の確認と Enterprise 機能の評価を先に行います

### チェックリスト

- [ ] 委任するタスクに機械検証可能な完了基準(テスト・CI)があるか
- [ ] main のブランチ保護・必須チェック・レビュー体制が整備されているか
- [ ] 学習利用の既定(セルフサーブ: オプトアウト方式 / Enterprise: 既定不使用)を確認し設定したか
- [ ] Secrets 機能で認証情報を管理し、スコープ(組織 / 個人)を適切に分けたか
- [ ] 支出制御(クォータ監視・Review の PR 単位上限・自動スリープ)を確認したか
- [ ] MCP サーバー追加が管理者ゲートで統制されているか
