# GPU・AI ハードウェアの基礎

## この記事の目的

セルフホストや調達の会話に参加できる最小限のハードウェア知識を持てるようになります。「なぜ GPU なのか」「VRAM とモデルサイズの関係」「なぜメモリ帯域が効くのか」「ハードの類型と調達の選択肢」を、選定判断に使える範囲で理解し、`このモデルはこの GPU に載るか・どこで買うか` を大づかみに見積もれる状態を目指します。詳細な理論ではなく、**インフラ選定の会話に必要な範囲**に絞ります。

- クラウド GPU・オンプレ購入のコスト比較を求められ、判断軸がほしいテックリード

## 本文

### 概要: なぜハードウェアの話が要るのか

API を借りて使う限り、ハードウェアは意識しません。しかし**セルフホスト**([セルフホスト推論の実務](self-hosted-inference.md))・**ローカル実行**([ローカル・オンデバイス LLM の実務](../03-implementation/local-and-on-device-llm.md))を検討した瞬間に、「どの GPU に載るか・いくらかかるか」が判断を左右します。本記事はその会話に必要な最小限の知識を、製品名でなく**類型と概算**で提供します。

### なぜ GPU か

LLM の処理は、大量の**行列の掛け算**の繰り返しです。

- GPU は、同じ計算を大量のコアで**並列に**行うのが得意で、行列積のような処理を CPU より桁違いに速くこなします
- LLM の推論・学習はこの並列行列積が中心のため、GPU(や同種のアクセラレータ)が使われます
- 「賢さ」の計算そのものより、**それを現実的な時間で回す**ためにハードウェアが要る、と捉えると位置づけが掴めます

### VRAM とモデルサイズの関係

セルフホストで最初にぶつかる壁が **VRAM(GPU の搭載メモリ)**です。モデルが VRAM に載らなければ、そもそも動きません。

- **概算式**: 必要な VRAM は、ざっくり「**パラメータ数 × 1 パラメータあたりのバイト数**」+ 実行時のオーバーヘッド(後述の KV キャッシュ等)で見積もります
- **精度でバイト数が変わる**: 1 パラメータあたりのバイト数は数値精度で決まります。FP16(半精度)なら約 2 バイト、INT8 なら約 1 バイト、INT4 なら約 0.5 バイトです
- **概算の例**: 70 億(7B)パラメータのモデルなら、FP16 で約 14 GB(7B × 2 バイト)、INT8 で約 7 GB、INT4 で約 3.5 GB が重みのおおよその目安です。これに後述の KV キャッシュや作業領域が上乗せされます
- **量子化で載せる**: **量子化(quantization)** — 重みを低い精度で表現してサイズを縮める手法 — を使うと、同じモデルをより小さい VRAM に載せられます。ただし精度を落とすほど品質が下がりうるため、**量子化後の品質は自社タスクで検証**します(下げれば必ず載るが、使い物になるかは別)

### メモリ帯域律速という直感

意外なポイントですが、LLM 推論の速度は**計算能力より「メモリ帯域」で決まる場面が多い**です。

- **なぜか**: テキストを 1 トークンずつ生成する推論では、毎トークンごとに大量の重みをメモリから読み出します。計算そのものより、**重みを読み出す速度(メモリ帯域)**が律速になりがちです
- **選定への含意**: 「演算性能(FLOPS)が高い GPU=推論が速い」とは限りません。**メモリ帯域と VRAM 容量**が、推論では効くことが多いです
- **バッチで効率が上がる**: 複数リクエストをまとめて処理(バッチング)すると、読み出した重みを使い回せて効率が上がります。これがサーバー側サービングの要点です([セルフホスト推論の実務](self-hosted-inference.md))

### ハードの類型

製品名は変化が速いため、**類型**で押さえます(具体の製品・世代は下記 TODO)。

| 類型 | 位置づけ | 主な用途 |
| --- | --- | --- |
| データセンター GPU | 大 VRAM・高帯域・高価。サーバー向け | 本格的なセルフホスト推論・学習 |
| コンシューマ GPU | 中 VRAM・入手しやすい | 小〜中規模モデルの実行・検証 |
| NPU・端末アクセラレータ | 省電力・端末内蔵 | オンデバイス推論(スマホ・PC) |

- **データセンター GPU**: 大きな VRAM とメモリ帯域を持ち、大規模モデル・高スループットに向きますが高価です
- **コンシューマ GPU**: 入手しやすく、量子化した中小モデルの実行・検証に使えます。VRAM が上限になりがちです
- **NPU・端末アクセラレータ**: スマホ・PC に内蔵され、省電力でオンデバイス推論を担います([ローカル・オンデバイス LLM の実務](../03-implementation/local-and-on-device-llm.md))

### 調達の選択肢と TCO

「GPU を使う」には調達方法が複数あり、**総保有コスト(TCO)**で比べます。

- **クラウド GPU(従量・予約)**: 使った分だけ払う/一定期間予約する。初期投資が小さく、需要変動に強い。長時間フル稼働だと割高になりやすい
- **専有・ホスティング**: 専有インスタンスや GPU ホスティングを借りる。クラウドとオンプレの中間
- **オンプレ購入**: 自社で GPU を購入・設置。閉域網・データ主権の要件や、常時フル稼働なら有利になりうるが、**初期投資と運用負担が大きい**
- **TCO で見る**: 本体価格だけでなく、電力・冷却・設置・保守・人件費・陳腐化(GPU は世代交代が速い)を含めて比べます。「買えば安い」は稼働率と運用体制次第です

### 「GPU を持つ」ことの運用負担

見落とされがちですが、GPU を自前で持つと**運用の仕事が増えます**。

- ドライバ・ライブラリのバージョン管理、故障対応、容量計画、電力・冷却、世代交代への追従が発生します
- クラウドはこの多くを肩代わりします。オンプレは自由度と引き換えに、これらを自社で担います
- 「原価が下がるはず」と始めても、運用負担・稼働率を織り込むと API を借りる方が安いことも多いです(損益分岐の詳細は [セルフホスト推論の実務](self-hosted-inference.md))

## 実務での注意点

### アンチパターン

- **演算性能(FLOPS)だけで GPU を選ぶ** → 推論はメモリ帯域・VRAM 律速のことが多く、期待どおり速くならない → VRAM 容量とメモリ帯域を重視する
- **VRAM を確認せずモデルを選ぶ** → そもそも載らず動かない → パラメータ数 × 精度で概算し、載るかを先に確認する
- **量子化すれば無条件で使えると考える** → 精度低下で品質が落ちる → 量子化後の品質を自社タスクで検証する
- **「買えば安い」で TCO を見ずにオンプレ購入する** → 電力・運用・陳腐化で結局割高になる → 稼働率と運用体制込みの TCO で比較する
- **GPU 保有の運用負担を見積もらない** → ドライバ・故障・世代交代の負担が想定外に重い → 運用の仕事が増えることを前提に判断する

### チェックリスト

- [ ] 対象モデルの必要 VRAM を「パラメータ数 × 精度」+ 余裕で概算した
- [ ] 量子化を使う場合、量子化後の品質を自社タスクで検証する計画がある
- [ ] GPU 選定で VRAM 容量・メモリ帯域を重視している
- [ ] 調達(クラウド従量/予約・専有・オンプレ購入)を TCO で比較した
- [ ] GPU 保有に伴う運用負担(ドライバ・故障・電力・世代交代)を織り込んだ
- [ ] セルフホストが API を借りるより本当に安いか、稼働率込みで確認した
