# 画像生成のプロダクト組み込み

## この記事の目的

画像生成を「デモ」ではなく**プロダクト機能**として組み込む際の、選定・制御・権利・安全の判断ができるようになります。ユースケース類型・選定軸・出力の一貫性管理・プロンプト資産管理・安全と権利(不適切生成の防止・商用利用・来歴表示)・人のレビューゲートを、実務判断として持ち帰れる状態を目指します。モデルの顔ぶれと権利の細目は変化が速いため、本記事は**選定軸と類型**に徹し、具体は確認先を示します。

## 対象読者

- 画像生成をプロダクト機能(素材生成・編集・バリエーション)として組み込む実装担当・プロダクト設計者
- 生成画像の商用利用・権利・安全性の判断を求められる立場の人

## 本文

### 概要: デモと本番の距離

画像生成は「1 枚それらしいものが出る」デモは簡単ですが、**プロダクト機能**にするには、一貫性・権利・安全・レビューの設計が要ります。本記事はこの「デモと本番の距離」を埋める判断を扱います。

> **最終確認日: 2026-07。** 本記事で触れる画像生成の提供類型・機能・商用利用条件・来歴機能は変化が非常に速いため、モデル名は挙げず類型で書きます。採用時は各社公式ページと利用規約で必ず再確認してください(後述の TODO)。

### ユースケース類型

画像生成の組み込みは、用途の類型で設計が変わります。

| 類型 | 内容 | 主な注意点 |
| --- | --- | --- |
| 素材生成 | ゼロから画像を作る(t2i) | 品質・指示追従・権利 |
| バリエーション | 既存画像から派生を作る | 一貫性・元画像の権利 |
| 編集 | 既存画像の一部を変える(インペインティング等) | 領域指定・自然な合成 |
| パーソナライズ | ユーザー・ブランドに合わせて生成 | スタイル固定・安全 |

- **編集(インペインティング)**: **インペインティング(inpainting)** — 画像のマスクした領域だけを再生成する編集 — や、画像の外側を拡張するアウトペインティングなど、ゼロ生成以外の編集機能も広く提供されています(2026-07 時点)。「作る」だけでなく「直す」用途を検討します
- **用途で選定軸が変わる**: 素材生成は品質・指示追従、編集は領域制御の精度、パーソナライズはスタイル固定と安全、と用途ごとに重視点が違います

### 選定軸

画像生成モデル・サービスの選定は、次の軸で行います(具体名は挙げず軸で判断)。

- **品質・指示追従**: 用途に足る品質か、プロンプトの指示にどれだけ従うか
- **制御性**: 領域指定(インペインティング)・参照画像・スタイル制御など、細かく制御できるか
- **速度・コスト**: 生成の速さと課金構造。大量生成なら特に効きます
- **提供形態**: API 型 / 統合サービス型 / オープンウェイト型(自ホスト)の 3 類型があります(2026-07 時点)。データを外に出せない・自ホストしたいなら、オープンウェイトを自環境で動かす選択もあります([セルフホスト推論の実務](../05-operations/self-hosted-inference.md)の画像版)
- **商用利用・来歴**: 後述の権利・来歴の条件が用途に合うか

### 出力の一貫性管理

プロダクトでは、**毎回バラバラな画像が出ては困る**ことが多いです。一貫性を管理します。

- **スタイル固定**: 同じトーン・画風で揃えたいなら、スタイルを指定・固定します。ブランド素材では特に重要です
- **シード・参照画像**: 再現性が要るなら、シード(乱数の種)の固定や参照画像の指定で、出力のブレを抑えます。「同じ入力で似た結果」を得る手段です
- **バリエーションの範囲**: 「どこまで揃え、どこを変えるか」を設計します。全部同じでは芸がなく、バラバラでは統一感がありません

### プロンプト資産の管理

画像生成のプロンプトも、テキスト同様に**資産**です([プロンプト管理とバージョニング](../03-implementation/prompt-management.md)の画像版)。

- **プロンプトを管理する**: 良い結果を出すプロンプト(スタイル指定・ネガティブ指定・参照)を、資産として記録・バージョン管理します。属人的な「うまくいく呪文」を暗黙知にしないことです
- **変更の影響を測る**: プロンプト・モデルを変えたら、出力がどう変わるかを確認します。モデル更新で作風が変わることもあります
- **再現性とセットで**: シード・パラメータもプロンプトと一緒に記録し、「あの画像をもう一度」に応えられるようにします

### 安全と権利

画像生成は、**安全と権利の論点**が重い領域です。ここを軽視すると、法的・レピュテーションの事故になります。

- **不適切生成の防止**: 暴力的・性的・差別的な画像、他者を害する画像の生成を防ぐガードレールを設けます([ガードレール](../06-security/guardrails.md)の画像版)。ユーザー入力によっては不適切な生成を試みられます
- **実在人物・商標**: 実在の人物(とくに無断での)・商標・著名キャラクターの生成は、肖像権・商標権・著作権の問題になります。これらを避ける・検知する設計を検討します
- **生成物の商用利用とライセンス**: 生成画像を商用利用してよいか・権利は誰に帰属するかは、**各社の利用規約で大きく異なります**(2026-07 時点で、年商の閾値・IP 補償の有無・オープンウェイトのライセンス層など、条件が分岐)。本記事は断定せず、使うサービスの規約と法務、規制の入口([業界別規制マップ](../09-business/industry-regulations-map.md))での確認を促します
- **来歴表示の潮流**: 生成画像に**来歴情報(Content Credentials / C2PA)や電子透かし**を付与する動きが広がっています(2026-07 時点で、クラウド API の主要各社は来歴・透かしを既定付与する方向)。一方、**オープンウェイトを自ホストする場合、来歴は自動では付かず、必要なら自分で付与を実装**する必要があります。生成物であることを示す責任をどう果たすかを設計に含めます

### 人のレビューゲート

プロダクトでは、生成物を**そのまま公開しない**設計が基本です。

- **公開前の人のレビュー**: とくに外部公開・ブランド利用する画像は、公開前に人がレビューするゲートを設けます([Human-in-the-Loop 設計](../02-architecture/human-in-the-loop.md))。不適切・権利侵害・品質不足を止める最後の砦です
- **リスクに応じたゲート**: 内部利用の下書きは軽く、外部公開・広告は厳しく、とリスクでゲートの強さを変えます
- **自動チェックと併用**: 不適切生成の自動検知(ガードレール)と人のレビューを組み合わせ、人が見る前に明らかなものは弾きます

## 実務での注意点

### アンチパターン

- **デモの 1 枚でプロダクト化を判断する** → 一貫性・権利・安全の設計を欠いたまま本番に出す → 選定軸・一貫性・権利・レビューを設計してから組み込む
- **商用利用の可否を確認せず本番利用する** → 規約違反・権利侵害になる → 使うサービスの規約と法務で商用可否・権利帰属を確認する
- **不適切生成のガードレールを設けない** → 有害画像・権利侵害画像が生成・公開される → 入力・出力のガードレールと人のレビューを設ける
- **オープンウェイト自ホストで来歴を付けない** → 生成物であることが示せず、透明性の責任を果たせない → 自ホスト側で来歴付与を実装する
- **生成物を人のレビューなしで外部公開する** → 不適切・権利侵害・品質不足がそのまま出る → 公開前の人のレビューゲートを設ける
