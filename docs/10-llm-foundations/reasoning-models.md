# 推論モデル(考える時間を使う LLM)

## この記事の目的

「応答の前に考える時間を使う」推論モデル(reasoning model)の**仕組みの直感**と、使いどころ・思考量の制御・プロンプトの変化を、実務判断にできるようになります。ライブラリのあちこちで断片的に触れてきた「thinking」「推論トークン」「reasoning effort」を、1 本にまとめて整理するのが目的です。

## 対象読者

- 推論モデルを使うか迷っている、または使っているが「なぜ速い問いで遅く・高くなるのか」を掴みたいエンジニア
- 「考えさせるモデル」に対して、通常モデルと同じプロンプト・同じ期待で臨んで戸惑っている人

### 仕組みの直感: 答える前に考えを書く

推論モデルは、[LLM はどうやってテキストを生成するか](how-llms-generate-text.md)で見た「**書き出した考察が、以降の予測の条件に加わる**」という性質を、製品として組み込んだものです。

- 通常のモデルは、質問に対していきなり答えを書き始めます
- 推論モデルは、答える前に**中間の思考(推論トークン)**を生成します。問題を分解し、候補を検討し、検証してから、最終的な答えを出します
- この「考える過程」は、段階的推論(chain of thought)というプロンプト技法([プロンプトエンジニアリングの基礎技法](../03-implementation/prompt-engineering-fundamentals.md))を、プロンプトで頼まなくてもモデルが自分でやる形にしたもの、と捉えると直感が働きます

要点は、**思考にもトークンを消費する**ことです。推論トークンは通常コストとして課金され、生成に時間がかかります。「考える時間」はタダではありません。

### 効くタスクと効かないタスク

推論モデルは万能ではなく、**向き不向き**があります。

| 向く(考える価値がある) | 向かない(考えても無駄・逆効果) |
| --- | --- |
| 多段の推論が要る問題(数学・計画・複雑なデバッグ) | 単純な事実検索・定型の抽出・分類 |
| 検証可能な問題(答えの正しさを確かめられる) | 低レイテンシが要る対話・大量処理 |
| 制約が多く、慎重な検討が要る判断 | 答えが一意で、考える余地がない問い |

判断の勘所は、**「考えれば質が上がる問題か」**です。答えが決まっている・検索すれば済む問いに推論モデルを使うと、遅く・高くなるだけで質は上がりません。逆に、腰を据えた推論が要る問題では、通常モデルより大きく精度が上がります([LLM の能力と限界の由来](capabilities-and-limits.md)の「向き不向き」の判断軸)。

### 思考量の制御とコスト・レイテンシ設計

推論モデルでは、**どれだけ考えさせるか**を制御できるのが一般的です(2026-07 時点)。

- **思考量の制御**: 多くのモデルは「努力(effort)の水準」や「思考の予算」といったパラメータで、考える量を段階的に調整できます。名称・仕様はモデルにより異なります([モデル間の違いと移行(横断比較)](../03-implementation/cross-model-prompting.md))
- **コスト・レイテンシとの直結**: 思考量を上げるほど、精度は上がりうる一方で**トークンコストとレイテンシが増えます**。思考量は「精度 ↔ コスト・速度」のつまみです
- **タスクで使い分ける**: 難しいタスクには高い思考量を、簡単なタスクには低い思考量(または通常モデル)を割り当てます。全タスクを最大思考量で回すのは、コストの無駄です([レイテンシ最適化](../05-operations/latency-optimization.md)・[コスト管理](../05-operations/cost-management.md))

### 考えすぎ(overthinking)

思考量を上げれば質が上がる、とは限りません。**考えすぎ(overthinking)**という失敗があります。

- 簡単な問題に高い思考量を割り当てると、モデルが無駄に長考し、コスト・レイテンシだけが増えます。ときには考えすぎて、単純な答えを外すこともあります
- とくに、指示が矛盾している・停止条件が曖昧・ツールアクセスが自由すぎる状況では、高い思考量が無駄な探索や品質の退行を招きます
- 対策は、**思考量を「品質の主ノブ」として上げ続けないこと**です。タスクに見合った水準を評価で見つけ、上げるのは精度向上が確認できるときだけにします

### プロンプトの変化: 手順を細かく書かない

推論モデルは、**プロンプトの書き方を変えます**([プロンプトエンジニアリングの基礎技法](../03-implementation/prompt-engineering-fundamentals.md)の「思考の誘導と推論モデル」の詳解)。

- **「ステップバイステップで考えて」は不要**: モデルが内部で推論するため、明示的な段階的推論の指示は効果が薄く、ときに内部の探索と衝突します
- **手順の細かい指定より、目標と制約を渡す**: 中間手順を細かく規定するより、**ゴール・強い制約・出力の要件**を明確にして、過程はモデルに任せるのが基本方針です
- **モデル別に作法が違う**: 具体的な制御(思考の促し方・抑え方)はモデルごとに異なります(モデル別の作法は [Claude](../03-implementation/claude-prompting.md) / [OpenAI](../03-implementation/openai-prompting.md) / [Gemini](../03-implementation/gemini-prompting.md) の各ガイド)

### 評価の注意: 思考は見えず、揺れる

推論モデルの評価には固有の注意があります。

- **思考の非公開**: モデルの生の思考は、公開されない・要約でしか見えないことが多いです。「なぜその答えになったか」を思考から追えるとは限りません
- **再現性**: 思考の過程は実行ごとに揺れます。[Agent 評価の基礎](../04-evaluation/agent-evaluation-basics.md)の複数回実行が、推論モデルではとくに重要です
- **コストと品質を同時に測る**: 思考量を変えたときの精度とコスト・レイテンシを両方見ます。「思考量を上げたら精度が 2 ポイント上がったがコストが 3 倍」を見逃さないようにします

## 実務での注意点

### アンチパターン

- **単純な検索・分類に推論モデルを使う** → 遅く・高くなるだけで質は上がらない → 考える価値のある(多段推論・検証可能な)問題に絞る
- **思考量を常に最大にする** → 簡単な問題で overthinking を招き、コスト・レイテンシが暴走する → タスクに見合った思考量を評価で見つける
- **「ステップバイステップで考えて」と指示する** → 内部推論と衝突し、効果が薄い → 目標・制約・出力要件を渡し、過程は任せる
- **通常モデルと同じ低レイテンシを期待する** → 思考にトークンと時間を使うため応答が遅い → レイテンシ要件のあるタスクでは思考量を下げるか通常モデルにする
- **1 回実行で評価する** → 思考が揺れるため結果を誤認する → 複数回実行し、コストと品質を同時に測る

### チェックリスト

- [ ] 推論モデルを、考える価値のある問題(多段推論・検証可能)に使っている
- [ ] 思考量(effort・予算)をタスクの難易度で使い分けている
- [ ] 思考量を「品質の主ノブ」として上げ続けていない(overthinking を避けている)
- [ ] 手順を細かく指定せず、目標・制約・出力要件を渡している
- [ ] 思考にもトークンコスト・レイテンシがかかることを見積もりに入れている
- [ ] 複数回実行で評価し、精度とコスト・レイテンシを同時に測っている
