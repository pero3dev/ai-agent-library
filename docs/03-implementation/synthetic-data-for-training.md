# 学習用合成データの実務

## この記事の目的

ファインチューニング(FT)・蒸留・選好調整に使うデータを LLM で生成する際の、**作り方の型と品質管理・落とし穴**を押さえられるようになります。「LLM に量産させれば安く学習データが作れる」という発想の危うさ(崩壊・単調化・汚染・権利)を理解し、実際に使える合成データを設計できる状態がゴールです。

## 対象読者

- FT・蒸留のための学習データを、LLM を使って作ろうとしているエンジニア
- 手元の学習データが足りず、合成で補うことの是非と作り方を判断したいエンジニア

## 前提知識

- [ファインチューニングと蒸留](fine-tuning-and-distillation.md) — FT・蒸留を「いつ選ぶか」の判断と手法(本記事はそのデータ作成側)
- [LLM の学習パイプライン(事前学習から選好調整まで)](../10-llm-foundations/llm-training-pipeline.md) — SFT・選好調整で挙動が決まる仕組み(なぜそのデータが要るか)

### 用途の類型

学習用合成データは、目的で作り方が変わります。

| 用途 | 作るもの | 使い先 |
| --- | --- | --- |
| SFT 用の指示応答 | 「指示 → 望ましい応答」のペア | 指示追従・形式・文体を教える教師ありFT |
| 蒸留(教師出力) | 上位モデルに解かせた出力 | 小型モデルに上位モデルの挙動を移す |
| 選好ペア | 同じ入力への「良い応答 / 悪い応答」の組 | 選好最適化(DPO 等)で優劣を教える |
| 既存データの拡張 | 少量の実データを言い換え・多様化 | 不足しているパターンの補強 |

用途を先に決めます。「とりあえず大量に生成」は、どの用途にも中途半端なデータを生みます。

### 生成の型

- **シード拡張**: 少数の良質な実例(シード)を種に、LLM でバリエーションを生成します。実データの分布を保ちやすい一方、シードの偏りが増幅されます
- **ペルソナ・条件の多様化**: 「初心者向け」「専門家向け」など視点・条件を変えて生成し、多様性を稼ぎます。単調化(後述)への対策になります
- **自己生成 + フィルタ**: モデルに大量生成させ、検証器で良質なものだけを残します。**フィルタの質が全体の質を決めます**(生成そのものより、選別の設計が本体です)
- **蒸留は教師出力の選別が要**: 上位モデルの出力をそのまま使わず、正しさ・形式を検証してから採用します

### 品質管理

合成データの品質管理は、生成より**選別と計測**が中心です。

- **検証器でフィルタリングする**: 生成物を、決定的な検証(スキーマ・テスト・照合)やモデル判定([LLM-as-a-Judge](../04-evaluation/llm-as-a-judge.md))で選別します。「生成は難しいが検証は易しい」構造を作れる用途ほど、合成データは有効です([ループ内フィードバックと検証器の設計](loop-feedback-and-verification.md))
- **多様性を測る**: 大量生成は、似たようなサンプルばかりになりがちです。重複・近重複の割合や表現の分散を測り、多様性が確保されているか確認します(単調なデータで学習すると、モデルの出力も単調になります)
- **人手サンプリング検品**: 全数は無理でも、無作為抽出した一定数を人が確認します。「もっともらしいが誤り」の合成データは、機械フィルタをすり抜けることがあります
- **正解ラベルの信頼性**: 生成器と同系のモデルで正解を付けると、両者が共有する誤りが検出できません(盲点の共有)。ラベルは独立した手段で検証します

### 劣化リスク: モデル崩壊と単調化

合成データ最大の落とし穴が、**自己出力での学習を繰り返すことによる劣化**です。

- **モデル崩壊(model collapse)**: モデルの出力を学習データにして次のモデルを訓練する、を繰り返すと、分布の裾(まれなパターン)が失われ、出力が平均的・単調に崩れていく現象です。合成データだけで世代を重ねるほど危険が増します
- **スタイルの単調化**: 単一モデルの出力で学習すると、そのモデルの言い回し・構造の癖に染まります。多様性の確保(ペルソナ多様化・複数ソース)が対策です
- **実データを混ぜる**: 崩壊を避ける実務的な指針は、**合成データを実データの代替ではなく補強として使い、実データを一定割合混ぜる**ことです。「実データが足りないから全部合成」は最も崩壊しやすい構成です

### 権利と規約: 教師モデル出力の利用条件

蒸留や合成データ生成に**他社の LLM の出力を使う**場合、その利用条件は提供者ごとに異なります。

- 一部の提供者は、自社モデルの出力を競合モデルの学習に使うことを利用規約で制限しています。制限の有無・範囲は提供者と時期で変わります
- **断定を避け、利用規約を必ず確認**します。契約面のチェックは [コンプライアンスとガバナンス](../06-security/compliance-and-governance.md) のベンダー契約確認と整合させます
- 自社モデル・オープンウェイトモデルの出力を使う、実データを使うなど、権利上安全な選択肢も検討します

### 評価データとの厳格な分離

学習用合成データで最も起こりやすい事故が、**評価データへの汚染**です。

- 生成した学習データが評価セットと重なると、評価が過大に出ます(見た問題を解けるのは当然です)。学習用と評価用の生成は**厳格に分離**します([評価データセットの構築と保守](../04-evaluation/evaluation-datasets.md))
- 同じシードから学習用と評価用を両方作ると、表現違いの実質同一データが混入します。シードのレベルで分けます
- 汚染は静かに起きます。「合成データで学習したらスコアが上がった」が、汚染による見せかけでないかを、独立した評価セットで確認します

## 実務での注意点

### アンチパターン

- **実データが足りないから全部合成で補う** → モデル崩壊・単調化で品質が劣化する → 合成は補強にとどめ、実データを一定割合混ぜる
- **大量生成すれば良いデータになると考える** → 似たサンプルの山になり、フィルタもかけない → 生成より選別(検証器フィルタ)と多様性計測に注力する
- **生成器と同系モデルで正解ラベルを付ける** → 共有する誤りが検出できない(盲点の共有) → ラベルは独立した手段で検証し、人手で抜き取り確認する
- **学習用と評価用を同じ生成・同じシードから作る** → 評価が汚染され、スコアが過大に出る → 学習用と評価用をシードのレベルで厳格に分ける
- **他社モデルの出力を規約確認なしに学習に使う** → 利用規約違反のリスク → 提供者の利用条件を確認し、契約チェックと整合させる
