# ケーススタディ: データ分析 Agent(もっともらしい誤りとの戦い)

> **注記:** 本記事は、本ライブラリの設計判断を 1 つの物語で通すために構成した**架空の事例(構成事例)**です。登場する組織・数値は実在のものではありませんが、各段階の判断と失敗は実際のプロジェクトで広く観察されるものに基づいています。

## この記事の目的

「聞けば数字を出してくれる」データ分析 Agent が、**動くのに信用されない**状態に陥り、そこから検証可能な構造に作り直して信頼を回復するまでを追います。もっともらしく誤る分析にどう気付き、どう構造で防いだかを、実務の判断として持ち帰れる状態を目指します。設計ガイドの正本は [データ分析エージェント](../13-domain-agents/data-analysis-agents.md) で、本記事はそれを物語で裏付けます。

## 対象読者

- 自然言語からの分析(Text-to-SQL・集計)を提供し、精度と信頼で苦戦しているエンジニア・データ担当
- 「デモは動いたが本番で信用されない」状態から抜け出す道筋がほしい人

## 本文

### 概要: 「動く」と「信用される」は別だった

題材は社内データに対する分析アシスタントです。「先月の地域別売上を出して」と聞くと SQL を書いて実行し、表とグラフを返します。デモは鮮やかに動きました。ところが本番で使い始めると、**現場が数字を信用しない**という壁にぶつかりました。原因は、実行が成功しているのに集計の意味が間違っている「もっともらしい誤り」でした。この事例は、その誤りに気付き、検証可能な構造へ作り直す過程です。

### 詳細: v1 — デモは動いたが、静かに間違えていた

初期構成は「スキーマを渡して SQL を生成・実行し、結果を可視化する」だけでした。問題は、**エラーが出ないまま誤る**ことでした([静かな失敗](../05-operations/observability-and-tracing.md))。

- **論理削除の無視**: `status = 9`(取消)の行を除外せず集計し、売上が過大に出ました。エラーは出ません
- **税抜・税込の混同**: 列名だけでは単位が分からず、税抜金額を税込のつもりで足しました
- **結合の取り違え**: 一対多の結合で明細が重複カウントされ、それらしい大きな数字が出ました

いずれも「実行は成功・グラフは綺麗・数字は間違い」です。現場が「なんとなく数字が大きい気がする」と気付いたのは幸運で、**気付かれずに意思決定に使われていたら**もっと危険でした。

### 詳細: v2 — スキーマに意味を与え、指標を一元化する

最初の改善は、モデルを賢くすることではなく、**与える文脈を整える**ことでした([データ分析エージェント](../13-domain-agents/data-analysis-agents.md))。

- **意味を添えたスキーマと分析用ビュー**: 物理名だけでなく「`status=9` は取消」「金額は税抜」といった業務上の意味を渡し、論理削除除外・結合済みの**分析用ビュー**を用意して、そこだけを対象にしました。誤りの多くが、このデータ整備で消えました
- **指標定義の一元化**: 「売上」「粗利」の定義をその場で組み立てさせるのをやめ、検証済みの SQL 断片(セマンティックレイヤーの入口)として渡しました。集計の取り違えの余地が構造的に減りました

### 詳細: v3 — 検証を二重にし、「わからない」を言えるようにする

文脈を整えても、誤りをゼロにはできません。そこで**検証を構造に組み込みました**([ループのフィードバックと検証](../03-implementation/loop-feedback-and-verification.md))。

- **既知値との突き合わせ**: 主要 KPI の月次値など、別経路で正しいと分かっている数字と自動照合し、ズレたら結果を出さずに差し戻す仕組みにしました
- **別経路集計での再現**: 重要な数値は、別の書き方の SQL でもう一度求めて一致を確認しました。一致しなければ、どちらかの前提が誤っています
- **クエリと条件を必ず併記**: グラフだけでなく、使った SQL・対象期間・フィルタを毎回添え、利用者が「何を数えた数字か」を検証できるようにしました
- **「わからない」を言えるように**: 確度が低い・前提が曖昧な問いには、無理に答えず「この条件では判断できません」と返す設計にしました([信頼度と較正](../04-evaluation/confidence-and-calibration.md))。もっともらしい嘘より、正直な保留のほうが信頼されました

### 詳細: v3 — 利用者教育で「鵜呑み」を防ぐ

構造だけでなく、使う側の期待値も調整しました。

- **鵜呑みを防ぐ**: 「この数字は補助であり、重要な意思決定では併記された SQL と条件を確認してください」と、UI と社内周知で伝えました。ツールを「答えを出す魔法」ではなく「検証できる下書き」として位置づけ直しました
- **正解つき評価セットで回帰を防ぐ**: 現場で実際に間違えたパターン(論理削除・税抜税込・重複結合)を、実行結果の一致で採点する評価セットに加え、モデル更新やプロンプト変更での再発を検知できるようにしました([評価データセット](../04-evaluation/evaluation-datasets.md))

### 設計判断: この事例の教訓

1. **最大の敵はエラーでなく「もっともらしい誤り」** — 実行が成功しても意味が誤る。エラーがないことを正しさと取り違えないのが出発点でした
2. **精度はモデルより文脈とデータ整備で決まった** — 分析用ビューと指標の一元化が、誤りの多くを構造的に消しました。モデルを替える前に、与えるものを整えました
3. **検証を二重化し、根拠を必ず見せた** — 既知値照合・別経路再現・SQL 併記で、誤りを実害にしない砦を作りました
4. **「わからない」と言えることが信頼を生んだ** — 正直な保留が、もっともらしい断定より信用されました

## 実務での注意点

### アンチパターン

- **デモが動いたことを本番品質と誤認する** → 静かな誤りが意思決定に紛れ込む → 既知値照合・別経路再現で意味を検証してから信用する
- **精度問題をモデルの賢さで解こうとする** → 論理削除・単位・結合の誤りは消えない → スキーマに意味を添え、分析用ビューと指標の一元化でデータ側を整える
- **グラフ・結論だけを見せる** → 利用者が何を数えた数字か検証できない → 使った SQL・期間・条件を必ず併記する
- **無理に答えを出す** → もっともらしい嘘が信頼を壊す → 確度が低ければ「わからない」と言える設計にする
- **架空事例の誤りパターンを網羅と誤解する** → 自社データ固有の落とし穴は別にある → 自社の「現場で実際に間違えるパターン」を評価セットに集める

### チェックリスト

- [ ] エラーの有無でなく、既知値照合・別経路再現で集計の意味を検証しているか
- [ ] スキーマに業務上の意味を添え、分析用ビュー・指標の一元化でデータ側を整えたか
- [ ] 結果に、使った SQL・対象期間・フィルタを必ず併記しているか
- [ ] 確度が低い問いに「わからない」と言える設計になっているか
- [ ] 現場で実際に間違えたパターンを、実行結果の一致で採点する評価セットに含めたか
- [ ] 利用者に「検証できる下書き」であることを伝え、鵜呑みを防いでいるか
