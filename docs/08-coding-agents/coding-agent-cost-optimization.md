# コーディングエージェントのコスト最適化

## この記事の目的

コーディングエージェントの消費(トークン・クレジット・クォータ)がどこで発生するかを理解し、**品質を落とさずに**消費を減らす判断ができるようになります。ツール非依存の原則を扱い、ツール固有の操作は各実践ガイドに委ねます。

なお本記事は、エージェントを**使う**側のコスト管理です。エージェントを**作る**側のコスト設計は [コスト管理](../05-operations/cost-management.md) を参照してください(原理は共通ですが、打てる手が異なります)。

## 対象読者

- 従量・クレジット型プランで消費が想定を超え始めたエンジニア
- チームのエージェント利用コストに予算責任を持つテックリード・EM

## 前提知識

- [AI コーディングエージェントの分類と全体像](coding-agents-overview.md)
- [コーディングエージェントへの依頼設計](coding-agent-prompting.md) — 依頼の質は消費に直結します

## 本文

### 概要: コストはどこで発生するか

コーディングエージェントの消費構造は、次の 1 行に集約されます: **消費 ≒ ループ回数 ×(毎ターン再送されるコンテキスト + 新規トークン)**。

エージェントは 1 回の依頼の中で「観測 → 思考 → 行動」のループを何十回も回し、**そのたびに会話履歴・ルールファイル・ツール出力を含むコンテキスト全体をモデルに再送**します。このため、(1) 無駄なループを増やす曖昧な依頼、(2) 肥大したコンテキスト、(3) タスクに対して過剰に高価なモデル、の 3 つが主なコスト源になります。ほとんどのツールで課金対象の大半は入力トークン側です。

### 課金モデル別の最適化方針

主要ツールの課金は 3 型に分かれ、最適化の力点が異なります(各ツールのプラン体系と料金参照先は [比較](coding-agents-comparison.md) の表 D、どの型に当たるかの詳細は各実践ガイドを参照)。

| 課金モデル | 例 | 最適化の力点 |
| --- | --- | --- |
| 定額 + 使用量上限(クォータ) | サブスクリプション型(時間窓・週次の上限) | 「上限に当たらない」ことが目標。高消費の操作(長大セッション・並列乱発)を減らし、枠内に収める。超過時の追加購入条件を把握する |
| 従量(トークン) | API キー接続、BYOK(Bring Your Own Key: 自前のモデル API キー持ち込み)の OSS 系 | 削減がそのまま請求減。本記事の全テクニックが直接効く。**上限・アラート設定が必須** |
| クレジット制 | トークン量 × モデル別レートをクレジットに換算する型 | 従量と同じ性質 + **モデル選択がレート差として直接見える**。無料側の機能(定額内の補完など)とクレジット消費側の機能の区別を把握する |

いずれの型でも、まず「**何がカウントされ、何がされないか**」を公式ドキュメントで確認するのが第一歩です(例: 補完は無制限でエージェントだけ消費する体系、読み取り操作は軽くツール実行ループが重い、など)。

### コンテキスト管理 = コスト管理

コンテキストの設計原則([コンテキストエンジニアリング](../02-architecture/context-engineering.md))は、使う側ではそのままコスト管理になります。

1. **セッションを長生きさせない** — 会話が長くなるほど毎ターンの再送量が増え、後半ほど 1 往復が高くなります。タスクが終わったら新しいセッションで始めるのが、品質(コンテキスト汚染防止)とコストの両方に効きます([依頼設計](coding-agent-prompting.md) の仕切り直しと同じ判断です)
2. **コンパクション(compaction)を理解する** — 多くのツールは上限接近時に履歴を要約・圧縮します。自動任せにせず、区切りの良いところで手動実行すると、要約の質と以降のターン単価の両方を制御できます
3. **大きな出力を安易にコンテキストへ入れない** — ログ全文・巨大 diff・依存ツリーの貼り付けは、以降の全ターンで再送され続けます。必要部分の抜粋を渡します
4. **ルールファイルの肥大は毎セッション課金される** — ルールファイルは全セッションの毎ターンに含まれます。「数百行以内」の目安([ルールファイルと設定](coding-agent-rules-and-config.md))はコスト面の根拠もあります
5. **探索をサブエージェントに委譲する** — 大量のファイル読み込みを伴う調査を別コンテキストのサブエージェントに任せ、要約だけを本体に戻す構成は、本体コンテキストの肥大を防ぐ代表的なテクニックです。ただし**本体の再送は減る一方、サブエージェント自身の消費が加算される**トレードオフがあり、多用すると単一エージェントより総消費が増えます(対応ツールと使いどころは各実践ガイド参照)
6. **ignore 設定で探索範囲を絞る** — 生成物・ベンダーコードを探索対象から外すことは、精度とコストの両方に効きます

また、多くのツールが**プロンプトキャッシュ(prompt caching)**を自動適用します([コスト管理](../05-operations/cost-management.md))。利用者が意識すべきことは 1 つで、**コンテキストの先頭側(システム・ルール)を安定させる**ことです。セッション途中でルールを書き換える運用はキャッシュを無効化します。

### モデルの使い分け

- **タスクの難易度と単価をマッチさせる** — 設計判断・難しいデバッグは高性能モデル、定型的な修正・機械的な変換は軽量モデル、という使い分けが基本です。多くのツールがセッション中のモデル切替やサブエージェント単位のモデル指定をサポートします
- **「安物買い」に注意** — 軽量モデルで失敗して高性能モデルでやり直すと、両方の消費を払うことになります。判断基準は単価ではなく「**タスク成功あたりのコスト**」です。迷ったら、まず高性能モデルで 1 回で終わらせる方が安いことは珍しくありません
- モデルの判断軸と用途別の使い分けは [モデル選定ガイド](../03-implementation/model-selection.md)、各ファミリーの特性・コスト帯は [主要 LLM の全体像](../03-implementation/llm-landscape.md) を参照してください(具体的なレートは各ツール・各社の料金ページが正です)

### 並列・自律実行の消費制御

自律性を上げるほど、人が消費を目視する機会が減ります。委任型・自動化の運用では構造的な歯止めを先に設定します。

- **暴走ループ** — 同じ失敗を繰り返すループは消費だけが積み上がります。試行回数・時間の上限を依頼に含める、進捗のない実行を打ち切る通知を設計する、などで対処します
- **支出上限を機能として使う** — 支出上限(spend limit)・月次クレジット上限・PR 単位の上限・非活動時の自動スリープなど、ツールが提供する上限機能を導入時に設定します(「あとで設定する」は事故の元です)
- **並列は消費も並列** — クラウド並列実行は時間を買う代わりに消費が同時多発します。並列度はレビュー能力([チーム導入](coding-agent-team-adoption.md))と予算の両方で律速されます

### 測定と可視化

- ツールの使用量表示(セッション内コマンド・ダッシュボード・メトリクス API)で、**誰が・何に・どれだけ**使っているかを見える化します。組織導入では予算アラートとセットにします([コスト管理](../05-operations/cost-management.md) の原則と同じです)
- 削減施策の評価は消費量だけでなく品質と併せて行います。消費が減っても手戻りが増えていれば逆効果です。「タスク成功あたりコスト」「介入回数」で見ます([評価](coding-agent-evaluation.md))

## 実務での注意点

### アンチパターン

- **常に最高性能モデルで全タスクを回す** — 定型タスクに高単価を払い続けることになります。→ タスク難易度で使い分け、サブエージェントに軽量モデルを割り当てます
- **単価の安さだけでモデル・プランを選ぶ** — 失敗とやり直しで総コストが逆転します。→ 「タスク成功あたりコスト」で測ります
- **1 つのセッションを何日も使い続ける** — 毎ターンの再送量が膨らみ、品質も劣化します。→ タスク単位で仕切り直します
- **上限・アラートなしで従量型を組織展開する** — 青天井の請求は個人の善意では防げません。→ 支出上限・予算アラート・ダッシュボードを導入条件にします
- **消費量だけを KPI に削減する** — 品質低下・手戻り増を見落とします。→ 品質指標(成功率・介入回数)とセットで評価します
