# AI 時代のエンジニアキャリア戦略

## この記事の目的

「AI に置き換えられるのでは」という漠然とした不安を、**根拠のある個人戦略** — 何を深め・何を任せ・何で価値を出すか — に変換できるようになります。将来の雇用を予測するのではなく、**価値の重心がどこへ移るか**という構造と、そこから導ける選択肢を整理します。

**本記事は、ライブラリの中で最も意見が分かれやすいテーマです。** 特定の職種の消滅や年収の予測といった**断定的な将来予測はしません**。構造と選択肢を示し、判断は読者に委ねます。

## 対象読者

- AI の進展に、キャリアの不安と関心の両方を持つエンジニア
- チームメンバーのキャリア相談に、ハイプでなく構造で応えたいリード・マネージャー

- 特別な前提はありません

## 本文

### 概要: 予測ではなく、価値の重心で考える

キャリアの議論は「AI が仕事を奪うか否か」に流れがちですが、それは予測の問題で、誰にも確実には分かりません。より実用的なのは、**「価値の重心がどこへ移っているか」**を見て、自分の投資先を決めることです。ここでは 3 つの問い — 何を任せ、何を深め、何で価値を出すか — に落とします。

### 価値の重心移動

観察できる傾向として、価値の中心が**「作る速さ」から、その周辺の判断・責任へ**移りつつあります。

- **コードや文章を生成する速さ**そのものの希少性は下がる方向にあります(AI が肩代わりしやすい)
- 代わりに、**仕様化(何を作るべきかを定義する)・検証(正しさを担保する)・判断(トレードオフを決める)・責任(結果を引き受ける)**の比重が上がります
- これは本ライブラリ全体の主張とも一致します — [PoC から本番への進め方](../09-business/poc-to-production.md)・[評価](../04-evaluation/agent-evaluation-basics.md)・[Human-in-the-Loop](../02-architecture/human-in-the-loop.md) が扱う「作った後」の難しさは、AI では相対的に重くなります

「速く作れる」だけでは差がつきにくくなり、**「正しく作るべきものを定義し、担保できる」**ことが価値になる、という重心移動です。

### 深める領域の選び方

すべてを深めることはできません。[スキルマップ](../00-overview/skill-map.md)の領域を、**市場価値と希少性**の軸で個人向けに読み直します。

- **希少性が保たれやすい**: 原理の理解(なぜそう動くか)、ドメイン知識(業務・制度の深い理解)、設計判断・アーキテクチャ、検証・品質保証、対人・調整(要件を引き出す・合意を作る)
- **代替されやすい**: 定型的な生成、暗記で足りる知識、手順が決まった作業
- **自分の掛け算を作る**: 「AI が得意な生成」× 「AI が苦手なドメイン・判断・対人」の交点に、個人の希少性が生まれやすい

どれが正解かは人・市場で変わります。**「AI が肩代わりしにくく、自分が伸ばせて、需要がある」の重なり**を各自で見つけるのが、汎用の指針です。

### 「AI を使いこなす」の中身

「AI を使いこなせる人が生き残る」とよく言われますが、その中身は**道具の操作習熟を超えています**。

- **委譲の設計**: どのタスクを、どこまで任せ、どう成果を受け取るかを設計する能力([Workflow 型 vs Agent 型の使い分け](../02-architecture/workflow-vs-agent.md)の個人版)
- **検証の設計**: 任せた結果の正しさを、続けられる形で担保する能力([AI 出力の検証習慣](verifying-ai-outputs.md))
- **過信しない自制**: 任せきって技能や判断力を失わないバランス([オートメーションバイアスとスキル退化](automation-bias-and-deskilling.md))

つまり「プロンプトが上手い」より、**「委譲と検証と判断を設計できる」**ことが、使いこなしの実体です。

### 実績の示し方

AI で誰でも「動くもの」を作れる時代には、**「作れる」ことの証明の仕方**が変わります。

- 「動くデモ」だけでは差がつきにくくなります(誰でも作れるため)
- 代わりに、**難しい判断をした過程・品質を担保した方法・失敗から学んだこと**が、実力の証明になります
- ポートフォリオも「何を作ったか」に加えて「なぜその設計にしたか・どう検証したか」を語れると強い

### 陳腐化への備え

技術の陳腐化に対して、**最も減価しにくい投資は原理の理解**です。

- フレームワークや特定モデルの使い方は速く変わります。一方、[Agent ループ](../01-concepts/agent-loop.md)・[LLM の能力と限界の由来](../10-llm-foundations/capabilities-and-limits.md)といった原理は変わりにくい
- これは本ライブラリが「概念(01)・LLM 基礎(10)を飛ばすな」と繰り返す理由でもあります([学習ロードマップ](../00-overview/learning-roadmap.md))
- 具体技術は「今使うために」学び、原理は「長く効かせるために」学ぶ、という二層で投資すると減価しにくくなります

### ハイプとの付き合い方

キャリアの話題は煽られやすく、**断定的な予測ほど当たりにくい**ものです。

- 「N 年で職種 X が消える」式の予測は、根拠が弱いまま不安を煽ります。真に受けて極端な行動を取らない
- 一方で「何も変わらない」という否認も危険です。**構造の変化(重心移動)は認めつつ、具体の予測には距離を置く**のが健全です
- 焦って表面的なスキルを追うより、重心移動に沿って**判断・検証・原理**へ投資する方が、多くの将来シナリオで効きます

## 実務での注意点

### アンチパターン

- **「AI に奪われる/奪われない」の二択で考える** → 予測に賭けて極端な行動を取る、または否認して備えない → 「価値の重心がどこへ移るか」の構造で考え、判断・検証・原理へ投資する
- **生成の速さだけを磨く** → 最も代替されやすい部分に投資してしまう → 仕様化・検証・判断・責任という重心側を厚くする
- **「プロンプトが上手い = 使いこなし」と捉える** → 道具の操作に矮小化し、委譲・検証の設計力が育たない → 委譲と検証と判断の設計を「使いこなし」の中身と捉える
- **原理を飛ばして流行の道具だけ追う** → 陳腐化が速く、投資が短命に終わる → 原理(01・10 章)への投資を長期の軸に据える
- **断定的なキャリア予測を鵜呑みにする** → 根拠の弱い予測に振り回される → 構造の変化は認め、具体の予測には距離を置く
