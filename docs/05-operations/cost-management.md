# コスト管理

## この記事の目的

Agent のコストがなぜ事前に読みにくいのかを構造から理解し、**計測(タスク単位の単位経済)→ 削減(キャッシュ・履歴圧縮・モデル使い分け)→ 上限(多層の予算ガード)** の 3 段階でコストを制御できるようになります。

## 対象読者

- Agent の本番運用でコストに責任を持つエンジニア・テックリード
- 「PoC では気にならなかったコストが、スケールさせたら読めなくなった」チーム

## 前提知識

- [Agent ループ](../01-concepts/agent-loop.md) — コストを積み上げる実行構造
- [コンテキストエンジニアリング](../02-architecture/context-engineering.md) — 入力トークンを構成する要素の設計
- [メモリと状態管理](../01-concepts/memory-and-state.md) — 履歴圧縮の手法

## 本文

### 概要: なぜ Agent のコストは読みにくいか

単発の LLM 呼び出しと違い、Agent のコストは 3 つの変動要因を持ちます。

1. **ループ回数が入力に依存します** — 同じ機能でも、簡単なタスクは 2 ステップ、厄介なタスクは 15 ステップになります
2. **履歴を毎回再送します** — LLM はステートレスなので、ループの各ステップで過去の全履歴を入力として送り直します。ステップが進むほど 1 回あたりの入力が膨らみ、**総入力トークンはステップ数に対して二乗的に増えます**
3. **本文以外もトークンです** — システムプロンプト・ツール定義・ツール結果がすべて入力トークンとして毎回課金されます。ツール定義が大きい構成では、ユーザーの入力より「固定費」の方が大きいことも珍しくありません

この構造から、Agent のコストは多くの場合**出力ではなく入力トークンが支配的**です。

### 詳細: まず計測 — タスク単位の単位経済

削減の前に、どこにコストが掛かっているかを計測します([可観測性とトレーシング](observability-and-tracing.md) のトレースにトークン数を記録するのが前提です)。

- **タスク 1 件あたりのコスト**を基本単位にします(「API 費用の月額合計」では、原因の特定も価格設定の判断もできません)
- **平均ではなく分布**で見ます。コストは裾の重い分布になりやすく、少数の暴走・迷走セッションが合計を支配していることがよくあります(P95 と最大値を見る)
- ユーザー・テナント・機能単位で集計できるように、トレースに属性を付けておきます

### 詳細: 削減の主要手段

| 手段 | 効くもの | 注意点 |
| --- | --- | --- |
| プロンプトキャッシュ | 毎回再送される固定部分(システムプロンプト・ツール定義)と伸びていく履歴 | キャッシュは前方一致。固定部分をプロンプトの先頭に集め、可変部分(日時・ユーザー情報)を後ろに置く構造が前提 |
| 履歴の圧縮・要約 | 長いセッションの入力膨張 | 圧縮で失った情報による品質劣化とのトレードオフ([メモリと状態管理](../01-concepts/memory-and-state.md)) |
| モデルの使い分け | すべてのステップに最上位モデルが必要なわけではない(分類・抽出などの部品) | 品質評価とセットで判断する。安いモデルで成功率が下がれば、再試行と人手対応で総コストは逆に増える |
| ツール結果の絞り込み | ツールが返す JSON の不要フィールド | 取りすぎない設計はツール側の責務([ツール定義の設計](../03-implementation/tool-definition-design.md)) |
| バッチ処理 | 即時性が不要なオフラインタスク(評価スイートの実行を含む) | ベンダーのバッチ API は割引がある代わりに応答が非同期 |

プロンプトキャッシュ(prompt caching)は Agent ループと特に相性が良い仕組みです。ループでは「先頭(システムプロンプト・ツール定義・これまでの履歴)は変わらず、末尾に新しいステップが足されていく」ため、各ステップの入力の大部分がキャッシュにヒットします。

### 設計判断: 上限は 3 層で設計する

コストの上限は 1 か所ではなく、影響範囲の異なる 3 層で掛けます。

| 層 | 仕組み | 守るもの |
| --- | --- | --- |
| タスク単位 | 最大ステップ数・タスクあたりトークン予算([Agent ループ](../01-concepts/agent-loop.md) の停止条件) | 1 タスクの暴走 |
| ユーザー・テナント単位 | 期間あたりの利用予算・レートリミット | 特定ユーザーによる占有・悪用 |
| システム全体 | 日次・月次予算のサーキットブレーカ([インシデント対応](incident-response.md)) | 請求額の事故 |

上限に達したときの挙動も設計します。黙って切り捨てるのではなく、「ここまでの結果と、なぜ中断したかを報告して停止する」が原則です([エラー処理・リトライ・フォールバック設計](../02-architecture/error-handling-and-retries.md))。

## 実務での注意点

### アンチパターン

- **月末の請求書で初めて気づく** → 集計が月次・全体のみでは、原因特定も対策もできない → タスク単位の計測と日次の可視化・コストレートのアラートを入れる
- **単価の安いモデルに変えて品質を測らない** → 成功率が下がり、再試行・人手対応を含めた総コストはむしろ増える → 評価スイート([回帰テストと CI 組み込み](../04-evaluation/regression-testing.md))で品質とコストをセットで判断する
- **キャッシュを意識しないプロンプト構造** → 可変部分(日時・ユーザー名)が先頭にあるとキャッシュが毎回無効になり、割引が効かない → 固定部分を先頭に集める構造にしてからキャッシュを有効化する
- **上限なしの自律ループ** → 1 セッションの暴走・少数ユーザーの占有が予算を食い尽くす → タスク・テナント・全体の 3 層で上限を掛ける

### チェックリスト

- [ ] タスク 1 件あたりのコストを計測できる(トレースにトークン数がある)
- [ ] コストを平均でなく分布(P95・最大)で見ている
- [ ] プロンプトが「固定部分が先頭」の構造になっており、キャッシュが有効
- [ ] モデル使い分け・履歴圧縮の変更は評価スイートで品質を確認してから入れている
- [ ] タスク・テナント・全体の 3 層の上限がある
- [ ] 上限到達時の挙動(報告して停止)が設計されている
- [ ] コストレートのアラートがあり、対応手順に接続されている
